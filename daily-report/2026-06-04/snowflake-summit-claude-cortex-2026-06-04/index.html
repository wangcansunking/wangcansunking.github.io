Snowflake 把 Claude、千问微调和 MCP 一起缝进了企业数据栈

当地时间 6 月 1 日下午 5 点，旧金山 Snowflake Summit 26 的开场主舞台上，坐着两个人：Snowflake CEO Sridhar Ramaswamy，和 Anthropic 总裁 Daniela Amodei。两人聊的不是某个新模型多强，而是一件更具体的事——企业的智能体，应该直接长在数据所在的地方。

这场大会真正值得开发者看的，不是数据仓库本身又快了多少，而是三件事被同时摆上台：Claude 成了 Snowflake 两个核心智能体产品的基础模型；一个叫 Cortex Training 的服务，让你在托管 GPU 上直接微调千问（Qwen）、Mistral 这类开放模型；公司还宣布拟收购一家做 MCP（模型上下文协议）网关的公司 Natoma。

来源：Snowflake Newsroom「Snowflake Makes AI Real at Snowflake Summit 26」配图

把这三件事连起来看，结论很清楚：数据平台正在把「Claude + 开放模型托管微调 + MCP」缝进企业数据栈，让开发者不必再为了跑一个智能体、微调一个模型、接一个工具，而把数据来回搬三趟。这篇就沿着这条主线，把三件事拆给做 AI 工程的同行看，最后聊聊它对国内的数据平台和国产模型托管有什么对位启发。

Claude 成了 CoWork 和 CoCo 的基础模型

先说最直接的一件事：Snowflake 这次把两个核心产品都改了名，而且都把 Claude 摆到了基础模型的位置。

  • Snowflake Intelligence 改名 CoWork：定位是「知识工作者的个人智能体」，你用自然语言问业务问题，它自己判断要哪些数据、从公司散在各处的 Snowflake 环境里取出来、再给答案；
  • Cortex Code 改名 CoCo：定位是「让你更快开发的编码智能体」，这次新增了桌面端（CoCo Desktop）、移动端、Slack 机器人、VS Code 插件和 Excel 插件；
  • 两者背后，Claude 都是被点名的基础模型——按 Snowflake 与 Anthropic 的口径，Claude 模型驱动 CoWork 和 CoCo，Anthropic 成了这套智能体平台的基础模型方。

来源：Snowflake Newsroom「Snowflake CoCo Redefines Enterprise AI Development」2026-06-02

这里有个口径要说清楚，免得读着读着就夸大了。Snowflake 的 Cortex AI 一直是多模型平台，台上能点的名字包括 Anthropic、OpenAI、谷歌（Google）、Meta、Mistral、DeepSeek，这次还新加了 SpaceX 旗下的模型。所以准确的说法是：Claude 是 CoWork 和 CoCo 的主力基础模型，而不是「唯一」。但放在这家公司多供应商的盘子里，Anthropic 被反复点名、被定位成这套智能体体验的底层，分量已经很重。

为什么是 Claude？两家公司去年 12 月 3 日就签过一份 2 亿美元、为期多年的合作，把 Claude（包括 Claude Sonnet 4.5、Claude Opus 4.5）通过 Amazon Bedrock、谷歌云 Vertex AI、微软 Azure 铺给 Snowflake 的上万家客户。这次 Summit 是那份合作往「投入生产」方向的推进——重点从「能不能用」变成「怎么在企业里稳定地用」。

CoCo 这个编码智能体的势头也被点了出来：它是 Snowflake 历史上增长最快的产品，官方在合作稿里给的数字是超过 7100 名用户。对一个数据平台自己的编码智能体来说，这个采用速度说明一件事——当智能体就长在数据仓库旁边、能直接读到表结构和业务上下文时，开发者真的愿意用。

Snowflake 产品负责人 Christian Kleinerman 的说法是：CoCo 让有经验的开发者做事「快得多、也容易得多」，同时给分析师、懂数据的业务人员也打开了一扇门，让他们能开始搭数据流水线、自动化和 AI 应用。

这一节的结论：Claude 在 Snowflake 这里不是被「集成」，而是被放到了智能体体验的底层——问答取数靠它、写代码靠它，连一项正在做的代码安全能力（识别、评估、修复漏洞，带人工复核）也建在 Claude Code 上。

Cortex Training：在数据所在地微调千问、Mistral

第二件事，是这次最值得 AI 工程同行细看的——Cortex Training。

它解决的是一个很多人都踩过的麻烦：你想用自有数据微调一个开放模型，但数据躺在数据仓库里，训练得在另一套 GPU 平台上做，于是你得先把敏感数据搬出去，训完再把模型搬回来。中间这两趟搬运，既是治理风险，也是工程摩擦。

Cortex Training 的做法是把训练这步直接放进数据所在地：

  • 托管 GPU：Snowflake 提供完全托管的 GPU 基础设施，你不用自己拼一套分布式训练集群；
  • 微调开放模型：明确点名支持千问（Qwen）和 Mistral 系列等开放权重模型，用你自己的专有数据做微调；
  • 数据不出门：训练就在数据已经在的地方进行，不需要把敏感数据搬到外部环境；
  • 支持强化学习等技术：让工程团队能用强化学习这类方法，做出在特定推理任务上比通用 API 模型更准的领域模型。

来源：Anthropic「Snowflake and Anthropic announce $200 million partnership」公告配图 2026-06-01

工程上还有一个被反复强调的点：多租户 GPU 调度。按 Snowflake 给的初步客户评估口径，同样一笔 GPU 预算，能跑出多达两倍的训练次数。换句话说，把多个训练任务在 GPU 上更密地排布，让每一块卡的空闲时间更少——对预算有限、又想多试几轮微调的团队，这是实打实省钱的工程优化。

值得多想一层的是「在数据所在地训练」这句话的工程含义。过去做一次企业微调，链条大致是这样的：从数据仓库导出训练集、脱敏、传到训练平台、起 GPU 集群、训练、评估、把权重导回、再接回线上推理。每一段都要写脚本、对权限、过审计，敏感数据在这条链上要落好几次盘。Cortex Training 把训练这一步搬到数据已经在的地方，省掉的不只是几趟搬运，还有围绕这些搬运的一整套合规手续——对受监管行业（金融、医疗、政务）来说，「数据不出库」往往不是性能问题，而是能不能做这件事的前提。

也别把它想成万能。托管微调换来的便利，代价是你被绑在这个平台的 GPU 供给和模型清单上：能微调哪些开放模型、用什么训练方法、调度优先级如何，都由平台说了算。对要极致掌控训练细节的团队，自建集群仍然更自由。Cortex Training 真正瞄准的，是那批「有自有数据、想微调一个领域模型、但不想养一支训练基础设施团队」的企业——这个人群在国内同样庞大。

这里要补一句行业判断。Cortex Training 背后的姿态，是 Snowflake 在主动给自己定位：它不去硬碰基础模型这条赛道，而是去做连接模型、企业数据、业务上下文和治理的那一层。Claude 这种闭源前沿模型负责「最强的通用推理」，千问、Mistral 这些开放模型负责「在你自己数据上微调出的领域专长」——两类模型各管一段，平台把它们都收在同一个数据治理框架里。

引用大会上一位客户（Resolve AI CEO Spiros Xanthos）的话，把这个分工讲得很直白：通用 API 模型会一直变好，但它有上限。言下之意是，到了某些领域推理任务上，用自有数据微调出来的专用模型，能补上通用模型够不到的那段。

说到底，Cortex Training 让「微调开放模型」从一件要跨平台搬数据的重活，变成了在数据仓库里点一下就能做的事——这对国产开放模型的托管思路，恰好是个很有参照价值的对位（后面细说）。

拟收购 Natoma：把 MCP 网关做成数据栈的内置治理层

第三件事，是 5 月底就放出、这次在大会语境里被串起来的一笔收购意向——Snowflake 已签署最终协议，拟收购 Natoma，一家做企业级 MCP 平台的公司。

先把 Natoma 是什么讲清楚。MCP（模型上下文协议）是现在智能体接外部工具、外部应用的通用接法。Natoma 做的是一个集中式 MCP 网关：当一个智能体要调用某个工具时，网关在「工具调用」这一层就把身份、策略、审计三件事卡住——谁发起的这个动作、他有什么权限、这个动作到底允不允许，全都看得见、管得住。

把它放进 Snowflake 的盘子里，逻辑就顺了：

  • 收购完成后，Snowflake 客户能用一个经过验证的 MCP 服务器库，把 CoWork 和 CoCo 安全地连到企业的各类应用系统；
  • Snowflake 会因此拥有一个原生集成的、面向智能体和 MCP 工具调用的身份与治理层；
  • 配合 CoWork 本身已经支持的、通过开源 MCP 接入企业应用的能力，等于把「智能体接工具」这件事，从各接各的，收进了一个统一的企业级网关。

补几个事实背景：Natoma 成立于 2024 年，团队 27 人，创始人 Pratyus Patnaik 此前把一家叫 atSpoke 的工作流公司卖给了 Okta。这笔收购目前是「拟收购 / 已签署最终协议」状态，Snowflake 没有披露金额，也没说预计何时完成。这个状态要标清楚——它还没正式落定。

来源：自制对位图，依据 Snowflake Newsroom、Anthropic 公告、SiliconANGLE 报道整理（Snowflake Summit 26）

把三件事并到一张图上看，主线就更立体了。智能体层用 Claude 做基础模型；自有模型层用 Cortex Training 在托管 GPU 上微调开放模型；工具接入层用 Natoma 的 MCP 网关统一治理。三层都在回答同一个问题：怎么让开发者在数据所在地，把智能体、微调、接工具这三件事就地做完，而不是为每一件单独搬一趟数据。

顺带说一句：别和 Databricks 的大会搞混

这里插一个容易混的点。Snowflake 和 Databricks 是两家不同的公司，各开各的大会——本文讲的是 Snowflake 自己的 Snowflake Summit 26（6 月初，旧金山）；Databricks 的 Data + AI Summit 是另一家公司在 6 月中旬办的活动。两家虽然都在「数据 + AI」这条线上较劲，但这次的 Claude、Cortex Training、Natoma 都是 Snowflake 这边的动作，别张冠李戴。

至于 Adaptive Compute、Cortex AISQL、Openflow 这些名字，这次大会也一并发了——它们更偏数据引擎和数据接入本身，跟 AI/智能体的关系是「让上面那三件事跑得更顺」，本文就不展开了，知道它们是同一场大会的配套即可。

国内对位：在数据所在地跑智能体，是一道值得抄的题

把镜头转回国内。Snowflake 这套打法，对国内的数据平台和国产模型托管，是一个很清楚的对位参照。客观摆几个对照点：

来源：自制对照表，依据 Snowflake Newsroom 与 SiliconANGLE 报道整理

第一，智能体长在数据旁边这件事，国内是分着做的。 国内很多数据平台和数仓产品，目前还是「数据栈是一套、AI 助手外接大模型 API 是另一套」，权限和治理各管各的。CoWork 这种「问答即取数、智能体直接读业务上下文」的形态，把分析助手和数据栈合到了一层——这是国内数据平台可以对照的一个方向。

第二，开放模型托管微调，国内的牌面其实很厚。 Cortex Training 微调的是千问、Mistral 这类开放权重模型，而千问本身就是国产开放模型里最被海外平台采用的一个。也就是说，海外数据平台正在把国产开放模型当成「可微调的领域模型底座」来托管——国内的云厂商和数据平台，手里握着千问、DeepSeek、GLM 这些开放模型，做「在数据仓库里就地微调国产开放模型」这件事，起点并不比别人低。

第三，MCP 网关这层治理，国内还在补位。 国内 MCP 接入这两年起得很快，但多是各接各的；像 Natoma 这种「在工具调用层统一卡身份、策略、审计」的企业级网关，国内做数据治理的团队还在往这个方向走。Snowflake 用一笔收购把这层补齐，等于提前告诉同行：智能体一旦真在企业里跑起来，工具调用的统一治理是绕不过去的一环。

这三点不是要给谁开行动清单，而是把对照摆清楚：海外数据平台正在做的「就地跑智能体、就地微调开放模型、就地治理工具调用」，每一项国内都有对应的底子——国产开放模型、活跃的 MCP 接入、成熟的数仓产品。把它们缝到一起，是一道国内同行完全有资本去解的题。

写在最后

回到开头那张同台的画面。Snowflake 把 Claude 放到智能体底层、把千问和 Mistral 收进托管微调、把 MCP 网关买进数据栈——这三步拼出来的，不是「又一个集成了大模型的数据产品」，而是一个更朴素的判断：企业 AI 的下一程，比的不是谁的模型分数高，而是谁能让智能体、微调、工具调用，都在数据所在的地方就地完成。

对做 AI 工程的我们来说，这是个挺让人踏实的信号。它说明前沿模型（Claude）和开放模型（千问、Mistral）不是二选一，而是各管一段、收在同一套治理里；也说明「数据不出门就能微调」这种听起来很理想的工程形态，已经有人把它做成了产品。国内的开放模型、MCP 接入和数仓产品都在手上，这道题我们有底子去接——下一程，值得期待。
