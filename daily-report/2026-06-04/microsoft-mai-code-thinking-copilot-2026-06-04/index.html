微软把自研模型塞进 Copilot：MAI-Code 接管编码，MAI-Thinking 对标 Sonnet

当地时间 6 月 2 日的 Build 2026 上，微软一次性发布了一批以 MAI（Microsoft AI）命名的自研模型。最值得关注的有两款：一款叫 MAI-Code-1-Flash，专门为写代码而生，已经悄悄出现在 VS Code 里 GitHub Copilot 的模型选择器里；另一款叫 MAI-Thinking-1，是微软第一款自己从头训练的推理模型。

过去几年，提到「微软的 AI」，大多数人脑子里浮现的还是 OpenAI 的模型——Copilot 背后跑的是 GPT，Azure 上卖的也是 GPT。这一次不一样了。

微软这次真正做的，是在模型这一层正式补上自己的牌：编码用自己训的 MAI-Code-1-Flash，推理用自己训的 MAI-Thinking-1，并且直接把编码这款塞进了开发者每天都在用的 Copilot。 这篇文章讲清楚三件事：这两款模型各自做到了什么、微软为什么选择在模型层自己下场、以及它对每天在 Copilot 里写代码的国内开发者意味着什么。

先看第一款，也是离开发者最近的一款。

MAI-Code-1-Flash：一款直接进了 Copilot 选择器的编码模型

来源：微软 AI 官方发布页 microsoft.ai 2026-06-02

MAI-Code-1-Flash 的定位很清楚：一款为 GitHub Copilot 和 VS Code 量身打造的编码模型，目标是又快又便宜。它的几个关键事实如下：

  • 参数规模：总参数 137B，激活参数 5B——也就是说，它是一款稀疏混合专家（MoE）模型，每次推理只点亮其中一小部分参数，换来更低的成本和更快的响应；
  • 训练方式：微软称这款模型「由微软端到端构建，使用干净且经过适当授权的数据」；
  • 可用渠道：正在向 VS Code 里的 GitHub Copilot 个人用户推送，既出现在模型选择器里，也进了默认的自动选择器；
  • 核心特性：会自适应地分配思考预算——简单请求保持简洁，复杂任务才多花推理算力。

这里最关键的一点是「进了 Copilot 选择器」。它不是一个挂在网页上让你去试用的样例，而是直接出现在你每天写代码的那个下拉框里，和你已经熟悉的那些模型并排站着。

51.2% 与省下的 60% token：两个值得记住的数字

来源：微软 AI 官方发布页 microsoft.ai 2026-06-02 基准对照图

编码模型好不好，最终要看任务跑分。MAI-Code-1-Flash 给出的两个数字值得记住：

  • SWE-Bench Pro 拿到 51.2%。SWE-Bench Pro 是衡量模型解决真实软件工程问题能力的基准，题目来自真实代码仓库里的问题。作为对照，微软把它和 Claude Haiku 4.5 放在一起——后者在同一基准上是 35.2%。也就是说，在这个面向「轻量快速」一档的对位里，MAI-Code-1-Flash 领先了约 16 个百分点。
  • 同等难度问题，最多省 60% 的 token。微软在 SWE-Bench Verified 上的说法是「以最多减少 60% 的 token 解决更难的问题」。token 直接对应 API 成本和响应延迟，省 token 意味着同样一笔预算能跑更多次，或者同样一次调用更快返回。

把这两个数字放在一起看，MAI-Code-1-Flash 的卖点就清楚了：它不去硬碰最顶尖的大模型，而是在「轻量、快速、便宜」这一档里，把通过率和成本同时往好的方向推。

来源：微软 AI 官方发布页 microsoft.ai 2026-06-02 通过率与 token 用量对照图

对每天在 Copilot 里写代码的人来说，「又快又省」这一档其实是高频刚需。很多日常补全、改 bug、写测试的活儿，并不需要动用最贵最重的模型；一个响应够快、单次便宜、又能把 SWE-Bench Pro 这种真实任务做到五成的模型，恰好卡在这个高频区间里。

MAI-Thinking-1：约 1T 总参数、256K 上下文的自研推理模型

来源：微软 AI 官方发布页 microsoft.ai 2026-06-02

如果说 MAI-Code-1-Flash 是「轻」的一头，那 MAI-Thinking-1 就是「重」的一头——微软第一款自己从头训练的推理模型，瞄准的是最复杂的问题。它的关键参数如下：

  • 规模：约 1T 总参数、35B 激活参数的稀疏混合专家（MoE）模型；
  • 上下文：256K token 的上下文窗口，按微软的说法，足够装下一份约 600 页的文档；
  • 训练方式：微软称它「从零开始训练，没有蒸馏第三方模型」，使用「干净且经过商业授权的数据，并在预训练中排除了 AI 生成内容」。

这里有一句需要特别说明的话。微软强调 MAI-Thinking-1「没有蒸馏第三方模型」，这是微软自己的官方表述。从外部看，这类训练数据来源的细节短期内很难逐字核验——它更应该被理解成微软的一种立场声明，而不是已经被第三方坐实的结论。本文如实转述微软的说法，也提醒读者对这类自述保持平常心。

SWE-Bench Pro 与 Opus 持平，盲测里被偏好于 Sonnet 4.6

推理模型的成绩单更值得逐项看：

  • 软件工程任务上与 Claude Opus 4.6 持平。在 SWE-Bench Pro 上，微软称 MAI-Thinking-1 与 Claude Opus 4.6「不相上下」。Opus 是 Claude 系列里最强的一档，能跟它在这个基准上打平，已经是顶尖梯队的位置。
  • 数学推理 AIME 表现亮眼。AIME 是美国数学邀请赛，常被用来衡量模型的数学推理能力。MAI-Thinking-1 在 AIME 2025 上拿到 97.0%，在 AIME 2026 上拿到 94.5%。
  • 盲测中被用户更偏好于 Claude Sonnet 4.6。微软称，在覆盖各种使用场景的 1276 项任务的盲测里，用户更偏好 MAI-Thinking-1 而非 Claude Sonnet 4.6。这是一项人评而非自动跑分，意味着「读起来更顺、答得更合用」这件主观感受上，它至少站稳了脚跟。

需要客观说明的是：这套盲测是微软自己组织、用自己的题目和评测人做的，「被偏好」是微软的官方结论。它能说明微软对这款模型有底气，但和第三方独立评测不是一回事，读者把它当作「厂商自评中的强信号」来看更稳妥。

至于可用性，MAI-Thinking-1 目前在 Microsoft Foundry 上以私有预览形式开放，面向部分早期合作伙伴，随后会在 MAI Playground 上进入公开预览。换句话说，它还没像编码那款一样直接进到普通开发者的工具里，但路径已经摆出来了。

这张表传递的信息很直接：微软在模型层补上的是两块互补的拼板——一块是「高频、轻量、便宜、直接可用」的编码模型，已经送到了开发者手边；另一块是「低频、重型、对标顶尖」的推理模型，先在受控范围里跑起来。

这两款之外，微软同场还发布了图像模型 MAI-Image-2.5 和转写模型 MAI-Transcribe-1.5。但对开发者而言，编码和推理这两款才是真正改变日常体感的那两块。

微软为什么选择自己训模型

理解这件事，关键是看清微软的处境与动机。这里给出几条偏分析性的判断：

  • 第一，把成本握在自己手里。 编码补全是 Copilot 上最高频的调用之一，高频意味着每一分 token 成本都会被放大。自己训一款 5B 激活、最多省 60% token 的小模型来接住这部分高频流量，等于把最大的一块成本攥回了自己手心。
  • 第二，把能力的节奏握在自己手里。 过去 Copilot 的模型能力，很大程度上跟着合作方的发布节奏走。手里有了自研模型，微软就能按自己的产品规划去调能力、调价格、调发布时间，而不必处处看别人脸色。
  • 第三，给企业客户一个「数据来源可说清楚」的选项。 微软反复强调「干净且经过授权的数据」「没有蒸馏第三方模型」，这套说法对在意合规与数据来源的企业客户特别有吸引力——它给了企业一个能把训练数据来源讲清楚的选择。

把这三条合起来看，微软并不是要在模型层「取代」谁，而是要让自己在编码、推理这两个最核心的场景里，有一套完全由自己掌控的选项。模型这一层，微软终于有了自己的牌。

对国内开发者意味着什么

回到最实际的层面：每天在 Copilot 或 VS Code 里写代码的国内开发者，会受到什么影响？这里客观摆几个对照点，不替你做决定。

第一，模型选择器里会多一个选项。MAI-Code-1-Flash 正在向 Copilot 个人用户推送，你的模型下拉框里很可能会多出一个「又快又省」的档位。它不替代你现在惯用的模型，而是在高频、轻量的活儿上多给一个选择。

第二，「轻量快速」这一档的竞争更热闹了。MAI-Code-1-Flash 对位的是 Claude Haiku 4.5 这一档，而国内开发者熟悉的轻量编码模型里，千问 Coder、DeepSeek、GLM 系列也都在这个区间持续发力。多一个强力玩家入场，对每天用编码模型的人来说是好事——可选项更多，性价比的水位也会被一起抬高。

第三，推理那款暂时还离得远一些。MAI-Thinking-1 目前是私有预览、面向早期合作伙伴，普通开发者还摸不到。但它把「自研推理对标顶尖闭源模型」这条路径摆了出来，值得放进观察清单，等公开预览时再实际上手感受。

总的来说，对国内开发者，这件事眼下最直接的就是「Copilot 里多了一个又快又省的编码选项」，不需要你立刻改变什么工作习惯，试一下、对比一下即可。

写在最后：模型这一层，竞争变得更好看了

回到开头那句核心：微软这次真正做的，是在模型这一层正式补上自己的牌——编码用自研的 MAI-Code-1-Flash 直接进 Copilot，推理用自研的 MAI-Thinking-1 对标顶尖。一边是已经送到开发者手边的「又快又省」，一边是先在受控范围里跑起来的「重型对标」，两块拼板互补，姿态清楚。

对我们这些每天靠编码模型干活的人来说，这其实是个让人踏实的信号：当又一个有实力的玩家把自研模型摆上桌，「轻量快速」这一档的选择更多了，性价比的水位也会被一起抬高。海内外的编码模型你追我赶，受益的最终是每天敲代码的我们。

不必急着站队，也不必焦虑。把新选项加进你的对比清单，跑两个真实任务比一比，让数据替你说话——这一程，值得期待。
