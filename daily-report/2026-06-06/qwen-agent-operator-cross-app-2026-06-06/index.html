通义千问开始替你办事：从会聊天到会订票

2026 年春节，全国差不多每十个人里就有一个人，在千问 App 上下过一单。这个数字背后藏着一件比"又一个对话模型"更值得看的事：通义千问从一个回答问题的助手，变成了一个能真的把事办完的助手。你说一句"订明天去上海的高铁，到了订家近的酒店"，它会自己去查车次、比酒店、走到付款那一步停下来等你点头。

这篇文章只想说清楚一件事：通义千问今年最大的变化不是模型又强了多少分，而是它把"办事"这个动作真正接通了——一头是普通人用嘴指挥的千问 App，一头是开发者手里默认带 100 万 token 上下文的 Qwen3.6-Plus。 前者解决"AI 能不能替我跑腿"，后者解决"AI 能不能替我写完一整个项目"。两件事是同一件事的两张脸，我们一张一张看，再把它和百度、字节，以及海外的 Claude、OpenAI 摆到一起对照。

说句话就能办成什么：先看千问 App 这一头

千问 App 今年上线的「AI 生活助理」，核心能力是用一句话把一件需要跨好几个应用才能完成的事走完。点外卖、订酒店、找餐厅、订机票、办政务，这些场景被打通成一条可以连续执行的流程，而不是把你导到某个小程序门口就撒手。

按公开报道和实测，目前接通情况大致是这样：

  • 餐饮外卖：接入饿了么，从选餐厅、下单到支付能连续走完；
  • 出行住宿：接入飞猪订机票、订酒店，接入高德做行程与位置筛选；
  • 支付与政务：接入支付宝，既能完成付款，也能办签证、查社保、查公积金、查户口等约 50 项民生政务事项，由支付宝里的政务智能体"晓政"承接办理；
  • 暂未接通：淘宝购物等部分场景仍在接入中，微信、抖音等非阿里体系的应用接口未通。

这条能力线带来的市场反馈很直接。据 QuestMobile，千问 App 在 2026 年 3 月的月活跃用户达到约 1.66 亿，同比涨了约 41 倍，在国内 AI 原生应用里排到第二，仅次于字节豆包。换句话说，"能办事"不只是技术演示，它实打实地把一批用户从"偶尔问两句"变成了"经常使唤"。

这一头的关键变化是：千问 App 把 AI 从"告诉你怎么办"推到了"直接帮你办"，而且办的是出行、吃饭、政务这些每天都要碰的真实事务。

跨应用怎么打通：主 Agent 拆活，子 Agent 干活

很多人第一反应是，这是不是又一个把按钮包一层的语音助手？不是。千问把"办一件事"拆成了一套有分工的智能体协作，这也是它和"看屏幕点点点"那条路线最大的不同。

它的做法可以拆成三层：

  1. 主 Agent 负责理解和规划：把"订高铁顺便订个近的酒店"这种模糊需求，拆成有先后顺序的几步——先定到达时间，再按到达点筛酒店，最后统一结账。
  2. 多个子 Agent 各管一摊：出行子 Agent 去飞猪和高德查车次、订票；住宿子 Agent 按位置筛酒店；支付子 Agent 走支付宝下单扣款。每个子 Agent 在自己的领域里能独立决策、执行，还能回头校正。
  3. 接口层走结构化协议：子 Agent 与各应用之间，靠 MCP（模型上下文协议）和 A2A（智能体间协作协议）这类结构化接口通信，而不是靠截屏识别再模拟点击。

这套"接口调用"的路线，决定了千问办事的几个特点：在阿里自家应用之间跑得稳、跑得快，因为接口是现成的、约定好的；但跨到非阿里体系的应用就会遇到接口没打通的墙——这也是为什么微信、抖音这类应用暂时进不来。能力边界画得很清楚：接口通到哪，事就能办到哪。

这一节的结论是：千问的跨应用代办不是障眼法，而是一套主从分工的智能体加结构化接口；它的长处是稳，它的短处是只能在接通了接口的应用圈里办事。

边界在哪：哪些环节它绝不替你拍板

一个会自己花钱、自己提交政务材料的助手，最让人在意的是"它会不会擅自做主"。千问在这里的设计是把人留在关键节点上。

从实测反馈看，几个边界是明确的：

  • 付款前要确认：下单、扣款这一步会回到用户，金额和支付方式由人点头，AI 不自己拍板花钱；
  • 关键信息会追问：需求不全时，主 Agent 会主动追问——比如"几点到、预算多少、住哪个区"，补齐了才往下走，而不是猜一个就执行；
  • 政务材料人来核：办签证、查公积金这类涉及个人敏感信息和正式材料的事，最终提交由人确认，AI 负责把路径和步骤铺到你面前。

把这几条放在一起看，千问的取舍很清楚：让 AI 承担"跑流程"的体力活，把"花钱、提交、担责任"的决定权留给人。 这不是能力不足，而是一个面向亿级普通用户的助手必须有的克制——真正的可靠，不是敢替你做所有决定，而是知道哪些决定不该自己做。

这一节的结论是：千问办事的可靠性，建立在"该停的地方停下来"上；付款和提交这种不可逆的动作，始终由人拍板。

开发者这一头：Qwen3.6-Plus 把上下文拉到 100 万 token

普通用户感受到的是"会办事"，开发者拿到的则是另一件东西——一个默认就带超长上下文、代码智能体能力大幅提升的模型 Qwen3.6-Plus。它在 2026 年 4 月 2 日正式上线，更早在 3 月底就在 OpenRouter 放出了免费预览版。

来源：通义千问官方博客 Qwen3.6-Plus 发布页 qwen.ai/blog（标题「Towards Real World Agents」，2026-06 截取）

官方给这次发布定的主题是「Towards Real World Agents」——走向真实世界里能干活的智能体，这和 App 那一头"会办事"的方向是同一句话的两种说法。

来源：OpenRouter Qwen3.6 Plus 产品页 openrouter.ai/qwen/qwen3.6-plus（2026-06 截取）

几组已经能核实的数字，先摆清楚：

  • 默认 100 万 token 上下文：一次能吞下约八本小说体量、或者约两千页文档的内容，整个代码仓库连着依赖关系一起读进去不用反复切片；
  • 代码智能体能力贴近顶配：在 SWE-bench Verified 这个仓库级真实修 bug 的榜单上拿到 78.8 分，对照 Claude Opus 4.5 的 80.9 分，差距收到约两分以内；
  • 价格友好：输入约每百万 token 0.29 到 0.33 美元、输出约每百万 token 1.95 美元，比同档海外顶配模型低不少；
  • 架构是混合路线：把高效的线性注意力和稀疏专家混合路由结合起来，在长上下文下兼顾推理质量和速度。

需要诚实标注一句：阿里官方称其整体表现与 Claude Opus 4.5 "相当"，但跨过多个评测维度的全面排名仍待更多独立第三方验证；GLM-5、Kimi K2.5 等同档国产模型目前缺少完全同口径的公开分数，所以上图里没有强行把它们并排进来比高低。这条线我们只认能核实的数字。

对开发者来说，100 万 token 上下文加上跃升的代码智能体能力，意味着什么？意味着一个能读完整仓库、自己规划多步、连续调用工具修一整条 bug 的编程助手，价格还压在每百万 token 几毛钱的水平上。这正好和 App 那一头形成呼应：普通人用一句话办生活的事，开发者用一个长上下文模型办工程的事，背后都是同一种"智能体能连续把事做完"的能力在往前走。

这一节的结论是：Qwen3.6-Plus 用 100 万 token 默认上下文和接近 Claude 顶配的代码智能体分数，把"会办事"从生活场景延伸到了开发场景。

阿里、百度、字节，三条不一样的办事路

把视野放大，会发现"让 AI 会办事"是 2026 年国内大厂共同的方向，但阿里、百度、字节走的是三条很不一样的路。

来源：IT之家《阿里 Qwen3.6-Plus 模型发布，编码智能体能力全面跃升》ithome.com（2026-06 截取）

对位维度 阿里 · 通义千问 百度 · 文心 字节 · 豆包
主力形态 千问 App 一句话办事 ＋ 开放 API Operator 代办 ＋ 千帆平台 豆包 App 助手 ＋ 扣子平台
代办怎么做 接口调用（MCP / A2A） 看屏幕点击（Computer Use 视觉操作） 平台搭智能体、多端分发
最先打通 自家生活服务（出行 / 政务 / 外卖） 通用网页（免接口直接操作） 内容与办公（抖音 / 飞书）
开发者抓手 千问 API、100 万上下文 千帆企业级平台、私有化部署 扣子 ＋ 火山引擎、开源加托管

三条路各有各的赌注：

  • 阿里走"接口代办"：依托自家庞大的生活服务体系，把出行、外卖、政务这些高频刚需先打通，长处是稳、是贴近日常，代价是出了阿里的应用圈接口要一个个接；
  • 百度走"视觉操作"：红手指 Operator 用 Computer Use 路线，不依赖接口，直接看懂网页结构再操作，长处是理论上什么网页都能动，挑战是视觉理解的稳定性和速度；
  • 字节走"平台分发"：扣子加火山引擎，主打把搭好的智能体一次开发、分发到抖音、飞书等多个入口，长处是入口多、生态融合强，重心更偏开发者和企业侧搭建。

没有谁的路一定对——它们其实在分头验证"AI 办事"的不同可能性。阿里赌的是"先把最高频的生活刚需办稳"，这条路最贴近普通人每天的真实需要，也最容易让一个亿级用户的助手真正长出黏性。

这一节的结论是：国内三家不是在做同一个产品的三个复制品，而是在用接口、视觉、平台三种不同方式，各自回答"AI 怎么替人办事"这道题。

海外那一头：Claude 和 OpenAI 也在抢"代办"

把镜头转向海外，会发现"让 AI 替人操作"同样是 2026 年的主战场，代表是 Claude Computer Use 和 OpenAI Operator。把它们和千问放到一起看，能更清楚国内这条路的位置。

海外两家走的主要是"看屏幕操作"的通用路线：

  • Claude Computer Use：截至 2026 年初在 macOS 桌面以研究预览形态开放，能打开应用、填表格、上网查资料，对没有接口的老旧软件也能动。它在 OSWorld 这个桌面任务榜单上的完成率从 2024 年的约 14% 升到约 44%，两年里翻了约三倍，屏幕理解的准确率也被推到了商用可用的门槛之上。
  • OpenAI Operator：随 ChatGPT Pro 提供，主打网页端的浏览、订票、填表这类任务；OpenAI 还在 4 月推出了可在后台并行跑多个会话的 Codex 后台 Computer Use，把桌面自动化往专业开发者方向推。

把两条路线并排，差异就清楚了：

  • 海外偏"通用视觉操作"：好处是不挑应用、什么界面都能试着动，难点在稳定性和速度，而且这条路目前更多落在开发者和专业用户手里；
  • 千问偏"接口结构化代办"：好处是在打通的应用圈里又快又稳、面向亿级普通用户，难点在接口要一个个接、出了自家体系就受限。

这不是谁高谁低，而是两种工程哲学：海外押注"先让 AI 能看懂任何屏幕"，国内押注"先让 AI 把最高频的事办得足够稳"。 而千问真正的差异化优势，在于它背靠的是一个能让普通人每天用、且真的有交易闭环的生活服务体系——这一点是单纯的屏幕操作能力换不来的。

这一节的结论是：海外用"通用视觉操作"探边界，千问用"接口稳办高频事"扎深度；两条路都在把 AI 从会说话推向会做事。

回到那个开头：从会聊天到会办事，是同一步的两张脸

我们从"每十个人就有一个在千问下过单"开始，绕了一圈，可以回到这篇文章最想说的那句话了。

通义千问 2026 年最值得记住的变化，不是模型榜单上又前进了几名，而是它把"办事"这件事真正接通了：

  • 普通人这一头，是千问 App 用一句话订高铁、点外卖、办政务，主 Agent 拆活、子 Agent 跨应用执行、付款前回到人手里拍板；
  • 开发者这一头，是 Qwen3.6-Plus 默认 100 万 token 上下文、SWE-bench Verified 78.8 贴近 Claude 顶配、价格压到每百万 token 几毛钱，让"读完整仓库连续修一条 bug"成为日常工具。

这两张脸，其实是同一种能力在生活和工程两个场景里各长出了一副样子——AI 不再只是给你一个答案，而是能连着把一串动作做完。国内有阿里的接口代办、百度的视觉操作、字节的平台分发分头探路，海外有 Claude 和 OpenAI 在通用屏幕操作上推进，大家其实是在一起把"AI 会办事"这件事往前抬。

对我们这些每天和这些工具打交道的人来说，这是个挺好的时刻：能用嘴指挥的助手越来越能干，能写代码的模型越来越长记性、越来越便宜，而且都触手可及。路已经铺到"会办事"这一程，往前看，值得多一点期待，也值得一起往前走。
