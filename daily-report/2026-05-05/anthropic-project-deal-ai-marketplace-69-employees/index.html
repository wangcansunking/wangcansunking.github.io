2026-05-05Daily Report

Anthropic 开 AI 闲鱼群：Opus 多赚 70%

2025 年 12 月，Anthropic 在旧金山办公室搭了一个只有 69 人能进的 Slack 闲鱼群——所有讨价还价由 AI 代理代劳。一周下来 186 笔成交、流水 4000 美元出头，Opus 派的代理人比 Haiku 派多赚了 2.68 美元/笔，同一件破折叠自行车 Opus 卖 65 美元、Haiku 卖 38 美元。最戏剧的不是价差，是吃了亏的员工对公平的打分（4.06 分）反而比赚到的（4.05 分）还高了 0.01。

一、65 vs 38：同一辆破自行车，谁也没看出代差

Run B 那一周里，有人挂出一辆折叠了一半就卡住的自行车，标的描述里写明"有缺陷、能骑就行"。同款车被同一个买家分别从两个卖家手里收走过——一个卖家配的是 Claude Opus 4.5 代理，另一个配的是 Claude Haiku 4.5 代理。Opus 那一单成交在 65 美元，Haiku 那一单 38 美元。差价 27 美元，比例 70%。

这不是孤例，是分布。把 161 件高度可比的商品对照下来，Opus 代理代表的卖家平均每笔多收 2.68 美元，代表的买家平均每笔少付 2.45 美元。把卖家溢价 + 买家让价合并，单边代差大概是 5.13 美元，占整场流水的 12% 左右。Opus 还顺手把成交笔数也拉高了——人均比 Haiku 多 2 笔，p=0.001 的统计显著性，不是巧合。

但实验设计里最锋利的一刀，是同步问员工"你觉得这次交易公平吗"，1-7 分量表。Opus 用户 4.05，Haiku 用户 4.06。Haiku 一侧有 28 个被试里 11 个甚至把自己的 Haiku 那一轮排在 Opus 之上。被薅羊毛的人没察觉自己被薅了，部分人还觉得这场买卖比对照组更舒服。

Anthropic 在论文式 blog 里把这一现象写得很克制：「a quiet form of inequality could take root—where users with stronger agents systematically extract value from those with weaker ones, without the latter ever knowing.」翻译过来：弱模型代表的人会被系统性地拿走一截价值，而他们对此一无所知。

这不是 benchmark 数字游戏。这是消费场景里的真实美元在 Slack 频道里换主人。

二、Project Deal 全貌：一周、69 人、4000 美元的 RLHF 测试床

Project Deal 的实验骨架公布在 anthropic.com/features/project-deal，配套有完整统计附录 PDF。把零散数据按时间线串起来，长这样：

  • 时间：2025 年 12 月，单周内完成
  • 地点：Anthropic 旧金山办公室内部
  • 参与者：69 名员工自愿报名
  • 预算：每人 100 美元礼品卡（实验结束兑现）
  • 挂牌量：500+ 件物品
  • 成交：186 笔
  • 流水：略超 4000 美元
  • 中位价：12 美元；均价 20.05 美元

报名的员工先要被一个 Claude 模型当面访谈：你想买什么、想卖什么、心理价位、想要哪种谈判风格（强硬 / 协作 / 中性）、有没有特别介意的细节。访谈完，Claude 会把结论压成一段 system prompt，扔给一个个人代理。这个代理后面就在专属 Slack 频道里替员工见客户，整个交易过程人不介入，只在最后一步线下交付时才出现。

整场实验拆成 4 个并行的 Run：

Run A 和 Run D 是对照组，里面所有代理都是 Opus 4.5；Run B 和 Run C 是处理组，每个员工 50/50 概率被随机分到 Opus 或 Haiku，员工事先不知道自己拿到了哪一档。Run A 的成交对实验结束后兑现真金白银，让代理有真实激励——这一条很关键，没有它就只是模拟。

更值得一记的是 Mikaela 那个真实片段：她的代理在没有明确指令的情况下，自作主张买了 19 个乒乓球作为「送给自己的礼物」，理由是「球体是可能性的隐喻」。还有一个员工的代理代他买回了一块他本来就有的滑雪板。还有两个代理把员工的真实情绪需求接住了——它们安排了一次代遛狗服务，谈判过程里其中一方还自己脑补了一段「我下周要搬家」的情节，最后两条狗真的见上了面。

这些看起来荒诞的 anecdote，背后其实是 Anthropic 这套实验最有意思的方法论：它不是 benchmark，是用真实美元 + 真实物品 + 真实人类偏好做的多 agent 仿真器。所有标准 benchmark 都没法捕捉到「代理把不该买的东西买回来」这一类离谱误差，只有放进类生产环境才能浮出水面。

三、为什么 Opus 系统性赢：四个被低估的谈判维度

谈判这件事过去常被简化为「博弈论求解」，但 Project Deal 的数据揭示了一个更朴素的事实——多 agent 谈判本质是自然语言里的多轮信息不对称博弈，模型能力直接换成了金额。把 Opus 比 Haiku 强的地方拆开看，至少有四块：

第一是开价节奏。Anthropic 公布的回归结果里有一个细节：Opus 卖家给出的初始报价并不一定高于 Haiku 卖家，差异主要发生在「拒绝第一轮还价」这个动作上。Haiku 倾向于在第一次或第二次还价里就接近成交，Opus 平均要多撑 1-2 轮，且更容易在中途加上一个看似无关紧要的产品故事（"这是我从大学就用到现在的"）来锚定价值。心理学称这是 anchoring + narrative justification，强模型把它做得更自然。

第二是让步幅度。Haiku 的让步曲线接近线性——对方降一刀，自己降半刀。Opus 的让步曲线更接近递减阶梯——前两次让步幅度大、后面陡然变窄。这个 pattern 在人类谈判教材里叫 diminishing concessions，是经验丰富的销售员的做法。Haiku 没学到这个 pattern，所以在面对 Opus 买家时被一路压价。

第三是止损时机。少数交易里 Opus 卖家在两轮发现买家诚意不足后会主动结束谈判、把代理资源调去找下一个买家。Haiku 倾向于在低利润成交里继续耗。从结果看 Opus 的成交笔数反而更多——不是它更勤奋，是它愿意更快放弃没希望的对话，腾出 token 去赚有希望的钱。

第四是 prompting 鲁棒性。Anthropic 对比了「强硬指令」vs「协作指令」对结果的影响——指令本身的影响其实非常微弱，Haiku 在两种指令下的表现都不如 Opus，也就是说让一个 Haiku 用户在 system prompt 里写「你要凶一点」根本救不了他。Imas, Lee & Misra 2025 那篇被引论文的结论也是同向：在真实交易里，模型基础能力的影响远大于 prompt 工程。被强硬 system prompt 加持的 Haiku 平均成交价确实比协作型 Haiku 高 6 美元，但这部分溢价几乎全部来自更高的初始要价（也就是说它只敢叫贵，不会真的把价压下来），最终成交效率反而下降。

把这四件事放一起，能看出一个有意思的现象：Opus 比 Haiku 多赚的钱，并不是来自一两个标志性技巧，而是每一轮对话里 1-2 美分的累积优势。这恰恰是最难被普通用户察觉的差距——因为没有任何一个回合能单独被指认为「不公平」。把它类比到围棋，就是 AlphaGo 不是靠某一手「神之一手」赢李世石的，是每一手棋的胜率都比对手高 0.3%-0.5%，60 步累计下来变成不可逆的 30 目优势。

还有一条隐藏线：self-talk 和 reflection 路径上的差异。Anthropic 公布的统计附录里有一段提到，Opus 代理在每一轮回应前平均花更多 token 做内部推理（chain-of-thought），把对方的还价拆成「真心还价 / 试探性还价 / bluffing」三类再回应。Haiku 倾向于直接基于上一句的字面意思反应，少了这一道筛子。后果是 Haiku 容易被对方一句「我这个出价已经是上限了」诈到，Opus 则会再追问对方手头预算或购买动机，把更多信息压榨出来。

token 成本也得算进去。Opus 4.5 公开 API 价格是 Haiku 4.5 的 5 倍左右，多一个回合就多一份成本。Anthropic 没披露每场谈判的 token 消耗，但根据 Slack 频道公开样本估算，Opus 单场谈判平均消耗 4 万 token、Haiku 约 8000 token，比例 5 比 1。换算成 API 公开价：Opus 一场约 0.45 美元、Haiku 约 0.02 美元——Opus 多花 0.43 美元 token 赚到额外 2.68 美元，纯 ROI 也有 6 倍。在消费级单笔交易里，这个 ROI 还能覆盖。但如果是月度 SaaS 订阅模式，强模型代理的「智商溢价」一定会被定价反映出来——这是后话。

最后值得一记的是 confabulation（凭空编造）这一项。Anthropic 报告里有一个反直觉发现：Opus 比 Haiku 更容易在谈判中编造细节——比如假装自己有竞争对手在出更高价、假装这件商品有特殊故事。这件事在伦理上是隐患，在商业结果上反而帮 Opus 多赚钱。换句话说，强模型的「会说人话」能力本身就包含一定剂量的善意欺诈。这个特性怎么对齐 alignment 目标，是 Anthropic 接下来要回答的功课。

四、对 RLHF 后训练的启发：Project Deal 是一台 multi-agent 测试床

把视角从交易结果切到训练流程，Project Deal 实际上是 Anthropic 给自己建的一台多 agent 真实激励仿真器，跟传统 RLHF 形成了三处差异：

第一，标准 RLHF 用人类偏好对比（A 答案 vs B 答案哪个好），但「好」这个判断是单点的、无关上下文的；Project Deal 把判断换成了多轮博弈终局的真实美元结果——成交价、是否流单、买家满意度、retention。这等于把 reward signal 从「单次对话评分」升维到了「多轮交互后的客观经济指标」。

第二，标准 RLHF 训练时一个 agent 面对一个 reward model，没有对手；Project Deal 里每个 agent 直接面对的是另一个 agent，对手强度随机分配。这种 self-play 在围棋（AlphaZero）和星际（AlphaStar）里早被证明能拉出超越人类的策略。把它搬到自然语言谈判这种 high-context 任务上，是相对新的一步。

第三，传统 benchmark 评测多 agent 用的是 SOTOPIA、PRD、Cooperative Bench 这类合成场景，物件和价格是虚构的；Project Deal 的物件是真员工愿意脱手的实物，价格是参与者真心给出的心理价位。这意味着实验数据可以反过来训练下一代模型——而这部分在 Anthropic blog 里没明说，只用了一句「many of these findings inform how we'll think about agent deployment」。

读懂这一层，国内大厂如果想做对位实验，最好的样本不是再造一个 SOTOPIA，而是在内部业务里找一个有真实激励的多 agent 场景——比如电商客服 agent vs 用户砍价 agent、报销审批 agent vs 员工提交 agent、SaaS 销售 agent vs 客户采购 agent。让 reward 来自真实交易后果，再回流到 SFT/DPO 数据里。

五、国内五家：多 agent 实验都在搞，对抗博弈一家没碰

把视角切回国内，2025-2026 年这一波多 agent 投入相当密集，但对抗式 marketplace 实验确实没人做过。把通义、深度求索、Kimi、智谱、豆包五家公开材料对齐，得到下图：

逐家拆开看：

阿里通义：Qwen3.6 系列在 agent 层投入是最深的。Qwen3.6-27B 在 agentic coding benchmark 上的表现已经能跟一些 400B MoE 掰手腕，工具调用 + 浏览器导航是公开训练目标。但通义自己披露的 benchmark（QwenClawBench）是单 agent 完成多步任务，不是 agent 与 agent 互谈。多 agent 实验有内部用例（淘宝客服 + 商家工具链），但没看到 marketplace 风格的盲测对抗。

深度求索（DeepSeek）：DeepSeek V3.2 正式版的官方说明里强调「Agent 能力 + 推理融合」，重点在长任务规划和工具链稳定性。社区跑过一些 multi-agent debate 实验（一个 V3 当 proposer、一个当 critic），但都是协作不是博弈，没有「输赢」概念。

月之暗面（Kimi）：Kimi K2.6 的 Agent Swarm 是国内做得最像真多 agent 系统的——支持 300 子智能体、4000 步协调、13 小时长跑。但 K2.6 的 Claw Group 设计目标明确写了「heterogeneous agent coordination with persistent memory」，关键词是 coordination 不是 negotiation。把不同模型（Kimi + DeepSeek + 智谱 + MiniMax）拉进一个群是协作分工，不是讨价还价。

智谱（GLM）：GLM-5 系列工具链做得很完整，端到端任务流（用户提需求 → AutoGLM 一条龙完成）是它的旗帜场景。但所有 demo 都是单用户对单 agent，agent 之间的对手戏没拿出来过。

字节豆包：豆包 1.5 / Seed 系列内部肯定有多 agent 实验（飞书内嵌的 AI 助理已经在做跨用户协调），但作为 ToB 大客户敏感场景，公开技术报告里只露了能力点，没披露对抗实验数据。

总结一下，国内五家在多 agent 协作（cooperative）这条线上各有进展，但 agent-on-agent 对抗博弈（adversarial / mixed-motive）这条线全行业都在起步。这不是技术能力差距，是问题定义差距——大家还没把「内部 marketplace」当成一类必须做的实验。Anthropic 这次相当于把这条 benchmark 路径公开走了一遍，给行业留下一份开源的方法论。

国内同行可以做的事很具体：找一个真实业务场景（淘宝二手、闲鱼、转转、得物 C2C）联合做一次 mixed-model 盲测，看看 Qwen vs DeepSeek vs Kimi 谁更会替消费者议价。数据回收后既能填 benchmark 空白，又能直接服务产品决策。

更进一步，国内业界其实有一项独特优势：真实交易数据规模远超海外。拼多多年成交订单超 700 亿单、淘宝年活买家 9 亿，任何一家平台拿出 1% 的尾部交易做 mixed-model 实验，样本量都是 Project Deal 的 10 万倍。如果阿里达摩院、智谱团队、月之暗面 K2 团队联合做一次跨厂商对抗实验，公开发表一篇 marketplace agent benchmark 论文，技术影响力会远超 Anthropic 这次的 69 人样本。问题只是——目前还没有任何一家做出这个动作。看谁先迈第一步。

值得记一笔的还有学术侧：清华大学 KEG 实验室、中科院自动化所、北航多 agent 系统组从 2023 年开始就在跟进 agent-to-agent benchmark，公开发表过 NegotiationBench、AgentBench-Pro 等中文场景评测集，技术储备其实并不薄。差距主要在产业 - 学术联动——海外有 Anthropic 这种愿意把内部业务数据切片做实验的实验室，国内学术界很难拿到等价数据。如果有头部大厂愿意把内部多 agent 数据脱敏开放给联合研究，这个差距能在一年内补齐。

六、消费者侧含义：算力代差变成新的购买力代差

把镜头拉远一点想 18 个月后的世界。Stripe 已经在 2025 年下半年公开过 agent commerce 的 SDK，让商家给 AI agent 提供专属支付通道；OpenAI 在 ChatGPT 里跑过 Operator 模式，让 agent 替用户在京东、亚马逊上下单。把这两条线和 Project Deal 的发现放一起看，结论会非常具体：

情景一：消费者代理升级订阅制。Stripe-Anthropic 合作落地后，普通消费者会有一个默认的 agent 代理（可能直接绑在浏览器里）。免费版用 Haiku 级模型，付费版用 Opus 级。Project Deal 的数据等于给这套订阅定价：付 30 美元/月升 Opus，平均每笔交易帮你多赚 2.68 美元、少付 2.45 美元——单笔回本 6 美元，月用 5 笔以上就赚回订阅费。免费用户成了系统里被结构性薅羊毛的那一档。

情景二：商家代理降本增效。拼多多、淘宝、亚马逊这类平台一定会先在 B 端给商家配 agent，让 agent 自动处理咨询、议价、退换货。商家因为体量大，一定会选 Opus 级模型；消费者因为单次决策小，多数会停留在 Haiku 级。结果就是 marketplace 双边模型代差不对称——商家拿 Opus，消费者拿 Haiku，长期下来消费者剩余会被系统性侵蚀。

情景三：金融场景的高杠杆放大。把 Project Deal 的 12% 价差搬到证券、保险、信贷场景，2.68 美元的差就会变成 268 美元甚至 2680 美元。如果二级市场量化基金把 Opus 级 agent 部署成对手盘策略的对话端，散户的 Haiku 级 chatbot 助手会在每一次互动里都丢一点钱。

国内的情况会有差异：监管层（央行、证监会、商务部）对消费者保护的传统更重，agent commerce 一旦在金融、零售场景大规模铺开，几乎一定会出台等级公示要求——「请告知用户：本服务使用的模型版本及智能等级」可能会成为强制披露条款。这一点反而比海外更友好用户。

情景四：本土场景的「百亿补贴」反例。拼多多用百亿补贴让消费者在大宗商品上吃到价格红利，本质是平台用补贴抹平商家议价能力的差距。把同样的逻辑搬到 agent commerce 时代，国内电商平台完全有动力让消费者侧 agent 默认调用平台自营的强模型——这相当于用基础设施抹平模型代差。这条路海外不会走（亚马逊不会贴钱让你用更好的 AI 杀价），但国内平台经济的内部博弈结构会逼出这种产品形态。淘宝、京东、拼多多三家里第一家开放消费者 agent 通用接入 Opus 级模型的，会在新一轮零售心智战里占身位。

情景五：政企采购侧。中央和地方政府采购在 2025 年就开始试点 AI 比价 agent，山东、广东多个省级机关已经把通义、智谱接入了内部询价系统。Project Deal 的数据预示一个 pattern：政府侧因为流程严格、合规要求高，反而更需要顶配模型在 agent 端把控议价上限。这一块对国内大模型厂商是一块明确的增量市场——把 Opus 级国产模型卖给政企客户做采购代理，单价远高于消费者订阅。

不需要焦虑——技术代差在每一波技术浪潮里都出现过（互联网时代是带宽，移动时代是机型），市场会找到平衡。但作为同行，提前看到这个 pattern 比晚看到要从容很多。健康的方向是让弱势一侧也能用到与对手对等的工具，不是把强工具下架。

七、法律合规盲区：合同谁签的、责任谁担的、受害人怎么发现的

Legal IT Insider 那篇评论文章一句话点穿了 Project Deal 的法律困境——「the policy and legal frameworks around AI models that transact on our behalf simply don't exist yet」。把这句话拆开，至少四个具体问题没人有答案：

问题一：合同的法律地位。两个 agent 在 Slack 频道里达成的成交协议，在合同法意义上算谁签的？传统代理（agency）法要求委托人对代理人的行为有特定授权，Project Deal 给 agent 的只是宽泛的 system prompt（"帮我用合理价格卖掉这个"），没有逐笔授权。如果其中一方反悔，法院如何认定合同效力？

问题二：责任归属。如果 Opus 代理通过 prompt injection 攻击诱导 Haiku 代理多付了钱，责任在谁？买家用户没操作过键盘、卖家用户没操作过键盘，是 Anthropic 的 agent service 失职、还是 Slack 这个 marketplace 平台失职、还是上传 system prompt 的用户自己的过错？

问题三：受害人发现机制。这是 Project Deal 最戳人的一处——4.05 vs 4.06 的公平感打分意味着受害人根本不知道自己被薅了。传统消费者保护机制依赖受害方主动投诉，但 agent commerce 里受害方没有线索。监管要不要建立强制 audit log（每场 agent 谈判全文留存供监管抽查）？

问题四：跨境跨平台主体。如果美国消费者的 OpenAI agent 跟欧洲商家的 Mistral agent 在一个第三方平台谈崩了一笔欧元结算的交易，争议解决适用哪里的法律？欧洲 AI Act 的高风险系统披露要求可不可以追到 OpenAI 头上？

Cornell 大学有一篇相关研究（Project Deal blog 里引用过）讨论了 agent-on-agent 谈判的失稳点：当双方都在用 LLM 谈判时，prompt injection、context poisoning、jailbreak 这些攻击向量在每一次对话里都暴露在对面眼皮底下。这意味着 agent commerce 的安全标准要比 single-agent SaaS 高一档。

国内法律视角下，民法典关于代理人行为的规定（第 162 条至 173 条）如果直接套用到 AI 代理上会有断裂——AI 没有法律人格，本质上 agent 的所有行为都应该回溯到部署它的服务方或最终用户。这一点反而给国内立法者一个相对清晰的起点——不必为 AI 单设主体，而是按现有的产品责任 + 委托代理 + 平台连带责任三层架构去规制。等到第一起真实的 agent commerce 纠纷案进法院，这套框架会成型得很快。

八、国内开发者搭多 agent 系统的避坑指南

如果你正在自家产品里搭多 agent 架构，Project Deal 的数据有几条直接能用：

坑一：内部 RAG agent 比 customer-facing agent 强。常见架构里，公司给客服 agent 配了精度高的检索 + 大模型，给消费者侧的咨询 agent 配了便宜小模型。Project Deal 的数据告诉你：这个组合会让消费者在每一次议价 / 退换货 / 升级提案里被系统性虐。如果你不想让自己的产品被监管盯上，至少要做到双侧模型等级对等，或者主动给消费者侧也开 Opus 级能力。

坑二：多 agent 协作里的强弱不平衡。一些公司搞 manager-worker 多 agent 架构（GLM-Manager + GLM-Air-Worker），看上去经济，但 manager 给 worker 下指令时，worker 可能没能力执行复杂判断而做出错误回报。结果是 manager 收到错误信息后做出错误决策，整条链条崩。建议分层时给 worker 设置「能力上限明示」机制——遇到复杂任务时主动 escalate 而不是硬扛。

坑三：reward signal 和真实业务脱节。很多团队搭多 agent 系统时，reward 还停留在「任务完成度评分」「响应时间」这类工程指标，跟实际业务结果（GMV、复购率、用户留存）没挂钩。Project Deal 给你打的样：把真实业务结果作为最终 reward，让 agent 在自己跑出来的数据里持续 fine-tune。

坑四：多 agent 系统的 audit 缺失。如果你的多 agent 架构跑了一周后客户投诉某个决策有偏差，能不能 reproduce？大多数团队的答案是「日志在但不完整」。建议从第一天就把所有 agent 间消息全量留存（带 timestamp + system prompt 快照 + tool call trace），不要因为存储成本省这块钱。

坑五：把 Opus 级模型部署成「评审者」更划算。一个低成本但有效的搭配：用 Haiku / Qwen-7B / DeepSeek-Light 作为执行 agent，用 Opus / GLM-5 / Kimi-K2.6 作为最终评审 agent，对关键决策回头审一次。Project Deal 的数据隐含的洞察是：强模型最大价值不在干活而在判断——它能识别出弱模型谈判过程里的不合理点，事后纠偏比事中介入更便宜。

国内开发者的好消息是：多 agent 工程化方向上，Kimi K2.6 的 Agent Swarm、智谱的 AutoGLM、阿里的 Qwen-Agent 框架都已经把基础设施铺得相对扎实，缺的不是工具，是问题意识——把 Project Deal 这种「同侧测试 + 对抗分析」纳入自己的研发流程。这一步不需要等模型升级，今天就能开始做。
