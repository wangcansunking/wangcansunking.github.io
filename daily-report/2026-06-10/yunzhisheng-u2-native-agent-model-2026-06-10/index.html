云知声 U2：2660 亿参数干 1.2 万亿的活

国内做基础大模型的厂商，过去两年的叙事主线几乎是同一个方向：参数往大了堆，价格往低了打。万亿参数成了第一梯队的入场券，每 token 几厘钱成了发布会上最响的掌声。云知声（港股 09678）6 月 8 日发布的新一代通用大模型 U2，偏偏在这两件事上都没跟。它的总参数只有 2660 亿，价格也不是它海报上最大的字。它把卖点压在一个国内厂商很少单独拎出来讲的指标上——智能密度：用 2660 亿参数，做出接近 1.2 万亿参数模型的活，参数效率约为后者的 5 倍。

这篇想讲清楚三件事：智能密度到底是个什么工程取舍，它和"参数堆叠"差在哪；U2 把 Agent 能力做进基座，跟外面那一圈"大模型外挂 Agent 框架"的做法有什么本质区别；以及对一个国内开发者来说，这个模型适合拿来干什么、不适合干什么。看完你大概能判断：这条路线不是营销话术包装的小聪明，而是一个有真实工程代价、也有真实收益的方向选择。

一句话先说清楚：U2 用三分之一的参数，对标万亿模型

先把云知声官方和媒体报道里的核心数字摆清楚，再解释它们意味着什么。

指标 U2 数据 口径与来源
总参数量 2660 亿（近 3000 亿）MoE 架构 云知声官方，2026-06-08 发布
对标性能 约 1.2 万亿参数模型水平 官方主张，参数效率约 5 倍
思考 token 消耗 单任务消耗大幅下降 官方定性，混合思考机制带来

复杂工作流 自主拆解推进 100+ 步 官方，原生 Agent 能力
GPQA 硬核推理 接近甚至超过部分超大模型 多家媒体报道，单篇未给统一分数

有一点要说清楚：GPQA Diamond、GDPval 这类评测，不同媒体报道给出的数字口径并不完全一致，官方发布通稿里也更多是"前排""接近甚至超越超大模型"这类定性表述。所以本文不把某一个具体分数当成铁板钉钉的结论，只取一个站得住的判断——在硬核推理和办公任务这两类评测上，U2 用远小于对手的体量，打出了同一梯队的成绩。这个判断有多家报道交叉支撑，也是这次发布真正的看点。

数字背后的那句话其实很朴素：同样一道题，别人调动一万亿参数去算，U2 调动两千多亿就把它做对了。如果这个结论在真实任务上稳定成立，那它改变的不是某个榜单名次，而是跑这个模型要花多少卡、多少电、多少钱。

智能密度不是新词包装，是一组具体的工程取舍

云知声创始人黄伟把这次的方法论概括成一个公式：AI 商业价值 = 智能密度 × Token 价值。听上去像发布会金句，但拆开看，智能密度落到工程上是三件能动手做的事。

第一件，激活参数而不是总参数。 MoE（混合专家）架构的关键在于，2660 亿是总参数，但每次前向计算真正被"点亮"的只是其中一小撮专家。稠密模型每算一个 token 都要把全部参数过一遍；MoE 模型则像一个分科室的医院，来什么病人只叫对应科室的医生,其余科室在休息。所以衡量推理成本要看的是激活参数，不是总参数。U2 的取舍是：把总参数控制在 2660 亿这个不算大的量级，但通过路由把每次激活压得很低，于是单位推理成本能压到显著低于同尺寸稠密模型的水平。

第二件，快慢思考融合，少推理但推到点上。 U2 用了一套混合思考机制，官方描述是"可控隐空间展开与熵感知切换"——名字很拗口，意思不复杂：模型自己判断这道题该用力想还是随手答。简单问题走快速通道，直接给答案，不浪费 token 去演一遍冗长的思维链；复杂问题才切到深度推理模式。它在隐式推理和显式推理之间动态切换，目标是"减少无效推理步骤和冗余中间文本"。官方对这套机制给出的口径是"大幅降低单任务 Token 消耗"，没有公布统一的百分比数字。对长期跑推理任务的人来说，单任务 token 砍下来是真金白银——它省的不是排名，是账单。

第三件，少 token，深思考。 这是前两件事合起来的结果。把激活压低、把无效推理砍掉，剩下的算力就能集中花在真正需要想的地方。云知声给这套理念起的口号是"高智能密度 × 高 Token 价值"——同样花一个 token，要让它承载更强的能力。

这三件事放到一起，智能密度这个词就不空了。它不是"我们模型很聪明"的另一种说法，而是一组明确的设计选择：宁可总参数小一点，也要把每一份激活算力的产出比做高。

这条路线的代价也很实在。MoE 的路由训练比稠密模型难调，专家负载不均、训练不稳定都是真实的坑；快慢思考的切换判断错了，简单题想太久或难题答太快，都会掉效果。云知声选择吃下这些工程难度，去换推理成本和 token 效率，这是一个有取舍、不是只有好处的决定。

把它放进对比：稀疏激活在海外早成主流，参数堆叠仍是国内主旋律

把 U2 放到更大的图景里看，智能密度这条路并不孤单。

海外这两年最主流的效率路线，正是稀疏激活的 MoE。从 Mistral 早期把 MoE 做成开源标配，到后来一系列前沿模型都用"总参数很大、激活参数很小"的配方，核心逻辑和 U2 完全同源：解耦模型的"知识容量"和"单次计算成本"。总参数决定模型能装多少知识，激活参数决定每次推理烧多少卡。

海外厂商靠这套配方，让一个账面上几千亿上万亿的模型，跑起来的成本只相当于一个小模型。U2 做的事在方法论上和这条国际主线是对齐的，区别在于云知声把"参数效率"直接提到了产品叙事的最前面，当成发布会的主标题来打。

这件事值得多说一句。同样是用 MoE，海外前沿模型更多是把它当成一个降本的底层技术，藏在工程细节里，对外讲的还是能力和榜单；云知声反过来，把"用更小的体量做同等的活"这个结论本身当成产品的第一卖点。这种叙事选择背后其实是一种判断：随着大模型从尝鲜走向日常高频使用，决定一个模型能不能被长期用下去的，越来越不是它在某个榜单上多几分，而是它每跑一次到底划不划算。把参数效率摆到台面上讲，等于赌的是"算账"这件事会成为下一阶段竞争的主战场。

国内这边，过去的主流叙事更偏向两个极端：一头是参数往万亿冲，用绝对体量证明实力；另一头是价格往地板打，用每百万 token 几毛钱抢市场。这两条路都很猛，但都绕开了一个问题——模型本身的效率有没有提上去。价格战压的是厂商的毛利，不是模型的算力消耗；参数竞赛堆的是体量，不必然带来单位算力的产出提升。U2 的差异化就卡在这里：它不靠补贴把价格打下来，而是想靠模型效率本身把成本结构改掉。

路线 代表做法 成本来源 风险
参数堆叠 总参数冲万亿，账面体量取胜 训练与推理双高 体量不等于效率，推理贵
价格补贴 每百万 token 价格打到地板 厂商让利换市场 压毛利，难持续
智能密度（U2 / 海外稀疏激活） 总参数适中，激活压低 + 快慢思考 模型效率本身改善 MoE 路由难调，工程门槛高

这张表想说的不是谁对谁错。参数堆叠在追求绝对能力上限时仍有意义，价格补贴在抢用户时也确实管用。但如果你关心的是"长期跑下来一个 token 到底烧多少算力"，那智能密度这条路指向的方向更可持续——它改的是分母，不是靠贴钱改分子。U2 把国内的注意力往这个方向拉了一把，这件事本身是有价值的。

原生 Agent：把"会干活"刻进基座，而不是外面套一层框架

这次发布另一个被反复强调的点，是 U2 的 Agent 能力是原生的。这个"原生"值得掰开讲，因为它和市面上大多数 Agent 方案的差别，恰恰是一个开发者会真切感受到的差别。

过去一年最常见的 Agent 做法是外挂框架：底座是一个为"生成文本"训练出来的大模型，外面再包一层编排框架（规划、工具调用、记忆、纠错都写在框架里），框架通过反复给模型发 prompt 来驱动它一步步干活。这套方案能跑，但有结构性的别扭——模型本身没被训练去"完成任务"，它被训练去"接着说话"。于是工具调用要靠 prompt 硬掰格式，多步规划要靠框架在外面盯着，一步错了要靠框架兜底重来。模型是个很会聊天的大脑，但手脚是后装的。

U2 的思路是把这套能力往基座里塞。官方的说法是"为执行而设计"，区别于传统的"为生成而设计"。具体讲，工具调用、状态管理、多步规划、任务拆解这几件事，是在训练阶段就当成模型的原生能力来练的，而不是推理时靠外部框架临时拼。落到能力上，U2 能把一条完整的任务链串起来：

  • 需求理解——搞清楚用户到底要什么
  • 任务规划——把大目标拆成可执行的步骤
  • 环境交互——读取上下文、调用外部资源
  • 工具调用——原生地用工具，而不是硬凑格式
  • 过程纠错——中间步骤错了能自己发现并修
  • 结果验收——交付前自己核对一遍

云知声给的能力指标是：自主拆解并推进 100+ 步复杂真实工作流。官方举的一个例子是"生成客户季度投资报告"——这一个任务里就包含调研、分析、计算、生成结构化报告好几个环节，模型要自己排好顺序、一步步做完、最后交出成品。从"给出答案"走到"完成任务"，这是这次发布想讲的核心跃迁。

原生和外挂的差别，对开发者意味着什么？外挂框架的好处是灵活、换底座方便，坏处是每多一步就多一次"模型不一定听话"的风险，100 步工作流里只要有几步格式崩了、规划跑偏了，整条链就断。原生 Agent 把这些能力训进权重里，理论上链路更稳、更省 prompt 工程，但代价是这套能力和模型绑死了——你没法像换底座那样轻松替换它的规划逻辑。

两条路各有适用面：要做高度定制的复杂编排，外挂框架的可控性仍然重要；要做开箱即用、长链路稳定执行的任务，原生 Agent 的下限更高。U2 押的是后者，也就是把"长链路别断"当成第一优先级。

这模型适合干什么：给国内开发者的判断

抛开发布会的措辞，落到"我该不该用"，可以这样判断。

它擅长的场景，是长链路、多工具、要把活真正做完的任务。 复杂办公（季度报告、数据整理成结构化文档）、软件工程类的多步操作、需要反复调研和工具协同的深度研究——这些任务的共同点是步骤多、中间状态需要维护、一步错可能满盘皆错。U2 原生 Agent 的设计正是冲着这类场景，100+ 步工作流的能力指标也是为它们准备的。如果你手上的活是"丢一段需求，希望模型自己拆解、自己调工具、自己交成品"，那这条路线值得试。

它的成本特性，对长期高频跑推理的人友好。 单任务思考 token 的下降、激活参数低带来的单位成本下降，这些收益不是在演示里体现的，而是在你每天跑成千上万次调用、月底看账单时体现的。任务越重、调用越密，智能密度省下来的算力就越显眼。

云知声也同步在自有的 Token Hub 公有云 MaaS 平台开放了 U2，开放标准化 API，按 token 调用付费——这意味着不用自己部署也能直接调。从云知声公开的财报和经营数据看，大模型业务正在从过去的项目制交付转向按 token 计费的服务模式，2025 年大模型业务收入约 6.1 亿元、占总营收过半，是这次技术发布背后的商业底气。

这些经营数字口径来自公司财报与媒体报道，放在这里是为了说明它有真实业务在跑、不是只能演示。具体的 ARR 增速、月度规模等数字不同来源口径不一，开发者选型时不必把某个具体数值当决策依据——真正该看的还是模型在你自己任务上的表现。

它不是万能的，也不必神化。 智能密度路线吃的是 MoE 和快慢思考的工程红利，这些技术海外前沿模型也在用，U2 的价值在于把它做出来、做进产品、并且在国内把这套叙事讲清楚了。但参数效率 5 倍、对标万亿模型这些主张，最终要在你自己的真实任务上验证才算数——榜单和通稿能给你一个值得一试的理由，给不了你直接照搬的结论。

写在最后：国内大模型多一条不靠堆参数的路，是好事

当所有人都在卷参数和卷价格的时候，云知声 U2 选了一条更难走、但更可持续的路——不靠把模型堆大来证明实力，也不靠贴钱把价格打下来抢市场，而是从模型效率本身下手，让每一份算力、每一个 token 的产出比更高，再把 Agent 的"会干活"直接刻进基座。

这条路有它的工程代价，MoE 难调、快慢思考切换难判、原生 Agent 和权重绑死，都是要实打实吃下去的难题。但它指向的方向是对的：AI 真正大规模用起来的那天，决定胜负的不会是谁的参数账面更大，而是谁能用更省的算力把更难的活干完。U2 把国内的注意力往这个方向拉了一把，让"智能密度"这个本该是主线的指标，第一次被一个国产模型当成主标题来打。

对每天跟模型打交道的国内开发者来说，多一条不靠堆参数的技术路线，就是多一个能上手验证的选择，也多了一种把成本算明白的可能。
