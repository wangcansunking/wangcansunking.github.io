昆仑万维 SkyClaw v1.0 限免登场：国产 Agent 模型第一次把 OpenClaw / Claude Code 兼容做进训练目标

2026 年 5 月 26 日下午量子位、IT 之家、新浪科技、CSDN、网易科技几乎同时挂出昆仑万维天工 AI 的发布稿，主角叫 SkyClaw v1.0。看完所有原文之后，最值得拎出来说的不是「对标 Opus 4.6」这种营销标语，而是另一个技术细节：这家公司声称在 mid-training 阶段就把大量复杂 Agent 任务并入训练数据，并明确把 OpenClaw、Claude Code、Hermes、Nanobot、Codex 五套 agentic harness 当成训练目标来对齐。这是国产模型公开材料里第一次把「harness 兼容」当成训练目标本身写进通稿，而不是发完模型再去靠 OpenAI 接口糊一层兼容皮。

30 秒速览

  • 模型：SkyClaw v1.0（旗舰）+ SkyClaw v1.0-lite（轻量），昆仑万维天工 AI 团队 5 月 26 日发布，上下文窗口百万 token。
  • 价格（正式开放后）：旗舰输入 0.5 元 / 百万 token、输出 4 元 / 百万 token；lite 输入 0.3 元 / 百万 token、输出 2 元 / 百万 token。现阶段限免开放 2 至 4 周。
  • 官方对标：在 OpenClaw 相关 Agent 任务上「接近」DeepSeek V4 Pro、Claude Opus 4.6、Qwen3.6 Plus；超越 Minimax 2.7、DeepSeek V4 Flash、Qwen3.6 35B A3B 等更小档。
  • 训练方法：大规模 mid-train + 高质量合成任务 SFT + 端到端强化学习；昆仑万维强调 mid-training 阶段就注入复杂 Agent 任务，强化学习阶段在自建 Claw 环境中跑真实工具调用、文件编辑、测试执行回路。
  • Harness 兼容：声明可在 OpenClaw、Hermes、Nanobot 等主流 Agent 环境中运行，同时适配 Claude Code、Codex 等代码 Agent 框架；其中只有 Nanobot 在通稿里写明「已率先完成对 SkyClaw-v1.0 的适配」，其余通过 OpenAI 兼容接口接入。
  • 入口：天工 Skywork 平台（tiangong.cn）5 月 22 日已接入；开发者通过 APIFree（apifree.ai）拿 API Key 直接调用，OpenAI 兼容协议。

核心论点：SkyClaw v1.0 跟此前国产 Agent 模型不一样的地方在「训练阶段就把工具调用当主任务」，但「逼近 Opus 4.6」「全球第一梯队」这种说法目前只有定性描述、没有公开 benchmark 数字，国内开发者最值得做的事不是急着替换主力模型，而是趁限免期间在自己的 OpenClaw / Claude Code 工作流里跑一遍真实任务，再下结论。

这套路线如果走通，对国产 Agent 生态的影响会比单纯多了一个便宜模型大得多——它意味着「Agent 能力可以通过专项训练做出来，不需要等通用大底座升级」，这条技术路径如果跑通，会让国内更多中型厂商有勇气加入 Agent 赛道，而不是只能仰望阿里 / 字节 / 百度 / DeepSeek 这几家头部做完底座之后再下沉应用。这对国内 AI 生态多样性是好事。

一、昆仑万维这次发的到底是什么

先把事实摆清楚。5 月 26 日下午，昆仑万维天工 AI 团队推出两个新模型：

  • SkyClaw v1.0：旗舰版本，对应通稿里反复出现的「百万 token 上下文 Agent 模型」「复杂工具调用」「多轮任务执行」「代码生成」「文件编辑」「交互式应用构建」「研究型数据分析」七个能力定位。
  • SkyClaw v1.0-lite：轻量版本，参数规模官方未公开，但根据通稿描述定位是「同体系下的小模型」，价格大约是旗舰的 60%，能力定位是「同等任务范围下的低成本替代」。

发布方式有两个口子。一个是 ToC 侧的天工 Skywork 平台 tiangong.cn，5 月 22 日就已经把 SkyClaw v1.0 接入了平台后端，普通用户直接登录就能用，配套 Skill 保存与定时任务设置（量子位通稿明确提到）。另一个是 ToD 侧的 APIFree（apifree.ai），开发者注册之后拿 API Key 直接调，目前免费，限免期 2 至 4 周。这个安排显然是有节奏的——5 月 22 日先把 ToC 平台接上做内测和稳定性观察，5 月 26 日同步开放 ToD 的 API 入口和宣发，把开发者口子打开。

值得提一句的是，这是昆仑万维天工 AI 团队继年初天工 Skywork 通用智能体平台升级之后第一次单独推大模型本身，而且是直接对准 Agent 这一个赛道。这跟此前同一团队的产品节奏（通用智能体平台为主、模型本身只是底座）是个明显的方向转折——这一次模型本身被推到台前，而且打的是「能干活、能调工具、能跑长 Agent 链路」的卖点。

昆仑万维这个公司在国内 AI 圈一直是一个稍微特殊的存在——主营业务是海外社交（StarMaker、Opera）和游戏，天工 AI 是它从 2023 年开始投入的 AI 业务线。跟阿里、字节、百度这种主营业务就是云 + 模型的厂商比，昆仑万维做 AI 是「副业」性质，反而没有那么重的「模型必须给主营业务带来什么 KPI」的包袱，可以做相对纯粹的技术选择。这一次押 Agent + 开源 harness 兼容的路线，跟它的公司画风是吻合的——不跟阿里 / 字节正面比通用模型规模，找一个垂直方向把技术做深。

另一个值得提的背景是模型起的名字。SkyClaw 这个命名跟 OpenClaw 的呼应非常明显——OpenClaw 是当前国内最火的开源 Agent harness（每天有大量国内开发者把它当 Claude Code 平替用），SkyClaw 直接拿 Claw 后缀做产品名，本身就在向 OpenClaw 用户群体发信号。这是 marketing 也是技术站队——昆仑万维选择把自家模型的命名空间跟开源 Agent 生态绑定，而不是另起一个 Sky 系列的独立品牌。

这个命名选择对国内开发者其实有实际意义。OpenClaw 用户群体本来就是「跑 agent 工作流的国内技术深度用户」，他们对工具调用、长上下文、harness 兼容性的敏感度远高于普通对话用户。SkyClaw 直接对准这群用户，能让营销资源不浪费——不用花钱教育普通用户什么是 Agent，直接告诉已经在用 OpenClaw 的人「这是更适合你的模型」。这种精准的目标客户定位，在国产模型发布稿里不算多见，多数厂商还在试图同时讨好 ToB、ToC、ToD 三类客户。

二、定价：和 Opus 4.6、DeepSeek、千问对比

定价是这次发布最容易出歧义的部分，几家媒体写法不一，需要把官方数字单独拎出来。

SkyClaw 自家两档（正式开放后）：

  • SkyClaw v1.0：输入 0.5 元 / 百万 token，输出 4 元 / 百万 token
  • SkyClaw v1.0-lite：输入 0.3 元 / 百万 token，输出 2 元 / 百万 token

当下是限免状态——昆仑万维通稿原话「即日起，SkyClaw-v1.0 与 SkyClaw-v1.0-lite 开放 2 至 4 周免费试用」，意思是开发者拿 APIFree key 在限免期内白嫖，到期后才是上面的挂牌价。

把这个数字放到国内主流 Agent 模型的价格梯队里看，整体大致是这样：

几个口径需要说清楚：

  • DeepSeek V4 标准版：阿里云百炼挂牌输入 1 元 / 百万 token、输出 2 元 / 百万 token（参考阿里云百炼上线 DeepSeek-V4 公告原文）；这是 DeepSeek 现在的主力档位。
  • DeepSeek V4 Pro：通稿里被昆仑万维拿来当对标，价格据通稿描述「SkyClaw v1.0 输入价是 V4 Pro 的 1/24」，倒推 V4 Pro 输入价约 12 元 / 百万 token、输出价同步推算约 96 元 / 百万 token（量子位文中比较口径，原文未直接给 V4 Pro 数字，本文按通稿提供的倍数关系倒推，存在 ±20% 不确定性）。
  • Qwen3.6 Plus：官方折扣价大致输入 2 元 / 百万 token、输出 8 元 / 百万 token（阿里云百炼挂牌四五折后口径，全模型 4.5 折促销同时期生效）。
  • Claude Opus 4.6：Anthropic 官方价输入 $15 / 百万 token、输出 $75 / 百万 token，按 7.2 汇率折算约 108 元 / 百万 token 输入、540 元 / 百万 token 输出，是 SkyClaw v1.0 输入价的 200 倍以上。

定价这里要给 SkyClaw 一句公道话——0.5 元输入 / 4 元输出的挂牌价，在国内 Agent 能力可用档位里确实压得很低，比千问 3.6 Plus 还便宜一个量级；但同时也要把营销话术拎出来——通稿里「价格只有 DeepSeek V4 Pro 的 1/24」这种描述，比较对象是 V4 Pro 这个最贵旗舰，而不是 DeepSeek V4 标准版（输入价 1 元，比 SkyClaw 只贵一倍而不是 24 倍）。对国内开发者来说，主力替代场景的对比锚点是 V4 标准版，不是 Pro。

另一个需要拎清楚的细节——通稿里没有写「限免之后会不会再降价」「2 至 4 周限免到期之后是直接转挂牌价还是有过渡折扣」「企业批量采购有没有 SLA 价」。这些是开发者实际决策时最关心的问题，但目前都还是空白。按国产模型这一两年的常规节奏推断，限免期之后大概率会有一段「8 折 / 7 折」的过渡折扣期，再之后才走挂牌价；但具体怎么落地需要等昆仑万维后续公告。

把 SkyClaw 的定价跟它自己声称的「逼近 Opus 4.6」对照看，最值得记下的一笔账是这样的：如果 SkyClaw 实测能力真的在 OpenClaw 任务上达到 Opus 4.6 的 80% 以上，那 200 倍价差对开发者来说意味着同样预算可以多跑 200 倍的任务量，长 Agent 链路的 token 烧钱压力可以直接归零。但「80% 能力 + 1/200 价格」这个组合成立的前提是能力侧得到独立验证，而不是只听昆仑万维一面之词。这就是为什么下一节要单独拆 mid-training 这件事。

三、mid-training 注入：这是真技术细节还是营销话术

发布稿里反复强调的一个技术点：SkyClaw 把大量复杂 Agent 任务在 mid-training 阶段就注入训练数据，不像多数国产 Agent 模型那样在 SFT 阶段才补。这个点值得单独拆开看，因为它决定了 SkyClaw 是真做了路线差异化，还是只是改了个营销 narrative。

从量子位转引的官方说明看，SkyClaw 训练管线大致是这样：

  1. Pre-training：常规通用语料预训练，参数规模官方未公开。
  2. Mid-training：在这一阶段把 OpenClaw 风格的 Agent 任务环境（文件读、代码改、检索、测试、页面观察）以合成数据形式并入训练，而不是放到后续 SFT。
  3. 合成轨迹 SFT：高质量合成 Agent 轨迹监督微调，原文特别强调「过滤低质轨迹」，避免学到无效工具调用、错误观察解读、偏离目标这三类问题。
  4. 端到端 Agentic RL：在自建的 Claw 环境中做强化学习，让模型反复执行任务、观察反馈、处理失败、继续修正动作。

这套路线，跟开源社区已经看到的几个公开方案（比如某些 Qwen 衍生 Agent 模型的训练管线、DeepSeek 强化学习论文里的工具使用部分）相比，差异主要在「mid-training 就注入」这一步——多数公开方案都是预训练只用通用语料，Agent 能力是 SFT + RL 阶段才加。

这是真技术细节，还是改名的 SFT？ 没有官方完整 technical report 之前不好下结论，但有两条间接信号支持这是真技术差异化：

  • 通稿里反复强调「训练第一天起目标就是干活」这种 narrative，并且把它写进了量子位、CSDN、网易等多家通稿；如果只是 SFT 改名，公关稿不会冒险写「mid-training」这个专业术语，因为这词在业内有明确含义、会被同行盯着。同一个时间段如果有任何懂训练的研究员发出来说「这其实就是高级 SFT」，对昆仑万维品牌是直接的负面影响——敢用这个词就说明内部有底气。
  • 强化学习阶段用自建 Claw 环境跑真实任务回路这一点，跟通用 SFT 完全是两套基础设施投入。能搭起「执行任务 → 观察反馈 → 处理失败 → 继续修正」的端到端环境，本身就需要把工具调用、文件系统、代码执行器、网页观察四套基础设施做完，不是 GPU 多了就能做的事。这套基础设施跟 OpenClaw 在用户端跑的工具调用是同构的——昆仑万维可能直接拿 OpenClaw 的开源代码作为 RL 环境的基础，模型在训练时见到的工具协议跟它部署后在用户端见到的工具协议是同一套。这种「训练分布跟部署分布对齐」的做法，在学术上有明确的理论支持，效果通常显著好于「训练用一套合成数据、部署见另一套真实数据」的传统路线。

但同时，「逼近 Opus 4.6」这个说法目前没有任何公开 benchmark 数字支撑。通稿原文是「在 OpenClaw 相关任务上接近 DeepSeek V4 Pro、Claude Opus 4.6、Qwen3.6 Plus」——「接近」这个词在中文技术发布稿里通常意味着「在某个内部测评上的某些子项达到对比模型的 80-90% 区间」，不是「跟它打成平手」。

把数字真正坐实，需要看两个东西：一是昆仑万维有没有跟进发完整 technical report，二是国内独立测评（量子位 Bench、SuperCLUE、OpenCompass）后续给出的 OpenClaw 任务横评分数。这两件事大概率会在限免期内的 2 至 4 周内出来。在那之前，「逼近 Opus 4.6」需要在所有引述时打个保留——这不是怀疑昆仑万维，是国产模型发布稿历来都需要让独立测评跟一轮才能下结论的常规姿态。

另一个值得提的判断角度：参数规模。昆仑万维通稿原话「参数量相对较小，通过专项训练而非堆积参数实现能力」——这是一个非常重要的暗示。「相对较小」相对于谁？相对于 DeepSeek V4 Pro 这个万亿参数级别的旗舰，SkyClaw v1.0 可能是 100B 量级或更小的模型，但通过 mid-training 注入 Agent 任务 + 端到端 RL 把 Agent 能力压榨到接近 V4 Pro 的水平。这条路线如果走通，对国内 AI 算力供给紧张的现状是有意义的——意味着不一定要堆万亿参数才能做出可用的 Agent 模型，专项训练 + 小参数同样能在垂直场景达到旗舰水平。这跟 DeepSeek 当年用 V2 / V3 走「高效专家混合」路线打开局面的逻辑有些相似——不靠规模靠技术。

四、五套 agentic harness 兼容：真适配还是 OpenAI 兼容糊一层

通稿里第二个反复出现的点：SkyClaw v1.0「深度适配」OpenClaw、Claude Code、Hermes、Nanobot、Codex 五套主流 agentic harness。这个说法值得拆穿一下，因为「适配」在国产模型发布稿里通常有两种完全不同的含义。

含义一（弱）：OpenAI 接口兼容。任何提供 OpenAI 兼容协议的模型理论上都能通过修改 base_url 接入 OpenClaw、Claude Code 等支持自定义 endpoint 的 harness。这一档「适配」的工程难度等于零，本质上是 harness 本身的能力，跟模型训练完全无关。

含义二（强）：在模型训练阶段就针对特定 harness 的工具调用协议、提示词模板、消息历史格式做了对齐。这意味着模型在该 harness 中的工具调用成功率、回合切换稳定性、长链路任务完成度会显著高于「只是 OpenAI 兼容」的对接方式。

SkyClaw v1.0 的通稿写的是哪一档？把所有公开通稿的相关表述梳理一遍：

  • 量子位通稿：「深度适配 OpenClaw、Claude Code、Hermes、Nanobot」——「深度适配」措辞偏向含义二，但没说明具体技术细节。
  • IT 之家通稿：「可在 OpenClaw、Hermes、Nanobot 等主流 Agent 环境中运行，同时适配 Claude Code、Codex 等代码 Agent 框架」——「运行」与「适配」分开写，OpenClaw 系算运行环境，Claude Code 系算适配框架。
  • 网易科技通稿：明确点出「Nanobot 已经率先完成对 SkyClaw-v1.0 的适配，只需填入 API 密钥即可使用」——这是唯一一家被点名「率先完成适配」的 harness。
  • 其余几家通稿：均使用「兼容主流 Agent 环境」「适配主流 Agent 框架」这类弱口径表述。

按现有信息推断真实情况大致是：

  • Nanobot：双方有明确合作，Nanobot 侧做了 SkyClaw 的 API key 直配，是含义二（强）的适配。
  • OpenClaw / Claude Code / Hermes / Codex：SkyClaw 在训练阶段对齐了这几套 harness 的工具调用模板（这部分如果属实，是含义二（强））；接入路径走 OpenAI 兼容协议（这部分是含义一（弱））。
  • OpenRouter：通稿明确写了「计划接入」，目前尚未上线。

对国内开发者最实用的判断方法：在限免期内拿真实工作流跑一遍。如果训练阶段对齐到位，OpenClaw / Claude Code 里跑长 Agent 链路的工具调用成功率会和 Anthropic 自家模型差不多（80% 以上）；如果只是 OpenAI 兼容糊一层，跑过 10 轮以上的工具调用就会出现协议格式漂移、tool_call_id 混淆、消息历史错乱等典型问题。这事不需要等官方测评，自己跑半小时就能知道答案。

可以拿来做基线对比的几个真实测试方法：

  1. 让模型在 OpenClaw 里改一个 React 组件的 props 类型，要求改完跑测试、改完观察输出、根据错误信息继续修改。中等复杂度任务（5-8 轮工具调用），如果成功率低于 70% 基本可以判断 harness 适配不到位。
  2. 让模型读一个 100KB 量级的代码库（几十个文件），找出某个 bug 的源头并修复。这个任务考的是「长上下文 + 多轮文件读取 + 工具调用」的综合能力，是 Agent 模型最难的场景之一。SkyClaw 既然把百万 token 上下文 + Agent 任务对齐两件事都做到了，理论上应该在这个场景表现不错。
  3. 让模型在 Claude Code 里做一次完整的「读 issue 描述 → 定位代码位置 → 修改 → 跑测试 → 提 PR」全链路，看一次性通过率。这个是 Agent 模型最实际的部署场景，跟 Anthropic 自家模型的差距能直接告诉你 SkyClaw 现在到了哪个水平。

跑完这三组测试，对 SkyClaw 实际能力的判断会比看任何 benchmark 都准。

五、OpenClaw + SkyClaw 实测路径

讲点工程实操层面的——拿到 APIFree 的 SkyClaw API Key 之后，怎么在 OpenClaw 里把主力模型切到 SkyClaw v1.0。

按目前 OpenClaw 已经支持自定义 endpoint 的常规配置思路，路径大致是：

  1. APIFree 平台注册 → 拿到 API key（apifree.ai 入口，OAuth 登录走主流国内身份）
  2. OpenClaw 配置文件里指定 base_url 与 model name：
    • base_url：apifree.ai 提供的 OpenAI 兼容 endpoint（具体 URL 需要登录后台拿）
    • model：skyclaw-v1.0 或 skyclaw-v1.0-lite
    • api_key：APIFree 颁发的 key
  3. 启动 OpenClaw → 第一次工具调用观察 tool_call 协议格式是否对齐（重点看 tool_call_id、parallel_tool_calls 字段、function arguments 是否符合 OpenAI 规范）
  4. 跑一组真实任务（建议 5-10 轮的多轮编辑场景，比如让 Agent 改一个 React 组件的 props 类型 + 跑测试 + 提交 PR）观察长链路稳定性

实测之前没办法替读者下结论，但有一个有用的对比锚点：DeepSeek V3 时代刚出来的时候，OpenClaw + DeepSeek 的工具调用成功率大致在 60-70% 区间，比 Anthropic 自家模型低 15-20 个百分点；DeepSeek V4 出来之后这个差距收窄到 5-10 个百分点。SkyClaw v1.0 如果真的在 mid-training 阶段做了 harness 对齐，理论上首次出场的成功率应该接近 V4 现在的水平，而不是 V3 那个起点。这是个可以拿来验真的标尺。

补充一个工程层的细节——APIFree 这个平台本身在国内开发者圈不算太知名（相比百度千帆、阿里百炼、火山方舟这些 ToB 平台）。昆仑万维选择 APIFree 作为开发者侧的主要分发渠道，可能是因为 APIFree 提供 OpenAI 兼容协议 + 免费试用模式 + 国内 IP 可访问 + 不强制实名注册这四项组合在国内目前还是稀缺的。对个人开发者来说门槛低（不用提交营业执照、不用走企业认证），对 SkyClaw 自己来说获客成本也低（不用搭一整套自家的开发者平台）。这个选择是务实的，但也意味着 SLA 保障、并发上限、企业服务这些 ToB 配套目前还没到位——大规模生产部署还需要等昆仑万维后续推出正式的企业 API 入口。

六、和国内同档横评

把 SkyClaw v1.0 放到 2026 年 5 月底国内 Agent 模型档位里看，对比维度按「定价、Agent 任务能力、上下文长度、harness 兼容、试用门槛」五项展开。

SkyClaw v1.0 vs 千问 3.6 Plus（阿里）：

千问 3.6 Plus 走的是 base-model 路线，把通用能力做强后让 Agent 能力作为下游应用层适配。SkyClaw 反过来——base-model 规模可能比千问 3.6 Plus 小（昆仑万维通稿原话「参数量相对较小，通过专项训练而非堆积参数实现能力」），但 Agent 任务在训练阶段就当主任务来对齐。两条路线哪个更对，看应用场景：如果是通用对话 + 多模态 + 偶尔工具调用，千问基本盘更稳；如果是高密度工具调用 + 长链路文件编辑 + Agent 工作流，SkyClaw 的定位天然更对。

阿里这边的策略一直是大底座 + 多生态覆盖——千问从 0.5B 到 235B 各档参数都有，云、端、边全场景覆盖。这是大厂打法，资源储备深、能扛长期投入。昆仑万维这种中型厂商打不出这套大而全的牌，反而被迫聚焦——把所有的训练资源都压在 Agent 这一个方向上，反而可能跑出比阿里更深的垂直能力。从竞争结构看，这种「大厂做底座、中厂做垂直」的分工对国内开发者其实是好消息——可以拿千问做基础能力、SkyClaw 做 Agent 任务，两者互补而不是非此即彼。

SkyClaw v1.0 vs DeepSeek V4 标准版：

DeepSeek V4 标准版输入价 1 元 / 百万 token、输出 2 元 / 百万 token，参数规模大、通用能力强，工具调用能力在 V4 这一代已经追上 GPT-4-class 水平。SkyClaw v1.0 输入价是 V4 的一半（0.5 元），但输出价是 V4 的 2 倍（4 元 vs 2 元）——意味着 SkyClaw 在「短输入长输出」（典型 Agent 工作流就是这个 pattern：吃几千 token 的上下文，吐几万 token 的工具调用 + 推理链）场景下价格反而比 V4 贵。对国内开发者来说，DeepSeek V4 的性价比目前依然是最稳的选择，SkyClaw 的价格优势主要在「长上下文 + 短输出」场景，比如长文档总结、长代码库导航这类。

SkyClaw v1.0 vs GLM-5.1 Air（智谱）：

GLM-5.1 Air 走的是「小模型 + Agent 应用」路线，价格区间和 SkyClaw 接近。差异在于 GLM 有更成熟的国内企业客户基础（智谱有大量 ToB 案例），harness 兼容主要靠 ZhipuAI 自家 agent SDK；SkyClaw 押的是 OpenClaw / Claude Code 这一档开源 harness。对个人开发者来说 SkyClaw 路径更顺（不用学新 SDK）；对企业用户来说 GLM 路径更稳（有合同有 SLA）。

智谱在国内做 Agent 应用是「从上往下打」——拿大量企业项目反推模型能力，模型本身可能不是国产最强但「能落地」这件事做得最深。SkyClaw 是「从下往上打」——把模型本身的 Agent 能力做到位，然后等开源 harness 生态来用它。这两条打法在国内市场不冲突，目标客户也不重合（智谱目标是企业 IT 决策者，SkyClaw 目标是个人开发者和小团队技术负责人）。

SkyClaw v1.0 vs Kimi K2.6（月之暗面）：

Kimi K2.6 长上下文能力强（同样百万 token 量级），但价格比 SkyClaw 贵一档（输出价 ~12 元 / 百万 token），定位偏向「长文档处理 + 阅读型 Agent」。SkyClaw 定位更偏向「工具调用 + 编辑型 Agent」。同一千万 token 上下文场景下，让模型做总结 / 摘要 / 问答用 Kimi 顺手，让模型改代码 / 跑工具 / 多轮 Agent 用 SkyClaw 性价比更高。

SkyClaw v1.0 vs MiniMax 2.7 / Minimax-M2（稀宇科技）：

MiniMax 2.7 是 SkyClaw 在通稿里明确点名「超越」的对手，价格按通稿描述 SkyClaw 不到一半。MiniMax 的路线一直是「大参数 + 通用能力」，Agent 能力不是它的主打。在 Agent 任务上 SkyClaw 应该确实有优势，但 MiniMax 在多模态、视频、语音等非 Agent 维度的覆盖更广——如果工作流不止是工具调用，MiniMax 仍然是合理选择。

把 5 个国产同档放在一起看，会发现一个有趣的事实——没有哪一家在五个维度（base-model 能力、Agent 能力、长上下文、价格、harness 兼容）上同时做到最强。每家都有自己的偏科。这对国内开发者来说反而是好事：可以根据自己的实际工作流挑最匹配的那一家，而不是被绑死在某个万能模型上。2024-2025 年「一个模型打通所有场景」的旧思路在 2026 年开始失效，多模型、多供应商的工作流配置反而成了主流。

横评一句话总结：国产 Agent 模型 2026 年 5 月已经分化出 4 条清晰的赛道——千问的 base-model 大底座路线、DeepSeek 的通用 + 性价比路线、GLM 的企业 Agent 路线、SkyClaw 的开源 harness 原生兼容路线。哪条路线最终更主流目前看不清楚，但对个人开发者来说，SkyClaw 这条路线最贴近日常的 OpenClaw / Claude Code 工作流，限免期内值得试一试。

七、给国内开发者的判断

不写「行动建议」式的说教（按之前的反馈，国内开发者不需要被告知该做什么）。把场景列出来，让自己判断要不要换。在做判断之前，先把几个事实摆清楚：SkyClaw 限免期是 2 至 4 周（不是永久免费）；正式价虽然便宜但比 DeepSeek V4 标准版输出价贵一倍；harness 兼容只有 Nanobot 是明确确认的强适配；benchmark 数字目前完全缺失，需要等独立测评跟一轮。

把这四条放在一起看，「值得换」的场景实际上很窄——但确实存在。下面分两组列：

适合在限免期内试试 SkyClaw 的场景：

  • 已经在用 OpenClaw / Claude Code 跑日常开发工作流，想找一个国内可访问、不需要科学上网的备用模型
  • 工作流以工具调用为主（编辑代码、跑测试、读文件、调浏览器），上下文经常超过 100k token
  • 主力模型当前是 Claude Sonnet / DeepSeek V4，想看看国产 Agent-native 路线在自己的实际任务上表现怎么样
  • 公司有合规要求，必须用国内厂商的模型 API

目前还不太适合切换的场景：

  • 工作流主要是通用对话 + 偶尔代码 + 多模态输入，base-model 综合能力比 Agent 能力重要——这种场景千问 3.6 Plus 或 DeepSeek V4 标准版更稳。
  • 已经深度依赖 Claude Code 的 specific 行为（比如某些 prompt 工程套路、某些工具调用 pattern），切换成本可能比性价比收益高。
  • 对 benchmark 数字有硬要求（比如客户 SLA 里写了「模型必须在 SuperCLUE Agent 榜上 top 3」之类）——SkyClaw 当前没有公开 benchmark，得等独立测评。
  • 生产环境对 SLA、并发上限、企业账户、私有部署有要求——APIFree 当前是个人开发者向的开放平台，企业级配套需要等昆仑万维后续推出正式企业 API。

值得放在观察名单里的几条信号：

  • 限免到期之后定价怎么走（直接挂牌还是有过渡折扣？官方有没有针对国内开发者的长期低价承诺？）
  • 完整 technical report 什么时候出（能不能正式公开 mid-training 数据规模、参数量、合成轨迹来源、Claw 环境基础设施细节？）
  • 独立测评跟进情况（量子位 Bench、SuperCLUE、OpenCompass 这一档独立测评对 SkyClaw 的 OpenClaw 任务横评分数）
  • 开源进度（通稿明确写了「计划逐步开源各版本」，但没说时间表——按惯例国产模型开源会先开 lite 版，再开旗舰版，时间窗口 3-6 个月）
  • 第二代模型节奏（如果 v1.0 反响好，v1.5 或 v2.0 大约会在 3 个月内跟进，看节奏能看出昆仑万维这条路线的内部优先级）

这五个信号在限免期内会逐渐落地，到 6 月中下旬应该能拼出更完整的判断。

收尾：国产 Agent 模型的训练路线开始分化

把 SkyClaw v1.0 这件事放到更大的国产模型坐标系里看，最值得记下的不是定价、不是 harness 列表，而是国产 Agent 模型的训练路线第一次出现明显分化——千问押 base-model、DeepSeek 押通用 + 性价比、GLM 押企业 Agent、SkyClaw 押开源 harness 原生兼容。四条路线背后是四种对「Agent 能力到底是怎么训练出来的」的不同答案。

2024 年到 2025 年初，国产模型的主流叙事是「跟 GPT-4 比一比、跟 Claude Opus 比一比」，路线选择高度趋同——大家都在堆参数、提通用能力、扩上下文。到了 2025 年下半年，DeepSeek 用 V3 / V4 走出了「专家混合 + 性价比」的差异化；2026 年初，千问 3.x 系列继续把 base-model 路线做深，做到通用能力国产第一档；GLM-5.x 系列把企业 Agent 应用做扎实，拿下大量 ToB 客户。SkyClaw v1.0 这次的发布，等于宣告国产 Agent 模型的第四条路线正式登场——把 mid-training + 端到端 Agentic RL + 开源 harness 兼容这三件事一起做。

这四条路线背后其实是四种行业判断：

  • 千问押 base-model：相信「通用能力强了 Agent 自然就强」，把资源压在通用底座上。
  • DeepSeek 押性价比：相信「成本足够低就能改变市场结构」，把技术红利转成价格优势。
  • GLM 押企业 Agent：相信「ToB 客户的需求是 Agent + SLA + 合规」，把模型 + 服务一起打包。
  • SkyClaw 押 harness 兼容：相信「Agent 能力是训练阶段对齐出来的，不是部署阶段拼出来的」，把训练分布跟部署分布对齐做成核心壁垒。

哪条路线最终更主流、哪条会被市场淘汰，2026 年底之前看不清楚。但多条路线同时存在这件事本身就是国产 AI 生态成熟的标志——以前只有一条路（跟海外旗舰对标），现在有四条路可以走，而且每条路都有人在做、都有真实用户在用、都有真实收入在跑。这是健康的市场结构。

更让人有信心的是，SkyClaw 这次愿意把「mid-training 注入 Agent 任务」这种相对冒险的技术细节写进通稿——不管最终的 benchmark 跑出来是真接近 Opus 4.6 还是只接近 DeepSeek V4 Pro，至少国产模型团队开始敢于公开自己的路线选择，而不是只甩参数量和价格。这种状态下，竞争会逼出技术进步，国内开发者最终拿到的工具会越来越好用。

最后多说一句——对国内开发者最实用的建议不是「赶紧切换到 SkyClaw」，也不是「持观望态度等等再说」，而是保持多模型工作流的灵活性。2026 年的国内 AI 生态已经过了「赌一个万能模型」的阶段，更聪明的做法是让自己的工具链同时支持 4-5 个国产模型 + Claude / GPT 海外模型，每个具体任务选最合适的那一个。SkyClaw 这次出现给这种多模型工作流又多了一个高性价比的选项——这是好事，不是负担。

限免期 2 至 4 周，对国内开发者来说就是一个低风险的窗口期——拿真实任务跑一遍、判断 SkyClaw 在自己的工作流里值不值得留下来。这事不需要等官方测评、不需要看 SuperCLUE 榜单、不需要听任何评论员推荐，自己跑半小时就有答案。这是 2026 年国产 AI 生态最让人欣慰的一点——选择权真的回到开发者手里了。
