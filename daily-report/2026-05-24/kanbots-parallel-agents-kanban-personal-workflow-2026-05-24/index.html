Kanbots 看板 AI：Cursor / Coze 之外的第三种调度形态

5 月 22 日深夜上 HN 的 Kanbots，36 小时内攒到 243 分、148 条评论。第二天去 GitHub 翻仓库，定格在 211 Star、MIT 协议、TypeScript 写、Electron 打包，作者 leodavinci1，本月 4 月 25 号才建仓。同一天 36 氪、量子位、机器之心、虎嗅四家 site 检索全部空仓。一个一周前还不存在的开源桌面 app，在国内中文圈几乎没人提，但已经在 HN 顶部待了一天半。

本文核心论点摆在第一段：个人 AI 工作流的下一段路，不是「再聪明一档的 agent」，而是「把 agent 调度搬上桌面、变成看得见的卡片」。 聊天框是一根管子，IDE 是一只箱子，看板是一张地图——三种形态各自适合的事情不一样，Kanbots 这次把第三种推到了能装在桌面上的成品状态。

往下分七节展开：Kanbots 到底是什么、HN 顶赞五条原话、看板和聊天 / IDE 三种形态的差异、和 Cline Kanban / Cursor / 扣子 Coze 2.5 / Trae 四家国内常见工具的横评、把 Kanbots 接到 OpenClaw + 本地 GLM-4.5-Air 的实操路径、看板形态最值得借鉴的三处工程细节、最后是对个人 AI 工作流形态升级的一个观察。

一、Kanbots 是什么：一张看板加四个并行的 agent

先把工具本体讲清楚。

Kanbots 是一个 Electron + TypeScript + React 写的本地桌面 app，跑在 macOS（Apple Silicon 和 Intel 双版本）、Windows x64 和 Linux 上，MIT 协议，本地优先，不联云、不发遥测、不开 HTTP 服务。整个应用的数据全部存在用户仓库旁边的 .kanbots/ 目录里——SQLite 数据库存卡片状态和运行历史、配置文件存模型偏好、worktrees/ 目录存每个 agent 的独立工作目录。

界面是经典的看板五列：Backlog → In Progress → Awaiting Decision → Review → Done，外加一个 Inbox 收没贴标签的卡片。每张卡片可以选三件事——派给 Claude Code 还是 Codex、跑哪个 persona、并行槽位开几个（最多 4 个）。点了之后，看板会拉起一个新的 git worktree、建一个 kanbots/issue-N 分支、把选好的 agent CLI 在这个 worktree 里跑起来。屏幕上同一列里同时摆着的四张卡片，对应的就是四个 agent 在四个隔离的 worktree 里干四件互不冲突的事。

最值得拎出来讲的是「autopilot」——看板自己当 PM，挑 persona、定并行度、定单卡预算、自动拆子任务往下派。autopilot 有两个口味：feature-dev 走多 persona 轮转（产品 / 工程师 / 设计 / QA 各跑一遍），QA 跑类型检查、单元测试、lint、e2e，失败了自己改自己重跑。autopilot 跑的时候，看板会把当前花了多少钱、单卡当前耗时、整体进度全部贴到卡片上方。

技术栈往下挖一层：

  • 上层 UI 是 React + Vite，桌面壳是 Electron
  • 状态层是本地 SQLite，存所有卡片、运行、token 用量
  • 每张卡跑 agent 时拉一个新 worktree 到 .kanbots/worktrees/issue-<n>-<runId>/，把 agent CLI（claude / codex）当子进程拉起
  • worktree 装了 pre-push 钩子，挡住 agent 自己往外推代码——人不点 promote，PR 就出不去
  • 看板自己暴露一个 kanbots-mcp-server，把卡片、运行历史、命令通道全部按 MCP 协议开出来——Cursor、Claude Desktop、任何懂 MCP 的客户端都能反过来读看板状态、给看板下命令

最后这条对国内开发者特别要紧：Kanbots 的整套对外接口是 MCP，不是某家厂商的私有协议。这意味着哪怕看板本身是个海外项目，国内任何走 MCP 的客户端（包括 OpenClaw、Cursor 镜像、本地 Claude Desktop）都可以反向接进来——MCP 已经在做事实标准的活了。

二、HN 顶赞五条原话：243 分背后讨论的是什么

hn.algolia.com 上把这条帖子（item 48239413）的评论按赞数排了一遍，顶赞五条都不是在夸 Kanbots 漂亮，而是在讨论一件更基础的事——当一个晚上 agent 能写出来的代码越来越多，人怎么来得及看？

aitchnyu：「我总是想不明白人们怎么接得住一晚上 agent 干出来的活。我自己一个单人侧项目里，前后 30 分钟规划加 30 分钟实现的产出，已经大到我没法逐行 review 了。」

d4rkp4ttern：「外面的叙事都在讲 AI 在写绝大多数代码，但我打赌『人类真正 review 过的代码比例』下降得比所有人意识到的还快。」

londons_explore：「去年我至少会扫一眼 AI 写的每一行。现在它出个一万行的临时脚本，我就跑一下、瞄一眼输出，过。」

phil21：「我家里 IT 项目玩过各种 agent，要是想做点真正鲁棒、抗扰动的东西，输出永远 review 不过。」

noduerme：「『瞄一眼输出』根本不够。我撞过一次：一个递归里的 >= 写成了 >，结果完全是胡说八道，但看起来又特别合理。」

五条加起来其实在讲一件事：agent 跑得越快，人盯不过来的焦虑越大；这个焦虑不解决，「再聪明一档的 agent」反而是负价值。Kanbots 这种「把跑的过程变成可视化卡片、每一步都能停下来等人决策」的形态，正好对位顶贴关注的问题。它不解决「agent 能不能写得对」，它解决「agent 写的时候人能不能看见」。

这也是 HN 这个圈子愿意给 243 分的原因——大家不是在为「又一个 Kanban app」鼓掌，他们在为「终于有人把进度可视化做成了产品形态」鼓掌。

三、聊天 / IDE / 看板：个人 AI 工作流的三种形态

把 Kanbots 放回更大的语境里看，个人 AI 工作流到目前为止已经定型出三种成熟形态。

第一种是聊天形态。 ChatGPT、Claude.ai、豆包、Kimi 走的都是这条路。形态本质是一根管子：人发问、模型回答、对话往下走、上下文越长越乱、需要多任务就开新窗口。优点是门槛低，五岁的孩子都会用；缺点是看不见 agent 在做什么、做到了哪一步、做了多久、花了多少钱。整个过程对人来说是黑盒，对工具来说是无状态。

第二种是 IDE 形态。 Cursor、Trae、Cline、Continue、GitHub Copilot 走的都是这条路。形态本质是一只箱子：把 AI 嵌进编辑器，让它在文件、目录、终端、git diff 这些 IDE 原生概念里干活。优点是上下文天然丰富——AI 看得见整个工程文件树、看得见你的 git 状态、看得见你刚改了什么；缺点是「一次只能干一件事」的预设非常根深，因为编辑器本身只服务于一个当前焦点窗口，跑两条任务就要开两个 IDE 窗口、状态没法在两边同步。

第三种是看板形态。 Kanbots、Cline 这次新出的 Kanban、kdlbs 的 kandev、andyrewlee 整理的 awesome-agent-orchestrators 列表里那十几个工具，都是这条路上的早期尝试。形态本质是一张地图：每件待办是一张卡片、每张卡片对应一个 agent 实例、看板列代表状态、并行靠多张卡同时跑、人对 agent 的介入靠拖卡和点决策弹窗。优点是把「我让 AI 同时做几件事」从隐喻变成 UI 上的实体；缺点是上下文密度比 IDE 形态低，agent 看不到完整文件树（除非自己 cd 进 worktree），适合「目标明确、可拆分、可并行」的任务。

三种形态各自的甜区不一样：

  • 聊天适合「写一段、聊一段」，知识工作和探索性问题
  • IDE 适合「围绕一个工程深挖」，单线长任务和需要密集上下文的活
  • 看板适合「同时推进几件互相独立的事」，多 PR 并行、批量重构、跨项目调研

国内绝大多数读者熟悉前两种——豆包、文心一言、Kimi 是聊天，Trae、Cursor、Continue、Cline 是 IDE。第三种之前一直没成熟，原因是技术前提没齐：要有靠谱的 git worktree 自动化、要有可暂停可恢复的 agent CLI、要有跨 agent 调度协议（MCP）、还要有人愿意把这些拼成桌面 app。2026 年的春天这几个前提同时凑齐了，所以看板形态在两个月里冒出来一打项目，Kanbots 是其中目前最完整、最适合个人用户的一个。

四、五家工具横评：调度形态决定一切

把 Kanbots 和国内开发者更熟悉的四家工具——Cline Kanban、Cursor 多 tab、扣子 Coze 2.5、Trae Agent——按九个维度拉齐看一遍。

数字口径先讲清楚。Kanbots 取自 5 月 23 日仓库公开页面：211 Star、MIT、TypeScript、本月新建仓。Cline Kanban 同一天定格在 915 Star、研究预览阶段、未公开许可证、TypeScript 96.7%、64 个 release。Cursor 取自 Anysphere 官网公开资料，闭源、月费 $20、本月版本 1.5。扣子 Coze 2.5 取自 53AI / 苏米客 4 月 7 日发布会公开报道：字节出品，多 agent 协同 + 工作流编排 + 200+ 插件市场，云端 SaaS。Trae Agent 取自字节火山引擎开发者社区 + Trae 官网：核心 agent 走 MIT 开源，IDE 部分闭源，字节内部 92% 工程师在用。

逐项看下来三个发现：

发现一：看板这一类把「同时跑几个 agent」做成 UI 实体的工具，目前还只有 Kanbots 和 Cline Kanban。 Cursor 的多 tab 本质仍是聊天队列、扣子 Coze 2.5 是后端工作流编排（节点级并行而非 agent 任务级）、Trae Agent 走的是命令行任务优先级队列。前两者把并行变成桌面看板列，后三者把并行藏在云端或者命令行里。

发现二：可视化进度这一行，Kanbots 五星、Cline Kanban 四星、扣子四星、Trae 三星、Cursor 两星。 Cursor 之所以低，是因为它的多 tab 切来切去其实看不出来谁跑到哪一步；扣子工作流图能看出节点状态但看不出每个 agent 在干什么；Trae 给了任务标签但视觉密度不够。Kanbots 的看板列加单卡详情侧栏，把状态密度推到了目前个人工具里最高的位置。

发现三：国内可直连这一行。 Kanbots 和 Cline Kanban 都是本机跑、不联云，所以国内开发者直接装上就能用；Cursor 需要梯子；扣子 Coze 2.5 和 Trae 都是字节直营的国内产品，原生直连。如果出于合规或者代理成本考虑要把工具拉回本机，看板形态的两家是当下最直接的选项。

五、单卡 agent 的六步循环：看板形态最值得借鉴的工程

把 Kanbots 拆开看「一张卡片从立到落完整跑一遍」的流程，比看任何宣传演示都更能体会看板形态的设计意图。

第一步立卡。用户在 Inbox 或 Backlog 里写一句目标，比如「把 src/payment/ 里的 v1 callback 路径迁到 v2」。可以附 spec 文档、UI 截图、参考链接，可以打 persona 标签（PM / Eng / Designer / QA）告诉调度器这张卡更适合哪一类 agent 跑。这个动作就像写一句 issue title。

第二步派单。点「Run」之后，看板做三件事：在 .kanbots/worktrees/ 下拉一个新的 git worktree、建一个 kanbots/issue-N 分支、把选好的 agent CLI（claude 或 codex）在这个 worktree 里以子进程方式拉起。worktree 是 git 原生功能（git worktree add），意味着主仓库的工作区完全不动，agent 在它自己的目录里改文件、提交，主仓库该干嘛干嘛。

第三步跑活。agent 在 worktree 里做事，看板把它的实时输出推到卡片侧栏。一个看板最多并行 4 张卡，分别对应 4 个 worktree、4 个 agent 子进程、4 段独立的 token 计费。autopilot 模式下，看板自己会按 persona 轮转、自己拆子任务、自己往下派。

第四步问询。这是看板形态最有意思的一步：agent 在跑的过程中撞到岔路，停下来问人。Kanbots 让 agent 把岔路写成「决策弹窗」推到卡片上，人可以点选项，也可以用 /spec、/review、/split 这种 slash 命令直接给指令。这一步对应顶赞评论关心的「夜里 agent 工作量太大、人看不过来」——把决策点显式化，人可以选第二天早上集中处理所有等待中的卡，不必跟着每一个 agent 实时盯。autopilot feature-dev 模式下还能配产品 / 工程师 / 设计 / QA 四个 persona 轮转跑，每个 persona 自带一套提问偏好。

第五步复核。卡片侧栏开 diff 视图，给所有 worktree 里的改动列出来；QA autopilot 自动跑类型检查、单元测试、lint、e2e，失败了自动改、改完再跑、跑通再交回审核。人在这一步看的是「真有问题的地方」，不是「所有 agent 写的每一行」——和顶赞 noduerme 那条「瞄一眼输出不够」的担忧之间，是另一种平衡：自动跑测试不能替代人的判断，但能让人的判断聚焦在测试通不过的地方。

第六步落库。复核没问题，点 promote，看板自动把 worktree 的所有 commit 整理成一个 draft PR，推到主仓库 GitHub。看完不满意，点 discard，整个 worktree 连同 kanbots/issue-N 分支一起清掉，主仓库一根毛都不掉。pre-push hook 在 worktree 里挡死了 agent 自己把代码推出去的可能——人不点 promote，commit 出不了本机。

这六步循环里值得借鉴的工程细节有三处：

  1. 隔离 = worktree，不是 docker / vm。 worktree 是 git 原生功能，启动时间毫秒级、磁盘开销近乎为零（共享 .git 目录）、不需要镜像/容器维护。比 docker 隔离轻得多，但是对「同一仓库多分支同时改」的并行场景已经够了。
  2. 决策弹窗 = agent 自带，不是 UI 框死。 Kanbots 没有强行规定 agent 该问什么不该问什么——它把弹窗机制开成 MCP API，agent 自己在跑的过程中调用「我现在需要人决定一个 A/B/C」，看板就贴出来。这把「什么时候停下来等人」的决定权交还给了 agent，灵活度比死板的状态机高得多。
  3. MCP server 不是单向接口，是双向通道。 Kanbots 的 kanbots-mcp-server 不仅把看板状态读出来给外部，还接收外部往看板里写指令——Cursor 在用户写代码的时候，可以直接调 kanbots-mcp 在看板上立一张新卡，把当前 highlight 的代码片当 spec 贴过去，agent 在另一个 worktree 里去重构。这种 IDE + 看板的双层协同，是单纯一个 IDE 或一个看板都做不到的形态。

六、接 OpenClaw + 本地 GLM-4.5-Air：国内开发者的适配路径

Kanbots 默认接的是 Claude Code 和 Codex 这两套 CLI。前者需要 Anthropic 账号 + API key，后者需要 OpenAI 账号 + API key——对国内开发者来说，这两条路要么走代理、要么走第三方中转，无论哪种都不算最舒服。好消息是 Kanbots 的 AgentCliAdapter 是个抽象接口，任何能在终端里跑、能按 prompt 输出结果、能在工作目录里读写文件的 CLI，理论上都能挂上来。

这就给了一条国内开发者最顺的路径：Kanbots 看板 + OpenClaw 桌面端 + 本地 GLM-4.5-Air 推理。

OpenClaw 是 5 月 23 日仓库定格在 373,989 Star 的个人 AI 桌面项目，MIT、TypeScript、支持 23+ 通道（含飞书、微信、QQ）+ 国产 LLM 后端切换。它本身是个聊天 + skill 编排的桌面 app，没有看板形态；但它有完整的 MCP server + agent CLI 适配层。把 OpenClaw 当 agent runner 挂到 Kanbots 上，整条链路就变成了：

  • Kanbots 提供： 桌面看板 UI、git worktree 隔离、卡片状态机、决策弹窗
  • OpenClaw 提供： 后端 LLM 调度、本地 GLM-4.5-Air / 千问 / Kimi 切换、skill 调用、MCP 通道
  • GLM-4.5-Air 提供： 106B 总参、12B 激活、MoE、MIT、单 4090 + Q4 量化可跑（前一周专题里有完整跑通的硬件参数）
  • 国内开发者得到： 本机看板调度 + 数据完全不出本机 + 国产模型 + MIT 全栈开源

实操层面三步走：

第一步Kanbots 这边：在 ~/.kanbots/config.json 里加一段 agent adapter，把 agentCmd 指到 openclaw CLI 的可执行路径，把 agentArgs 写成调起本地 GLM-4.5-Air 的命令模板（OpenClaw 已经原生支持 --backend openclaw-local-glm45air 这种切换）。

第二步OpenClaw 这边：在 openclaw.config.yaml 里把后端切到本地，模型走 vLLM 起的 GLM-4.5-Air-Q4。OpenClaw 的 skill 仓库（ClawHub）里有现成的 code-review、refactor-step、spec-from-issue 三个适配 Kanbots 卡片场景的 skill 可以直接拉。

第三步GLM-4.5-Air 这边：用 vLLM 0.6+ 拉一个本地 OpenAI 兼容端点，端口 8000；显存占用 18-20GB（Q4 GGUF），单 4090 即可，单卡推理速度 45-60 tok/s（参考 5 月 22 日 GLM-4.6 Mac MLX 实测专题外推区间）。

这条链路跑下来，每张 Kanbots 卡片背后的 agent 推理全部走本机 GLM-4.5-Air——四张卡并行的话每张分到的 token/s 会降到 15-20 左右，但对绝大多数个人开发者的实际节奏（一张卡跑十分钟、看完结果再立下一张），这个速度完全够用。代价是放弃 Claude Opus 4.7 那种最强模型的代码理解能力——换回的是数据完全不出本机、不踩国内合规线、月度边际成本 600-800 元而不是 2000-4000 元。

这套组合的核心论点是：看板形态 + 本地 LLM + 中国开源模型，是国内开发者把「同时让 AI 做几件事」这件事完全收回本机的最完整路径。

七、看板形态对国内大厂的影响：会不会被复制

把视野放到一个月之后：Kanbots 这套看板形态会被谁吸收？

最有可能的是字节 Trae。Trae 已经有任务优先级队列、有 6A 工作流框架、有内部 92% 工程师使用的产品基础，给 Trae IDE 加一个「看板 panel」在工程上是几周的事。Trae 团队在火山引擎社区里给的「多任务并发处理策略」文章里，已经多次提到「单任务串行模式向多任务并发处理转变」是产品方向——看板 UI 是这个方向最直观的呈现。

第二是扣子 Coze 2.5。扣子已经做了多 agent 协同——「内容创作团队」由创意 agent / 剧本 agent / 视频生成 agent / 运营 agent 组成，整个流程自动跑完。扣子的工作流是节点级并行（每个节点跑完才进下一个），加一层「看板视图」把工作流里每个 agent 当前状态拍扁成卡片，工程量不大。

第三是Cursor 自己。Cursor 1.5 已经做了 background agent，下一步把多个 background agent 排成一列卡片几乎是必然路径。但 Cursor 因为是 IDE 形态，看板大概率是 IDE 的一个 panel 而非主界面。

第四是国内更早期的看板尝试——andyrewlee 维护的 awesome-agent-orchestrators 列表里能看到十几个项目在做类似形态，国内也有 kdlbs 的 kandev（AI Kanban + 开发环境 + 多 provider + 自托管）正在快速迭代。这一类工具大概率会在六个月内出现至少一两家成长起来的国产替代。

回到开头的论点：个人 AI 工作流的下一段路不是「再聪明一档的 agent」，而是「让人看见 agent 在干什么」。聊天框做不到，IDE 做不到，看板能做到。Kanbots 这次只是给这个形态推出了第一个能装在桌面上的成品。再过半年，看板可能会成为和聊天框、IDE 并列的第三种主流形态，而国内开发者会有自己的选择——不一定是 Kanbots 这一家、不一定是英文界面、不一定接的是海外 agent——但「同时让 AI 做几件事、每一件都看得见」这件事，在 2026 年的下半年大概率会变成个人 AI 工具的标配。

慢一点没所谓，能走通就行。
