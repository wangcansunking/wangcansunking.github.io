OpenAI 的 AI 安全平台：8 家 partner 双押

5 月 12 日，OpenAI 发布 Daybreak——一个把 GPT-5.5 拆成三档、再接进 8 家全球安全龙头的 AI 安全平台。GPT-5.5 是通用版，所有开发者都能用；GPT-5.5 + Trusted Access for Cyber 是受信任版，给已经验证过身份的防御方做漏洞分诊、补丁验证、检测工程；最高一档 GPT-5.5-Cyber 是受控红队验证版，只给授权账号做红队、渗透测试、隔离验证，不开放公开下载。绑在 Trusted Access 计划里的 8 家明示 partner——Cisco、Cloudflare、Palo Alto Networks、CrowdStrike、Fortinet、Oracle、Akamai、Zscaler——覆盖了从网络设备、边缘 CDN、端点检测、防火墙到零信任接入的全球安全工具栈。

这家公司选择走的路线，跟一周前 Anthropic 的 Glasswing 项目正面撞了上去。Anthropic 是用一档不公开的旗舰 Claude Mythos Preview 压全场，宣布 1 个月扫出 1 万多个高危漏洞、把 wolfSSL 一个证书伪造漏洞抓到 CVE 编号、给 Firefox 找出 271 个漏洞——但 Mythos Preview 本身不对外开放。OpenAI 这次直接亮了反面姿态：模型拆三档、SKU 明示、partner 名字摆出来、Codex Security 当作威胁建模代理写进开发者管道。

更有意思的细节藏在 partner 列表里。Cloudflare、Palo Alto Networks、Oracle 三家全球安全龙头同时押 Anthropic 和 OpenAI 两边——Cloudflare 上个月在 Glasswing 跑出了 2000 个 bug（其中 400 个高危），这个月又出现在 Daybreak 的 8 家明示 partner 列表里；Palo Alto Networks 在 Glasswing 期间释放的 patch 量是平时的 5 倍以上，同时也是 Daybreak 端点合作方；Oracle 在 Glasswing 公告里被 Anthropic 点名「修复速度快了好几倍」，同时是 Daybreak 的基建合作方。这种双押姿态意味着：对全球安全龙头而言，AI 安全工具已经不是「选一个 vendor 押注」的问题，而是「多模型并行评估 + 谁家先发现就用谁家」的常态。

国内安全大模型这条赛道上，奇安信、360、启明星辰、阿里云、腾讯五家都有自己的产品——奇安信 QAX-GPT 安全大模型加 AI 燔石漏洞挖掘 agent、360 大模型安全卫士、启明星辰九天·泰合安全大模型加安星智能体、阿里云通义千问安全侧、腾讯混元安全侧——产品形态齐了，但目前没有出现「模型分三档 + 8 家以上联盟接入」这种产品姿态。差距不在模型本身，而在产品组织方式：海外把 AI 安全做成多家共建的开放平台，国内还停留在 1 厂 1 客户的封闭项目制。这个差距是这篇文章想拆开讲清楚的对象。

一、Daybreak 平台架构：三档 SKU 把红队能力关在受信账号里

先把 Daybreak 这个名字说清楚。Daybreak 不是模型名——它是平台名，是 OpenAI 旗下专门面向 AI 安全场景的产品线，对应 Anthropic 那边的 Glasswing 项目（Glasswing 也是项目名不是模型名）。绑在 Daybreak 平台里跑的模型有三层 SKU，每一层对应不同的访问策略和能力边界。

第一档：GPT-5.5（通用版）。和你今天在 ChatGPT、API 里用的 GPT-5.5 完全一致，带标准安全护栏。用来做通用开发、知识工作、代码补全，没有任何额外的安全权限提升。任何账号都能调，价格走 OpenAI 标准 API 价目表。

第二档：GPT-5.5 + Trusted Access for Cyber（受信任防御版）。给已经通过 OpenAI 身份验证的防御方使用，覆盖的是「白帽」侧的全部任务：安全代码评审、漏洞分诊、恶意软件分析、检测工程、补丁验证。这一档的核心限制是「verified defensive work in authorized environments」——你必须证明你是在自家授权范围内做防御工作，不能拿这一档去攻击别人的系统。这一档的入口是 OpenAI 的 Trusted Access for Cyber 计划，目前明示加入的就是 Cisco、Cloudflare、Palo Alto Networks、CrowdStrike、Fortinet、Oracle、Akamai、Zscaler 这 8 家。

第三档：GPT-5.5-Cyber（受控红队验证版）。这是最值得拆解的一档，也是最容易被外行误解成「OpenAI 把红队工具开放下载」的一档——错的。GPT-5.5-Cyber 是「permissive model for red teaming, penetration testing, and controlled validation」，确实把通用版那些「不要写漏洞利用代码、不要生成恶意载荷」的护栏放松了一部分，但它不对外公开下载，所有调用都走 OpenAI 自己的 API，账号级控制 + 强身份验证 + 调用日志全留存。读这一档要带两条限定词：一是受控，所有调用都在 OpenAI 监控之下；二是验证，主要任务不是「批量写攻击载荷」而是「在隔离环境里验证一个漏洞到底能不能利用」。OpenAI 在所有三档上都明确禁止四件事：凭证窃取、隐蔽持久化、恶意软件投放、未授权利用。

下面这张三家平台架构对比图把核心维度摆全。注意 Microsoft Security Copilot（2024 年发布）这一列是放进来做对照的，它不是 AI 安全平台对位赛道的当期玩家，是上一代 GPT-4o 时代的产品。

平台架构这一层之外，Daybreak 还有一个绑死的代理框架叫 Codex Security。Codex 是 OpenAI 那个已经服役多年的代码代理，原来主要做代码生成；这次 OpenAI 给 Codex 加了一条「Security」侧的工作链路，专门做四件事：

  • 从代码仓库构建可编辑的威胁模型——不是跑一遍通用扫描器那种「列 1000 条 CVE」，而是基于具体代码库构建一份「这个项目最可能被攻击的路径有哪些、攻击者会从哪里入手、哪些代码段是高危」的可编辑模型，开发者可以基于这份模型继续追问。
  • 分析现实攻击路径——重点是「specific to that codebase, not generic checklists」，意思是 Codex Security 会跨多个函数、多个文件、多个依赖去追一条具体攻击链路，而不是机械跑 OWASP Top 10。
  • 在隔离环境里验证漏洞——找到疑似漏洞之后，Codex Security 会把它放到隔离环境（也就是不接触生产）跑一遍，确认这个漏洞是「真能利用」还是「告警噪音」。
  • 生成补丁 + 强制人工评审——补丁不会直接合到主分支，而是「proposed directly in the repository with scoped access and monitoring」，必须走人工评审才能落地。

OpenAI 官方原话是「reduces hours of vulnerability analysis to minutes」——把过去几个小时的漏洞分析压到几分钟。这条数字目前没有可对照的 CVE 战绩支撑（Daybreak 5 月 12 日刚上线，OpenAI 还没像 Anthropic 那样亮 1 万漏洞的数字），需要后续几个月看 partner 实战反馈。

二、8 家 partner 阵营：从网络设备到零信任接入的全球安全工具栈

8 家明示 partner 这个数字，比起 Anthropic Glasswing 公开的「约 50 家 partner」看起来少。但要看 OpenAI 这 8 家挑的角度——它不追求「数量多」，追求「覆盖全球安全工具栈的每一层」。下面按工具栈位置把这 8 家排开看。

网络设备 / 防火墙层：Cisco + Fortinet。Cisco 在企业网络设备里是过去 30 年的事实标准，路由、交换、IDS/IPS 设备装机量横跨绝大多数大型企业内网；Fortinet 是下一代防火墙（NGFW）领域第一梯队，全球装机量第二。这两家进来意味着 Daybreak 能拿到的攻击面信号是「企业级网络设备 + 防火墙日志」级别，是 Codex Security 做威胁建模时最底层的数据来源。

端点检测层：CrowdStrike + Palo Alto Networks（Cortex XDR 线）。CrowdStrike Falcon 是端点检测 EDR 赛道的当期龙头，2024 年那场「全球蓝屏事故」让所有人都记住了它的装机面之广；Palo Alto Networks 既是网络安全也是端点检测的双线龙头，Cortex XDR 是它的端点产品。这两家进来，Daybreak 拿到的就是「员工电脑、服务器主机、容器」级别的端点信号。

边缘 / CDN 层：Cloudflare + Akamai。Cloudflare 把全球 20% 的互联网流量过自家边缘节点，Akamai 是 CDN 赛道的老牌龙头。这两家进来，Daybreak 拿到的是「外部攻击流量」级别的信号——DDoS、爬虫、漏洞扫描、Web 应用层攻击全部经过它们。

零信任接入层：Zscaler。Zscaler 是云原生 SASE / 零信任网络访问（ZTNA）赛道的第一梯队，企业员工远程接入公司内网的流量全走它的全球节点。这一家进来，Daybreak 拿到的是「员工身份 + 应用访问」级别的信号。

基建 / 身份层：Oracle。Oracle 进来这件事乍看有点违和——它不是传统意义上的「安全公司」。但 Oracle 在企业基建里的位置是「数据库 + ERP + 身份目录」，全球大企业的核心业务数据有一半绕不开 Oracle Database 或 Oracle Cloud Infrastructure。Oracle 进 Daybreak 意味着 Codex Security 在做威胁建模时，能拿到「核心业务数据库 + 身份目录」级别的访问。

8 家加在一起，Codex Security 跑威胁建模时能调用到的信号，从外部流量（Cloudflare / Akamai）→ 网络设备（Cisco / Fortinet）→ 端点（CrowdStrike / Palo Alto）→ 接入身份（Zscaler）→ 核心基建（Oracle），完整覆盖了一条企业被攻击的全链路。这是 OpenAI 这次 partner 阵营设计的核心——不追求广度，追求把企业攻击链路的每一节点都接通。

需要补充的还有 OpenAI 自己提到的「20+ 家 partner 生态」。第一档 partner 之外，Daybreak 周边还围着 Intel、Okta、Netskope、Snyk、Gen Digital、Semgrep、Socket、Qualys、Rapid7、Tenable、Trail of Bits、SpecterOps、SentinelOne 等十多家——覆盖 SAST 静态分析、供应链审计、攻击面管理、事件响应、攻击模拟等周边赛道。这一圈不在第一档 partner 列表里，意味着它们不算 Trusted Access for Cyber 计划的核心成员，但已经在跟 Codex Security 做集成。

Akamai 那边的 CSO Boaz Gelbord 在 Help Net Security 的报道里给了一段引语：「Frontier models are fundamentally changing vulnerability management, and early access enables us to adapt proactively. The adoption of these capabilities will be critical for enterprise security teams.」——前沿模型正在从根本上改变漏洞管理这件事，提前接入让我们能主动适应。这一段表态比平台架构本身更值得读：8 家 partner 不是被 OpenAI 拉来站台的，是各家自己判断「AI 安全工具会成为企业安全采购的下一个标准件」，所以选择尽早接入做 co-design。

三、与 Anthropic Glasswing 对位：Cloudflare / Palo Alto / Oracle 三家「双押」的产业含义

5 月 8 日 Anthropic 发布 Glasswing 更新——约 50 家 partner、1 万多个高危漏洞、wolfSSL CVE-2026-5194、Firefox 271 个漏洞。5 月 12 日 OpenAI 发布 Daybreak——8 家明示 partner、三档模型、Codex Security 工作链路。两家在 5 月这一周内正面对位。

更有意思的对位不在数字上，在 partner 名单上。

Cloudflare。在 Anthropic Glasswing 公告里被点名「找出 2000 个 bug，400 个高危」，并且强调「false positive rate better than human testers」——假阳性率低于人类测试员。同一家公司，5 月 12 日又出现在 OpenAI Daybreak 的 8 家明示 partner 列表里，覆盖「边缘 / CDN 层」。

Palo Alto Networks。在 Glasswing 公告里被点名「释放 patch 量是平时的 5 倍以上」。同一家公司，又在 Daybreak 列表里负责端点检测线。

Oracle。在 Glasswing 公告里被 Anthropic 直接点名「修复速度是过去几倍」。同一家公司，又在 Daybreak 列表里负责基建身份层。

为什么这三家会同时押 Anthropic 和 OpenAI 两边？读这件事有三条逻辑可以放进来。

第一条，AI 安全工具的发现能力还在快速演化期，没人想押单家。今天 Anthropic Mythos 在 wolfSSL 上抓到 CVE，明天可能就是 OpenAI GPT-5.5-Cyber 在另一个开源项目上抓到。Cloudflare 不会因为 Mythos 这次找出 2000 个 bug 就关掉 GPT-5.5-Cyber 那条调用通道——下一个月可能反过来。

第二条，两家的产品形态在互补。Anthropic Mythos 是单档不公开模型，主要价值在「破坏性扫描」——一次性扫一个开源大项目（Firefox、wolfSSL、Linux 内核）找出全部高危漏洞；OpenAI 的三档分层 + Codex Security 是「持续集成式」——绑在开发者管道里，每次提交代码顺手做威胁建模。两条工作模式都有自己的位置，全球安全龙头在不同业务里都用得上。

第三条，多模型对照已经是企业级 AI 应用的常态。这个判断不光适用于安全赛道，整个企业级 AI 工具采购，今天的姿态早不是「选 OpenAI 还是 Anthropic」，而是「同一个工作流，让 Claude 跑一遍、GPT-5.5 跑一遍，看哪家结果好」。AI 安全这条赛道因为「漏报一个就可能丢全部数据」的高代价，多模型对照评估的需求更急迫。

CyberScoop 那篇报道里有一段总结说得直白：OpenAI 是「taking direct aim at Anthropic's tightly restricted Mythos model, offering a more open — but still carefully gated — path to AI-powered cyber defense」——OpenAI 选择走「比 Anthropic 开放、但仍精心管控」的路线。对照 Anthropic 自己说的「no company—including Anthropic—has developed safeguards strong enough to prevent such models from being misused」——没有任何公司（包括 Anthropic）开发出了强到能阻止滥用的护栏——两家在「红队能力到底要放出来多少」这个核心问题上路线选择不同，但都走到了「不公开下载、账号级管控」的同一种姿态上。

四、模型分层 vs 单一旗舰：两家路线分歧的工程权衡

到这里把 OpenAI 的「三档分层」和 Anthropic 的「单档不公开」这两种路线选择放到工程角度拆一拆——两条路线各有优劣，没有谁绝对赢谁。

OpenAI 三档分层路线的优点：

  • 入口宽。普通开发者用第一档 GPT-5.5 做日常代码评审、合规检查，已经够用；不需要等审核也不需要申请，调 API 就行。
  • 能力梯度清晰。第二档 Trusted Access 对应「我是甲方安全团队，要做内部漏洞分诊」，第三档 GPT-5.5-Cyber 对应「我是甲方的红队团队或外部渗透测试方，要做受控验证」。每一档对应的业务场景明确，定价和访问控制可以分开走。
  • 生态可扩展。三档 SKU 让 OpenAI 可以基于此跟 partner 共建——比如 Cisco 接的是「网络设备日志接 GPT-5.5 第二档做异常检测」，CrowdStrike 接的是「端点信号接 GPT-5.5-Cyber 第三档做隔离验证」，每一家 partner 接入哪一档可以独立设计。

OpenAI 三档分层路线的代价：

  • 管控复杂度高。三档 SKU 意味着 OpenAI 要维护三套访问控制系统，账号身份验证、调用日志、审计追溯都要分档跑。Anthropic 用「Mythos Preview 全部受控、不公开」一条路绕开了这个复杂度。
  • 滥用风险更分散。第二档和第三档的滥用边界比单档不公开模型更模糊。比如一个甲方安全工程师在第二档下「分诊」的过程里如果调用方式不当，可能逼近第三档「红队」的能力——这种边界 OpenAI 要靠账号级管控 + 调用模式监控来兜底。

Anthropic 单档不公开路线的优点：

  • 管控简单粗暴。Mythos Preview 全部走「受邀加入 + 模型不公开下载」一条路，没人能从公开渠道拿到模型权重，最大的滥用风险被一刀切掉了。
  • 能力上限可以做得更高。因为不需要面向公众做安全护栏，Mythos Preview 在内部测试场景里能力上限可以放得更宽。1 万漏洞 / Firefox 271 个 / wolfSSL CVE 这些战绩，部分来自这种「不受护栏约束」的能力释放。

Anthropic 单档不公开路线的代价：

  • 入口窄。普通开发者拿不到 Mythos Preview，没法在日常代码评审里直接用——只有 50 家受邀 partner 能用。这意味着 Anthropic 在「日常代码安全」这个最大市场里被 OpenAI 第一档卡住了。
  • 生态扩展受限。50 家 partner 之外的厂商如果想接入 Glasswing，要走 Anthropic 的 onboarding 流程；OpenAI 那边任何想加入 Daybreak 的厂商，只要愿意接受第二档 / 第三档的访问条款，就能直接申请。

两条路线放在一起看，能解释为什么 Cloudflare / Palo Alto / Oracle 这种全球安全龙头选择双押——它们不光要 Mythos Preview 那种「全力扫描破坏性发现」的能力，也要 GPT-5.5 三档那种「绑在开发管道做持续集成」的能力。前者解决「我有 1 万行祖传代码积压了一堆历史漏洞，需要一次性扫一遍」的问题，后者解决「我每天提交几百次代码，每次提交都顺手做威胁建模」的问题。两个问题在大企业里都存在，所以两边都要押。

五、国内 5 家安全大模型横评对比：奇安信 / 360 / 启明星辰 / 阿里云 / 腾讯

把视线拉回国内。AI 安全大模型这条赛道国内不是没有玩家——五家厂商各自的产品都已经上线，但姿态跟海外两家差距明显。

奇安信。奇安信的安全大模型叫 QAX-GPT，已经通过国家网信办「双备案」（生成式 AI 服务备案 + 大模型算法备案）。绑在 QAX-GPT 上的漏洞挖掘 agent 叫「AI 燔石」，是奇安信技术研究院主导的产品，定位是「漏洞终结者」——自动识别真假风险、自动生成 PoC/EXP 一键验证、把「发现-验证-修复」做成自动化流水线。在 DataCon（数据安全竞赛）的评测里，QAX-GPT 排名国内领先，2026 年 4 月还登顶 CyberSec-Eval 评测榜八项第一。

360。360 的安全大模型在国内综合能力评测里被赛迪顾问列为第一梯队，主打「数据和算力优势」。今年 9 月 360 推出「大模型安全卫士」，把安全大模型的能力面向自家客户矩阵（360 杀毒、360 安全大脑、360 浏览器）做集成。

启明星辰。启明星辰的安全大模型叫「九天·泰合」，绑在上面的智能体叫「安星」。今年 2 月启明星辰发布 MASB（Model Access Security Broker，大模型访问安全），定义了大模型时代的访问安全标准。客户主要是电信运营商、能源央企等政企甲方。

阿里云安全。阿里云的安全大模型走「云捆绑」路线——通义千问的安全侧能力集成进阿里云云安全中心，主要解决 AIGC 内容安全、deepfake 检测、anti-misinformation 这一类下游应用场景。

腾讯。腾讯的安全大模型基于混元大模型做安全微调，主要服务腾讯云客户，能力侧重 deepfake 检测、内容审核。

五家产品都在，但放到海外两家旁边对位有三个差距。

差距 1：产品形态停留在「单档不分层」。OpenAI 用三档 SKU 把红队能力关在受控账号里，国内五家目前都是单档单产品——奇安信 AI 燔石就是漏洞挖掘 agent，没有进一步分出「通用版 / 受信任版 / 红队验证版」三档；启明星辰、阿里云、腾讯也都是单产品形态。这不是技术做不到，是产品组织方式没走到这一步。

差距 2：partner 阵营是「闭环」不是「联盟」。OpenAI 8 家 partner 跨 Cisco / Cloudflare / CrowdStrike / Palo Alto 等不同公司，是「联盟式开放平台」。国内五家走的是「自家产品矩阵 + 自家客户」的闭环——奇安信的 partner 是奇安信自家客户，360 的 partner 是 360 自家产品矩阵，启明星辰的客户是电信运营商内部部署，阿里云和腾讯是云客户。没有出现「奇安信安全大模型接 360 端点信号 + 启明星辰网络日志」这种跨厂商联盟。

差距 3：CVE 战绩没有单口径披露。Anthropic Glasswing 公告里 1 万漏洞 / wolfSSL / Firefox 271 个这种数字，是把 AI 模型在漏洞挖掘上的成绩单独拎出来做公开口径披露。国内五家厂商的 CVE 战绩不是没有，奇安信的安全研究院年报里有大量自研漏洞披露，但「AI 模型直接挖出来的 CVE」这一项的单口径披露目前没有看到。这件事的差距不是技术差距，是披露习惯差距——国内安全厂商习惯把 CVE 算在「企业整体安全研究战绩」里，没有像 Anthropic 那样把 AI 战绩单拎一档讲清楚。

值得一提的是，国内监管角度其实在加速这件事的发生。国家网信办对生成式 AI 服务的「双备案」要求、对大模型算法的安全评估要求，给国内安全大模型铺了一条「合规先行」的赛道。奇安信的 QAX-GPT 已经过双备案，360、启明星辰也都在跟备案流程做适配。这条赛道的合规壁垒是国内厂商的天然优势——海外两家想进国内市场，要重新走整套备案流程。

六、AI 安全工具从单 vendor 到多模型评估常态：海内外路线对比下的国内机会

回到本文开头那个核心论点：AI 安全工具已经从「选一个 vendor」变成「多模型对照评估」的常态。这个判断在海外两家的 partner 重叠里已经被验证——Cloudflare、Palo Alto、Oracle 三家全球安全龙头同时押两边，不是出于战略试探，是出于「这俩家产品形态互补、都用得上」的实务判断。

对国内安全大模型厂商来说，这件事意味着两层机会和一层挑战。

第一层机会，国内厂商之间的「联盟式」整合已经具备技术基础。奇安信在漏洞挖掘上的 AI 燔石、360 在端点检测上的安全卫士、启明星辰在网络访问安全上的 MASB——这些产品的能力面其实是互补的，不冲突。海外 Daybreak 把 8 家不同赛道的安全厂商绑在一个平台里跑，国内五家如果愿意做类似的联盟，技术上是可行的。

第二层机会，单口径 CVE 披露这件事是低成本高收益的产品姿态升级。奇安信、360 的安全研究院每年发现的真实 CVE 数量不少，把其中「AI 模型直接挖出来」的那一部分单拎一档做季度披露，几乎不增加额外工作量，但能直接对位海外两家的产品姿态。Anthropic 那 1 万漏洞的数字之所以引发全球关注，不只是数字本身大，是它单口径披露的姿态给「AI 安全」这件事的产业认知做了一次重塑。

挑战则是模型分层 SKU 这件事需要时间。OpenAI 把 GPT-5.5 拆成三档不是一天做出来的——这背后涉及访问控制系统、账号身份验证、调用日志审计、滥用边界监控等一整套工程。国内五家厂商目前的产品工程化程度还不到这一步，要做到「通用版 / 受信任版 / 红队验证版」三档分层 SKU，需要从访问控制到合规审计的整套基建升级。

但从产业演化路径看，这件事是大概率会发生的。国内安全大模型走过「单产品上线 → 国家备案 → 客户落地」三步之后，下一步必然是「面向更大客户范围的能力分层 SKU」。海外两家这次在 5 月一周内对位发布，给国内厂商提供了一份非常清晰的产品姿态参考。

最后一句话留给国内同行：海外 AI 安全平台这一仗刚开打，今天 OpenAI 和 Anthropic 走到了「8 家 partner + 50 家 partner」「三档分层 + 单档不公开」两条不同路线，但都在「平台化 + 多模型对照 + 受控访问」这个大方向上同步推进。国内五家厂商的技术基础和合规壁垒都在——剩下的只是产品姿态怎么摆开的问题。
