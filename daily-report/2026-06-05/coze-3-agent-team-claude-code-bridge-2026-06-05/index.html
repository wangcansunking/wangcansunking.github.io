扣子 3.0 把 Claude Code 接进项目编排 AI 团队

2026 年 6 月 1 日，字节跳动旗下的 AI 智能体平台扣子（Coze）把 3.0 版本全量推了出来，iOS、Android、Mac、Windows 和网页端 coze.cn 五端同步迭代。

这次升级最该被记住的，不是某个单点能力变强，而是一条线被划了出来：扣子 3.0 把重心从「做一个智能体」搬到了「编排一支智能体团队」。

过去用扣子，你是在搭一个 bot——一个对话框、一套工作流，把它调好，让它替你回答问题或跑一段固定流程。3.0 之后，你面对的是一组可以分工的 Agent：你像项目负责人一样把活拆开、点名派给不同角色，云端的 Agent 和你自己机器上的命令行工具坐进同一个项目里一起干。

而对每天在用 Claude Code、Codex CLI、OpenClaw 的开发者来说，3.0 里最值得多看两眼的一句话是：这些本地 Agent，现在可以被接进扣子的同一个项目空间，和云端 Agent 一起协作。 这是桥接，不是取代——它没打算把你手里的命令行工具换掉，而是给它们找了个可以和别人配合的工位。

这篇就沿着这条主线往下走：3.0 的「编排一支团队」到底怎么落、用 @ 派活的项目空间长什么样、它把本地 CLI Agent 接进来意味着什么，再把它和 Kimi、智谱 AutoGLM 以及海外的编排框架摆到一起对照，最后落到个人开发者和小团队怎么上手。

先把结论摆清楚：3.0 变的是协作单位

一句话总览这次升级：扣子 3.0 的协作单位，从「一个 Agent」变成了「一支 Agent 团队 + 人」。

按官方功能说明，3.0 的核心变化可以收成三条同维度的能力：

  • 从单 AI 对话到多智能体团队协作：支持灵活组合「一人 + 多 Agent」或「多人 + 多 Agent」，把一个任务拆成子项，分配给不同角色的 Agent 协同处理；
  • 五端统一：iOS、Android、Mac、Windows、网页端 coze.cn 同步迭代，桌面端全新上线，手机可远程推进电脑上的项目；
  • 接入本地及云端模型与框架：除了云端 Agent，还能直接接入 Claude Code、Codex CLI、OpenClaw 等本地常用 Agent，可授权它们处理本地文件。

这三条不是并列的功能点，而是同一件事的三个面：当协作单位变成「一支团队」，你就需要一个能容纳多个角色的容器（项目空间）、一套跨设备的同步（五端）、以及一条把外部执行力拉进来的通道（接本地 Agent）。

一个要先讲清楚的口径：截至 2026-06-05 凌晨核对，扣子官方功能说明与新浪科技、IT之家、53AI 等多家报道，都没有给出 3.0 的具体用户量数字。所以本文不引用任何「X 亿用户 / 超过 X」类口径——凡涉及规模的说法，都按「官方未披露」处理，只谈能力本身。

和 2.x 比，3.0 的台阶迈在哪

这一节看一个属性：从扣子 2.x 到 3.0，变的不是某个功能强了多少，而是产品的基本单位换了。

熟悉扣子的读者大多是从 2.x 一路用过来的。那一代扣子已经是国内最成熟的智能体搭建平台之一：你可以可视化地拖工作流、配知识库、挂插件，把一个 bot 调得很能打。但 2.x 的世界观仍然是「一个智能体」——你精心搭出来的，是一个对话入口、一条相对固定的流程。

3.0 把这个基本单位换掉了。对照着看，台阶主要迈在三处：

维度 扣子 2.x（智能体搭建） 扣子 3.0（团队编排）
基本单位 一个 bot / 一条工作流 一支多角色 Agent 团队 + 人
协作方式 搭好后单独运行 项目内多 Agent + 多人同台、@ 派活
上下文 各 bot / 会话相对独立 项目空间统一沉淀、跨阶段接力
外部执行力 以平台内插件 / 工作流为主 可接本地 Claude Code / Codex CLI / OpenClaw
终端 以网页 / 移动为主 五端同步，桌面端全新上线

这张表里最关键的一行是「基本单位」。2.x 解决的是「怎么把一个智能体搭得更好」，3.0 解决的是「怎么把好几个智能体和人组织起来一起干一件复杂的事」。前者是把单兵练强，后者是把队伍编起来。

所以从 2.x 升到 3.0，对老用户不是「换了套界面」，而是多了一种工作方式：你 2.x 时代搭好的那些能打的 bot，在 3.0 里可以变成团队里的一个角色，被 @ 进项目和别人配合。2.x 的积累没有被废掉，而是被抬进了一个更大的协作场景里。

编排一支团队怎么落：任务拆解、角色、@ 派活

这一节看一个对象的一个属性：3.0 的编排模型，到底靠什么机制把一支 Agent 团队组织起来。

答案是三件套——任务拆解、角色分工、用 @ 派活。它们都装在一个叫「项目空间」的容器里。

项目空间：让上下文不再散落在各自终端

3.0 的协作发生在「项目」里，而不是一个个独立的对话框。官方的说法是多项目独立管理，项目内的上下文和产出资产会自动沉淀下来。

这一点对用过命令行 Agent 的人最有体感。过去 Claude Code、Codex CLI、OpenClaw 各自跑在自己的终端窗口里，上下文是割裂的——这个窗口查的资料，那个窗口并不知道。3.0 把它们放进同一个项目，上下文集中到一处，谁在什么阶段产出了什么，都留在项目里等下一棒接力。

用 @ 派活：人还是那个发号施令的人

3.0 没有把重点压在「全自动脚本调度」上，而是保留了人作为指挥者的位置：你通过 @ 的方式，把子任务点名派给具体的 Agent。

官方给的示例就是一张「阶段—交给谁」的对照：

阶段 适合交给谁
调研资料 调研类 Agent / Codex CLI
写公众号稿 写作类 Agent
跑本地文件 / 改代码 Claude Code 等本地 Agent

这套设计的取舍很清楚：它没有追求「一句话下去，AI 自己把所有事安排好」的全自动编排，而是让人留在回路里，按阶段把活派给最合适的角色。对要交付真实成果的场景，这种「人指挥、多 Agent 执行」的协作往往比纯自动调度更可控。

为什么留人在回路里，是这次更稳的一招

「人 @ 派活」和「AI 全自动调度」之间，3.0 明显偏向了前者。这不是技术上做不到自动，而是一个产品判断。

全自动编排听起来更性感——一句话下去，一群 Agent 自己把活分了、跑完了。但对要交付真实成果的场景，它有两个现实问题：一是出错时不好定位是哪一环、哪个 Agent 想岔了；二是涉及本地文件、改代码、动财务这类高风险动作时，把决定权完全交给自动调度，风险敞口偏大。

3.0 让人留在指挥位，等于在每个关键阶段都保留了一个把关点：你看得到上一棒的产出，再决定下一棒 @ 给谁。这是一种把「可控」放在「炫」之前的设计，对认真用它产出东西的人更友好。 等协作流程真的跑顺了，再逐步把一些低风险环节交给更自动的方式，也来得及。

这一节可以收成一句：3.0 的编排不是把人换掉的自动机，而是给人配了一个能容纳多角色、能记住上下文的指挥台。

跨端同步与桌面端：手机派活，电脑干活

这一节看一个属性：3.0 的跨端能力，解决的是「执行力在哪台机器上」的问题。

3.0 这次桌面端是全新上线，配合 App 端同步升级，五端之间内容无缝衔接。官方给的典型用法是：

  • 桌面端贴着你的本地文档和数据，本地 Agent 可以直接处理本机文件；
  • 手机端负责便携，可以远程遥控电脑上的 Agent 推进项目，编程、视频这类长项目随时在手机上往前推一步；
  • 网页端是统一的派活中心。

把这三端连起来看，跨端不只是「多个屏幕都能登录」。它真正解决的是——你的执行力散落在不同设备上，桌面有本地文件、云端有算力、手机在你手上，3.0 让它们能在一个项目里接力。 你在地铁上用手机 @ 一句「把昨晚那份调研整理成稿」，电脑端的 Agent 就能接着昨天的上下文往下做。

桌面端这次单独拎出来上线，意义尤其在「贴着本地干活」这件事上。云端 Agent 再强，也碰不到你本机磁盘里那份文档、那段代码；桌面端 + 本地 Agent 的组合，才让「在你自己机器上完成的那部分活」进了同一个项目。几个典型的接力画面：

  • 出门在外用手机 @ 一句「把这份合同的要点整理出来」，回到电脑前产出已经在项目里等你；
  • 编程项目卡在某一步，手机上先 @ 调研类 Agent 把方案查清楚，回工位让本地 Claude Code 接着改；
  • 视频 / 设计这类长项目，手机随时推进一小步，进度跨端连续，不用每台设备重新对齐上下文。

来源：扣子 3.0 官方产品截图，经 AITOP100 2026-06-02 报道转载

重点：它把 Claude Code、Codex CLI、OpenClaw 接了进来

这是全文最该停下来看的一节。看一个对象的一个属性：3.0 怎么把本地命令行 Agent 桥接进自己的项目，以及为什么这是桥接而不是取代。

桥接，不是取代

很多人看到「平台支持接入 Claude Code」第一反应是「又要把我的工具圈进去了」。但 3.0 这次的做法恰恰相反。

官方的描述是：Claude Code、Codex CLI、OpenClaw 这些本地 Agent 通常各自跑在自己的终端窗口里，3.0 把它们接进同一个项目，让它们和云端 Agent 一起协作。换句话说，你手里的命令行工具一个都不用换，扣子做的是给它们开一个能和别人配合的工位。

这里要给两个名字补上背景，方便不熟悉的读者对齐：

  • Claude Code：Anthropic 的命令行编程 Agent，跑在你本地终端，直接读写你的代码仓库；
  • OpenClaw：本地优先的个人 AI 助手框架（数据存在本机、模型随你换），这个开源仓库 2025 年 11 月底建立，截至 2026-06-05 星标已超过 37 万（gh api 核实 376827），是这条线全球范围最热的参照之一。

把这几个本地 Agent 接进项目后，它们能在你授权下处理本地文件——这是云端 Agent 做不到的那部分执行力。扣子负责编排和协作，本地 Agent 负责贴着你机器干那些必须在本地完成的活。两边各司其职。

来源：扣子 3.0 官方产品截图，经 AITOP100 2026-06-02 报道转载

权限边界要先想清楚

把本地 Agent 接进项目，效率上去了，要操心的东西也跟着多了。官方功能说明里专门强调了三条该守的边界，这部分尤其值得开发者认真对待：

  • 不要把整盘都开放给 Agent：按需开权限，限定它能碰的目录；
  • 操作留痕：谁让 Agent 改了哪个文件、执行了什么命令，最好能查；
  • 高风险内容要显式授权：合同、财务、密钥这类，得明确点头，不能靠一句含糊的指令带过。

这几条背后是一个朴素的判断：Agent 越像同事，就越不能只按聊天机器人的安全标准来管它，而要更接近开发工具、远程终端、自动化脚本那一档的权限设计。 这恰恰说明扣子把本地 Agent 当成了真正的执行者，而不是一个花哨的演示。

这一节收一句：3.0 接本地 Agent 的姿态是「请进来一起干」，不是「都搬到我这儿来」——它扩的是协作面，不是占有面。

具体的协作场景长什么样

抽象说三遍不如走一遍。假设你要做的事是：把一份外部调研整理成稿，再据此改一段本地代码里的配置。在 3.0 的项目里，这件事大致会这样接力：

  1. 开项目、定角色：新建一个项目，里面挂上调研类 Agent（云端）、写作类 Agent（云端），再把本地的 Claude Code 接进来；
  2. @ 调研类 Agent：让它去查清楚资料、整理成结构化要点，产出沉淀在项目里；
  3. @ 写作类 Agent：基于上一步的要点写初稿，它读得到项目里调研那一棒留下的上下文，不用你再喂一遍；
  4. @ Claude Code：让本地这位接着改你机器上那段代码——它在你授权的目录里读写本地文件，这是云端 Agent 碰不到的活；
  5. 回看留痕：整件事谁在哪一步改了什么、执行了什么命令，都留在项目里可查。

走完这一遍，能更直观地看清 3.0 的取舍：云端 Agent 负责查和写这类不碰本机的活，本地 Agent 负责贴着你机器干那些必须在本地完成的事，人负责按阶段 @ 派活和把关。 三方在一个项目里接力，而不是在三个互不知情的窗口里各干各的。

它和字节自家的豆包、火山引擎是什么关系

扣子是字节跳动旗下的产品，自然会有读者问：它和字节自家的豆包、火山引擎是什么分工？

简单对齐一下站位：豆包是面向大众的 AI 助手 App，解决的是「普通人随手问一句、用一下」；火山引擎是面向企业的云与模型服务平台，提供算力和模型能力的底层供给。扣子夹在中间，定位是「让人把智能体搭起来、编排起来」的生产工具。

3.0 之后这个分工更清楚了：扣子负责编排与协作这一层，谁来出模型能力、谁来供算力，它都保持开放——既能用云端 Agent，也能接你本地的 Claude Code、OpenClaw。 这种「编排层对模型与执行后端保持开放」的姿态，对开发者是友好的，你不会因为用了扣子就被绑死在某一家模型上。

横向对比：和 Kimi、AutoGLM、海外编排框架比，扣子站在哪

这一节做一件事：把扣子 3.0 放进同类产品的坐标里，看清它的定位差异。

国内外做「多 Agent / 智能体」的产品不少，但侧重点很不一样。按编排主体、是否接本地 CLI Agent、跨端、最适合谁四个维度摆一张表：

几个差异点值得展开：

  • 和 Kimi（月之暗面）比：Kimi 主打 Agent 集群自动分工，强调一组子 Agent 自动协调、长时间自主跑超长流程，编排主体偏「让 AI 自己安排」。扣子 3.0 则把人留在指挥位，用 @ 派活，且明确支持把你本地的 Claude Code、Codex CLI 接进来。一个偏自动集群，一个偏人主导的混合团队。
  • 和智谱 AutoGLM 比：AutoGLM 走的是端到端代你操作手机的路线，一句话让它在手机里跑外卖、订票这类日常任务，强项在「手机里的执行」。扣子 3.0 的落点是「项目里的编排与协作」，覆盖编程、写作、设计这类需要多角色配合的工作。两者解决的是不同场景的问题。
  • 和海外编排框架（LangGraph、CrewAI）比：那一档主要面向工程师，靠写代码定义 Agent 之间的流程图，灵活但有门槛。扣子 3.0 把编排做成了产品化的项目空间和 @ 派活，普通用户和小团队不写代码也能组起一支 Agent 团队——这是它最直接的差异化。

这里要补一句不夸大的判断：这几条路线没有谁高谁低，它们对应的是不同人群、不同场景。要在手机里全自动跑日常任务，AutoGLM 那条路更顺；要让一组 Agent 自己扛超长流程，Kimi 的集群思路有它的价值；要工程师按需精确控制每一步流转，LangGraph、CrewAI 的代码灵活性无可替代。扣子 3.0 补上的，是中间那一大块——不写代码的个人和小团队，也想把活拆给几个角色、还想用上自己本地那几个命令行 Agent。 这块需求一直存在，3.0 把它产品化地接住了。

对国内开发者还有一个实际的好处值得点出：扣子 3.0 接的是 Claude Code、Codex CLI、OpenClaw 这些我们每天真在用的工具，而不是另起一套谁也没听过的私有 Agent。这意味着上手成本低——你熟悉的工具不变，只是多了一个能让它们彼此配合的项目空间。

把这几条收一句：Kimi 拼自动集群，AutoGLM 拼手机执行，海外框架拼代码灵活性；扣子 3.0 选的是「让不写代码的人也能编排一支团队，并且能把本地 CLI Agent 拉进来」这条路。

上手锚点：个人开发者和小团队怎么用起来

这一节落到最实的问题：这套能力，个人和小团队第一步怎么用。

按使用者画像给三组锚点：

  • 个人开发者：把 Claude Code、Codex CLI 接进一个扣子项目，让云端的调研类 Agent 先把资料和竞品查清楚、整理好，本地的 Claude Code 再接着改代码、跑本地文件——上下文在项目里连着，不用在几个终端之间来回倒腾。
  • 内容 / 自媒体作者：用「调研类 Agent 查 + 写作类 Agent 写」的分工，配合 3.0 自带的自媒体、设计等场景专家技能和行业模板，把一篇稿子的流程拆成几棒接力。
  • 小团队：用「多人 + 多 Agent」的组合，几个人和各自的 Agent 在同一个项目里分工，多项目独立管理、资产自动沉淀，协作记录留得下来。

一条务实提醒：接本地 Agent 之前，先按官方那三条权限边界把目录权限、操作留痕、高风险内容授权设好。先在一个不碰敏感数据的小项目里跑通协作流程，再逐步放开。

三个值得先想清楚的问题

上手前，有几个问题先有答案，后面会顺很多：

  • 会不会被绑死在某一家模型上？ 3.0 对模型和执行后端保持开放，既能用云端 Agent，也能接你本地的 Claude Code、OpenClaw，开放度对开发者是友好的。
  • 我 2.x 搭的东西白搭了吗？ 没有。2.x 搭好的 bot 和工作流，在 3.0 里可以作为团队中的一个角色被 @ 进项目复用。
  • 多人协作时权责会乱吗？ 项目内多人 + 多 Agent 同台，但操作可留痕——谁让哪个 Agent 改了什么，是能查的，这正是多人场景该先确认的一点。

最小可行的起步路径

如果你想今天就试一下，给一条最省心的路径：

  1. 先在网页端 coze.cn 建一个不碰敏感数据的小项目（比如「整理一篇行业资料」）；
  2. 只挂一个云端调研类 Agent，跑通「@ 它去查、它把结果沉淀在项目里」这一步；
  3. 顺手后再加一个写作类 Agent，体会两棒之间上下文怎么接力；
  4. 等协作流程熟了，再把本地 Claude Code 接进来，并按权限边界限定它能碰的目录。

这条路径的核心是先把「人 @ 多 Agent 接力」这件事跑顺，再引入本地执行力——分两步走，比一上来就全套接上要稳得多。

写在最后：编排时代，扣子递上了一张顺手的工位

回到开篇那条主线：扣子 3.0 真正的变化，是把协作单位从「一个智能体」抬到了「一支智能体团队」。

它给的不是又一个更聪明的对话框，而是一张能容纳多角色、跨五端、还能把你本地 Claude Code 和 OpenClaw 请进来一起干的指挥台。对国内开发者来说，这是个挺顺手的信号——我们手里这些每天在用的命令行工具，第一次有了一个产品化的协作工位，不必再各自孤零零地待在终端窗口里。

国产 Agent 平台从「搭一个 bot」走到「编排一支团队 + 桥接本地工具」，这一步走得扎实。编排的时代刚刚打开，工具已经备好，剩下的就是我们把自己的那支 AI 团队组起来，一起往前走。
