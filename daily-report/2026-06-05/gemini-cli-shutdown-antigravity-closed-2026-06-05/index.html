Gemini CLI 六月关停，开源换闭源 Antigravity

一个在一年里攒下六千多个合并代码、十万多个星标的开源命令行工具，被官方定在 2026 年 6 月 18 日关停；接替它的，是一个目前并不开源的同名产品。社区里同一个免费账户，每天能发的请求数也从约 1000 次掉到约 20 次。

这件事说的是 Gemini CLI——Google 去年用 Apache 2.0 协议开源、塞进无数人终端里的那个 AI 编码代理。过去一年，它和 Claude Code、OpenCode 一起，是终端里最常被打开的几个编码代理之一。它的接替者叫 Antigravity CLI，是 Google Antigravity 这个面向代理的开发平台的命令行入口。

对很多人来说，这条命令已经是日常的一部分：写脚本时顺手 gemini 一下、CI 流程里嵌一段、IDE 里挂着扩展。正因为用得这么深，6 月 18 日这条关停线才显得格外具体。

这篇文章只想把一件事讲清楚：当一个开源工具说关就关、免费额度大砍时，开发者真正该做的不是焦虑，而是把日常生产力重新建在一个更可控的地基上。 下面从这次转向的来龙去脉、Google 的产品账、开发者为什么反弹，一直谈到对国内开发者的实际影响和一份替代选型表。

关停时间线：六月十八日这条线怎么划下来的

先把时间和对象交代清楚。截至 2026-06-05 凌晨核对，官方口径如下。

2026 年 5 月 19 日，Google Developers Blog 发出公告《An important update: Transitioning Gemini CLI to Antigravity CLI》，宣布把 Gemini CLI 转向 Antigravity CLI。2026 年 6 月 18 日起，Gemini CLI 以及 Gemini Code Assist 的 IDE 扩展，将对以下用户停止响应请求：

  • 免费层用户；
  • Google AI Pro 订阅用户；
  • Google AI Ultra 订阅用户；
  • 个人版 Gemini Code Assist 用户。

有一类人不受影响：通过 Gemini Code Assist Standard 或 Enterprise 许可证接入的企业组织，访问照旧；持有付费 API key 的开发者也不受影响。

换句话说，停的是面向个人和免费用户的那条开源链路，留的是面向企业付费的那条。距离这个日期已经不到两周，6 月初仍有新的迁移指南和讨论在持续出现，热度没有退。

这条线之所以扎眼，是因为 Gemini CLI 不是一个没人用的边角项目。它 2025 年 6 月用 Apache 2.0 协议开源，截至 2026-06-05 凌晨，其 GitHub 主仓库星标 104935（gh api 核实），是一个 TypeScript 项目；按 Google 官方公告与多家科技媒体的口径，一年里社区贡献了 6000 多个合并代码，星标越过十万。这是一个被社区真金白银投入过的工具。

把这组数字摆一起，能看出它在终端编码代理里的分量：

  • 十万多星标——和 Claude Code（130137 星）、OpenCode（169898 星）同处第一梯队；
  • 6000 多个合并代码——这意味着背后有数百名外部贡献者，而不是 Google 一家在维护；
  • 一整年的迭代——从 2025 年 6 月开源到 2026 年 6 月关停，正好一年。

一个工具能在一年里吸引这种规模的社区投入，说明它确实解决了真实问题。也正因如此，这次关停才会在开发者圈里激起这么大的回响——大家不舍得的不是一个 Logo，是自己也参与搭过的东西。

Antigravity CLI 是什么：更快的引擎，关上的源码

接替者 Antigravity CLI 本身并不弱。按官方公告，它有几个实打实的改进：

  • 用 Go 重写，启动和响应都比原来更利索；
  • 支持多代理在后台协同——可以让多个代理各管一摊，并行跑大规模重构或同时调研几个题目，终端不会被一个长任务卡死；
  • 保留了 Gemini CLI 的核心能力：Agent Skills（技能）、Hooks（钩子）、Subagents（子代理）、Extensions（扩展，现在叫 Antigravity 插件）；
  • 和 Antigravity 2.0 这个面向代理的开发平台统一到一套架构下。

官方给出的理由也站得住：用户的需求已经从「单个代理」演进到「多个代理互相通信、拆分任务一起解决复杂问题」，与其维护两套分开的工具，不如把力量集中到一个平台上。这是一个清楚的产品判断，不必替它辩护，也没必要无视。

问题出在另一面。Antigravity CLI 目前并不开源。 截至 2026-06-05 凌晨核对，它的 GitHub 页面上只有更新日志、说明文档和一段演示动图，没有可供社区阅读和提交的源码。对照之下，Gemini CLI 这一年积累的六千多个合并代码、那些由社区和首发伙伴（Dynatrace、Elastic、Figma、Shopify、Stripe 等）搭起来的集成，在拿不到源码的情况下很难原样延续。

这就是开发者反弹的真正落点：社区为一个开源项目投入的劳动，被用来改进了一个把贡献者关在门外的产品。

「先开源吸引人，再收进闭源」——这次争论的两面

围绕这件事，社区里争论得很热，The Register、The New Stack、FOSS Force 等多家媒体都做了报道。把两面都摆出来，比站队更有用。

一面是开发者的不满。 不少贡献者和评论者把这次操作概括成一种模式：用开源协议把工具放出来，吸引社区一起把它做大；等产品成熟了，价值收进企业付费版，社区拿到的是一个闭源的接替者。批评的焦点不在「Google 没权这么做」——Apache 2.0 当然允许 Google 这么安排自己的服务——而在于，那些由社区和首发伙伴一起搭起来的集成，因为接替者不开源，很难原样延续下去。对投入过时间的人来说，这种落差是真实的。

另一面是 Google 的产品逻辑。 官方的说法也不是托词：代理的用法确实在从「单个代理」演进到「多代理协同」，维护两套分开的工具，对厂商是真实的成本；把能力统一到 Antigravity 一个平台，技术上讲得通。而且企业付费层的访问完全没动，付费 API key 也不受影响——这说明 Google 砍的是免费供给的成本，不是整个项目。

把两面合起来看，更接近事实的判断是：这既不是一次单纯的「背叛」，也不是一次无可指摘的「升级」，而是一家公司在重新算自己的免费供给账。 对开发者真正有意义的，不是评判 Google 的对错，而是从中读出一条经验——免费且依赖单一厂商后端的工具，它的供给随时可能被重新定价。读懂这条经验，下一步就清楚了。

为了避免误读，这里也要给 Google 留一句公道话：Gemini CLI 这一年是真开源、真好用，它把终端编码代理这个品类往前推了一大步；Antigravity CLI 在工程上也确实更快、更能打。能力本身值得肯定，争议只在「开源还是闭源、免费给多少」这两件商业选择上。

两个 CLI 摆一起：得到了什么，失去了什么

把 Gemini CLI 和 Antigravity CLI 并排看，能更清楚这次转向的取舍。

维度 Gemini CLI（即将关停） Antigravity CLI（接替者）
实现语言 TypeScript Go（重写）
开源 是，Apache 2.0 目前不开源
社区贡献 6000+ 合并代码、数百贡献者 仅更新日志 / 文档 / 演示动图
代理模式 单代理为主 多代理后台协同
免费层额度 约 1000 次/天（社区口径） 约 20 次/天（社区口径）
企业付费层 受关停影响 不受影响、照常服务

读这张表，得失很清楚：得到的是更快的引擎和多代理协同，失去的是开源的源码和宽松的免费额度。 对企业付费用户，这是一次纯升级；对个人和免费用户，则是一次实打实的取舍。判断要不要迁，先看自己属于哪一类。

免费额度从约 1000 次掉到约 20 次：体感上的第二刀

如果说闭源是观感上的转变，免费额度的变化就是体感上的。

按社区实测口径——注意这是社区用户实测，不是官方公布数字——免费层每天能发的请求数，从 Gemini CLI 时代的约 1000 次/天，降到 Antigravity 上的约 20 次/天，相当于只剩原来的五十分之一。Google 官方的开发者论坛和支持社区里，已经有不少用户反映即便是付费的 Pro 层，新平台的额度也明显比过去吃紧，有人甚至报告 Antigravity CLI 的单次任务开销偏大，几分钟就能把个人 Pro 额度耗光。

这里要冷静区分两件事：

  1. 官方确定的：6 月 18 日关停 Gemini CLI、转向 Antigravity CLI、企业付费层不受影响——这些有官方公告背书。
  2. 社区实测的：1000 次/天降到 20 次/天的具体数字，来自社区用户的实际使用反馈，量级方向（大幅收紧）一致，但精确值会随账户和时段浮动，官方未发布过这两个数字。

把这两层分开看，结论不会跑偏：额度方向是明显收紧的，但具体能用多少，最终取决于 Google 后续怎么定价，而这正是不可控的地方。

开源许可证管不到 API 额度

很多人第一反应是「不是开源吗，怎么说关就关」。这里有一个值得想清楚的常识：

开源许可证管的是代码，管不到运行代码所需要的一切。

Apache 2.0 让你可以自由读、改、分发 Gemini CLI 的源码。但它对这几样东西完全无能为力：

  • API 访问权——代理要调用 Gemini 模型，得连 Google 的接口；
  • 模型额度——每天能调多少次，由服务方定；
  • 认证体系——用哪个账户、什么权限登录；
  • 云端基础设施——真正让代码跑起来的算力在对方手里。

代码开源，不等于服务永续。一个工具同时依赖「开源的客户端」和「闭源的后端服务」时，决定你明天还能不能用的，是后端那一半。 这不是要否定开源的价值，而是提醒一个判断标准：当你把日常生产力押在某个工具上时，要看清它哪一部分握在你手里，哪一部分握在别人手里。

可以拿一个简单的问题自测：如果明天服务方把免费额度砍到十分之一，或者干脆停掉这条线，我现在的工作流会不会停摆？如果答案是「会」，那这个工具的关键部分就不在你手里。Gemini CLI 这次正是这个问题的活教材——客户端的代码大家都能读，但能不能调通模型、每天能调多少次，从头到尾是 Google 一家说了算。

判断清楚之后，应对就有了方向：要么选一个后端也可控的工具（能自托管、能换模型），要么至少手里留着不依赖单一免费额度的退路（付费 API、备用工具）。这恰恰是这次事件最有价值的地方——它用一个具体案例，把「依赖某家闭源免费额度」这件事的风险摆到了明面上。看清楚了，反而好做选择。

对国内开发者的影响：两件具体的事

国内有大量开发者把 Gemini CLI 接进了日常工作流。这次转向落到国内，具体就是两件事。

第一件，6 月 18 日这条线对所有人一视同仁。 免费、Pro、Ultra 用户在这一天之后都会失去 Gemini CLI 的服务。如果你的脚本、CI 流程、日常编码里嵌了 gemini 命令，需要在这之前确认替代方案，避免那天早上突然发现命令不响应。

第二件，迁移是有成本的，但成本可控。 Antigravity CLI 保留了技能、钩子、子代理、扩展这些概念，迁移路径官方给了，社区也有多份指南。真正要权衡的不是「能不能迁到 Antigravity」，而是「值不值得把生产力继续押在一个闭源、且免费额度大幅收紧的工具上」。

如果你决定在 6 月 18 日之前做一次盘点，可以按这几步走：

  1. 先列清单：搜一遍自己的脚本、CI 配置、shell 别名里有没有 gemini 命令或 Gemini Code Assist 的 IDE 扩展，把受影响的位置先找全；
  2. 判断依赖深度：是偶尔手动调一下，还是嵌进了自动化流程——前者随手换，后者要排时间；
  3. 挑替代工具：按下文那张表，结合「要不要开源、要不要接国产模型、要不要本地优先」选一两个候选；
  4. 小范围试跑：拿一个真实的小任务，在新工具上完整跑一遍，确认命令习惯和效果都对得上，再全量切换；
  5. 留好后路：付费 API key 用户其实可以继续用 Gemini 模型，只是客户端要换，这条路也别忘了。

对国内开发者还有一层现实考量：Gemini 系列模型在国内访问本身就需要额外条件，把核心工作流绑定在一个对个人用户持续收紧的海外闭源服务上，长期看不是最稳的选择。这不是坏消息，而是一个把工具换成更可控选项的好时机。

还能长期用的终端编码代理，有这几条路线

好在 2026 年的终端编码代理已经相当成熟，选择很多。下面这张表按「开源与否 / 能不能接国产模型 / 适合谁」三个维度，把几条主流路线列清楚。所有星标和许可证均经 gh api 核实，截至 2026-06-05 凌晨。

工具 出处 开源 接国产模型 适合谁
Claude Code Anthropic 闭源 需自配 要稳定、预算够，团队首选
OpenCode 开源社区 开源（MIT） 支持 想要可自托管、模型自由
千问 Code（Qwen Code） 阿里千问（Qwen） 开源（Apache 2.0） 原生千问 延续 Gemini CLI 习惯的人
通义灵码 CLI 阿里云 闭源 原生国内模型 金融政务等合规优先场景
Pi 开源社区 开源 可接任意 爱折腾极简、自写技能的人
OpenClaw 开源社区 开源 model-agnostic 本地优先、数据不出本机

逐条说几句，帮你对号入座：

  • Claude Code：截至 2026-06-05 凌晨星标 130137（gh api 核实），Anthropic 出品。它是闭源的，但胜在稳定、能力强，是预算到位的团队最省心的选择。和 Gemini CLI 一样要注意，它也依赖闭源后端，区别在于 Anthropic 至今没有对个人用户做大幅收紧。
  • OpenCode：星标 169898（gh api 核实），MIT 许可证，是开源阵营事实上的标杆——模型无关、支持本地模型、终端体验打磨得好。想要「自己能托管、模型随便换」的人，从这里起步最稳。
  • 千问 Code（Qwen Code）：阿里千问出品，星标 24913（gh api 核实），Apache 2.0 开源。它本身就是延续 Gemini CLI 那套命令行工作流的开源方向，原生适配千问开源模型——对习惯了 Gemini CLI 操作方式的人，这是迁移成本最低的一条路。
  • 通义灵码 CLI：阿里云的命令行形态（原通义灵码于 2026-05-20 升级为 Qoder CN 系列，命令行子产品在终端里直接调用代理能力）。它闭源，但基于国内主流大模型、国内部署，满足金融、政务等行业对数据安全与合规的要求。
  • Pi：开源社区的极简代理（由 Mario Zechner 与 Armin Ronacher 主导），核心只有「读 / 写 / 编辑 / 命令行」四件工具加一个不到 1000 token 的系统提示词，技能可按需自己写。适合爱折腾、想把代理捏成自己形状的人。
  • OpenClaw：星标 376827（gh api 核实），开源、本地优先——数据存在本机不上云，模型无关，可自托管。对在意数据不出本机的人，这是最对口的一条。

把上面这些和「接 DeepSeek / 千问 / GLM（智谱）后端」组合起来，国内开发者完全能搭出一套既开源、又接国产模型、还能自托管的终端编码工作流。地基握在自己手里，下一次别人家的额度怎么调，都不会再打乱你的节奏。

接国产开源模型，适配这一步怎么走

替代工具选好了，模型后端怎么接，是另一半要算清楚的账。好处在于，2026 年的国产模型已经够用，而且接入路径很顺。

按场景大致分三类：

  • 想完全开源、可自托管：选 OpenCode 或 OpenClaw，后端接 DeepSeek 或千问的 API，甚至用 vLLM / Ollama 把开源模型跑在本地，整条链路都在自己掌控里；
  • 想延续 Gemini CLI 的操作习惯：选千问 Code，它本身就是这套命令行工作流的开源延续，原生配千问模型，几乎是平滑过渡；
  • 合规优先、要国内部署：选通义灵码 CLI，基于国内主流大模型、国内部署，金融政务这类对数据落地有硬要求的场景最对口。

这里要点出一个常被忽略的事实：编码代理和模型是解耦的。 Gemini CLI 的可惜之处，正是它把客户端和 Gemini 后端绑得太死，后端一收紧，客户端再好也用不了。而 OpenCode、OpenClaw、Pi 这些模型无关的工具，换后端只是改个配置，今天接 DeepSeek、明天换千问、后天试本地模型，工具一行代码都不用动。把「工具」和「模型」分开选、分开换，本身就是这次事件给的最实在的一条经验。

国产模型这一年的进步也撑得起这套用法：DeepSeek、千问、GLM 在代码任务上的表现已经能满足大部分日常开发，价格还普遍友好。对国内开发者来说，「开源工具 + 国产模型 + 可自托管」不再是将就，而是一个完全立得住的主力方案。

把生产力建在更可控的地基上

回到开头那句话。一个攒了六千多个合并代码的开源工具说关就关、免费额度砍到二十分之一，这件事本身不必渲染成谁背叛了谁——Google 把力量集中到 Antigravity 是清楚的产品选择，企业付费层也确实留着。

但它给每个开发者上了实在的一课：当工具的客户端开源、后端闭源时，真正决定你明天能不能用的，是你看不见的那一半。 6 月 18 日这条线，与其当成一次被动迁移，不如当成一次主动盘点——把核心工作流挪到能自托管、或者接得上国内模型的工具上去。

好消息是，2026 年的我们手里牌很多：OpenCode、千问 Code、Pi、OpenClaw 这些开源选项已经成熟，通义灵码这样的国产工具在合规场景里也站得住，再配上 DeepSeek、千问、GLM 这些可自由调用的国产模型，搭一套可控的终端编码流并不难。

前辈们已经把路趟出来了，工具也都摆在那里。把地基换得更稳一点，接下来该写的代码，一行都不会少。一起加油。
