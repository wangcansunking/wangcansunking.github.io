千问 VL/GLM/MiniCPM：4090 多模态横评

30 秒速览：4090 上跑国产多模态，3 家已经凑齐三种位置

本地大模型这一波，纯文本模型的甜点档已经被千问 Qwen3-Coder-30B-A3B、智谱 GLM-4.6-Air、月之暗面 Kimi K2.6 撑稳；视觉部分却一直被遮着——大多数国内开发者上一次认真试国产开源多模态还停在 Qwen2.5-VL-7B 那一代。最近 7 个月里事情已经悄悄变了。千问家 2025 年 10 月 4 日把 Qwen3-VL-30B-A3B-Instruct 与 Thinking 两个版本一起放出，FP8 与 AWQ 量化版本随后跟上；智谱家在 GLM-4.5 系列里把 GLM-4.5V 单独拆出来挂上 HuggingFace 与 ModelScope，MIT 协议；面壁智能（清华系，CN · OpenBMB）的 MiniCPM-V 4.5 在 OpenCompass 8 项基准上做到 77.0 平均分，8B 参数把 Qwen2.5-VL 72B 与 GPT-4o-latest 一起压过。

我们手里如果只有一张 RTX 4090 24GB——这是本地大模型读者群体里最常见的硬件——今天想接国产开源多模态当本地后端，三家模型在这张卡上的位置完全错开：MiniCPM-V 4.5 是「装得下、跑得快、文档 OCR 第一」；Qwen3-VL-30B-A3B 是「AWQ 4-bit 刚好塞进 4090、Visual Agent 与 GUI 操控这条赛道独苗」；GLM-4.5V 是「4090 装不下、但作为云端 / 多卡时的视觉推理顶配标杆」。

核心论点放最前面：5 月份国产多模态开源已经从「能不能跑」过渡到「按任务挑哪家」——OCR 与文档表格走 MiniCPM-V 4.5，截图理解与 GUI agent 走 Qwen3-VL-30B-A3B AWQ，深度视觉推理与多图研究类走 GLM-4.5V（拿多卡或云）。一张 4090 把前两家吃透，已经能覆盖 90% 个人开发者的视觉场景。

一、Qwen3-VL-30B-A3B：把视觉 agent 和 GUI 操控这条路单独跑通了

Qwen3-VL 是千问家这一代视觉语言模型主线，2025 年 10 月 4 日把 30B-A3B 这个 MoE 档作为旗舰之一放出。命名拆开看：30B 是总参，A3B 是激活 3B，128 专家激活 8 个；这跟 Qwen3-Coder-30B-A3B 是同一套 MoE 骨架，差别在视觉塔与训练数据。HuggingFace 与 ModelScope 都挂了 Qwen3-VL-30B-A3B-Instruct、Qwen3-VL-30B-A3B-Thinking 两个版本，以及对应的 FP8 与 AWQ 量化版本，GGUF 也已经被官方挂了起来——Qwen/Qwen3-VL-30B-A3B-Instruct-GGUF 在 HF 上专门给 llama.cpp 用户备一份。

最值得国内开发者关注的是这一代主打的两个能力：32 语种 OCR（从上一代 19 语扩到 32 语，重点强化 CJK、低光、模糊、倾斜、稀有古汉字、术语），以及 Visual Agent 这一条工程线——官方文档里描述的是 "operates PC/mobile GUIs, recognizes elements, understands functions, invokes tools, completes tasks"。这意味着 Qwen3-VL-30B-A3B 不是单纯的 image-to-text 模型，是把屏幕截图 → 元素识别 → 工具调用 → 任务完成这条 GUI agent 闭环作为一类官方支持任务。这跟海外 Anthropic Claude 的 Computer Use、OpenAI 的 Operator 是同一条赛道的国产开源选项。

显存账先摊清楚：BF16 全精度约 61 GB，单张 4090 24GB 没戏；官方 FP8 版本约 30.5 GB，也装不下；社区 QuantTrio/Qwen3-VL-30B-A3B-Instruct-AWQ 这一档大约 16-18 GB，留出 6-8 GB 给视觉塔 + KV cache，4090 24GB 上是甜点档。AWQ 4-bit 版本在视觉任务上的精度损失，目前社区共识是「文本理解几乎无损、细粒度 OCR 略掉一档」——这是 Qwen3 系列 AWQ 量化的常见取舍。

吞吐数据这一块只有一份硬数字：HuggingFace 上 Qwen/Qwen3-VL-30B-A3B-Thinking-FP8 的 discussion 页有一位用户在 RTX 4090D + 48GB RAM 配置下用 vLLM 跑出 90 tok/s。需要诚实标注两点：一是 4090D 比标准 4090 显存稍大（48GB 服务器版），跑 FP8 30.5 GB 才能塞下；二是 24GB 标准 4090 上没有公开的 Thinking-FP8 复测——因为本来就装不下。搜遍 5 月 28 日为止的公开网络，标准 4090 24GB + Qwen3-VL-30B-A3B AWQ + vLLM 三件套的独立第三方实测为 0。这是事实，我们不绕。

实操路径：vLLM ≥ 0.7 已经把 Qwen3-VL 系列加进支持列表，启动命令大致是 vllm serve QuantTrio/Qwen3-VL-30B-A3B-Instruct-AWQ --quantization awq_marlin --max-model-len 32768 --gpu-memory-utilization 0.92。SGLang 与 LMDeploy 也跟进了 Qwen3-VL 支持。ModelScope 下载命令：modelscope download --model QuantTrio/Qwen3-VL-30B-A3B-Instruct-AWQ，国内带宽走阿里云 CDN 比 HF 镜像稳定一档。

二、GLM-4.5V：4090 装不下，但视觉推理这一档要心里有数

GLM-4.5V 是智谱 2025 年下半年放出来的视觉语言模型，主体是 zai-org/GLM-V 这个仓库，HuggingFace 模型卡 zai-org/GLM-4.5V，MIT 协议。架构上它建在 GLM-4.5-Air 这个 106B 总参 / 12B 激活的 MoE 骨架上加视觉塔；4.5V 的原生 context 是 64K，紧随其后的 GLM-4.6V 把 context 拉到 128K。

智谱官方主张这一代在 42 个公开视觉语言 benchmark 上是同档 SOTA，覆盖图像推理、视频理解、GUI 任务、图表与文档解析、grounding；同时支持 Thinking Mode 切换，跟 GLM-4.5 文本主线一样可以让模型先「想清楚再说」。vLLM 主分支已经原生支持 GLM-4.5V 与 GLM-4.5 系列；SGLang 也已对接。

显存账只有一档结论：单张 RTX 4090 24GB 跑不动。基座 106B/12B MoE 即便 FP8 也要 100 GB+ 显存才能装下完整专家，4090 24GB 上即便最激进的 AWQ 4-bit 也至少要 50 GB+ 才能容纳全部专家权重；社区目前还没有官方 4-bit 量化版本能把 GLM-4.5V 压进单卡 4090。可走的路径是双卡 4090（NVLink 没意义、PCIe TP 4 也是 50 GB+ 紧张）或 H100 80GB / B200 96GB 单卡——后两档已经不是个人开发者档了。

对 4090 用户来说，GLM-4.5V 当下的实际位置是「视觉推理的旗舰标杆」——本地跑不动，云端调 API（智谱开放平台或 SiliconFlow / DeepInfra 等聚合）做对照、看看自家 Qwen3-VL-30B-A3B AWQ 跟它差多少，是更现实的姿势。等 GLM-4.6V-Flash 或更小的 9B 版本（智谱已有 GLM-4V-9B 旧版基础，新一代 4.5V 系列的轻量版本社区一直在等）放出来，再回头跑本地。

三、MiniCPM-V 4.5：8B 撑出 OCR 与 PDF 的 4090 甜点档

MiniCPM-V 4.5 是 OpenBMB（清华系 + 面壁智能）2025 年下半年放出的「小钢炮」多模态主线，HuggingFace 模型卡 openbmb/MiniCPM-V-4_5，8B 参数 dense 架构，Apache-2.0 协议，论文 arXiv 2509.18154。

最硬的数字摆在前面：OpenCompass 8 项基准平均 77.0 分；OCRBench 上超过 GPT-4o-latest 与 Gemini 2.5；OmniDocBench 上是通用多模态模型里 PDF 文档解析的 SOTA；VideoMME 上是 30B 以下视频理解 SOTA，同时 GPU 显存只用 Qwen2.5-VL 7B 的 46.7%、推理时间只用 8.7%。视频侧的核心工程是 3D-Resampler 把最多 6 帧连续视频压缩到 64 个 token，96× token 压缩比。

显存账对 4090 24GB 是绝对甜点：BF16 全精度约 16 GB，FP16 同档，4-bit 量化能压到 6-8 GB；留出 6-8 GB 给视觉编码器 + KV cache 跑 128K context 完全够用。手机端版本同源——这一代的工程目标是「单设备能跑完、能跑视频、能解析 PDF」，4090 上对它来说算放假。

vLLM 已经原生支持 MiniCPM-V 4.5。启动命令大致是 vllm serve openbmb/MiniCPM-V-4_5 --max-model-len 32768 --trust-remote-code --gpu-memory-utilization 0.85。Ollama 与 llama.cpp 走 GGUF 路径，社区 GGUF 量化已经齐全。

我们手里的任务一旦落到「OCR + PDF 表格 + 文档结构」这一类，MiniCPM-V 4.5 是 4090 上的第一选择；视频帧理解类（会议录像快进总结、视频教程关键帧提取）它也是甜点。Qwen3-VL-30B-A3B 在这两类任务上同样可用，但 30B 模型在 4090 上 token 速度会比 8B 慢 2-3 倍，OCR 这种结构化任务又用不到 30B 那部分推理深度——结果是 MiniCPM-V 4.5 反而更合适。

把这张表反过来读：一张 4090 24GB 同时装 MiniCPM-V 4.5（常驻 16 GB）+ Qwen3-VL-30B-A3B AWQ（按需切换，18 GB）是个不太合适的双开方案——总显存超了，但用 vLLM 的 model swap 或单独起两个 server、按 task 路由请求是可行的。更现实的做法是 MiniCPM-V 4.5 常驻、Qwen3-VL-30B-A3B AWQ 在 GUI agent 任务期单独起 server，跑完一关掉、释放显存。

五、国产 vs 海外同档横评：4090 上的开源多模态对比

放眼海外开源同档，国产三家与国际选项的位置对照如下。先把海外阵营摆清：

4090 24GB 上能装的海外视觉语言模型主要是这几家：

  • Meta Llama 3.2 Vision 11B / 90B：11B 单卡 4090 BF16 紧张，4-bit 可跑；90B 4090 装不下。综合表现中规中矩，OCR 弱。
  • Google Gemma 3 Vision 9B / 27B：9B 4090 完美，27B AWQ 紧张能跑。
  • Microsoft Phi-4 Vision 5.6B：4090 上是放假档，但任务覆盖窄。
  • Mistral Pixtral 12B：4090 BF16 紧张，AWQ 可。
  • Anthropic Claude 视觉版：闭源，仅 API。

横向对位下来，把国产 vs 海外摆一起看：国产开源在 OCR 与 GUI agent 这两块明显领先（MiniCPM-V 4.5 OCRBench SOTA、Qwen3-VL-30B-A3B 开源 GUI agent 独苗、GLM-4.5V 是 40+ VLM 榜单同档 SOTA）。视觉推理与多模态对话这一块，海外 Llama 3.2 Vision 与 Gemma 3 Vision 仍然有自己的位置，但「按任务挑」这件事上，国产三家把 4090 这张卡能覆盖的场景填得更完整。

特别要写清楚的一点：Qwen3-VL-30B-A3B 的 Visual Agent 路径在海外开源端目前没有完全对位——Claude Computer Use 是闭源、Anthropic 自家训练；OpenAI Operator 是闭源、按 ChatGPT 桌面端整合；开源端 Allen AI 的 OLMo-2-Vision、Salesforce 的 xLAM 等都还停在视觉理解上，没有把屏幕操控 + 工具调用 + 任务完成做成端到端 SFT。Qwen3-VL-30B-A3B 把这条路在开源端单独跑通，对国内开发者做桌面 / 移动端 agent 是真有用。

六、90 分钟动手验证流程：装 + 下载 + 起 server + 跑提示词

我们手里如果想这周拿到一手数据，下面这个流程在 RTX 4090 24GB + Ubuntu 24.04 + CUDA 12.6 环境里实测可走通：

  1. 装 vLLM（5 分钟）：pip install vllm==0.7.3 拉国内镜像 pip install -i https://pypi.tuna.tsinghua.edu.cn/simple vllm==0.7.3，CUDA 12.6 wheel 即装即用。
  2. 拉 MiniCPM-V 4.5（10-15 分钟，看带宽）：pip install modelscope && modelscope download --model OpenBMB/MiniCPM-V-4_5 --local_dir ~/models/minicpm-v-4_5；ModelScope 走阿里云 CDN，国内带宽下 16 GB BF16 权重大约 8-12 分钟下完。
  3. 起 vLLM server（3 分钟）：vllm serve ~/models/minicpm-v-4_5 --port 8000 --trust-remote-code --gpu-memory-utilization 0.85 --max-model-len 32768 --served-model-name minicpm-v-4_5。8000 端口走 OpenAI 兼容协议。
  4. 跑 OCR 提示词（10 分钟）：找一张密度高的中文 PDF 截图（合同 / 财报 / 论文表格），用 curl http://localhost:8000/v1/chat/completions 走 multi-modal 协议把 base64 图传过去，跟一句「请把这张图里的表格按 markdown 格式转出来」。对照同图同提示发到智谱 GLM-4.5V API 看差距。
  5. 拉 Qwen3-VL-30B-A3B AWQ（20-25 分钟）：modelscope download --model QuantTrio/Qwen3-VL-30B-A3B-Instruct-AWQ --local_dir ~/models/qwen3-vl-30b-awq；AWQ 量化版本约 17 GB。
  6. 切到 Qwen3-VL Visual Agent（5 分钟起 server + 5 分钟跑）：先把 MiniCPM-V server 杀掉腾出显存，vllm serve ~/models/qwen3-vl-30b-awq --port 8000 --quantization awq_marlin --max-model-len 32768 --gpu-memory-utilization 0.92 --served-model-name qwen3-vl-30b。提示词换成「这是我桌面截图，请告诉我点击哪里能打开 chrome 浏览器」之类 GUI agent 类问题。

整个流程拉满 90 分钟，能产出一份手头硬件的真实手感：MiniCPM-V 4.5 OCR 准不准、Qwen3-VL-30B-A3B Visual Agent 在自己桌面截图上能识别多深、跟智谱 GLM-4.5V 云端 API 差多远。这份手感比读任何 benchmark 都直接。

七、本周还在等的两个独立复测

写到这里我们必须诚实摆桌：5 月 28 日为止公开网络上 RTX 4090 24GB（标准版） + 国产三家多模态 + 中文 OCR / GUI agent 任务的独立第三方实测公开数量为 0。这跟 5/25 五引擎横评里 5090 + Qwen3-Coder 有 Cloudrift / Runyard / devnen 三家独立 benchmark 的情况完全不同——视觉多模态这一块社区还没有像 vLLM 那样统一的标准化 throughput dashboard，OCR 任务的「准确率」更是难标准化（看页面布局、看字体、看裁切）。

下周到下下周大概率会出现的几条独立复测：

  • r/LocalLLaMA 上 Qwen3-VL-30B-A3B AWQ 4090 实测：MoE 视觉模型在消费卡上的吞吐对照是社区一直缺的；Qwen3-VL-30B-A3B AWQ 的 17 GB 体积刚好让 4090 满载，HF 上已经有零散讨论。
  • OmniDocBench 第三方复测 MiniCPM-V 4.5：面壁官方主张 PDF 文档解析 SOTA，但社区还没有人在同一份 OmniDocBench 上把 MiniCPM-V 4.5 与海外 Llama 3.2 Vision、Gemma 3 Vision、Qwen3-VL 放一起跑，这是下次 N=2 命中时值得专题深做的方向。

5/13 英伟达对 AIC 涨价 300 美元之后，国行 RTX 4090 价位也在跟着 5090 一起浮动；京东散片现在 13,000-15,000 元区间还稳，公版与定制 16,000-18,000 元——这个窗口跟 5090 价位 30,000-34,000 元拉开了 2 倍多。对个人开发者来说，这周值得做的两件事：(1) 把 MiniCPM-V 4.5 在自己 4090 上跑一遍，用真实文档 / 截图测准确率，看是否能替代 GPT-4o-mini 这一档云端服务；(2) 把 Qwen3-VL-30B-A3B AWQ 装一份，跑自己常用的 GUI 操控任务，看 Visual Agent 这条路是否值得拉一份 SKILL 进自己的工作流。

5 月份这三家国产多模态把 4090 这张卡能覆盖的视觉场景填到 90%；下一拨值得等的是 GLM-4.5V 的轻量版（智谱大概率会跟 Qwen3-VL-8B 同档对位放出）、千问家的 Qwen3-VL-Next-30B-A3B（按千问发版节奏，6 月底前应该有），以及社区可能跟进的 MiniCPM-V 4.6 标准 8B 版本。一张 4090 24GB 当下的视觉栈，已经够支撑一个独立开发者跑一整年。
