Karpathy 65 行 CLAUDE.md 拿下 14 万 stars

65 行 Markdown，凭什么是 AI Coding 生态最星仓库之一

一月二十六号晚上，OpenAI 创始成员、特斯拉前 AI 主管、Eureka Labs 创始人 Andrej Karpathy 在 X 上发了条吐槽 LLM 写代码的长帖——四类反复犯的错，一条没遮掩。第二天，多伦多开发者 Forrest Chang（Jiayuan Zhang，multica.ai 创始人）把那条推文逐条落地成 65 行的 CLAUDE.md，扔进新建的 GitHub 仓库 multica-ai/andrej-karpathy-skills。

到今天五月二十二号，仓库 117 天累计 142,922 stars、14,657 forks，774 人订阅。这是当日 GitHub 公开 API 的数字，不是搜索引擎缓存。在 AI Coding 工具这条生态里，一个不含一行代码、只装一份 Markdown 行为契约的仓库，绕过了所有 vibe coding 神器、绕过了所有终端集成、绕过了所有 IDE 插件，硬把 star 拉到这个段位。它的核心论点也很硬：模型再强，没有可被读、被守、被检查的行为契约，agent 就是个跑偏概率高的实习生；写好这 65 行规则，比再换一个基座更划算。

整篇文章把这件事按四件子事拆开：仓库与时间线、四条铁律的中英对照、铁律一一对位 Karpathy 抱怨的病灶、国产八家 AI Coding 工具的适配现状。最后落到国内开发者最关心的那一段——这套规则在 OpenClaw 这种本地 AI 助手主项目上怎么挂、怎么验证，给一份可直接复制的精简中文版。

铁律原文与精准中文翻译

仓库 CLAUDE.md 文件 65 行，包含一个开篇说明、四条规则与一段结尾验证语。下面是四条规则的原文与翻译，全部对照原文件一字不动。

规则一：Think Before Coding · 先想清楚再动手

Don't assume. Don't hide confusion. Surface tradeoffs.

Before implementing:

  • State your assumptions explicitly. If uncertain, ask.
  • If multiple interpretations exist, present them - don't pick silently.
  • If a simpler approach exists, say so. Push back when warranted.
  • If something is unclear, stop. Name what's confusing. Ask.

中文：

不假设。不掩饰困惑。把权衡摆到桌面上。

动手实现之前：

  • 明确写出你的假设。不确定就提问。
  • 存在多种解读时，全部呈现——不要默默选一种。
  • 更简单的方案存在时，主动说出来。必要时反驳。
  • 搞不清楚就停下。把不清楚的点命名出来。然后问。

规则二：Simplicity First · 能简单就简单

Minimum code that solves the problem. Nothing speculative.

  • No features beyond what was asked.
  • No abstractions for single-use code.
  • No "flexibility" or "configurability" that wasn't requested.
  • No error handling for impossible scenarios.
  • If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

中文：

解决问题的最小代码量。不写任何投机性代码。

  • 不做超出需求的功能。
  • 一次性代码不要做抽象。
  • 没人要的"灵活性""可配置性"不要加。
  • 不为不可能发生的场景写错误处理。
  • 写了 200 行而其实 50 行能解决的——重写它。

自问一句：「资深工程师会不会说这写得太复杂了？」如果会，简化。

规则三：Surgical Changes · 外科手术式改动

Touch only what you must. Clean up only your own mess.

When editing existing code:

  • Don't "improve" adjacent code, comments, or formatting.
  • Don't refactor things that aren't broken.
  • Match existing style, even if you'd do it differently.
  • If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:

  • Remove imports/variables/functions that YOUR changes made unused.
  • Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

中文：

只动该动的。只清理你自己制造的麻烦。

在改既有代码时：

  • 不要"顺手改进"相邻代码、注释或格式。
  • 没坏的东西不要重构。
  • 匹配既有风格，哪怕你觉得另一种更好。
  • 注意到无关的死代码，提一句即可——不要删。

当你的改动产生孤儿（被本次变更弃用的 import / 变量 / 函数）时：

  • 删掉你的改动造成的弃用。
  • 不要去删已经存在的死代码——除非有人要求。

检验标准：每一行改动都应当能直接追溯到用户的请求。

规则四：Goal-Driven Execution · 目标驱动执行

Define success criteria. Loop until verified.

Transform tasks into verifiable goals:

  • "Add validation" → "Write tests for invalid inputs, then make them pass"
  • "Fix the bug" → "Write a test that reproduces it, then make it pass"
  • "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:

1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

中文：

先把成功标准定清楚。循环执行直到验证通过。

把任务翻成可验证的目标：

  • 「加校验」→「先写覆盖非法输入的测试，再让它过」
  • 「修 bug」→「先写一个能复现 bug 的测试，再让它过」
  • 「重构 X」→「保证重构前后测试都通过」

多步任务先写简短计划：

1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]

强成功标准让 agent 自己跑完一轮。弱标准（「让它能跑」）只能反复回头问你。

结尾验证语

These guidelines are working if: fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.

中文：

这套规则起作用的迹象是： diff 里多余的改动变少；因为过度复杂而重写的次数变少；澄清性问题出现在实现之前，而不是在出错之后。

处方与病灶：规则不是平行清单，是对位映射

很多人读这份 CLAUDE.md 时把它当成一份并列的好习惯清单。这是误读。这四条规则不是平行的，是 Karpathy 那条 X 推文里点名的四类病灶的处方，一一对位。下面把它们摆到一张映射矩阵里。

病灶一 → 规则一。 Karpathy 原话：「模型会替你做错误假设，然后顺着假设跑下去都不查一查；它们不管理自己的困惑、不寻求澄清、不抛权衡、不反驳。」处方是 Think Before Coding——要求 agent 把假设显式说出来，遇到歧义全部摆开，不确定就停下来发问，而不是猜一种跑下去。

病灶二 → 规则二。 原话：「它们真的喜欢把代码和 API 复杂化，抽象膨胀，不清理死代码……一件 100 行能搞定的事，能给你堆 1000 行。」处方是 Simplicity First——要求最小可工作代码，不主动加灵活性 / 可配置性，不写不可能发生的错误分支，200 行能压到 50 行就压。

病灶三 → 规则三。 原话：「它们有时还会改/删自己没看懂的注释和代码，作为副作用——哪怕跟当前任务正交。」处方是 Surgical Changes——每一行改动必须能追溯到用户的请求，只清理自己变更产生的孤儿，无关的死代码顶多提一句，不动手。

病灶四 → 规则四。 Karpathy 这一条没在原帖中直接列出，但他的整体框架里多次出现「应当先写测试再写代码」「应当让 agent 在 verify 闭环内跑」。处方是 Goal-Driven Execution——把任务翻成可验证目标（先写测试 → 让它过），多步任务先写计划，每步带验证方法，让 agent 能自循环不打扰人。

读到这里很多人会反问：这些不都是常识？资深工程师本来就这么做？没错，正因为是常识，更值得明确写到一份 agent 默认会读的契约文件里。LLM 的训练目标是最大化用户满意度，而不是最大化「不动无关代码」——常识需要被 explicitly stated 才能进 agent 的 prompt 上下文。65 行 Markdown 干的就是把常识固化成 agent 的硬约束。

八家 AI Coding 工具横评：谁能直接吃下这份 65 行

接下来到大家最关心的一段：这套规则在国内能用的工具上怎么落地？我把 2026 年五月还在持续更新的国产 AI Coding 工具横评了一遍，每家工具的官方文档与各自社区帖（V2EX / 知乎 / Linux.do / GitHub issues）都看过一遍。

把上图里的关键结论展开说：

通义灵码（阿里）： 没有原生 CLAUDE.md 支持，但提供项目级 system prompt 的 .tongyirc 文件位置。把 65 行规则原文粘进去即可，IDE 重启生效。难度：一行 import。国内直连，无需 VPN。这是国内开发者最低门槛的路径之一。

Trae（字节）： 工作区 setting 里有 system prompt 文本框，UI 表单操作。需要把规则粘到框里。难度比通义灵码稍高一些——文本框有长度提示，太长需要分段折叠。但同样国内原生直连。

Qoder（字节海外版）： 提供 Workspace Rules 文本框，可直接粘入规则。部分功能需登录海外账号，国内访问有概率被中转环节卡掉。属于「能用但麻烦」段。

Cline（开源跨国）： 这是最爽的一家——原生读取仓库里的 CLAUDE.md 文件，文件名都不用改，规则零适配。模型走自配 API，可以接 DeepSeek / 通义 / 智谱国产基座，绕过模型层访问问题。

Cursor 国内版（Anysphere）： 原生支持 .cursorrules 与 CLAUDE.md 双文件，文件名重命名即可，规则直接生效。模型层需 VPN 或中转，但规则层零摩擦。Cursor 是最早在 Forrest Chang 仓库 README 里被官方列出适配指引的 IDE。

千问 Code（Qwen Code）： 阿里推出的 CLI 工具，按官方文档采用 QWEN.md 作为接近规范的项目级规则文件。结构与 CLAUDE.md 极为相似，可以做一行 sed 替换适配。属实验性阶段——文档承认这是早期版本。

文心快码（百度）： 没有 CLAUDE.md 类文件支持，但提供项目级偏好设置的 UI 表单（暗色模式、代码风格、抽象偏好这类选项卡）。把四条铁律拆成四组偏好填进去，需要手工搬运，是八家里嫁接最累的。

智谱 GLM-Code（基于 Codegeex）： Codegeex Settings 里有 system prompt 字段，可以把规则拼成一段灌进去。智谱发的 GLM-Coder 系列基座质量在国产里偏前列，规则 + 国产强基座的组合在国内场景里性价比高。

把八家一字排开看：原生支持 CLAUDE.md / cursorrules 的只有 Cline 和 Cursor 国内版两家；千问 Code 用 QWEN.md 等价文件；通义灵码、Trae、Qoder、智谱 GLM-Code 走 system prompt 注入；文心快码需要拆成 UI 表单偏好。如果你今天想立刻试，最省事的路径是把 65 行原文存到仓库根目录的 CLAUDE.md，用 Cline 或 Cursor 国内版打开就行。

配置文件比模型更重要：一个月内五次出现的元层判断

把 multica-ai/andrej-karpathy-skills 放在最近一个月的全球 AI Coding 工具事件里看，会发现一件有意思的事——「写好规则比换基座更划算」这个元层判断，不只这一家在喊。

四月二十八号 Dirac Harness 提出「regression harness > model upgrade」——把回归测试集做扎实，比追每周一更的基座更值。五月十五号 Claude SMB 15 个 agent workflow 全部围绕 prompt 工程组织，没有任何一个新模型。五月十六号 Anthropic 推出 Legal Pack 十二个 plugin，把法律行业 SOP 写成 plugin 而不是去训一个法律基座。五月十八号 Academic Research Skills 走的是同一条路——学术写作的元约束写成 skill，让通用模型跑学术任务。五月十九号本仓库的 OpenClaw + Qwen3 个人工作流专题给出了同样的实测结论：本地 Qwen3-Coder 30B 配上一份精心写的 SKILL.md，能解决八成 Claude Code Max 的活儿。

到了五月二十二号这一家——65 行 CLAUDE.md，14 万 stars。论点收口：在 2026 年的 agent 时代，配置文件层的边际回报远高于换基座的边际回报。基座质量已经过了「能不能完成任务」的门槛；现在的差距在「这个 agent 守不守规矩」。守规矩的契约写在哪？写在 Markdown 文件里。

这件事对国内开发者尤其重要。我们没有 GPT-5 / Claude Opus 4.7 这种顶级闭源基座的成本优势，但我们能写出和海外同行同样质量的 65 行规则，这是个零成本可复制的工具。Karpathy 这条线的故事其实在告诉我们：你不用焦虑「下一个基座我们追不上」——把规则层写扎实，是中国开发者今天就能拿满的功夫。

OpenClaw 嫁接：两条路径，三十行精简版

OpenClaw 是本仓库的主项目，本地 AI 助手方向的开源 TypeScript 工程。从五月十九号专题的实测看，OpenClaw + 本地 Qwen3-Coder 30B A3B 在 RTX 4090 上跑一周共 256 次 agent 调用，月度边际成本 430 元，覆盖写作 + 财务两条主线。把 Karpathy 这份 65 行 CLAUDE.md 嫁接到 OpenClaw，有两条路径。

路径 A，挂到全局规则。 复制 65 行原文到仓库根，~/.openclaw/config.yml 添加 global_rules: - source: ./CLAUDE.md / apply_to: [code_edit, plan] / enforce: strict / language: zh-CN。重启 CLI，规则全局生效。验证方法是让 agent 改一行无关代码——按规则三应当被拒。OpenClaw 与 Claude Code 同源，一改双吃。

路径 B，拆成四个 skill。 把四条铁律拆成独立的 think-before-coding / simplicity-first / surgical-changes / goal-driven 四个 skill，放到 ~/.openclaw/skills/<name>/SKILL.md。按场景启用——写新代码全开；改 bug 只开 ④；review 别人 PR 只开 ②③。这种路径的好处是颗粒度更细，团队多人协作时可以 fork 到 internal git 做 PR 修订。

在 OpenClaw + Qwen3-Coder 30B A3B 本地 4090 的实测样本里（作者一周 256 次 agent 调用统计），挂同一份 CLAUDE.md 之后，agent 改无关文件次数从 11/100 降到 1/100。这跟海外 Tech Times、Level Up Coding 给出的描述方向一致——「fewer unnecessary changes in diffs」是 Karpathy 给出的第一条验证迹象，作者自己跑出来的数字也确认了这点。

最后留一份给国内开发者的精简中文版 CLAUDE.md。把它存成 CLAUDE.md 放进项目根，Cline / Cursor 国内版直接读；通义灵码 / Trae / Qoder 把它粘到 system prompt；千问 Code 重命名为 QWEN.md；OpenClaw 走路径 A。

# 项目协作约定（基于 Karpathy 四条铁律精简版）

## 1. 先想清楚再动手
- 写出你的假设；不确定就先问。
- 有多种解读，全部摆出，不要默默选一种。
- 更简单的方案存在，主动说；必要时反驳。
- 搞不清楚就停下，命名不清楚的点，发问。

## 2. 能简单就简单
- 不做超出需求的功能。
- 一次性代码不抽象。
- 没人要的灵活性、可配置性不加。
- 不为不可能发生的场景写错误处理。
- 200 行能压到 50 行就压。

## 3. 外科手术式改动
- 不顺手改无关代码、注释、格式。
- 没坏的东西不重构。
- 匹配既有风格，哪怕你觉得另一种更好。
- 无关死代码提一句，不删。
- 只删本次改动产生的孤儿 import / 变量 / 函数。
- 验证：每一行改动都能追溯到用户请求。

## 4. 目标驱动执行
- 「加校验」→ 先写非法输入测试，再让它过。
- 「修 bug」→ 先写复现测试，再让它过。
- 「重构 X」→ 保证重构前后测试都通过。
- 多步任务先写计划：步骤 → 验证方法。
- 强成功标准让 agent 自循环；弱标准只能反复问你。

## 验证
- diff 里多余改动变少。
- 因过度复杂而重写的次数变少。
- 澄清问题出现在实现之前，而不是出错之后。

这三十行是 Karpathy 原版 65 行的中文精简，删掉了重复说明，保留全部硬约束。复制粘贴即可启用。

给中国开发者的判断

总结一下这件事的价值。Karpathy 的四条铁律是从五年 OpenAI / 特斯拉 / Eureka Labs 一线 AI 工程师视角沉淀出来的「LLM 写代码反复犯的错」处方。Forrest Chang 用 65 行 Markdown 把它落地。117 天 14 万 stars 不是营销，是几十万开发者用脚投票投出来的——这套规则真的让他们 diff 干净了、重写少了、澄清前置了。

我们今天遇到的局面是：国内基座（千问、文心、GLM、DeepSeek）写代码的能力已经跨过门槛，但 agent 守规矩这件事在国产 AI Coding 工具里普遍不如海外重视。八家横评里只有 Cline 和 Cursor 国内版做到了原生支持。这是中国 AI Coding 生态接下来最值得补的一段——不是再训一个更大的基座，而是把规则层做扎实，让通义灵码、Trae、千问 Code 都默认读 CLAUDE.md。
