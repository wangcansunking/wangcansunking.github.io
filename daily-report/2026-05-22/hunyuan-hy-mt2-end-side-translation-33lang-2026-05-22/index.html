腾讯混元 Hy-MT2：440MB 跑 33 语种，手机离线翻译时代真到了

这件事的位置

五月二十一日下午，腾讯混元在自家官方公众号、量子位、新浪科技同时发出一条消息：Hy-MT2 翻译模型三档同时开源，1.8B、7B、30B-A3B 三个尺寸覆盖从手机到服务器的完整谱系。同步上线了一款叫「腾讯 Hy 翻译」的微信小程序，把这套能力交到了普通用户手里。

如果只读到这里，很容易把它当成「又一篇国产开源翻译模型发布」的常规公关。但这次有三件事不太一样。

第一件事是 1.8B 这一档真的塞进了手机。1.25-bit 极致量化之后体积只剩 440 MB，比 Hy-MT1.5 同尺寸版本推理还快 1.5 倍，苹果、高通、联发科三家主流手机芯片都能跑。这件事的意义是：从今天起，国内开发者第一次有了一个不依赖云端 API、不收按字符费、可以完全离线工作的翻译模型，而且它的权重本身是开源的。

第二件事是腾讯一次开源了完整谱系。同代厂商发布翻译模型时通常先放一档作为试水，腾讯这次直接把 1.8B、7B、30B-A3B 三档加上对应的 FP8 与 GGUF 量化版本全部丢到 HuggingFace 与 GitHub，仓库 Tencent-Hunyuan/Hy-MT2 实测开放访问，截至发稿前已经收到 97 颗星。三档定位也分得很清：1.8B 端侧轻量、7B 桌面均衡、30B-A3B 服务端专业。

第三件事是民族语言这一块。33 种主流语种之外，模型卡上还列了粤语、藏语、维吾尔语、蒙古语、哈萨克语 5 种少数民族语言或方言。这一组覆盖在主流厂商的翻译产品里几乎是空白——Google Translate 与 DeepL 都没有藏语、维吾尔语的稳定支持，Meta NLLB-200 虽然语种多但偏服务端，覆盖中国少数民族的实际下沉应用稀缺。

本文按可独立核实的数字、腾讯混元官方技术报告口径、HF 模型卡 README、四份独立中文媒体把这件事的工程意义讲清楚，并把字节豆包翻译、阿里通义翻译、百度翻译、网易有道、Meta NLLB-200、Apple Foundation Models、Google Translate (Gemini) 七位邻居一起摆到桌面上看。这次的核心问题不是「Hy-MT2 是不是世界最强翻译模型」，而是「为什么这套三档加 1.25-bit 量化的组合是国产端侧翻译这件事的真正起点」。

三档模型：从口袋到机房一次铺到位

三档同发是这次发布最值得仔细看的部分。把它和过去半年国产翻译类发布对照：阿里通义在春节后只放出 Qwen-MT 一档，字节豆包翻译 API 始终维持云端单一形态，网易有道 Pro 只把端侧能力捆绑在词典笔硬件里。腾讯这次直接给了一份完整谱系。

Hy-MT2-1.8B 是这次最关键的一档。BF16 原始权重大约 3.6 GB，FP8 压到 1.8 GB，GGUF 2-bit 进一步压到 574 MB，最终走 AngelSlim 1.25-bit 极致量化定格在 440 MB。这个体积意味着任意一款 2024 年之后的中端及以上手机都能放进可用内存。新浪科技在报道里给出的描述是「可以轻松部署在主流手机芯片上支持本地推理」，并显式点名苹果、高通、联发科三家——也就是说从国产安卓主力到 iPhone，都在覆盖范围之内。

Hy-MT2-7B 走桌面 / 服务端均衡路线。HF 模型卡同时给了 BF16、FP8、GGUF 三种格式，FP8 体积约 7 GB，单卡消费级 GPU 可以承接，Apple Silicon 通过 llama.cpp 也能跑。这一档的定位与 Qwen-MT 7B、DeepSeek-V4-Pro 一类同代开源大模型对位，腾讯自评在这一档上「超 DeepSeek-V4-Pro / Kimi K2.6 fast-thinking」，但需要注意 fast-thinking 模式是这两款模型的低思考链路设定，并不是它们的全力档。

Hy-MT2-30B-A3B 是旗舰。30B 总参数、激活 3B 的 MoE 架构，FP8 体积约 30 GB，目标是「专业翻译」「企业级」「真实场景」。腾讯官方技术报告称这一档在真实场景超过 Gemini 3.1 Pro，在 FLORES-200 公开评测集上接近 Gemini 3.1 Pro。两句话需要拆开看：「真实场景」是腾讯自建评测集 IFMTBench 的结果，「FLORES-200 接近」是公开评测集的结果。前者需要时间等第三方复现，后者的差距大小公开报告里暂未给出单点 BLEU 数字，等 PDF 详细解读出来才好严格判断。

三档之间的取舍很清楚：要离线、要装进手机，选 1.8B；要桌面单机能跑、要平衡能力与成本，选 7B；要企业接入、对译文质量有专业要求，选 30B-A3B。一次性铺出三档，是国产翻译大模型这两年节奏最稠密的一次开源动作。

AngelSlim 1.25-bit：把权重「折叠」进 440 MB

整套发布里最有工程感的一点，是 1.25-bit 这个数字本身。它不是「1-bit」也不是「2-bit」，平均下来每个参数只占大约 1.25 个比特——比 INT4 节省约 3.2 倍空间，比开源社区主流的 2-bit GGUF 再省 1.6 倍。

直觉上，1-bit 意味着每个权重只能是 +1 或 -1，会丢失很多信息；BitNet 路线的研究证明 1-bit 在大模型上可行，但需要从头训练。腾讯这次走的不是 BitNet 路线，而是「1 bit 主码 + 共享 0.25 bit 缩放因子」的混合方案：权重分组后，每组用 1 bit 表达符号方向，再共享 0.25 bit 的组内 scale 用来恢复数值幅度。平均下来每个参数 1.25 bit，但每组共享 scale 保持了数值分布的平滑度，避免了纯 1-bit 在尾部权重上的极端误差。

ModelScope 模型卡披露这个技术内部代号叫「Sherry」，归属于腾讯的 AngelSlim 量化框架。和 llama.cpp 社区常见的 Q1_M、Q2_K 这一类通用 Round-to-nearest 量化相比，「Sherry」做了一件关键的事：针对翻译任务做了 SFT 校准——也就是说量化过程中保留了翻译相关的关键子空间，让 1.8B 模型在压到 440 MB 之后翻译质量损失被控制在能用的范围里。

腾讯官方对这一档的评价是「整体超越微软等主流商业 API」。这是厂商自评，需要按业务垂直跑过才能放心，但即便把这条结论的可信度打个对折，「440 MB 离线、不联网、不收按字符费、零 API 调用」这件事本身已经足够把端侧翻译的工程基线往前推一大截。

33 主流 + 5 民族：1056 条翻译方向

语种覆盖是这次发布的另一个核心维度。HF 模型卡 README 列出的完整清单包括中、英、法、葡、西、日、土、俄、阿、韩、泰、意、德、越、马来、印尼、菲律宾、印地、繁体中文、波兰、捷克、荷兰、高棉、缅甸、波斯、古吉拉特、乌尔都、泰卢固、马拉地、希伯来、孟加拉、泰米尔、乌克兰 33 种主流语种，再加粤语、藏语、哈萨克语、蒙古语、维吾尔语 5 种民族语言或方言。

ModelScope 模型卡把这个组合表述为「33 种语言，5 种方言/少数民族语言，1056 种翻译方向」。1056 的算法是 33 × 32——即 33 种主流语种两两之间任意互译，每条方向单独计数。5 种民族语言以加层形式叠加在这之上，让端侧本地翻译能力第一次延伸进民族文化场景。

这个组合的特殊之处在于覆盖结构。Meta NLLB-200 语种最多（200+），但定位偏学术与服务端；Google Translate 在 Gemini 加持下支持 70+ 实时语种，主要走云端；DeepL 与商业 API 集中在欧洲语系；Apple Foundation Models 走 iOS 内嵌路线，目前实时翻译覆盖大约十五到十九种。Hy-MT2 是这一组里唯一同时满足「端侧落地」「开源权重」「民族语言覆盖」三个维度的方案。

具体到中国市场，民族语言这一块覆盖能落到几个真实场景：新疆维吾尔自治区与哈萨克自治州的政务双语翻译、内蒙古的蒙汉混合教学材料、西藏与川甘青藏区的旅游业辅助、广东港澳与海外华语圈的粤语口头翻译。过去这些场景里要么没有开源工具可用，要么只能调用云端商业 API 按字符付费——现在 440 MB 装进任何一部手机就能离线工作。

横评：八家同代模型摆到一起看

把腾讯之外的七家邻居摆到桌面上：

字节豆包翻译：闭源云端 API，深度集成在豆包 App 与飞书机器人里。优势是与字节自家产品生态绑定紧、面向 C 端体验流畅；劣势是不开权重、按调用计费、无端侧版本。腾讯在 1.8B 这一档自评「超越微软等主流商业 API」时，潜在对位包含豆包翻译。

阿里通义翻译 / Qwen-MT：阿里 2025 年下半年开源了 Qwen-MT 一档作为试水，主推依然是云端通义翻译 API，绑定 1688、国际站等阿里跨境业务。语种覆盖 200 多种（商业 API 口径），但端侧版本目前没有真正可用的开源权重。

百度翻译 / 文心：纯云端 API，集成在百度搜索、百度输入法里，C 端覆盖广。开发者使用门槛低，但同样不开权重、不可端侧化。

网易有道翻译：国内最早把翻译能力做端侧化的厂商之一，但绑定在词典笔、有道学习机硬件里，软件 SDK 不公开。开源权重路径走不通。

Meta NLLB-200：54.5B MoE 模型，语种 200+，是当前开源翻译模型里语种最广的一档。FLORES-200 上 BLEU 比上一代翻译系统平均提升 44%，部分非洲与南亚语言提升超过 70%。但 NLLB-200 定位偏学术与服务端、CC-BY-NC 许可证限制商用，且没有民族语言、没有 1.25-bit 这种极致量化版本。

Apple Foundation Models：iOS 26 起把 Live Translation 集成进 Messages、FaceTime、Phone、AirPods Pro 3，端侧推理走 Apple Silicon。优势是终端体验丝滑，劣势是闭源、语种覆盖大约十五到十九种、生态封闭。

Google Translate (Gemini)：2025 年下半年 Google Translate 集成 Gemini 翻译模型，实时翻译覆盖 70+ 语种，Pixel 设备部分能力做端侧，但整体仍走云端为主。许可证闭源，权重不开放。

横评结论很清楚：Hy-MT2-1.8B 是这张表上唯一同时满足「端侧落地 + 开源权重 + 民族语言覆盖」三个维度的方案。Meta NLLB-200 语种最广但偏服务端、CC-BY-NC；Apple、Google 端侧丝滑但闭源；国内其他四家全部走云端 API 路径。

与 Hy-MT1.5 的演进：从「跟上」到「接住」

Hy-MT1.5 是腾讯混元在 2025 年底发布的上一代翻译模型，论文 arXiv 2512.24092，作者包括 Mao Zheng、Zheng Li、Tao Chen、Mingyang Song、Di Wang。当时的定位是「跟上 NLLB / Google Translate」，1.8B 端侧档已经开源，但量化深度只到 2-bit。

Hy-MT2 在这一基础上做了三件事的提升：

第一是量化深度推进。Hy-MT1.5 端侧版本量化到 2-bit GGUF 体积 574 MB，Hy-MT2 的 AngelSlim 1.25-bit 继续往下压到 440 MB，体积省了 23%，推理速度反而提升 1.5×。这意味着同样一部手机，跑同一档 1.8B 模型，Hy-MT2 启动更快、响应更顺。

第二是旗舰档加宽。Hy-MT1.5 没有 30B-A3B 这一档，旗舰版本主要走 7B。Hy-MT2 把 30B-A3B MoE 加进谱系，给企业级翻译留出了上方空间。

第三是评测路线变化。Hy-MT1.5 主要对标 NLLB-200 和上一代 Google Translate，腾讯报告中 Hy-MT2 把对位目标抬到 Gemini 3.1 Pro 与 DeepSeek-V4-Pro，并把翻译方向数明确报为 1056。

腾讯官方在仓库 imgs/main_result.png 里给出了一张主结果图（上图），里面包含 IFMTBench 与 FLORES-200 上 Hy-MT2 各档与若干对标系统的相对位置。需要提醒一句：图中数字是腾讯自评，第三方复现要等社区用 IFMTBench 数据集（已开放在 HF datasets/tencent/IFMTBench）跑一遍才能定。

评测口径透明这件事

横评的可信度，最终落在评测口径是否透明。这次腾讯混元有一件做得对的事：把评测数据集 IFMTBench 同步开源到 HuggingFace（datasets/tencent/IFMTBench），让第三方可以拿到同样的数据自跑。

需要保持冷静的几条：

  • 「1.8B 整体超越微软等主流商业 API」是腾讯自报；微软 Translator API 本身不公开权重，第三方复现要在同一组测试样本上做才公平
  • 「30B-A3B 真实场景超过 Gemini 3.1 Pro」中的「真实场景」就是 IFMTBench，第三方复现需要按这份数据集跑
  • 「FLORES-200 上接近 Gemini 3.1 Pro」是公开评测集结论，但具体的 spBLEU / chrF++ 单点数字 README 没列，要等 HY_MT2_0_Report.pdf 详细解读
  • 「7B / 30B-A3B 超 DeepSeek-V4-Pro / Kimi K2.6 fast-thinking」 中的 fast-thinking 是这两款模型的低思考链路设定，对位口径需要注意
  • 33 × 32 = 1056 条主流翻译方向 + 5 民族语言加层这套口径，腾讯自评与新浪、量子位、ModelScope 三家中文媒体口径一致

第三方独立复现可以等几天，但 IFMTBench 已经开放、权重已经全开、推理代码 README 给了完整示例——这套发布是国产翻译大模型这一年开源透明度最高的一次。

国内开发者：五条端侧 / 服务端落地路径

把这套谱系翻译成国内开发者真正能用的落地方式，大致有五条路径。

第一条是出海类 RN / Flutter App。1.8B GGUF + llama.cpp Android binding 是最直接的组合，小红书、抖音、TikTok、SHEIN 这类做出海评论翻译的产品，过去走云端 API 按字符付费，单条评论虽然便宜，量大之后云端账单不可控；现在 440 MB 模型直接打包进 App，离线翻译、零调用成本。粤语、英语、日语、韩语、东南亚六种主流语言之间互译，正好覆盖海外华语圈与东南亚市场。

第二条是企业 Bot / 文档翻译。7B 与 30B-A3B 服务端 vLLM 部署，对接飞书机器人、钉钉文档、企业微信群消息。这一档比云端商业 API 强在数据闭环：跨国公司、出海企业的内部翻译需求过去要么自建小模型质量不够、要么走 DeepL 商用 API 把数据流出境，现在拿到 30B-A3B 开源权重，自己部署在合规云上一次解决。

第三条是国产手机 NPU 直接接入。1.8B 1.25-bit GGUF 适配高通骁龙、联发科天玑、苹果 A 系列三家 NPU。这条路径的对位是 Apple Live Translation：苹果在 iOS 26 上把 AirPods Pro 3 做成了实时口译，但只覆盖十五到十九种语言、闭源；Hy-MT2 在国产安卓阵营里有机会以开源方案补上这块空白，做国产输入法、国产手机相机文字识别、国产 AirPods 替代。

第四条是教育与学习硬件。1.8B 离线 + 词典笔 / 学习机 / 翻译耳机，是网易有道、科大讯飞这一批厂商的传统主场。Hy-MT2 的特殊价值在于民族语言：藏汉、维汉、蒙汉双语教学材料的实时翻译过去几乎没有开源工具可用，1.8B 离线方案让这一块覆盖第一次有了起点。

第五条是跨境电商与外贸客服。7B 或 30B-A3B + 业务垂直微调，对位 DeepL Pro 与 Google Translate 商用 API。SHEIN、Temu、阿里国际站这类平台对翻译质量与稳定性要求高、量级大，按字符付费的云端 API 长期承压，自部署开源大模型是过去一年明确的方向。Hy-MT2 给出了一个起点合理的基线。

接入门槛实测在可控范围。HF 模型卡 README 给了完整的 transformers 推理示例代码，端侧走 llama.cpp + GGUF，服务端走 vLLM 与 LLaMA-Factory 训练支持，全套样例齐备。

端侧翻译时代的元层判断

把这次发布放在更长的时间尺度上看，有三个值得记录的趋势。

第一是**「翻译」这件事正在从云端 SaaS 回到设备本身**。过去十年翻译能力的演化路径是 PC 翻译软件 → 云端在线翻译 → 移动 App 内嵌 → 商业 API → LLM 翻译模型，整体走向是越来越集中在云端。AngelSlim 1.25-bit 这种极致量化技术成熟之后，翻译能力开始走逆向回归：模型本身回到手机、回到设备、回到完全离线的状态。这件事对用户隐私、对企业数据合规、对 C 端体验流畅度都是正向。

第二是国产开源翻译模型这一档第一次有了完整谱系。1.8B 端侧、7B 桌面、30B-A3B 服务端的三档铺位，加上 1.25-bit 极致量化与民族语言覆盖，让国产翻译这件事第一次具备了「从口袋到机房」的完整路径。今后一段时间，阿里通义、字节豆包、百度、智谱这些厂商大概率会在 6-12 个月内跟上类似的三档开源动作，国产端侧翻译生态会快速成型。

第三是民族语言进入主流开源大模型的覆盖范围。粤语、藏语、维吾尔语、蒙古语、哈萨克语五种语言在过去几乎是开源 LLM 翻译模型的盲区，Meta NLLB-200 虽然语种多但实际落地少。Hy-MT2 把这块覆盖明确写进模型卡，让民族地区的教育、政务、文旅场景第一次有了开源可控的端侧工具。这件事的社会价值，比单纯的 BLEU 分数更值得记录。

腾讯混元这次给国内开发者交付的，不只是一个翻译模型，而是一套「翻译能力作为基础设施」的开关。440 MB 装进任意一部手机，33 种主流语种加 5 种民族语言一次配齐，开源权重摆在 HuggingFace，连评测数据集都同步开放——手机离线翻译这件事，第一次有了真正可用的国产开源版本。这是值得记下的一天。
