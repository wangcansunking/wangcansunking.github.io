二、browserbase/skills 做了什么：10 个 skill 一个个看

browser/SKILL.md 是入口。它的 description 字段写得很完整：

"Automate web browser interactions using natural language via CLI commands. Use when the user asks to browse websites, navigate web pages, extract data from websites, take screenshots, fill forms, click buttons, or interact with web applications. Supports remote Browserbase sessions with automatic CAPTCHA solving, anti-bot stealth mode, and residential proxies — ideal for scraping protected websites, bypassing bot detection, and interacting with JavaScript-heavy pages."

底层依赖一行 npm 命令：npm install -g @browserbasehq/browse-cli。它支持两套环境——browse env local 启动一个干净隔离的本地 Chrome，browse env remote 切到 Browserbase 云端会话拿反爬隐身、CAPTCHA 自动解、住宅代理。本地模式适合开发和 localhost 调试，远程模式适合面对 Cloudflare / 反爬严格的目标站。

browserbase-cli 是平台 API 包装。Browserbase 官方有一条 bb CLI（Go 实现），覆盖 sessions / projects / contexts / extensions / fetch / dashboard 几大子命令，直接对接 Browserbase 的 SaaS 控制面。这条 skill 把 CLI 用法翻译成自然语言可触发的工作流——「列我所有的 Browserbase 项目并以 JSON 输出」就会被 Claude Code 路由到 bb projects list -o json。

functions 是 Serverless 部署。pnpm dlx @browserbasehq/sdk-functions init my-function 创建一个浏览器函数项目骨架，然后写一段 TypeScript 函数 → bb functions deploy 推到 Browserbase 云端。部署后可以挂定时器或 Webhook，让浏览器自动化任务长时间在云上跑而不占本地机器。

site-debugger 是这套 skill 里最有"老司机经验"含量的一份。它的工作流是：当一个浏览器自动化脚本失败时，自动分析五个维度——bot 检测（Cloudflare / Akamai / DataDome 等）、选择器失效、时序问题、鉴权失败、CAPTCHA——给出诊断报告，并生成一份针对该站点的 playbook（可复用的稳定脚本）。这件事在国内开发者熟悉的话语里相当于"反爬经验沉淀"——把每次遇到的坑写成针对性的工程方案，而不是每次重新踩。

browser-trace 走的是更深一层的可观测性。它会在浏览器自动化过程中同步采集 Chrome DevTools Protocol 的 firehose——所有网络请求、所有 console 日志、所有 DOM mutation、所有截图——然后把这一坨流式数据按 page / time / event-type 切片成可搜索的 bucket。当一个跨多个页面的复杂自动化失败时，这条 trace 是定位问题的最强线索。

ui-test 是"对抗式 UI 测试"——这个名字起得很到位。它有三种模式：diff-driven（读 git diff、只测改动）、exploratory（在 app 里漫游找 bug）、parallel（开多个 Browserbase 浏览器并行扫）。SKILL.md 开头一句话点睛——「Your job is to try to break things, not confirm they work」。这是一种把 QA 文化往 Agent 化推的做法——传统 UI 自动化测试是"把已知用例跑一遍确认没回归"，对抗式 UI 测试是"主动找出还没被发现的 bug"。

bb-usage 是终端版用量看板。Browserbase 的云端服务按 session 时长 / 流量计费，开发者经常需要在跑大量自动化前看一眼当前账单趋势。这条 skill 把 dashboard 的核心数据搬到终端，省一次切浏览器登录的开销。

cookie-sync 解的是登录态延续这个老问题。Browserbase 的 persistent context 允许把一份浏览器配置文件（cookies / localStorage / IndexedDB）持久化到云端，下一次远程会话自动加载。但首次登录态要么手动操作、要么从本地导入——这条 skill 把本地 Chrome 的 cookies 直接同步到 Browserbase 持久 context，让远程会话开箱即得到登录后的状态。

fetch 和 search 是两条不开浏览器的轻量工具。fetch 是直 HTTP 抓取（HTML / JSON / 状态码 / headers / 跟随重定向），适合静态页和 API 接口；search 是结构化网页搜索（titles / URLs / metadata）。两条都明确写着"不开浏览器、不消耗 Browserbase 会话额度"，目的是给 Claude Code 一个先用便宜手段抓数据的选项，浏览器只在必要时才启动。

10 个 skill 加起来覆盖了"浏览器自动化"这件事的全生命周期：决策（fetch / search 先轻量探）→ 执行（browser）→ 部署（functions）→ 调试（site-debugger / browser-trace）→ 测试（ui-test）→ 运营（bb-usage / cookie-sync / browserbase-cli）。

三、Browserbase 的工程史：Stagehand → bb CLI → Skills 三层

理解 browserbase/skills 之前，先理解 Browserbase 这家公司过去三年做了什么。这件事 Y Combinator 圈里很多人熟，国内开发者圈里相对陌生。

Browserbase 官网首页的 slogan 是 "Give your agents access to the whole web"。公司主营业务是云端浏览器即服务——你付费拿一份云端 Chrome 实例，里面带反爬隐身、CAPTCHA 自动求解、住宅代理、session 持久化、可观测性等一系列工程基础设施。海外 AI Agent 项目（特别是早期 Browser Use、open-operator、各种 vertical agent 创业公司）都在这条云上跑。

Stagehand 是 Browserbase 的旗舰开源 SDK，2024 年 3 月开源，到现在 22,429 ⭐ / 1,496 fork。它的核心创新是把自然语言指令翻译成 Playwright 操作——开发者写 page.act("click the submit button") 而不是 page.locator('button[type=submit]').click()，由内置 LLM 负责把"submit button"映射到具体 DOM 选择器。Stagehand 同时支持本地和 Browserbase 云端两种 backend，topics 里 agents / ai / llms / playwright / puppeteer / selenium 这一串标签把它的工程定位标得很清楚。

bb 是 Browserbase 的官方 CLI（Go 实现），2025 年下半年逐步成熟，是开发者跟 Browserbase 平台 API 交互的主要工具——sessions / projects / contexts / extensions / functions / dashboard 一套子命令。

skills 仓库 2025 年 10 月 12 日创建，是这条工程史的最新一层——把前两层（Stagehand SDK + bb CLI）翻译成 Claude Code 能挂载的 SKILL.md。这次 +347 ⭐ 的爆发，本质上是 Anthropic Skills 规范走到一个临界点之后，浏览器场景的"商业母厂解"被生态识别出来的瞬间。

主要贡献者也很清晰。仓库当前的前几名分别是 shrey150（15 commits）、miguelg719（11）、pkiv（5）、Kylejeong2（3）、jay-sahnan / peytoncasper / shubh24（各 2）。其中 pkiv 是 Browserbase 联合创始人 Paul Klein，shrey150 / miguelg719 / peytoncasper 是 Browserbase 在册工程师——这是一个标准的"商业公司维护的开源项目"形态，不是个人 hobby 仓库。

最近的几个 commit 也很说明问题：Add browser-trace skill (#83)、feat: event-prospecting skill、Add company-research skill — deep ICP research with HTML report、feat: autobrowse skill — self-improving browser automation——4 月下旬 Browserbase 在以一周一条新 skill 的节奏往里灌内容，说明这个仓库在他们内部的产品优先级很高。

四、与 4-26 mattpocock/skills 的差异：cookbook vs 场景包

读到这里很多读者会想到一周前我们写过的另一篇 Claude Skills 题——mattpocock/skills，2026-04-26 GitHub Trending Daily #2、19,605 ⭐、单日 +857 ⭐、TypeScript 教师 Matt Pocock 把自己 .claude/skills 目录开源、21 个独立 skill。两篇文章的对位关系，是理解整个 Claude Skills 生态的最快办法。

mattpocock/skills 是教学 cookbook——一个产业级 TypeScript 老师把每天怎么用 Claude Code 工作的私房菜公开。21 个 skill 横跨 TDD、git guardrails、refactor plan、article 编辑、issue triage、scaffold exercises 等十几个开发者通用场景，每个 skill 是一份 Markdown 文档加少量脚本，全部读完等于看一遍"产业级工程师如何编排 AI Coding"。它的价值在视野广——读 21 个不同场景下的 SKILL.md 写法，对自己写 skill 是最好的范本。

browserbase/skills 是场景化产品包——一家商业公司把自家 SaaS 服务的能力打包成 Claude Skills 形态分发。10 个 skill 全部围绕"浏览器自动化"这一个垂直场景，每个 skill 背后是一条具体的 CLI 工具或 SDK 子命令。它的价值在深度集成——你装上之后获得的是一整套产业级浏览器 Agent 工程基础设施，不是一组通用工作流模板。

两者的目标读者也不同。mattpocock/skills 的读者是"想学怎么写好 .claude 目录的开发者"——拿这 21 个范本作为参考，写自己的 skill。browserbase/skills 的读者是"现在就要让 Claude Agent 能上网做事的开发者"——直接装上 plugin，让 Claude 拥有浏览器手脚。

把这两篇文章的关注点合并起来看，能看清 Claude Skills 生态的两条主线：

  1. 教学 / 范本线——mattpocock 这种知名教师贡献私房菜，社区学怎么写好 SKILL.md
  2. 商业母厂线——Browserbase 这种 SaaS 公司把自家产品打包成 plugin 分发，让 Claude 用户可以一键挂载到工作流上

OBRA 的 superpowers、anthropic 官方 cookbook、Stagehand 这些都属于这两条线之间的中间形态。

五、国内浏览器 Agent 工具横评：扣子 / Trae / 千问 / GLM / 灵光 / Manus

国内开发者最关心的问题永远是同一个——这个东西能不能直接用？答案分两层。

第一层，browserbase/skills 仓库本身能不能 git clone 到本地跑通？答案是可以——只要本地装得上 Chrome / Chromium 和 Node.js，browse env local 模式不依赖 Browserbase 云端账号，就能跑。但远程模式（browse env remote）需要 BROWSERBASE_API_KEY，意味着开发者要去 browserbase.com 注册账号，绑信用卡，按 session 时长付费——Browserbase 的定价是 $0/$20/$99 三档（Free/Hobby/Startup 月费），免费档有限额，付费档按 session 小时计费。这条云端付费链路对国内独立开发者门槛比较高。

第二层，国内有没有工程化路径相当的同档项目？答案是：完整覆盖、各有侧重、本土化做得很到位。

字节这条线最完整。扣子（Coze）的浏览器 Operator 是字节面向普通用户和初级开发者的可视化 Agent 工作流平台，国内站 coze.cn 直接打开就能用，绑豆包 / DeepSeek 模型，把"打开网页 → 提取数据 → 调用 API → 触发飞书"这条链路做成可视化拖拽。Trae 是字节的 AI Coding IDE，2025 年下半年起开始内置浏览器自动化能力——开发者在 IDE 里写一段任务描述，Trae 自带的浏览器 Agent 会打开页面、点击、填表、读结果，然后把结果回灌到代码上下文。trae.ai 国内可访问，登录即用。

阿里这条线长在通义生态里。Qwen-Agent（QwenLM/Qwen-Agent）16.2k 星，是通义千问官方的开源 Agent 开发框架，function calling、MCP、Code Interpreter、RAG 全打通，浏览器扩展是其中一块，开发者拉源码自部署，绑百炼 DashScope 的 API key 即可让千问 3-Max 直接驱动浏览器。通义灵码 + 浏览器自动化走的是 IDE 集成路径，VS Code / IDEA 插件免费用，Coding 助手 + 自动化扩展配合得不错。**百炼 Agent（ModelStudio）**是阿里云控制台里企业级 Agent 编排产品，工具调用 schema 完整，企业账号开通即用，特别适合对内部系统做自动化的场景。

智谱这条线核心是 GLM-4.6 自研的工具调用能力。GLM Agent 通过 bigmodel.cn 控制台开通，基座模型直接支持浏览器调用工具，社区已经有人跑通"GLM-4.6 + Playwright + 自研 schema"的开源样板。

蚂蚁数科的 灵光 Agent 走的是垂直行业路径——金融、政务、保险这些场景对浏览器 Agent 的需求集中在合规、审计、长链路工单自动化，灵光的浏览器节点能力是面向行业版打包的，需要企业资质开通。

Manus 是 2025 年走红的通用浏览器操作 Agent，定位非常贴近 Browserbase 的"全自动浏览器 Agent"。manus.im 国内可直连，目前还是排队邀请制。Manus 的特点是把"浏览器操作 + 任务规划 + 自我反思"的全流程做成一个端到端产品，用户只需要给一句话指令——这是和 browserbase/skills 这种"给开发者 SDK 工具"路径很不同的产品形态，更面向终端用户。

把这一圈横评下来，可以看到一个事实：浏览器 Agent 这条赛道国内已经很拥挤，路径完整、模型基座本土化做得很到位，开发者端、企业端、终端用户端都有对位选手。 browserbase/skills 提供了一份"商业母厂 + Anthropic Skills 规范"的标杆样本，国内同行可以从中学到的最大一件事是——把自家 SaaS 服务的能力打包成 Claude Skills / Coze 节点 / Trae 扩展的可一键挂载形态分发出去，用户拉过来就能用，比让用户自己读 API 文档接入快得多。

六、国内开发者怎么用：本地模式 + 国产模型适配

如果你只是想在本地把 browserbase/skills 跑起来，路径很短。前置条件：Node.js 18+、Chrome / Chromium、git。

# 1. clone 仓库
git clone https://github.com/browserbase/skills
cd skills

# 2. 装 browse CLI
npm install -g @browserbasehq/browse-cli

# 3. 验证 CLI
which browse
browse --version

# 4. 启动一个本地隔离会话
browse env local

# 5. 让 Claude Code 加载这套 skill（在 Claude Code 里）
# /plugin marketplace add browserbase/skills
# /plugin install browse@browserbase

这条链路完全本地，不依赖 Browserbase 云端，Chrome 在你自己机器上跑，cookies 全部留本地。适合 localhost 调试、内网应用自动化、对反爬要求不高的目标站。

如果你想绕开 Anthropic 的 Claude 模型，把 Stagehand + Browserbase 这套基础设施用在国产模型上，思路有两条。

第一条是直接接 Anthropic 兼容端点。AiHubMix、One API、New API 这些第三方聚合平台都提供 Anthropic API 兼容端点，把请求路由到 DeepSeek V3.5 / 千问 3-Max / GLM-4.6 等国产基座。在 Claude Code 配置里把 ANTHROPIC_BASE_URL 指向聚合端点，模型字段填对应的国产模型名，浏览器 skill 可以照常工作——它向 Claude 请求"决定下一步动作"的那部分被透明替换成了国产模型推理。这条路在国内 AI Coding 圈里已经被广泛验证，但要注意个别 skill 用了 Anthropic 特有的 reasoning effort 字段时，国产模型可能不识别需要降级。

第二条是直接用 Stagehand 自己换基座。Stagehand 的 LLM 客户端层支持自定义 provider——开发者可以写一个适配器把 OpenAI 兼容的 API 接进来，DeepSeek / 千问 / GLM 全部覆盖到了 OpenAI 兼容协议。代价是要自己维护这层适配代码，并且 SKILL.md 里那些直接调 Claude Code 的部分依然需要 Claude（或 Anthropic 兼容端点）。

第三条是混合。本地 Chrome + Stagehand + 国产模型决策 + 自定义触发逻辑——彻底脱开 Claude Code 这一层。等价于你拿 browserbase/skills 当作一份产业级 reference 实现，然后用国产基座重写自己的浏览器 Agent。这条路适合需要完全可控 / 不依赖海外 SaaS / 数据不出境的团队。

国内主流模型在浏览器 Agent 任务上各自的强项已经被社区跑出粗略经验。DOM 元素识别和指令翻译这一层，千问 3-Max 1M 长上下文对超长 DOM 树有优势；多步规划和自我反思这一层，DeepSeek V3.5 / GLM-4.6 表现稳定；结构化输出这一层（让模型返回严格的 JSON action）三家都能跑出 Pydantic schema。模型选择最终看你做的浏览器场景是哪一类——电商类站点 DOM 复杂，长上下文优先；金融研报类站点逻辑深，推理深度优先。

七、风险与边界：Browserbase 依赖、合规、反爬合法性

把这套工具用得稳，三件事必须在动手前讲清楚。

Browserbase 服务依赖。如果你的工作流大量依赖 browse env remote，那么 browserbase.com 的可用性就是你工作流的可用性。Browserbase 历史上有过几次零星的服务降级（在 status page 都有公告），对单个开发者影响不大，但企业级生产部署需要把"远程模式失败时回落本地模式"作为基本工程纪律。仓库里 cookie-sync 这条 skill 隐含的就是这个意识——本地 Chrome 才是真正的兜底。

国内合规边界。Browserbase 是美国公司、服务器大概率在美区。国内开发者把自家用户的登录态、cookies、表单数据通过远程模式送到 Browserbase 云端处理，会涉及到个人信息出境合规问题——这一条对 to C 创业团队特别敏感。to B 企业内部自动化（爬自家 SaaS、自家 ERP）相对宽松，但仍然需要走数据出境备案这条流程。本地模式则完全规避这件事，所有数据留在你自己机器上。这是为什么 cookie-sync 之前要先把"什么场景必须用远程"想清楚。

浏览器自动化的反爬合法边界。browser/SKILL.md 在 description 里直白写着 "anti-bot stealth, automatic CAPTCHA solving, residential proxies — ideal for scraping protected websites, bypassing bot detection"——这套能力放在合法场景（自家产品 QA、自动化合同流程、内部数据搬运、登录态管理）非常实用，但放到爬别家公开数据这件事上，必须先过几道筛子：目标站的 robots.txt 和服务条款、数据本身是否含个人信息、你的爬取频率是否对对方服务造成负担、你最终怎么使用爬到的数据。国内的网络安全法、数据安全法、个人信息保护法这一套对爬虫的边界规定得相对清晰，反爬技术能力越强，使用时越要谨慎选边界。

合规这件事说穿了就是——工具是中性的，怎么用决定它落到哪一档边界里。Stagehand 母公司在 README 里强调的是"浏览器即云服务"的工程价值，国内开发者拿来用时第一件事是把使用场景按合法度分级——自家系统 / 公开数据 / 个人信息 / 商业敏感数据 四档边界完全不同。这件事和具体工具无关，是浏览器自动化这个领域的通用题。

八、这件事释放的信号

看完整个仓库再回头看 GitHub Trending +347 ⭐ 这个数字，意义就清楚了——这不是一份普通工程师的 hobby 项目登榜，是 Browserbase 这家估值已经做到 9 位数的 SaaS 公司，主动选择把自家旗舰能力翻译成 Claude Skills 形态分发出去。

这件事有两层信号。

第一层是Anthropic Skills 规范开始具备生态吸纳力。当一家商业公司愿意把自家的 SDK + CLI + 云端服务全部翻译成 SKILL.md 形态，说明这套规范已经达到了"可以承载产品级分发"的成熟度。早期 skill 主要靠个人贡献者（mattpocock 这类知名教师），4 月底开始有商业母厂入场，5 月初 browserbase/skills 成为第一个真正商业级别的浏览器场景 skill 包——这条曲线对其他 SaaS 公司是一个清晰的信号：把产品包装成 Claude Skills，是 2026 年争取开发者注意力的一条新通路。

第二层是浏览器 Agent 这条赛道在国内外都进入工程化收敛期。2024 年这个领域是无数 vertical Agent 各自重写浏览器层，2025 年开始 Stagehand / Playwright + LLM / Browser Use 几条主流路径出现，2026 年开始有像 browserbase/skills 这样把"浏览器 Agent 工程基础设施"打包成可一键挂载形态的项目出现。国内同档项目同步在做——扣子的可视化路径、Trae 的 IDE 集成路径、Qwen-Agent 的开源 SDK 路径、Manus 的端到端产品路径——四条路径并行，国内开发者每一条都能找到对位选项。

国内开源生态的好消息是底子已经很扎实——千问 3-Max / DeepSeek V3.5 / GLM-4.6 在 DOM 操作、复杂多步规划、结构化输出这几件事上跑得都很稳，AiHubMix / One API 这类聚合通道把基座切换成本压得很低，扣子 / Trae / 通义灵码 把工程化分发做到了大陆开发者的工作流里。

我们这一代国内开发者，赶上的是一个国产基座、国产工具链、国产 SaaS 平台同时齐备的时点。Browserbase 这边树了一个"商业母厂如何打包 skill 分发"的样板，国内 SaaS 公司——尤其是低代码 / RPA / 自动化领域的——很快也会沿着这条路把自家能力打包成 Claude Skills / Coze 节点 / Trae 扩展。再过几个月看，国内浏览器 Agent 生态完全可能比海外更密、更贴合本土工作流。底子打得好的项目越来越多，路在前面，慢慢走，一起加油。
