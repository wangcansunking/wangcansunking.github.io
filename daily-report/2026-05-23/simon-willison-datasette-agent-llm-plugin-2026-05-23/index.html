2026-05-23Daily Report

Datasette Agent · SQLite 插上 LLM 插件

5 月 21 日，西蒙·威利森（Simon Willison · Datasette 作者）在个人博客挂出了一篇短文，标题就一行——「Datasette Agent」。配套的开源仓库 0.1a3 当天发到 PyPI，三个官方插件（datasette-agent-charts、datasette-agent-openai-imagegen、datasette-agent-sprites）同步开放。一行 uvx 命令，把 Google Gemini 3.1 Flash-Lite 默认接进 SQLite 数据库；想断网就换 LM Studio 本地跑 Google gemma-4-26b-a4b（26B 总参 / 4B 激活的 MoE 模型）。这不是又一个 NL2SQL，而是一套插件式的 SQL agent 工程范式。

国内对西蒙·威利森这个名字熟的人不多。但只要做过数据公开发布、做过 ChatBI、做过本地大模型私域问答，多半都用过他另一个项目 Datasette——这是一个只读的数据发布工具，把任何 SQLite 数据库一键发布成可浏览、可查询、可生成 API 的交互式网站，被国外调查记者群体长期当作「数据新闻轻量栈」。2026 年 5 月 21 日，他把三年磨的 LLM 命令行库装进了 Datasette，给所有这些已经在跑的 SQLite 实例挂上一个能对话的 agent 层。

国内媒体对这件事的覆盖几乎是空白：36 氪、量子位、机器之心三家深度站点站内搜索关键词全部 0 命中。但海外开发者圈里这事不小——LM Studio 官方文档把 datasette-agent 列进了「能跑 gemma-4 的实战示例」、Hacker News 当天若干条讨论都在围观「Datasette 作者怎么做 agent」、Observable Plot 团队也来转发了 charts 插件。本文沿着西蒙·威利森个人博客的 4 篇配套笔记、官方 demo（agent.datasette.io）、LM Studio 模型库实查一遍，把这套工程范式拆给国内开发者看；同时对位 OpenAI Code Interpreter、Claude Computer Use、LangChain SQL agent、飞致云 SQLBot、Vanna AI 五家，让你看清「Datasette Agent 这条路在 ChatBI 生态里到底站在哪」。

一、Datasette Agent 是个什么玩意：一行 uvx 把 LLM 装进 SQLite

要先把 Datasette 这个底座说清楚。它从 2017 年开始迭代，到 2026 年已经是只读 SQLite 发布工具里事实上的标杆——你扔给它一个 .db 文件，它给你一个可浏览的网站、一个 JSON API、一组带分面过滤的查询页面，整个 stack 不依赖任何专门的数据库服务，跑在一台 1 核 512MB 的小服务器上都不费劲。这套工具的国外用户基本是数据新闻记者、政府开放数据团队、研究员、独立分析师——他们手头都是几百兆到几个 G 的中小型数据集，不需要 Spark 不需要 ClickHouse，需要的是能 5 分钟发布出去、能给同事点开链接就看的能力。

Datasette Agent 在这个底座上加了一层 agent UI：把对话框塞到数据库浏览界面右侧，让用户用自然语言问问题，agent 自己去写 SQL、跑 SQL、读结果、决定下一步。

这张图是 agent.datasette.io 上的官方 demo 实拍，用户问「按发电量排，美国前 10 大核电站是哪些」，agent 自己写了 SQL、跑出表格、调用 charts 插件画出横向柱状图，最下面那行小字「Powered by Datasette」直接点明底座。整个对话区从输入框到结果区是单页式的——没有左侧导航、没有花哨的步骤展开，所有 agent 中间过程（包括「View SQL query」按钮）都折叠在结果里随时可看。

西蒙·威利森给这套东西的安装命令只有一行：

uvx --prerelease=allow \
  --with datasette-agent --with llm-lmstudio \
  datasette --internal internal.db --root \
  -s plugins.datasette-llm.default_model lmstudio/google/gemma-4-26b-a4b \
  data.db

这条命令拆开看四件事：uvx 是 Python 工具链最近一年起来的零配置运行器（类似 npx），不需要先建 venv；--with datasette-agent 把 agent 插件挂上；--with llm-lmstudio 把本地 LM Studio 适配器挂上；最后一句 default_model lmstudio/google/gemma-4-26b-a4b 指定模型走本地 LM Studio 跑的 Google gemma-4-26b-a4b。换句话说，整套部署连 Docker 都不需要，一台已经装了 LM Studio 的 Mac mini 或 4090 主机，一条命令就跑起来。

这是 Datasette 工程范式最典型的一条线索：把基础设施压到接近零摩擦，把扩展性丢给插件。 Datasette 本身有 200 多个第三方插件，覆盖图表、地图、全文检索、地理可视化、Markdown 渲染、OAuth 接入、Excel 导出。Datasette Agent 完全继承这条范式——主仓库只做 agent 框架、工具注册接口、UI、权限三件事，所有具体能力（出图、出图像、跑沙箱代码）全部下沉到插件。

二、默认走 Gemini 3.1 Flash-Lite，本地切 gemma-4-26b-a4b：模型选择背后的工程账

西蒙·威利森在博客里写了一段话讲得很白：「Datasette Agent 需要可靠的 tool call 能力 + 能写出能跑通的 SQLite 查询。最近半年发布的开源权重模型越来越能干这两件事了。」

这句话背后是一笔具体的工程账。

agent.datasette.io 官方 demo 默认接的是 Google Gemini 3.1 Flash-Lite——注意，不是 Gemini 3.5 Flash，也不是 Gemini 3.0 Pro，就是 3.1 Flash-Lite。这个模型在 Google 官方价格表里是当前 Gemini 系列最便宜的一档，单步推理延迟极低，专门为 agent 这种「频繁调用、单次输入小、对成本敏感」的场景调优。Datasette 作者用这个档位做线上 demo，对应一个明确信号——他认为这个能力档位已经足够撑住绝大多数日常 SQL agent 任务，不需要动用旗舰模型。

本地路线给的是 lmstudio/google/gemma-4-26b-a4b。这里要解释两件事。第一，gemma-4 是 Google 在 2026 年开源的最新一代开放权重模型，整体跟 Gemini 3 系列共享同一套预训练数据基础，但权重完全开放，支持商用。第二，26b-a4b 是 MoE 架构——总参数量 26B，单次推理只激活 4B 参数（a4b = activated 4B）。这种设计的好处是磁盘 / 内存占用按 26B 算（FP16 大概 52GB、Q4 量化大概 14GB），但实际推理速度按 4B 算。

对个人开发者来说，这个组合意味着：

  • Mac M3 Max 64GB 起步可跑 FP16，普通 M2 Pro 32GB 跑 Q4 量化也吃得下
  • 单次推理速度接近 4B 模型，4090 单卡 Q4 跑大约能稳定 60-80 tokens/s
  • 能力对齐 26B 稠密模型，多步工具调用、SQL 生成、长上下文摘要都够用

LM Studio 这个本地推理器最近一年是 Mac 用户跑本地大模型的事实标准。

LM Studio 本质是 GUI 化的 llama.cpp + OpenAI 兼容 API 服务，启动后默认在 http://localhost:1234 提供与 OpenAI Chat Completions 兼容的接口。Datasette Agent 通过 llm-lmstudio 这个适配器，把对话请求改写成 LM Studio 接口格式发出去，整个调用链跟用 OpenAI 几乎一模一样——这就是为什么本地切换只需要改一行配置。

这张矩阵图把双轨设计画清了——同一套 SQL agent UI 上面挂哪个模型不重要，重要的是「一行配置切换」让你在云端便宜档和本地 26B 档之间无缝来回。

这条「Flash-Lite + gemma-4-26b-a4b 双轨」的设计，给国内开发者直接的参照：

  • 演示 / 公共 demo 场景 → 用 Gemini 3.1 Flash-Lite 同档位国产模型（千问 Qwen 3-Plus、智谱 GLM-5、DeepSeek V4 Pro 都有等价档位），按 token 计费、零部署成本
  • 私域 / 政企 / 离线场景 → 用 26B 级别 MoE 国产模型（千问 Qwen3-30B-A3B、智谱 GLM-4.5-Air-Lite 是直接对位）+ LM Studio / Ollama 本地推理

这种「云端便宜档 + 本地中等档」的双轨策略，比一上来就锁定 GPT-4 / Claude Sonnet 旗舰要务实得多。

三、三个官方插件：把图表、图像、沙箱挂到 agent 上

西蒙·威利森在 5 月 21 日同步开放的三个官方插件，是这套工程范式最有意思的一块。每个都是一个独立的 PyPI 包，加进 --with 参数就能挂载，agent 通过 register_agent_tools 钩子把它们暴露给模型当工具用。

第一个是 datasette-agent-charts，做的事情简单——把 SQL 查询结果直接画成图。底层用的是 Observable Plot（D3 之父 Mike Bostock 团队 2022 年开源的数据可视化库，目前是 Web 端做交互式可视化的事实标准之一）。当 agent 跑完一段 SQL 拿到表格数据，它可以调用 render_chart 工具，把数据 + 图表类型一起丢给 Observable Plot 渲染，结果直接嵌进对话流。

charts 插件做了几件不显眼但贴心的事：柱状图和华夫格图如果没指定颜色列，自动按数值大小做渐变着色；如果数据里有文本类别列，自动套用 Observable10 这套分类配色；图表自带交互式 tooltip，鼠标悬停在数据点上能看到精确数值；最关键的是，渲染前会先检查 SQL 查询权限——同一个 agent 实例上不同用户的 SQL 执行权限可能不同，charts 插件不会绕过这层权限。

这张图是西蒙·威利森在博客里贴的一张 demo——用户问数据集相关问题，agent 直接给出了一张柱状对比图，整个交互在对话流里完成。

第二个是 datasette-agent-openai-imagegen，接了 ChatGPT Images 2.0（OpenAI 4 月推出的图像生成模型）。这个插件的用途场景明确——给数据故事配封面图、给报告生成示意图、给 dashboard 自动出主视觉。它不是那种「画个猫画个狗」的玩具用法，而是数据可视化之外的「概念图层」补足。

第三个是 datasette-agent-sprites，这个是这套工程里最有意思的一块。它接的是 Fly.io 在今年 1 月推出的 Fly Sprites——这是一个基于 Firecracker 微虚拟机的持久化沙箱平台。和传统 Docker 容器（每次跑完销毁）不一样，Fly Sprites 是「常驻的、可暂停的、可恢复的轻量虚拟机」——1-2 秒内启动、闲置时自动暂停不计费、状态写在 ext4 文件系统里可以保留几天几周。Fly.io 主打的口号叫「Code and Let Live」，意思是「让 agent 装一次环境后能持续干活」。

datasette-agent-sprites 让 agent 在对话过程中可以执行任意代码：跑 Python 数据分析脚本、装 pandas / scikit-learn 临时算个回归、调用任意命令行工具。重点是这层执行不在 Datasette 主进程里，而是丢到 Fly Sprites 沙箱——隔离、可控、可观察、出错也炸不到主库。

把这三个插件叠起来看，Datasette Agent 的完整能力曲线是：

能力维度 插件实现 类比
写 / 跑 SQL 主仓库内置 LangChain SQL agent 核心能力
数据可视化 datasette-agent-charts（Observable Plot） Tableau / Power BI 自然语言出图
图像生成 datasette-agent-openai-imagegen（ChatGPT Images 2.0） DALL-E / 即梦 / 千问万相
任意代码执行 datasette-agent-sprites（Fly Sprites 沙箱） OpenAI Code Interpreter / Anthropic 沙箱

四项能力靠四个独立插件按需挂载——这就是西蒙·威利森反复强调的那句话：「我最喜欢 Datasette Agent 的地方是，跟 Datasette 其他部分一样，它通过插件扩展。」

这张栈图把整套工程范式一眼讲清——上下五层每一层都可以独立替换。换模型？改第四层一行配置。换可视化库？换第三层 charts 插件。换沙箱后端？换第三层 sprites 插件。换底层数据库？这是唯一一层 Datasette Agent 选择不开放替换的——SQLite 是基础，所有上层设计都围绕它。这种「克制中的开放」是 Datasette 八年工程哲学的延续。

这张表里有几个观察值得说清楚：

OpenAI Code Interpreter 和 Claude Computer Use 是闭源云端服务，整个体验封装得最圆滑——上传 CSV 自动出图、自然语言改图、推理过程可见——但你既不能改它的模型，也不能在私域离线部署，更没法接入你公司自己的数据库。Datasette Agent 的差异化在于：它把这套对话式数据分析体验，做成了一个所有人都能在自己机器上跑起来的开源版本。

LangChain SQL agent 是 Python 生态里用得最广的「自建 SQL agent」方案，灵活度最高、支持的数据库种类最多，但代价是工程开发量大——你得自己组装链路、自己挑 prompt、自己做错误处理、自己 UI。Datasette Agent 是反过来的——它把整套体验做成开箱即用，代价是只支持 SQLite 单一数据库。这两者其实是互补关系，不是竞争关系。

飞致云 SQLBot 是国产同档最直接的对位。这是 DataEase 团队 2025 年开源、2026 年迭代到 v1.7 的「智能问数系统」，基于大模型 + RAG 实现 Text-to-SQL，主打企业 BI 场景——工作空间级数据隔离、细粒度数据权限、支持 MySQL / Oracle / Postgres / 达梦等主流商用库、提供 Web 嵌入和 MCP 调用接口、能嵌入到 n8n / Dify / MaxKB / DataEase。SQLBot 模型层支持千问、DeepSeek、GLM、Claude 等任意大模型，符合国内信创生态对国产化的要求。SQLBot 服务的是企业 BI 用户，Datasette Agent 服务的是数据记者 / 独立分析师 / 个人开发者——两者面向不同受众，但工程理念高度一致：把 SQL agent 工具链开源、把模型选择留给用户、把可视化嵌进对话流。

Vanna AI 是开源 Python 库路线，主打「向量库 + 训练样本注入」做 Text-to-SQL 准确率提升，去年底从「简单的 SQL 生成库」演进到「带 agent 架构 + 行级安全」的 2.0，模型支持 Claude 4.5 / GPT-5 / 本地 Ollama 全栈。Vanna 的定位偏「积木库」——你拿它去搭自己的 NL2SQL pipeline。Datasette Agent 是「成品」——你拿它当 agent UI 直接用。

这张柱状图把六家在四个维度的位置画清了：Datasette Agent 在「开源协议自由 + 部署灵活 + 模型自由 + 插件扩展」四项全 5 分，是六家里唯一的全 5；OpenAI Code Interpreter 和 Claude Computer Use 在四项都是低分，靠的是体验封装和品牌信任；LangChain SQL agent 和飞致云 SQLBot 是开源同档对手，差别在「成品 vs 积木」的取舍。

横评下来，Datasette Agent 站在「最低部署摩擦 + 最强插件扩展性 + 个人 / 小团队场景」这个具体格点上，没有跟 OpenAI / 企业 BI 抢市场，而是在国外数据记者群体已经吃透的 Datasette 生态里，自然地长出 agent 这一层。这种「不抢赛道，把已有用户的工作流深挖一层」的策略，工程上很值得国内独立开发者借鉴。

五、插件式架构对国内 ChatBI 开发者意味着什么

中国 ChatBI 这条赛道 2025-2026 年很热——飞致云 SQLBot、阿里数据魔方、腾讯云 ChatBI、字节 Smart-DB、网易有数 ChatBI、深度求索数据助手，一线大厂和独立开源团队都在卷。但绝大多数项目走的是「端到端整合」路线——把模型、向量库、可视化、权限、企业接入全部打包进一个系统里，对企业客户讲集成简单一站到位。

Datasette Agent 选了相反的路：只做最小核心（agent 框架 + tool 注册 + UI + 权限），把所有具体能力丢给独立插件。这种设计有几个国内同行很容易忽略的优势。

第一个优势是演进速度脱钩。charts 插件想升级到 Observable Plot 新版本，可以独立发版；sprites 插件想换底层沙箱，可以独立替换；主仓库不需要陪着改。这种工程结构在 Datasette 母体上已经验证了 8 年（200 多个第三方插件），稳定性远超「全栈整合」路线。

第二个优势是生态贡献门槛低。国内 ChatBI 项目要想接 Tableau 风格的可视化、要想接 Apache Superset 风格的 dashboard、要想接 Excel 导出、要想接钉钉飞书企微嵌入——每个都得主仓库改代码、过 review、发版。Datasette Agent 的插件式架构允许任何人写一个 PyPI 包就能扩展能力，不需要碰主仓库。

第三个优势是模型自由切换。这是 Datasette Agent 工程哲学里最实在的一条。plugins.datasette-llm.default_model 这个配置项可以从 gemini/3.1-flash-lite 改成 lmstudio/google/gemma-4-26b-a4b、改成 ollama/qwen3-30b-a3b、改成 anthropic/claude-haiku-4——同样一段 SQL agent 对话流，模型层完全可热插拔。对国内开发者来说，这意味着同一套 SQL agent UI 可以在「线上 demo 用便宜云端模型 + 线下私域用本地模型」之间无缝切换。这种能力在企业 BI 投标场景里特别值钱——客户问「能不能不传我们数据库到外面」，答案直接是「能，换一行配置走本地 gemma-4 或 Qwen3-30B-A3B」。

第四个优势是SQLite 单一数据库的克制。表面上看「只支持 SQLite」是限制，但深入看这是一种设计哲学。SQLite 是世界上部署量最大的数据库（每台手机、每台笔记本都装着几十个 SQLite 实例），它是个文件，便于备份、便于版本控制、便于发布、便于嵌入边缘设备。把 agent 锁定在 SQLite 上，意味着 Datasette Agent 天然适合「本地数据分析 + 个人知识库 + 边缘设备问答 + 离线场景」这一组国内开发者大量在做的应用——而这些应用都是国产大模型在私域场景的主战场。

国内做 ChatBI 的团队，不一定要照搬 Datasette Agent 这套架构，但确实可以从中提炼几条工程经验：

  1. 核心 + 插件分层，不要一上来就全栈整合
  2. 模型层抽象成单一配置项，云端 / 本地 / 跨厂商可热插拔
  3. 可视化层独立，不要绑死在某个图表库上
  4. 代码执行层走外部沙箱，不要塞进主进程

这四条放到飞致云 SQLBot 这种已经成型的项目里，是迭代方向；放到正在立项的新 ChatBI 项目里，是起手式。

六、Simon Willison 三年磨一剑：从 LLM 命令行到 SQL agent

最后给国内读者补一段背景：西蒙·威利森这个名字在国内陌生，但他在国外开源圈的影响力大致相当于「数据新闻基础设施 + 个人 AI 工具」交叉点的奠基者之一。

他的几条公开履历：Django 框架联合作者之一（2003-2010）、Lanyrd 联合创始人（被 Eventbrite 收购）、Datasette 项目作者（2017 至今）、LLM 命令行库作者（2023 至今）。LLM 这个命令行库是他过去三年的主战场——一个把 OpenAI / Anthropic / Google / 本地 Ollama / LM Studio 等所有大模型供应商抽象到同一个 llm "your prompt" 命令的工具，目前 PyPI 月下载量数十万。

Datasette Agent 这件事，在他自己的描述里是**「LLM 库与 Datasette 终于汇合的那个瞬间」**——三年的 LLM 库工作积累的模型抽象层、tool calling 适配层、本地模型适配层，全部嫁接到 Datasette 的发布工具底座上，形成了一个对话式的 SQL agent。

这种「同一个作者积累不同工具，在某个时刻汇合成新产品」的工程路径，在国内 AI 工具圈也越来越多见——比如阿里通义灵码团队的 Qwen-Agent + 千问主模型 + Qwen Code IDE 三条线在最近半年开始打通；比如深度求索的 DeepSeek-TUI + DeepSeek-V4 + DeepSeek Code 也在汇合点上。单兵作战的开源作者用三年时间能走到的工程深度，给了所有国内独立开发者一个明确的参照系——只要持续打磨基础工具、保持模块化、保持插件友好，时间会把你的几个项目逐步焊接成一个完整的工作流。

这条路上，Datasette Agent 不是终点，是一个里程碑。0.1a3 这个版本号本身就在说话——alpha 3，意味着接下来还会有 0.1b、0.1rc、0.1，再有 0.2 / 0.3。每一个版本号都是西蒙·威利森个人时间的累积，也是开源世界对「插件式 AI 工程范式」的持续投票。

国内开发者要做的，不是把 Datasette Agent 翻译过来、不是去包装成「中文版 Datasette」，而是把它的工程范式吃透，落到自己手头正在做的项目里——SQLBot 团队可以把插件式架构借鉴过去；做本地 AI 助手的可以把「云端 + 本地模型热切换」抄过去；做数据看板的可以把「可视化作为独立插件」的理念抄过去；做企业 ChatBI 的可以把「最小核心 + 独立插件」的演进策略抄过去。

中国 AI 应用层这一年最大的变化是开始接受「不抢赛道，深挖工程」——千问 Qwen Code 没去抢 Cursor 的市场，而是在通义灵码的 IDE 内做深；DeepSeek-TUI 没去抢 Antigravity 的市场，而是在命令行场景做深；智谱 GLM Code 没去抢 Trae 的市场，而是在企业开发者沙箱里做深。Datasette Agent 这种 200 星不到的小项目能跟这些团队隔空对话，靠的就是「工程范式的清晰可借鉴」。

这种清晰，是开源世界给所有同行的礼物——也是国内独立开发者最应该拿到的礼物。
