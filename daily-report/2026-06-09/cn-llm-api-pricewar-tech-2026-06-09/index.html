DeepSeek 把输入价压到每百万 token 1 元：国产模型降价背后的技术账

DeepSeek 的 V4-Flash 现在输入每百万 token 1 元、输出 2 元，缓存命中的输入价压到 0.2 元；通义千问的 Qwen-Long 输入每百万 token 0.5 元；MiniMax M2 的输入折成人民币不到两毛。这些数字摆在一起，第一反应往往是"又在烧钱抢市场"。

但价格战这个词，容易把人引到错的方向。补贴确实有，免费额度也确实在发，可如果只是补贴，价格降下去厂商每一次调用都在多亏钱，这种打法撑不过两个季度。事实是国内厂商这半年六次下调 API 价格，其中三次明确写成"永久降价"——永久两个字意味着，新的低价是在新的成本线之上重新标的，不是亏本清仓。

这篇文章想回答的是一个技术问题：国产模型凭什么能把单 token 的成本真的打下来。 答案不在融资额，不在 CEO 的发布会口号，而在四笔可以逐条核对的技术账：MoE 稀疏激活让每次调用只算一小部分参数，稀疏注意力让长上下文不再按平方膨胀，推理引擎把同一张显卡的吞吐拉高一个数量级，国产算力让每张卡的采购和折旧成本往下走。把这四笔账算清楚，再回到"自建还是调 API、按量还是包月、长上下文和高频 agent 调用怎么选"的选型问题上，判断就有了依据。

价格口径说明：本文所有具体单价均截至 2026 年 6 月初，标注了输入价、输出价、缓存命中价的区别，以及按百万 token 计的口径。人民币价取自厂商官方定价页与权威报道，美元价取自第三方按量计价聚合平台（多家供应商最低价口径），两者因汇率和供应商差异不完全对应，正文会注明来源。模型迭代快，读到本文时请以官方定价页当前数字为准。

谈成本之前，先确认价格本身是真的。下面这张表把当前主力模型的输入、输出、缓存命中价列在一起，所有数字都标了来源口径。

几个需要先讲明白的点，否则容易被表面数字误导：

  • 输入和输出不是一个价。 几乎所有模型的输出价都明显高于输入价，常见是两到四倍。原因在推理机制里——输入 token 可以一次性并行处理（预填充阶段），输出 token 要一个接一个生成（解码阶段），后者更吃显卡时间。所以一个对话里输入长、输出短，和输入短、输出长，实际花的钱差很多。
  • 缓存命中价是这一轮降价里最狠的一档。 DeepSeek 把缓存命中的输入价压到未命中的 1/5（V4-Flash 是 0.2 元对 1 元），Kimi K2.6 的缓存命中保底价折成美元约 0.07，是全场最低之一。缓存命中指的是这次请求的开头部分（通常是系统提示词、固定的工具说明）和上一次请求重复，服务端不必重新计算，直接复用算好的结果。对 agent 这种每次都带一长串固定提示词的场景，这一档价是真金白银的差别，下文会专门算。
  • 不是所有模型都在降。 同一时间，月之暗面的 Kimi K2.6 把 API 定价相对上一代上调了约 58%。一个旗舰模型敢逆势涨价，恰恰说明价格不是被补贴绑死的——它反映的是这个模型在能力、调用成本、目标客群上的综合定位。这条反例很重要：它证明价格战的底层不是谁补贴得起，而是谁的成本结构和产品定位允许它定在哪个点。

把价格确认清楚之后，真正的问题来了：DeepSeek V4-Flash 输入每百万 token 1 元这个价，背后的单位成本到底由什么决定？接下来四节，一笔一笔算。

第一笔账：MoE 稀疏激活，每次调用只算 5% 的参数

这是国产模型成本能往下走的第一块，也是最容易理解的一块。

传统的稠密模型（dense model），参数有多大，每生成一个 token 就要把全部参数都算一遍。一个 700 亿参数的稠密模型，每个 token 的计算量就实打实压在这 700 亿上。MoE（Mixture of Experts，混合专家）换了一个思路：把参数拆成很多个"专家"小组，每次只激活其中跟当前 token 最相关的几组，剩下的不参与计算。

具体到数字。DeepSeek-V3 总参数 6710 亿，但每个 token 实际只激活约 370 亿，激活比例约 5.5%。也就是说，它对外是一个 6710 亿的大模型该有的能力，对内每次调用只付约 370 亿参数的计算账。到了 V4 这一代，总参数升到约 1.6 万亿，单 token 激活约 490 亿——总参数翻了一倍多，激活参数只增加了三分之一，稀疏度进一步拉大。

模型 总参数 单 token 激活参数 激活比例
DeepSeek-V3 6710 亿 约 370 亿 约 5.5%
DeepSeek-V4 约 1.6 万亿 约 490 亿 约 3%
同等能力稠密模型（参照） 全部参数都算 等于总参数 100%

这笔账的意义是：MoE 让"模型有多强"和"每次调用多贵"这两件事解耦了。 过去想要更强的模型，就得接受更高的单次推理成本，因为参数都得算。MoE 之后，厂商可以把总参数堆得很大来换能力，同时把激活参数控制得很小来压成本。一个 1.6 万亿的模型，单次调用的计算量只相当于一个约 490 亿的稠密模型——这中间省下的算力，直接变成了可以让利给用户的降价空间。

国内几家主力厂商现在普遍走 MoE 路线，这不是巧合。它是把单 token 计算成本真正压低的第一个、也是占比最大的一个技术原因。值得补一句：MoE 的工程难点在"负载均衡"——怎么保证 token 被均匀分到各个专家上，不让某几个专家过载、另一些闲置。DeepSeek-V3 用了一套不靠辅助损失函数的负载均衡策略来解决这个问题，这也是它能把 MoE 训练成本压到约 558 万美元（2.788M H800 GPU 小时）的关键之一。

第二笔账：稀疏注意力，长上下文不再按平方涨

如果说 MoE 解决的是"参数算多少"，那稀疏注意力解决的是"上下文长了之后怎么不被拖垮"。这是这一轮长上下文、agent 高频调用场景里最关键的一笔账。

标准注意力（attention）机制有个老问题：计算量随上下文长度的平方增长。上下文从 1 万 token 涨到 10 万 token，长度涨 10 倍，注意力的计算量涨约 100 倍。这就是为什么长上下文一直又慢又贵——不是线性变贵，是平方级变贵。Agent 场景偏偏最吃长上下文：要把工具说明、历史对话、检索到的资料全塞进去，动辄几万几十万 token。

稀疏注意力的核心思路是：不让每个 token 都去和前面所有 token 算关系，而是只挑真正相关的那一小部分算。国内两家的做法很有代表性，而且都有公开的技术报告可以核对。

DeepSeek 的原生稀疏注意力（NSA，Native Sparse Attention）。 它把注意力拆成三条支路并行：压缩（把一段 token 压成粗粒度的摘要）、选择（在细粒度上挑出最相关的若干块）、滑动窗口（保住近处的局部细节）。三条路同时看全局和局部。公开技术报告里的实测数据：在 64k 长度的序列上，前向计算最高加速约 9 倍，反向计算约 6 倍，解码阶段随序列变长最高加速约 11.6 倍。关键是 NSA 是"原生可训练"的——稀疏不是推理时硬截的近似，而是训练阶段就这么学的，所以在长上下文检索这类任务上不掉精度，甚至略好于全量注意力。

到了 V4，DeepSeek 进一步用了压缩稀疏注意力（CSA），把每 4 个 token 的键值缓存（KV cache）合并成摘要，再在摘要上挑最相关的若干条算注意力。效果是：在 100 万 token 的超长上下文里，V4-Pro 的计算量只有上一代 V3.2 的约 27%，键值缓存占用只有约 10%。换句话说，同样跑一个百万级长上下文请求，V4 烧的算力是上一代的约 1/4，占的显存是约 1/10。

MiniMax 的稀疏注意力（MSA）。 这条线的故事更曲折，也更说明问题。MiniMax 在 M2 这一代试过把线性注意力、滑动窗口这类次平方结构混进来，但发现一上规模，这些变体在推理任务上会严重掉链子，所以 M2 退回了标准全量注意力来保住能力。直到 M3，他们才拿出 MSA——在标准 GQA 结构上做块级选择，而且是在真实、未压缩的键值上选。早期硬件实测：长上下文下预填充阶段加速约 9.7 倍，解码阶段约 15.6 倍。

这条弯路恰恰是技术账真实的证据：稀疏注意力不是免费的午餐，做不好会掉精度，几家是真在工程上反复试出来的。 它带来的降价空间也是实打实的——长上下文的计算量和显存占用同时往下压，长文档、长对话、agent 这些重上下文场景的服务成本才有可能跟着降。

把前两笔账放一起看：MoE 省的是"每个 token 要算多少参数"，稀疏注意力省的是"上下文一长，注意力要算多少关系"。这两笔都是模型架构层面的省，省的是浮点计算量和显存占用本身。接下来第三笔，省的是另一个维度——同一张显卡，能不能多干活。

第三笔账：推理引擎，让一张显卡的吞吐翻一个数量级

模型架构省下的是单次调用的计算量，推理引擎省下的是显卡的闲置。这笔账普通用户看不见，但它实实在在决定了每张卡每天能服务多少请求，也就直接决定了摊到每个 token 上的硬件成本。

最有代表性的是 PagedAttention 这套机制。它解决的是显存碎片问题：键值缓存过去要占一整块连续显存，请求长度不一就会留下大量用不上的碎片，显存利用率常年卡在约 60%。PagedAttention 借了操作系统的分页思路，把键值缓存切成固定大小的小页、按需分配，显存利用率拉到 95% 以上。利用率从 60% 到 95% 是什么概念——同样一张显卡，能同时塞下的并发请求数量大幅增加，单卡吞吐随之上去。

配套的还有几项：

  • 连续批处理（continuous batching）：不等一批请求全部跑完再开下一批，而是谁先生成完谁就让位、新请求随时补进来，显卡不空转。这一项加上 PagedAttention，让推理吞吐相比早期实现有了数量级的提升。
  • 前缀缓存（prefix cache）：上一节价格表里那个"缓存命中价"，技术上就靠它。SGLang 这类引擎用基数树（RadixAttention）来管理键值缓存，自动识别不同请求之间共享的前缀（比如同一套系统提示词），复用已经算好的结果，缓存命中率能提到原来的三到五倍。系统提示词、固定工具说明这些每次都重复的部分，复用之后既省算力又省钱，缓存命中价才有底气定到未命中的 1/5。
  • 量化与投机解码：量化把模型权重从高精度压到低精度（比如 FP8、INT8），显存占用和计算量一起降；投机解码用一个小模型先猜几个 token，大模型一次性校验，减少大模型的串行步数。

这套引擎能力，国内厂商要么用开源的成熟方案（这类推理引擎已在数十万张显卡的规模上每天处理万亿级 token），要么在此基础上自研定制。它的意义在于：架构省下的算力，要靠引擎才能真正变成"每张卡多服务的请求数"。 模型再稀疏，引擎跟不上，显卡照样空转，成本照样降不下来。这笔账是把前两笔的理论节省，落到每天每张卡实际吞吐上的那一环。

第四笔账：国产算力，把每张卡的采购与折旧往下压

前三笔账都是软件层面的省——少算参数、少算注意力、少让显卡闲置。第四笔账动的是硬件本身：显卡从哪来、多少钱一张、能不能稳定供应。

这一年国产算力的进展，是这一轮降价里被低估的一块。华为昇腾 950PR 推理芯片 2026 年 3 月完成量产，FP4 精度算力达 1.56P，单卡算力相比英伟达 H20 提升约 2.87 倍，HBM 显存 112GB。DeepSeek V4 这一代已经做到能在昇腾平台上跑推理——它支持 FP4 精度，正好对上昇腾 950 的原生能力；公开报道里，由 8192 张昇腾 950 加速卡组成的超节点用于高并发推理，单位 token 成本下来了，吞吐上去了，API 价格才有了下调的空间。

这笔账分两层：

  • 采购成本层。 在外部出口管制的背景下，国产芯片的供应稳定性本身就是成本——能稳定拿到货、价格可控，比依赖受限的进口高端卡更可控。有报道测算，部分国产推理芯片的硬件成本约为进口高端卡的 60% 到 70%，这部分差价直接进了降价的空间里。
  • 软件配套层。 算力便宜没用，还得让模型在上面高效跑起来。DeepSeek V4 从英伟达的 CUDA 生态迁到昇腾的 CANN 生态，本身是一项不小的工程——它意味着万亿参数模型在国产算力上的推理链条真正打通了，不只是能跑，是能高效、高并发地跑。

需要客观说明的是，这笔账目前还没完全兑现。DeepSeek V4-Pro 现在 12 元/24 元的价格，官方明确说是"受限于高端算力产能的阶段性定价"——意思是高端算力的产能还没完全铺开，等下半年昇腾 950 的后续型号大规模部署，这一档价还有进一步下探的空间。所以第四笔账更像是一张"已经开始兑现、还在继续兑现"的账：它解释了为什么现在能降，也预示了为什么后面还能再降。

四笔账各占多少：一个定性的拆解

把四笔账放在一起看，单 token 成本的下降不是某一项的功劳，是四项叠加。这里给一个定性的拆解（不是精确百分比，各家配比不同，但量级关系是清楚的）：

技术账 省的是什么 量级感受
MoE 稀疏激活 单次调用激活的参数量 1.6 万亿模型只算约 490 亿，激活约 3%
稀疏注意力 长上下文的计算量与显存 百万 token 下计算量约为上代 27%、KV 缓存约 10%
推理引擎 显卡的吞吐利用率 显存利用率 60% → 95% 以上，吞吐数量级提升
国产算力 每张卡的采购与折旧 硬件成本约为进口高端卡 60%–70%，且在继续兑现

这张表想说明的核心判断是：国产模型的低价不是补贴堆出来的，是这四笔技术账叠加出来的真实成本下降。 MoE 和稀疏注意力是模型架构层面的省，决定了一次调用要烧多少浮点计算；推理引擎是工程层面的省，决定了显卡能不能不空转；国产算力是硬件层面的省，决定了显卡本身多少钱。四笔账方向一致，单 token 成本才能真的往下走，永久降价才站得住。

补贴当然也在，免费额度也在发——但那是获客的策略，不是定价的底。定价的底是成本线，成本线由这四笔技术账决定。这也是为什么 Kimi K2.6 敢逆势涨价：它的成本结构和产品定位允许它定在一个更高的点上，而不必跟着最低价走。

回到选型：开发者该怎么看这件事

把技术账算清楚，最终要落到选型上。对国内独立开发者和创业团队来说，价格降到这个水平，几个实际决策点变了。

自建还是调 API。 这是最该重新算的一笔账。过去自己买卡、自己部署推理，图的是长期成本可控。但当 DeepSeek V4-Flash 输入每百万 token 1 元、缓存命中 0.2 元这种价摆在面前，自建的盈亏平衡点被大幅推后了——你得调用量大到一定规模、且对数据本地化有硬性要求，自建才划算。对大多数中小团队，现阶段调 API 在成本和工程投入上都更合理。自建真正不可替代的场景，是数据不能出本地、或者有特殊定制需求这两类，而不是"为了省钱"。

按量计费还是包月。 关键看调用模式稳不稳。调用量波动大、有明显高峰低谷的，按量计费不浪费；调用量稳定且高、能预估的，包月套餐通常更省。这里特别要把缓存命中价算进去——如果你的应用每次请求都带一长串固定的系统提示词，缓存命中能让这部分成本降到 1/5，按量计费叠加缓存命中可能比包月还划算，得拿自己的真实调用日志算一遍。

长上下文场景怎么选。 如果业务重度依赖长上下文（长文档分析、超长对话、大量资料检索），优先看两点：一是模型用没用稀疏注意力（NSA/MSA/CSA 这类），用了的在长上下文下计算成本明显低，价格也更稳；二是专门的长文本模型（如通义 Qwen-Long 输入 0.5 元/百万 token）在纯长输入场景往往比通用旗舰更省。别拿旗舰模型硬扛长上下文，那是按最贵的档在付钱。

Agent 高频调用怎么选。 Agent 场景的特点是请求频繁、每次都带固定的工具说明和系统提示词、上下文长。这种场景缓存命中价就是命门——选一个缓存命中保底价低的模型（比如 Kimi K2.6 这类把缓存命中做到极低的），把固定提示词的部分尽量稳定下来提高命中率，整体成本能差出一截。同时长上下文那一档也要看稀疏注意力支持，两笔账叠加，agent 的单位调用成本才压得住。

场景 主要看什么 判断要点
自建 vs 调 API 调用规模 + 数据本地化要求 非"数据不出本地/特殊定制"，现阶段调 API 更合理
按量 vs 包月 调用是否稳定可预估 波动大选按量；稳定高量选包月；务必把缓存命中算进去
长上下文 稀疏注意力 + 专用长文本模型 别用旗舰硬扛长输入，长文本模型更省
Agent 高频 缓存命中保底价 + 稀疏注意力 固定提示词稳定化提命中率，两笔账叠加压成本

写在最后

国产模型这半年把 token 单价一路压低，看起来像一场谁也不肯先停的价格战。但把价格表和技术报告对着读，会看到一条更扎实的线：MoE 让每次调用只算约 3% 到 5% 的参数，稀疏注意力让百万级长上下文的计算量降到上一代的约四分之一、显存降到约十分之一，推理引擎把显卡利用率从 60% 拉到 95% 以上，国产算力把每张卡的成本压到进口高端卡的六七成，而且还在继续往下走。

这四笔账方向一致，单 token 成本才真的降了，永久降价才不是空话。低价不是亏出来的，是这几年实打实的工程一点一点抠出来的——稀疏注意力的几条弯路、MoE 负载均衡的反复调试、从 CUDA 到 CANN 的迁移，每一项背后都是真实的技术积累。对国内开发者来说，这意味着调一个能力对得上 Claude、GPT 量级的模型，单位成本已经低到可以放心把它用进高频、长上下文的真实业务里。把缓存命中、长文本模型、稀疏注意力这几个变量算进自己的调用模式，选型这件事，从来没有现在这么从容过。
