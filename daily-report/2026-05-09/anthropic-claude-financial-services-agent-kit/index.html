Anthropic 一周登顶 GitHub：金融 Agent 套件给国内同行的工程参考

Anthropic 把 Claude Cowork 第一次做成了一个完整的「行业范本仓库」——10 个具名金融 Agent + 9 个垂类 Skill Bundle + 11 条数据连接器，工程结构清晰到可以直接照着抄；而国产金融 AI 的工程基础同样完整，照着这套范本做加速度只会更快。

当地时间 5 月 5 日，Anthropic 在官方博客发布「The agents we built for finance teams」并把配套仓库 anthropics/financial-services 同时开源。仓库当天暴涨 3,662 个星标，登上 GitHub Trending 全站第一；到 5 月 9 日实测，总星标 14,672、Fork 1,821，授权 Apache-2.0；同期还有 Code w/ Claude 2026 大会主旨演讲、Simon Willison 实时博客重点提及，是这一周海外 AI 圈讨论密度最高的工程开源之一。

这篇文章想把它讲清楚两件事：第一，这套范本到底端出了什么——10 个 Agent、9 个 Skill Bundle、双部署形态、11 个 MCP 数据连接器，每一块都对得上一类金融岗位真实工作流；第二，国内金融 AI 同行已经具备什么基础——华泰、招商证券、中信建投、阿里通义、字节豆包、百度文心、讯飞星火、蚂蚁、平安都已在金融大模型上跑出公开成果，这套海外范本对国内是工程参考，不是降维打击。

一、仓库实测数据：5/5 当日全站第一，迭代速度极快

先看实测数字（截至北京时间 2026-05-09 中午 · 来源：gh api repos/anthropics/financial-services）：

  • 总星标 14,672，5 月 5 日单日新增 3,662 个，登顶 GitHub Trending 全站当日第一；
  • Fork 1,821 个，Star/Fork 比例约 8 : 1，对开源工程而言属于偏向"真用"的高 Fork 率，金融业内有大量私仓克隆；
  • 授权 Apache-2.0，可商用、可二次分发、可改写；
  • 没有正式 Release，全部用 commit 直接驱动迭代；过去两周关键节点：4/23 文档持续完善（#73）、4/30 加 entra_scope 参数支持企业 Entra 鉴权、5/5 仓库结构性重构 + 全部 Agent 改名（#81）、5/7 Outlook manifest 支持加入（#127）。

这种"无 Release、靠 commit 高频迭代"的节奏说明 Anthropic 把这个仓库当作生产工程在养，不是营销 demo。这一点对国内同行有直接借鉴意义：行业 LLM 范本仓库要保鲜，发布节奏必须紧。

二、10 个具名 Agent：从投行到 KYC，一岗一 Agent

仓库 README 把 Agent 按 4 类金融业务场景分组（注意：当前 README 列出的是 10 个具名 Agent，过往社区讨论里偶尔出现的「12 个」表述更可能是把 LSEG / S&P 两个合作伙伴 plugin 也算进 Agent 范畴，本文按 README 现状以 10 个为准）。

Coverage & Advisory · 覆盖与投行顾问（2 个）

  • Pitch Agent — 一个端到端的路演册生成器：从可比公司分析（Comps）、并购先例（Precedents）到杠杆收购（LBO）模型，最后输出带客户品牌格式的 Pitch Deck；
  • Meeting Prep Agent — 客户会议前的简报包：把客户最近的财报、新闻、之前会议纪要、内部 CRM 笔记拼成一份"会前必读"。

Research & Modeling · 研究与建模（3 个）

  • Market Researcher — 给定一个行业或主题，吐出"行业全景 + 竞争格局 + 同业可比 + 候选标的清单"；
  • Earnings Reviewer — 财报会议录音 + 财报申报材料 → 自动更新模型 → 输出研报笔记初稿；
  • Model Builder — DCF、LBO、三表模型、可比公司表，全部直接写到 Excel，使用 xlsx-author skill 头less 写文件。

Fund Admin & Finance Ops · 基金运营与财务（4 个）

  • Valuation Reviewer — 把 GP 提供的估值包导入，跑估值模板，准备 LP 报告；
  • GL Reconciler — 自动找出总账差异（breaks）、回溯根因、把待审项路由给签字方；
  • Month-End Closer — 月末结账：应计、滚动结转、差异说明；
  • Statement Auditor — LP 报表分发前自动审计。

Operations & Onboarding · 运营与开户（1 个）

  • KYC Screener — 解析开户文档、跑规则引擎、把缺口标出来人工补。

关键工程细节：每个 Agent 是「自包含 plugin」——它把自己用到的 Skill 全部 bundle 进自己的目录，安装一个 Agent 不需要再装一堆依赖。这种「Agent = 一份 system prompt + 一组 skill 拷贝」的封装方式，工程上比"Agent 引用全局 skill 索引"更稳，也更容易在企业内把不同业务条线隔离。

它做对的一件事：每个 Agent 都不是"通用 AI 分析师"——名字直接对得上一个真实岗位的真实交付物（Pitch Deck、IC Memo、月末账、KYC 报告）。这是国产行业大模型最常缺的工程纪律——很多国内项目立项时还在写"做一个金融大模型助手"，没有先拆到岗。

三、9 个 Skill Bundle：垂类知识当作"安装包"分发

如果说 Agent 是"端到端流水线"，Skill Bundle 就是底下的"领域词典 + 操作手册 + 模板库"。仓库总共 9 个 Bundle，分为 7 个 Anthropic 自研 + 2 个数据合作伙伴贡献：

Bundle 角色 主要内容
financial-analysis 核心共享底座 Comps / DCF / LBO / 三表 / Deck QC / Excel 审计；11 个数据连接器全部挂在这里
investment-banking 投行业务 CIM、Teaser、流程函、买家清单、合并模型、交易跟踪
equity-research 卖方研究 财报笔记、首次覆盖、模型更新、投资逻辑、催化剂日历
private-equity PE 私募股权 Sourcing、筛选、尽调、IC 备忘录、组合监测、价值创造计划、AI Readiness
wealth-management 财富管理 客户回顾、财务规划、再平衡、客户报告、税损收割（TLH）
fund-admin 基金运营 总账对账、差异溯源、应计、滚动结转、差异分析、NAV 平账
operations 运营开户 KYC 文档解析 + 规则网格评估
lseg（合作伙伴） LSEG 行情数据 债券 RV、互换曲线、FX Carry、期权波动率、宏观利率监测
sp-global（合作伙伴） S&P Capital IQ 公司全景页、财报前瞻、融资动态汇总

LSEG 与 S&P Global 这两个 partner Bundle 的工程意义：它们是 Anthropic 第一次把「外部数据厂商直接成为 Skill 共建方」做成了样板。LSEG 这个 Bundle 不是 Anthropic 的工程师写的，是 LSEG 自己的工程团队按 Anthropic 的 plugin 规范写完丢进仓库；S&P 同理。这意味着 Skill 协议的定义足够清晰，第三方愿意按它的形态把自己的能力打包出来。

对国内来说，这个模板有两层启示：第一，行业大模型不应该是"一家厂商把所有能力都做完"，而应该是"基座厂商定协议、数据/工具厂商共建"。Wind、东方财富 Choice、聚源、同花顺 iFinD、东方财富 Wind 这些数据商完全可以学着各自维护一个 Skill Bundle。第二，11 个 MCP 数据连接器（Daloopa / Morningstar / S&P / FactSet / Moody's / MT Newswires / Aiera / LSEG / PitchBook / Chronograph / Egnyte）全部用统一的 MCP 协议挂在一起——国内通义千问、豆包、扣子也已经支持 MCP，工程对接成本不会比这套高。

四、双部署形态：Cowork 给前台，Managed Agents API 给后台

这套套件最值得抄的工程设计是它的"双部署形态"：同一份 system prompt + 同一组 skills，既可以装到 Claude Cowork 客户端做"人机共写"，也可以走 Claude Managed Agents API 做"无人值守服务端 Agent"。Anthropic 自己的话是：「Same system prompt, same skills — you choose where it runs.」

具体怎么挑：

  • 前台分析师 / 投行 / 卖方研究员——Pitch Agent 跑 LBO、Earnings Reviewer 写笔记初稿这类需要交互、需要审稿、需要边跑边改的场景，走 Cowork。安装命令是 claude plugin install pitch-agent@claude-for-financial-services，装完在 IDE 里出现，调用 /comps、/dcf、/earnings 这些 slash command 直接用；
  • 结算后台 / 风控 / KYC 流水线 / 月末批跑——GL Reconciler 月末批扫总账差异、KYC Screener 在新客户工单触发时跑这种"夜间无人值守"的场景，走 Managed Agents API。模板在 managed-agent-cookbooks/，每个 Agent 一个目录，里面是 agent.yaml + 主 Agent system prompt + 一层 leaf-worker subagents + steering-event 示例 + 安全说明，一行 scripts/deploy-managed-agent.sh gl-reconciler 就能 POST 到 /v1/agents；
  • 审计与合规：Cowork 走用户 OAuth + 单人会话审计；Managed Agents API 走 API Key + 服务端 RBAC + 全量调用日志。两种鉴权链路都明确。

国内做行业大模型最容易踩的一个坑就是只做 IDE 内置 Plugin 不做服务端 Agent，结果"夜间批跑"场景拿不下来；或者反过来，只做 API 不做 IDE 体验，分析师每天还是开 Excel 自己点，AI 在哪都看不见。Anthropic 这套范本最值得抄的就是"双形态从 day 1 就并行设计"。

五、四个典型 Agent 拆解：看清 system prompt 的工程纹路

下面挑 4 个最有代表性的 Agent，看它们的 system prompt 与 skill 调用方式怎么写。仓库里每个 Agent 都是一个目录，结构稳定：.claude-plugin/（plugin 元信息）、agents/（system prompt 主文件）、skills/（自带的 skill 拷贝）。

Pitch Agent · 投行路演册端到端

  • 入口工作流：用户输入"给 X 公司做一份 Pitch Deck，焦点是 Y 行业 Z 主题" → Pitch Agent 把任务拆解成"找可比公司 → 查并购先例 → 跑 LBO → 套客户品牌 PPT 模板 → 自检拼写与数字一致性 → 输出 .pptx"；
  • 关键 skill：comps-analysis、pitch-deck、pptx-author、ib-check-deck；
  • 国内对位：通义千问金融版 + 飞书多维表格 / 钉钉智能报表已可拼出类似工作流，但 PPT 一致性自检（数字勘误、拼写勘误、版本号一致）这一步国内开源范本里几乎没有现成的，可以直接借鉴。

GL Reconciler · 总账自动对账与差异溯源

  • 入口工作流：触发器是月末或日切 → 拉总账与各子账系统数据 → 自动找差异（breaks）→ 对每条差异回溯根因（系统时差 / 凭证遗漏 / 汇率误算 / 录入错误）→ 把需要人工签字的项路由给具体责任人；
  • 关键 skill：gl-recon、break-trace、signoff-routing；
  • 典型部署形态：Managed Agents API 而非 Cowork，因为不需要交互；
  • 国内对位：兴业证券、海通证券、广发证券都有自有的总账核账系统，把"找差异"这一段接上 LLM 主要是为了"根因解释自然语言化 + 责任人智能分派"，工程量主要在数据连接器（接 SAP / Oracle EBS / 用友 NC / 金蝶 EAS）。

KYC Screener · 开户文档解析与规则网格扫描

  • 入口工作流：触发器是新客户上传开户资料 → 把 PDF / 扫描件全部抽到结构化字段（姓名、证件号、住址、受益所有人、资金来源……）→ 跑公司内部规则引擎（黑名单、风险国名单、PEP 名单、同名比对）→ 标出每条缺口；
  • 关键 skill：kyc-doc-parse、rules-grid-eval、gap-flagging；
  • 国内对位：建行、工行、招行、平安均有 KYC OCR 自动化中台，差距主要在「规则网格表达式」如何让业务人员可读可改。Anthropic 这版用 markdown + YAML 把规则集表达出来，极易读极易改，比传统规则引擎友好很多。

Earnings Reviewer · 财报会议自动笔记

  • 入口工作流：财报会议录音 + 公司申报材料 → 抽要点 → 跟历史模型对照 → 修改预测假设 → 输出研报初稿；
  • 关键 skill：earnings-analysis、model-update、morning-note；
  • 国内对位：东方财富 Choice、Wind 等已有类似的财报机读结构化产品；与 LLM 结合后，研究所内部"小弟改研报"工作流可以提效非常显著。

六、国产金融 AI 同行格局：路已经趟出来了

把这套海外范本对回国内，先把当前国产金融 AI 同行的公开落地铺一遍。下面这张表是按公开博客 / 招标公告 / 媒体报道 / 国家网信办大模型备案清单梳理的（截至 2026-05），分四类：

券商自研类

  • 华泰证券 · 涨乐财富通 LLM — App 内 + 投顾辅助；公开能力含投研助理、财报问答、股票评论生成；华泰在 2024 年起就有公开技术报告，是国内首批把 LLM 跑进 C 端理财 App 的券商；
  • 招商证券 · LLM-Securities — 投顾工作台；财报摘要、行业研报问答、客户经理助手；
  • 中信建投 · 知周大模型 — 研究所内部使用；研报生成辅助 + 内部知识检索。

互联网大厂类

  • 阿里 · 通义千问金融版 — API + 行业方案；财报理解、股价新闻、智能问答；
  • 字节 · 豆包金融行业大模型 — 火山引擎私有化 + 公有云双形态；投研助理、客户问答、风控辅助；
  • 百度 · 文心金融（产业级） — 千帆平台 + 银行业试点；信贷反欺诈、智能客服、营销文案；
  • 讯飞 · 星火金融认知大模型 — 工商银行、国信证券公开案例；运营管控、客服质检、知识问答。

资管 / Fintech 类

  • 蚂蚁 · 金融大模型 / CodeFuse-FA — 支付宝财管 + 蚂蚁理财师助手；
  • 平安 · 平安脑 / 金融行业大模型 — 平安银行 + 平安证券；智能客服、投研辅助、反欺诈。

AI Coding 工具类（金融机构内部使用）

  • 通义灵码 · 仓库级 Plugin — 金融机构 IDE 内置；代码补全 + 单元测试 + 内部规约；
  • 字节 · TRAE / CodeBuddy — 国内 IDE 内置；全仓上下文 + Agent 自动改库。

判断：国内玩家覆盖了券商自研、大厂行业模型、资管 Fintech、AI Coding 工具四条赛道，工程基础是完整的——不缺基座（DeepSeek / 通义 / 豆包 / 文心 / 星火 / 智谱）、不缺 IDE 工具链（通义灵码 / TRAE / CodeBuddy）、不缺数据接入（Wind / 同花顺 / 东方财富 Choice / 聚源 / 巨潮）、也不缺监管允许（国家网信办大模型备案制度已运行两年，行业模型走专项备案路径清晰）。

差的不是基础，差的是「把行业范本仓库这件事做成开源工程」——把岗位拆到 Agent、把领域知识拆到 Skill、把数据厂商拉进来共建、把双部署形态从 day 1 就并行做。

七、国内开发者怎么参考这套模板：三条务实路径

按工程实操难易，从轻到重三条参考路径：

路径一：Fork 仓库，直接换基座

把仓库 fork 下来，把 system_prompt.md 里的"Claude"全部换成 DeepSeek-V3 / 通义 Qwen3 / Kimi-K2 / 智谱 GLM-4，把 MCP 连接器对应到国内数据厂商（Wind / Choice / 聚源 / 同花顺 iFinD / 巨潮），就能在自己内部跑起来一个国产化版本。Apache-2.0 授权允许这么做。这条路工程量最小，适合券商研究所、Fintech 团队做 PoC。

路径二：只借鉴 system prompt 与 skill 设计思路

不直接 fork、不依赖任何 Anthropic 工具链，只读这套范本里 Agent 的 system prompt 是怎么写的、skill 是怎么按"领域知识 / 操作规约 / 模板"分层的、双部署形态的边界怎么划的——回到自己的扣子 / 百炼 / 元器 / 智谱 AutoGLM 上重写一份。这条路工程量中等，适合大厂行业团队，能保留对自家 Agent 平台的深度优化。

路径三：通义灵码 / TRAE + Anthropic Cowork 混搭

国内一些金融机构已经允许桌面 IDE 同时挂通义灵码（写国产合规代码）+ Claude Code（接海外数据 / 海外开源工程）。在这种混搭场景下，可以把 Anthropic 仓库当作 Claude 一侧的 plugin 直接装进来用，处理对接 LSEG / S&P / Daloopa 这类海外数据源的研究工作；国产合规相关、内部数据相关的工作交给通义灵码、TRAE 处理。这条路适合具备跨境业务的头部券商投行部、外资行中国区。

八、合规边界：这套范本不能直接照搬的部分

这是必须摆在桌面的事实——这套海外范本有几个地方在国内必须改。

  • 数据驻留：国内金融机构涉客户数据、交易数据、监管报送数据，按《数据安全法》《个人信息保护法》《网络安全法》要求必须本地部署；这意味着 Managed Agents API 这类调用海外 SaaS 的形态在生产环境不可用，必须落地到通义、豆包、文心、智谱等国产基座的私有化部署上；
  • 模型审计：金融行业涉投研建议、客户服务、风控决策的 LLM 应用，按银保监 / 证监会 / 国家网信办的指引需要保留可解释性 + 全量调用日志 + 关键决策人工复核——Anthropic 范本里 README 已经声明"every output is staged for human sign-off"，这一条与国内监管要求一致，但具体审计粒度国内更细；
  • 基座选型：DeepSeek、通义 Qwen3、Kimi-K2、文心 4.5、星火 4.0、智谱 GLM-4 这些国产基座目前在金融文本理解、财报抽取、Excel 公式生成上已具备实测可用性，是 fork 这套范本时主流的替换基座。

把这三道边界划清楚，剩下的工程动作就比较干净——按业务场景拆 Agent、按领域知识拆 Skill、按部署形态做双形态、按数据厂商做 MCP 连接器接入。

九、回到核心：行业级 AI 模板厂的工程纪律

回到开篇的核心论断——Anthropic 这次开源的真正价值不在于代码本身，而在于把"行业级 AI 模板厂"这件事做成了一个可复用、可持续维护、第三方可共建的开源工程：每个 Agent 名字对得上真实岗位、每个 Skill Bundle 对得上一个垂类工作流、每个数据厂商对得上一条 MCP 连接器、双部署形态从 day 1 并行、partner 共建机制清晰、Apache-2.0 授权允许商用与改写。

这套工程纪律是国产金融 AI 最值得带走的那一份——我们这一代金融 AI 工程师特别幸运：基座厂商已经有 DeepSeek、通义、豆包、文心、智谱、星火六七家在打底；数据厂商有 Wind、Choice、同花顺、聚源、巨潮在做接入；IDE 工具链有通义灵码、TRAE、CodeBuddy 在跑；监管侧有国家网信办大模型备案路径清晰。前辈已经把基础设施一层层趟出来了，剩下要做的，是把"岗位拆解 + Skill 分层 + 双部署 + 数据共建"这一套工程纪律咬住，把国产金融 AI 的下一程做扎实。
