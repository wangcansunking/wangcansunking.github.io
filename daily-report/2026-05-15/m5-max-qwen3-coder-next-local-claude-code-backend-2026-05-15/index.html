2026-05-15Daily Report

M5 Max 跑 Qwen3-Coder 79 tok/s 啥概念

thebytelab 把 Qwen3-Coder-Next 8-bit 在新款 Apple M5 Max 上跑出 79.3 tokens per second 的生成速度，提示处理（prefill）阶段冲到 1,887 tokens per second。这两个数字摆在一起，等于宣布开发机本地已经能扛起一个国产代码模型当推理后端——把它接在常用编码代理（如 Claude Code 风格的本地后端）背后，普通日常补全、改写、解释代码的体感，已经追平不少线上 API。对国内开发者来说，这条线意味着合规敏感场景里，模型权重和提示词都可以不出本机，整条链路一直停在统一内存（unified memory）里。

一、Apple M5 Max 关键规格：内存带宽是这次的胜负手

Apple 在三月发布的官方新闻稿写得很清楚：M5 Pro 与 M5 Max 是给「最吃硬件」的专业工作流准备的一代，3 月 4 日开放预订，3 月 11 日正式发售。规格层面值得记的几条：

  • GPU 配置：M5 Pro 起步 24 核，Max 顶配做到 40 核
  • 统一内存带宽：32 核 GPU 版本 460 GB/s，40 核 GPU 版本 614 GB/s
  • 统一内存容量：M5 Max 最高 128 GB
  • 神经网络引擎：16 核重新设计的 Neural Engine
  • 对位 M4 Max：llmcheck.net 实测在 LLM 工作负载上整体提升约 28%

llmcheck.net 把提升原因写得非常直白：

"M5 Max pushes approximately 28% more tokens per second than the M4 Max across LLM workloads, thanks to 600 GB/s memory bandwidth, improved GPU compute, and a redesigned Neural Engine." — llmcheck.net

翻成本地化的说法：M5 Max 在常见大语言模型工作负载上比 M4 Max 多吐出约 28% 的 tokens per second，主要靠 600 GB/s 量级的内存带宽、更强的 GPU 算力和重新设计的神经网络引擎三件套。这条结论的关键是「内存带宽」三个字——后面会反复回到它。

二、Qwen3-Coder-Next 跑分一览：MoE 架构和苹果硅是天作之合

DEV.to 的 Qwen3-Coder-Next 2026 完整指南把这款模型的来历写得很清楚：阿里巴巴在 2026 年 2 月发布，采用混合专家（Mixture of Experts，MoE）架构，总参数约 80B，单次推理激活约 3B，主打代码场景。MoE 不是新词，但放在苹果硅上意义不一样——它每步只激活一小部分参数，对内存带宽的吃法比稠密大模型友好得多。

thebytelab 在 M5 Max 上的实测数据汇总如下（生成速度 = generation tokens per second，提示处理 = prefill tokens per second）：

模型 量化精度 生成速度 提示处理
Qwen3-Coder-Next 8-bit 79.3 tok/s 1,887 tok/s
Qwen3.5-122B-A10B 4-bit 65.9 tok/s 1,239 tok/s

llmcheck.net 同机的横向跑分补齐了画面：

模型 量化精度 生成速度
Llama 3 8B Q4 82 tok/s
Mistral 7B Q8 68 tok/s
Qwen 3.5 30B-A3B Q4 58 tok/s
Llama 4 Scout Q4 32 tok/s
Llama 3 70B Q4 18 tok/s

llmcheck.net 把这个结论钉得很死：

"Prefer MoE models on Apple Silicon. Qwen3.5's MoE architecture was practically designed for this hardware." — llmcheck.net

转述一下：在苹果硅上优先选 MoE 架构的模型，Qwen3.5 的这套架构几乎就是冲着这类硬件设计的。这背后是同一条原理：

"LLM inference is memory-bandwidth bound, not compute-bound." — llmcheck.net

大语言模型推理的瓶颈是内存带宽，不是算力。MoE 模型每步只激活一小部分参数，正好把「需要从统一内存里搬过去喂给 GPU 的数据量」按比例砍小，于是同一块 M5 Max 跑 80B-A3B 的 Qwen3-Coder-Next，体感比跑稠密 70B 的 Llama 3 要爽两个量级。

三、70B 也能跑：18 tok/s 是什么概念

很多开发者最关心的不是 80B 的旗舰，而是「最大能塞下多大的稠密模型」。llmcheck.net 给了答案：M5 Max 跑 Llama 3 70B Q4 量化版本，生成速度 18 tokens per second。

参照系是上一代 M4 Max。Sean Kim 的博客（搜索摘要：原文 403 不可直访）测出 M4 Max 跑 Llama 3.1 70B 的生成速度为 20 tokens per second，并留下一句被反复引用的话：

"Running Llama 3.1 70B at 20 tokens per second on a laptop is unprecedented." — Sean Kim 博客（搜索摘要）

意思是在一台笔记本上以每秒 20 个 token 的速度跑 70B 级模型，过去是不可想象的。M5 Max 在更大的稠密模型上数字略有波动（18 vs 20），但量级一致，并且因为 MoE 优化的存在，绝大多数实际负载都不会卡在稠密 70B 上——这只是用来证明「上限够」。

放到日常体感上：

  • 18 tokens per second 对应中文输出约每秒 12–15 字，大致是一个流畅打字的人的两倍
  • 79 tokens per second（Qwen3-Coder-Next）已经超过绝大多数云端代码补全 API 的端到端体感
  • 1,887 tokens per second 的提示处理（prefill） 意味着塞一段 5,000 token 的项目上下文，三秒以内吃完

云端 API 的延迟瓶颈往往在网络 RTT 和排队上，本地推理把这两段直接削掉，整体响应曲线明显更平。

四、MLX 还是 llama.cpp：工具链选择决定终态速度

硬件再强，跑分也要看跑分的人怎么跑。llmcheck.net 给出的工具链建议非常直接：

"MLX delivers 20-50% higher throughput than llama.cpp for the same models" — llmcheck.net

同一份模型权重，用 Apple 自家的 MLX 跑，吞吐量比 llama.cpp 高 20%–50%。原因在于 MLX 是苹果硅原生框架，对统一内存、Metal 后端和 Neural Engine 调度做了深度对齐；llama.cpp 是跨平台路线，在通用性上有优势，但在苹果硅上要让出一截天花板。

另一条值得知道的事情，llmcheck.net 帮读者拆掉了苹果发布会上常见的口径模糊：

"Apple's '4x faster' claim specifically refers to prefill (prompt processing), not generation speed. The actual generation improvement versus M4 Max is modest—around 15%." — llmcheck.net

苹果宣传的「速度 4 倍提升」指的是提示处理（prefill）阶段，不是生成速度；和 M4 Max 比，生成速度真实涨幅约 15%。明白这条口径差，看官方海报就不会被「4 倍」三个字带偏——但 1,887 tokens per second 的提示处理本身已经够强，对一段五千字代码上下文几乎是秒进。

实战层面给开发机的建议：

  • 跑国产代码模型优先 MLX，吞吐量上限更高
  • 需要跨平台部署或 Linux 服务器迁移再选 llama.cpp
  • 关心提示处理速度的（比如代码代理一次塞整个文件）认准苹果硅的 prefill 强项

五、选机指南：32 核还是 40 核 GPU、128 GB 内存值不值

llmcheck.net 留下一条非常实用的选配公式：

"Your Unified Memory should be at least 1.5x the model file size at your chosen quantization level." — llmcheck.net

意思是统一内存容量至少要选模型文件在所选量化精度下大小的 1.5 倍。直接套到主流模型上：

目标模型 量化精度 模型大小估算 建议统一内存
Qwen3-Coder-Next 8-bit ≈ 40 GB 64 GB 起步
Qwen3.5-122B-A10B 4-bit ≈ 60 GB 96 GB 起步
Llama 3 70B Q4 ≈ 40 GB 64 GB 起步
Mistral / Llama 3 8B Q4–Q8 5–8 GB 32 GB 即可

具体到 SKU：

  • 32 核 GPU + 460 GB/s 带宽：日常跑 8B / 30B-A3B MoE，体验非常流畅，价位友好
  • 40 核 GPU + 614 GB/s 带宽：跑 80B MoE / 122B-A10B / 70B 稠密的首选，跑分天花板对齐 thebytelab 那 79 tok/s
  • 128 GB 统一内存：留给「想同时挂代码模型 + 嵌入模型 + 视觉模型」的多角色场景，单一代码模型用不上

简单一句：如果只想要一台「装上去就当本地 Claude Code 风格后端」的开发机，40 核 GPU + 64 GB 统一内存就是甜点位；想再激进一点上 128 GB，留给后面更大的 MoE。

六、对国内开发者意味着什么

把上面所有数字归到一句话：M5 Max 把「开发机本地就当代码模型推理后端用」这件事的门槛拉到了一个普通专业用户能接受的线上。Qwen3-Coder-Next 是阿里巴巴在 2026 年 2 月发布的 80B 总参 / 3B 激活 MoE 代码模型，权重开源，许可证宽松，配 MLX 工具链能在新款 M5 Max 上吐到 79 tokens per second——这是一个真正能撑日常编码代理工作流的速度。

更重要的是，整条链路完全停在统一内存里，模型权重不出本机，提示词和代码上下文也不出本机。对合规敏感场景（数据不能出网、行业监管限制、企业内网开发机），这条线比任何「我们提供企业版云端 API」的话术都更直接。开源国产模型 + 开源推理框架 + 一台开发机，三件套就齐了。

接下来值得关注的，是阿里巴巴 Qwen3-Coder-Next 后续的量化体积压缩，以及 MLX 社区对 MoE 调度的进一步优化——这两条线哪条先动，下一轮的开发机本地后端门槛就会再往下走一截。
