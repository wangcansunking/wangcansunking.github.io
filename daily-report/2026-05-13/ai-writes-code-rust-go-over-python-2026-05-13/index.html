AI 写代码该选 Rust 还是 Python：HN 868 评论吵翻

一、823 分、871 评论：一篇反 Python 长文为什么能在 HN 吵翻全场

先把数字摆出来：2026-05-11 美东时间 20:45，一位叫 Noah Mitchem 的工程师在 Medium 上贴出一篇博客《If AI writes your code, why use Python?》。投到 Hacker News 之后，到 5/13 早晨，这篇文章拿到 823 分、871 条评论（HN item 48100433 实测）。这是一个非常特别的比例——评论数比顶赞数还多 6%，在 HN 上几乎只有一种文章能跑出这种数字：论点足够刺激，让两个阵营的人都忍不住进去吵。

咱们写 Python 的工程师看完标题大概率第一反应是「又来一个唱衰 Python 的」。但点进去会发现，Mitchem 的论点其实比标题克制得多。他不是说「Python 死了」，他说的是这样一件事：Python 当年压过 C++、Java 的核心优势是「人写起来轻松」，但当今天主力代码越来越多由 AI 写，「人写起来轻松」就不再是最重要的指标了。强类型、紧编译反馈、运行时性能——这些对 AI 写代码更友好的指标，反而是 Rust 和 Go 的强项。

把这条核心论点拆开来看，需要回答四个问题：

  • 第一，为什么 AI 写 Rust / Go 比人写 Rust / Go 顺？
  • 第二，有没有真实的工程案例？
  • 第三，HN 上 871 条评论的真正分歧在哪里？
  • 第四，国内做 AI Coding 的同行，手上工具链对 Rust / Go 的支持到底怎么样？

这篇文章的结构就按这四个问题展开。先把答案剧透一句：对国内开发者而言，「AI 时代多了一条强类型路径」不是要不要选边站的问题，是手上工具已经具备能力、可以多一份选项的问题。咱们 Python 的位置仍然稳固，但身边的工具箱确实在变厚。

二、Mitchem 的核心论点：可读性曾经压过性能，AI 把这条曲线拉回来了

要理解 Mitchem 为什么说「AI 时代该选 Rust / Go」，得先回到 Python 当年是怎么打赢的。

2008 年到 2018 年这十年，Python 之所以从一门「胶水脚本」长成现代主力语言，核心理由非常清晰：人写起来轻松。for item in items: 比 Java 的 for (Iterator<Item> it = items.iterator(); it.hasNext(); ) 短一个数量级，json.loads() 比 C++ 的 boost::json 顺手一个数量级，pip install 比 Maven 配置 pom.xml 快一个数量级。当人写代码的时间是瓶颈，「写起来轻松」就是真金白银。

但 Mitchem 的观察是：这条「人友好」曲线，在 AI 写代码时代正在被抹平。

他的逻辑链是这样的：

第一，Claude Opus 4.7、GPT-5.5、Gemini 3.1、DeepSeek V4 这一代模型，在 SWE-bench Verified 上都跨过了 80% 这条线。这意味着 AI 写出能跑的代码，已经不是稀罕事，是默认事件。

第二，当 AI 是主要的「打字员」，「人写起来快不快」就不再是核心指标了。一段 Rust 代码，人写要 30 分钟，AI 写要 30 秒，跟 Python 30 分钟还是 30 秒已经没本质区别。

第三，反而是另一组指标变得关键：AI 写错的时候，编译器能不能立刻告诉它错在哪？ Rust 的强类型 + 借用检查器 + cargo check 反馈链，在这个维度上对 AI 异常友好。Mitchem 在文章里有一句话被 HN 反复引用：「The compiler feedback loop is so tight that models self-correct in real time. Every error message is a free training signal.」翻译过来就是「编译反馈循环紧到模型可以实时自我纠正，每一条编译错误都是免费的训练信号」。

第四，Python 因为是动态类型，AI 写错了往往要等到运行时、单测、甚至上线才会暴露。同样一段「校验用户订单」的逻辑，Python 版的类型错误是写出来要靠人手工加的 if-else，Rust 版直接嵌进类型签名——AI 想偷懒都偷懒不了。

第五，运行时性能这块，原本是 Python 的硬伤，但在「人写为主」时代不构成致命短板，因为人时间贵、机器时间便宜。可是当 AI 接管打字之后，机器时间反而被放大——24 小时跑着的 agent、上下文动辄几十 K token 的推理、海量的 GPU 调用，每一项都让运行时效率重新值钱。Mitchem 原文里的提法是「a harder language's runtime advantages compound every day you run the service in production」——「一门更难的语言，每运行一天就多积累一份性能复利」。

把这五条串起来，Mitchem 的论点就成立了：人友好这条曲线被 AI 抹平，机器友好这条曲线却被 AI 放大。Rust / Go / Swift 这种「学起来麻烦但跑起来稳」的语言，从此跟人的关系不再紧张，因为人不需要直接学，只需要让 AI 学就够了。

三、三个硬到不能再硬的工程案例

光讲逻辑没有说服力，Mitchem 在原文里搬出了一串真实的工程案例，每一个都是 2025-2026 这一年发生的事。这里挑最硬的三个，按时间顺序摆一摆：

第一个：Nicholas Carlini 用 16 个 Claude 并行写出一个能跑的 C 编译器。这位 DeepMind 出身的研究者，在 2025 年下半年公开了一个项目：把 16 个 Claude agent 并行起来，让它们协作写一个产线可用的 C 编译器。代码语言是 Rust，最终代码量约 10 万行，API 调用费用约 2 万美元。这个案例的关键不是「AI 能写编译器」——这个事 2024 年就有人证明过——关键是 16 个 agent 并行能协作出 10 万行不打架的代码，这背后非常依赖 Rust 的强类型签名作为「契约」，让不同 agent 之间不会改坏对方的数据结构。

第二个：微软用 Go 重写 TypeScript 编译器。这件事 TypeScript 之父 Anders Hejlsberg 在 2025 年 3 月正式官宣，内部代号 Project Corsa，目标是把 TS 编译器从 TypeScript 本身重写成 Go。结果出来很硬：VS Code 自己的 150 万行 TS 代码，原本编译要 78 秒，Go 版编译器跑完只要 7.5 秒，整整 10 倍。Microsoft devblog 2025-03 的官方说法是「a 10x faster TypeScript」。咱们国内同行做大型前端工程的，做过 monorepo 的，应该对 TS 编译耗时的痛深有体会——这是一个真实减痛的工程决策，不是噱头。值得注意的是，TS 团队评估过 Rust、C、C++ 后选了 Go，理由是「Go 在低层控制、GC、并发原语这三块的平衡点最好」。这也回应了 HN 上一种常见误解：选 Go 不等于反 Rust，两者在 AI 时代的位置是互补的。

第三个：Andreas Kling 两周把 Ladybird 的 JS 引擎从 C++ 改成 Rust。Ladybird 是一个独立的浏览器项目，从零做了好几年。2025 年下半年，主作者 Andreas Kling 用 AI 辅助，把 JS 引擎部分约 2.5 万行 C++ 代码，在 14 天内重写为 Rust。这个数字之所以惊人，是因为 JS 引擎是浏览器最复杂的几个子系统之一，传统团队做这种迁移要按年算。

把这三个案例摆一起的意义是：Rust 不是「适合学习」的语言，而是「适合 AI 协作」的语言。10 万行 C 编译器、150 万行 TS 编译器、2.5 万行 JS 引擎，这三种工程量级，都是过去人写要按年算的，现在 AI 配合强类型语言能在天或周级别完成。Mitchem 在文章里还补了一句：「Armin Ronacher 用 agent 把 MiniJinja 从 Rust 翻到 Go，10 小时全部完成，作者本人只花了 45 分钟，API 费用约 60 美元。」——这种数字，让「AI + 强类型语言」的工程节奏感变得很具体。

四、Python 生态正在被 Rust「换芯」：6 个咱们每天在用的工具

但更扎心的一件事，Mitchem 在文章里只点了一段，但咱们写 Python 的工程师其实早就感觉到了：Python 的工具链生态，正在被 Rust 大规模换芯。这不是未来时，是已经发生时。

把 Python 工程师每天大概率会用到的工具列一下：

  • Pydantic：FastAPI、LangChain、几乎所有现代 Python web / agent 框架的数据校验底座。v2 在 2023 年用 Rust 重写了核心 pydantic-core，根据官方 blog「Obtain a 5x speedup for free by upgrading to Pydantic v2」字面口径，简单模型 5× 提速、嵌套模型 17×、JSON 解析 20×、JSON 序列化 10×，复杂深嵌套结构能维持 4-10× 速度优势，整个量级是 5-50×。
  • Polars：对标 pandas 的数据框架，纯 Rust 实现。做数据分析的同行越来越多在新项目里直接用 Polars，因为它对大数据集的吞吐能拉开 pandas 一两个数量级。
  • HuggingFace tokenizers：训练、推理、SFT、RLHF 都要用到的分词器。底层 Rust 实现，跑 BPE 比纯 Python 快几十倍。
  • orjson：Python 里最快的 JSON 序列化库之一，纯 Rust 实现。FastAPI 项目把 json.dumps 替换成 orjson.dumps，能拿到几倍到十倍的接口耗时下降。
  • ruff：替代 flake8 / black / isort 的 lint 工具，Astral 团队出品，纯 Rust。在大型 Python monorepo 上比 flake8 快 10-100×，国内大厂的 Python 团队过去一年几乎都迁了。
  • uv：Astral 出品的包管理工具，对标 pip + poetry + pyenv，纯 Rust。Mitchem 在原文里点了一句很扎眼的数据：「uv saves Codex approximately one million compute minutes per week」——光是 OpenAI 自家的 Codex 工具链，每周省下来的算力就有 100 万分钟。

这一组工具的存在意义是：Python 工程师并没有离开 Python，但每天调用的高性能路径已经是 Rust。再加上 JetBrains 2025 年的开发者调查显示，Python 扩展库里使用 Rust 的比例从 27% 涨到 33%，一年涨了 6 个百分点——这条曲线还在加速。

Mitchem 借这个事实做了一个稍有挑衅的延伸：「既然 Python 生态最关键的高性能路径都已经是 Rust 了，咱们干脆直接写 Rust 不也行吗？」这条引申是不是合理可以争论，但「Python 工具链正在被 Rust 重写」这条事实，已经是无可争议的现状。

五、HN 871 条评论的真正分歧：五种立场全在这里

回到 HN 那 871 条评论上。这种规模的讨论，很难用「支持派 vs 反对派」两分法概括。仔细看下来，至少能分出五种立场，每一种都有一条 verbatim 顶赞引语作为代表。咱们国内同行如果想了解海外开发者社区当下对「AI + 编程语言选择」这件事的真实分布，下面这五条比看任何二手综述都直观。

立场一 · 强类型对 AI 极有用（支持派）

来自 tptacek（HN 老评论员，知名安全工程师）：

There are two working LLM axes. Critic strength: how much the language catches before runtime. Sensor strength: how good the empirical feedback loop is.

翻译：「让 LLM 跑顺有两条正交的轴。批评家强度：语言能在运行前抓住多少错。传感器强度：经验反馈循环有多紧。」这是支持派的核心理论框架——Rust 把这两条轴都拉满。

立场二 · Rust 跟 Python 一样顺（实操派）

来自 munro：

Lately I just have Claude build most things in Rust, it's really amazing. I tried Go, but I found it wasn't as good--Rust really does to me feel like Python.

翻译：「最近我基本都让 Claude 用 Rust 写东西，效果好得惊人。我试过 Go，但没有 Rust 顺。Rust 对我来说真的像 Python 一样。」这条引语的杀伤力在于，「Rust feel like Python」这种说法过去是反 Rust 派的笑话，现在被支持派当成日常体验。

立场三 · Go 是更稳的折中（折中派）

来自 0xbadcafebee：

I know a couple languages fairly well: C, Perl, Python, Bash. I never formally learned Go, but as a test of AI coding, I started some vibe coded projects in Go. It worked very well: the code is minimal, there's few dependencies, and it compiles down to a static app. ... 'language' isn't a barrier anymore.

翻译要点：「我熟悉 C / Perl / Python / Bash，没正式学过 Go，但用 AI 写 Go 测试项目效果很好——代码精简、依赖少、编译成静态可执行。在 AI 时代，『语言』已经不是壁垒。」这条折中派的关键判断是：Go 的「batteries included 标准库」让 AI 写出来的代码依赖少、可读性高，工程上比 Rust 更稳。

立场四 · Python 仍然不可替代（反方）

来自 deferredgrant：

Even if AI writes the first draft, humans still debug, review, and explain the code. Python's advantage is not just that it is easy to type.

翻译：「即便 AI 写初稿，最后还是人来调试、审查、讲解代码。Python 的优势不只是『打字快』。」这条反方观点强调：Python 的可读性优势不只对作者，对所有 review 它的人也有用。AI 时代的代码还是要给人看的，这条价值在 AI 接管打字之后并没有消失。

立场五 · 哲学反对（极端派）

来自 ulfw：

How is this comparable in any way? The recipient of the blog posts (all of us) can read English. None can read whatever this Logjam is. If AI writes code why not write it straight into assembler or binary? No need to compile an intermediate language if the end user (the machine) is running on binary not on Python, nor on Rust, nor on BASIC or Swift or any intermediary human-optimised language

翻译：「如果 AI 写代码不需要人读，干脆直接让 AI 写汇编或者二进制好了，何必停在 Rust？所有所谓『AI 友好』语言都是为人类设计的中间产物。」这条极端派的杀伤力在于把 Mitchem 的逻辑推到极端——如果代码不是给人读的，那连 Rust 也是多余的。

这五条立场放在一起，能看出 HN 社区的真实分布：支持 + 实操 + 折中 = 三种偏积极立场，加起来大概是评论池的多数，但反方与极端派也都拿到了显眼位置。换句话说，「AI 时代选 Rust / Go」这个判断在海外开发者社区已经从「少数派观点」走到「主流候选项之一」，但远没到「共识」级别。

把这件事翻译给国内同行，最有用的信号是：这场讨论的真正分歧不在于「Python 死不死」，而在于「咱们手上的工具链是不是应该多一条强类型路径」。这条路径在海外已经从理论走到工程，国内同行的工具链也已经基本具备。

六、国内 AI Coding 工具链 × Rust / Go：手上的牌已经够用

最关键的一节留给咱们国内的同行：DeepSeek Coder、千问 Code、通义灵码、豆包 Coder、Trae，这些工具对 Rust 和 Go 的支持到底怎么样？

把 2026-05 这个时间点的真实体验摆出来，避免假大空判断。

按工具拆开来看：

通义灵码（基于 Qwen Code）：阿里通义实验室出品。Qwen3-Coder 在 Rust 上的体验，根据 V2EX、知乎、X 上国内开发者的反馈，简单 cargo 工程能跑顺，借用检查、生命周期错误能修。Go 体验更好一点，标准库题型熟练。Python 是它的主场，没疑问。

DeepSeek Coder V3.2：深度求索出品。这一代 Coder 在国内开发者圈里的 Rust 口碑反而是几个里最突出的——长任务（比如「把这个 Python 包翻成 Rust」）的体验稳定，编译错误自我纠正能力强，能跑很长的「修一个错、编译、再修下一个」的链路。Go 上的并发、网络题也稳。

豆包 Coder（火山引擎）：字节跳动出品。火山引擎在主推这个产品时，明确把 TypeScript / Rust 作为重点目标，字节自家很多基础设施已经迁到这个组合上——咱们昨天聊的 Bun 6 天用 AI 把 Zig 翻成 Rust 那篇报道，背后的语言选择逻辑跟这条线是一致的。

Trae：字节系出品，IDE 内联体验。Rust 体验中等偏上，主要优势在 IDE 集成里的「写完一行立刻拿到编译反馈」这条路径，跟 Mitchem 强调的「紧反馈循环」契合。

国内通用对照：除了上面这四款国产工具之外，国内同行还可以通过 OpenRouter、Kimi 等中转接入海外模型，在 10K+ tokens 的长任务里持续维持类型一致性。这些海外工具在第二章已经讨论，本节只看国产工具自家在 Rust / Go 上的真实进度。

把这四个国产工具横向比，能得到三条对国内同行有用的判断：

第一，国内工具链对 Rust / Go 的支持已经从「能写」走到「能扛长任务」。一年前如果让国内 AI Coding 工具写一个 5000 行的 Rust 项目，大概率走到一半就会越改越乱。今天 DeepSeek Coder、豆包 Coder 这一档已经能跑通。

第二，强类型语言的「编译反馈循环」对国内模型同样有效。Rust 的 cargo check 反馈、Go 的 go vet 反馈，这些信号给国内模型纠错的帮助跟给海外模型一样大。这不是任何一家海外模型的专属能力。

第三，Python 仍然是国内 AI / 数据 / 脚本场景的主场，没有任何替代压力。DeepSeek、千问、豆包对 Python 的支持仍然是最稳的那一档。Rust / Go 是新增的选项，不是替代关系。

落到一句话：咱们国内开发者手上的牌，已经够用来做 Mitchem 文章里描述的那种「让 AI 写 Rust / Go」的工程实验。不需要等海外工具，不需要担心模型能力跟不上，工具链已经在了。

七、Python 的不可替代场景：另一半故事

最后回到一个克制的角度：Mitchem 的论点是「AI 时代多了一条强类型路径」，不是「Python 死了」。这一节必须摆出来——Python 仍然占着至少三块基本盘，这些盘子在可预见的未来不会被 Rust 抢走。

第一块：机器学习与深度学习研究。PyTorch 控制着大约 85% 的深度学习研究市场（Mitchem 自己原文里也承认这个数字）。整个 ML 研究生态——HuggingFace transformers、TRL、accelerate、deepspeed、vllm、SGLang，从模型定义到训练框架到推理引擎，最外层 API 全是 Python。这一层在 AI 时代不但不会衰退，反而是 AI 应用增长的直接受益者。

第二块：数据科学与分析。pandas、numpy、scipy、scikit-learn、matplotlib、seaborn 这一整套，加上 Jupyter Notebook 的交互范式，构成了数据团队、量化团队、金融团队、生物医药团队的日常。这些场景的特点是「探索性」——人来回试错、看图、改公式。这跟 Mitchem 强调的「AI 长任务写代码」是完全不同的工作模式。

第三块：脚本、胶水、教学。运维脚本、数据 ETL、自动化测试、小工具、教学示例，Python 的可读性优势在这些「人写人读」的小代码场景里仍然不可替代。Rust 在这些场景下的体验，即使有 AI 加持，仍然是杀鸡用牛刀。

把这三块基本盘摆出来，Mitchem 的论点就更精确了：不是 Python 让位，是「服务后端 + 系统软件 + 工具链」这一类原本属于 Python 的场景，正在分流到 Rust 和 Go。研究、数据、脚本三块，Python 仍然是主场。

对咱们国内同行的实操建议（这里只摆事实不摆动作清单）：

  • 做 ML 训练 / 微调 / 推理上层框架：Python，没有任何理由换
  • 做数据分析、Notebook 探索：Python，pandas 与 Polars 共存
  • 做 Web 后端 + agent infra：原来 Python 也能跑，现在 Rust / Go 多一个选项
  • 做高性能 CLI 工具 / 数据管道：Rust 更值得试
  • 做大型前端 monorepo 编译链：Go（TS 编译器走了这条路）值得跟看

八、回到核心论点

绕了一大圈，回到开头那条核心论点：Python 的可读性优势被 AI 抹平，强类型语言反而成为 AI 时代的最佳基础设施。

这条论点之所以引发 871 条评论的争论，是因为它同时挑动了两个心理按钮：一个是「我学了多年的 Python 会不会过时」，一个是「我一直不愿碰 Rust 的难学借口会不会被 AI 拿掉」。两种心理放在一起，自然会吵翻。

但如果把情绪拿掉，回到工程数据，剩下的事实其实很清楚：

  • Pydantic v2 用 Rust 重写跑出 5-50× 性能跃迁——这是 Python 生态自己做出的选择，不是别人逼的
  • TS 编译器迁 Go 跑出 10× 编译速度——微软在自家最大的 TS 仓库上验证过的真实减痛
  • Ladybird JS 引擎两周从 C++ 改 Rust——AI 辅助下，重写代码的工程节奏被压缩了一个量级
  • HN 上 823 顶赞 871 评论的分布——海外开发者社区已经在认真讨论这件事，不是边缘话题
  • 国内 DeepSeek Coder、通义灵码、豆包 Coder 等工具在 Rust / Go 上的体验已经够用——咱们手上的牌不缺

把这些信号串成一条逻辑链：AI 让强类型语言重新值得选，国内工具链也已经准备好，但 Python 的主场没有动摇。多一条路径，不等于换路径——这是 Mitchem 这篇文章最该被记住的那句话。

写到这里想说一句温暖一点的判断：咱们做 AI Coding 的同行，过去一年其实活在一个非常幸运的窗口里。一年前还在为「Cursor 能不能写好 Python」吵的圈子，今天已经在认真讨论「Claude Code 在 Rust 长任务里是不是比人更稳」。这种节奏感，是技术行业很久没见过的。多一门可选的工具语言，不是焦虑，是好消息——尤其是对那些一直被 Rust 的学习曲线劝退过的工程师，AI 给了一条「不必从零学起也能产出 Rust 代码」的合理路径。

国内同行的工具箱在变厚。能把工具用顺、把基础设施做扎实的人，永远是最稀缺的那一种。

九、几个值得追踪的后续信号

把 Mitchem 这篇文章读完，并不意味着判断就此封口。咱们国内同行如果想长期跟看这条线，有几个具体的信号值得加进观察清单：

第一，Python 工具链里下一个被 Rust 替换的组件是什么。这是最便宜的领先指标——Pydantic、Polars、orjson、ruff、uv 都已经走通，下一个候选可能是 SQLAlchemy 的查询层、Celery 的调度内核、或者 Airflow 的执行器。任何一个再换芯，都能给「Python 生态 Rust 化」这条趋势加一个新里程碑。

第二，国产模型在 Rust 长任务上的稳定性曲线。DeepSeek Coder、千问 Code、豆包 Coder 接下来几个月的迭代节奏，重点就看「能不能稳住 5000-10000 行 Rust 工程的连续编辑」。目前 V3.2 这一代已经接近能用，下一代如果再往前走半步，国内开发者「让 AI 写 Rust」就会从「值得尝试」变成「日常工具」。

第三，Anthropic / OpenAI / Google 自家用什么语言重写自家工具链。这件事的领先信号意义最强。微软已经用 Go 重写了 TS 编译器，OpenAI 的 Codex 工具链大量用 uv（Rust），如果 Anthropic 接下来公开把 Claude Code 的核心组件迁到 Rust，那基本可以认定「AI 公司自己内部也在按 Mitchem 这条判断走」。

第四，Stack Overflow 2026 调查会不会让 Rust 连续 11 年蝉联「最受喜爱语言」。2025 年 Rust 是 72% 喜爱度，Gleam 70%、Elixir 66%、Zig 64%——这些数字摆在一起，说明开发者社区对强类型 / 函数式 / 紧反馈这一组属性的偏好正在系统性上扬。

第五，国内大厂 Rust 招聘量曲线。字节、阿里、腾讯、美团、华为这五家如果在 2026 下半年同步开始增加 Rust 岗位招聘，那就是 Mitchem 判断在国内落地的最直接证据。火山引擎已经在打这张牌，其他几家是不是跟进，下半年就能看清楚。

把这五个信号放进自己的雷达，不需要每天盯。每过一两周扫一眼，就能持续校准判断。咱们国内同行真正的优势从来不是抢热点，是把热点变成持续可用的工具链——AI 时代选语言这件事，慢慢来，不慌。
