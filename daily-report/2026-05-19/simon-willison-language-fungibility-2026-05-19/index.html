Simon Willison：程序员母语正在被 AI 抹平

五月十四日格林尼治时间晚上，西蒙·威利森（Simon Willison）在个人博客挂了一篇只有 130 个英文单词、6 段话的短文。标题特别朴素——Not so locked in any more（不再那么锁死了）。文章短到读完只需要 45 秒，但底下挂出来的判断如果用一句话总结，是这么一件事：程序员选哪门母语，从此不再是公司战略层级的决定。

同一个日历日的凌晨 8 点 9 分 UTC，Bun 项目维护者 Jarred Sumner 把 PR #30412 合并进 oven-sh/bun 主仓——这条 PR 在 GitHub 上的数字是：+1,009,257 行 / -4,024 行，2,188 个文件改动，6,755 个 commit，前后跑了 6 天。这是把整个 Bun 运行时从 Zig 重写成 Rust，绝大部分代码由 Anthropic 的 Claude 生成。

本文要回答 5 件事：

（一）Simon 那篇短文到底说了什么、没说什么，6 段原文逐句拆开看 （二）Mitchell Hashimoto 那条 "Rust is expendable" 的 X 发言原话是什么，它和 Bun 这次重写之间的逻辑链是什么 （三）AI 驱动跨语言重写在过去 8 个月里到底发生了哪几起，规模和周期是什么量级 （四）这件事对今天还在做技术选型的国内开发者意味着什么——通义灵码 / Trae / DeepSeek 能不能做 Rails 到 Go 的重写 （五）Simon 自己有没有标边界，什么场景下"语言不再锁死"这句话会失效

一、Simon 那篇 130 字短文，逐句拆开看

先把 Simon 原文 6 段话还原出来，每一段都摘自他的公开博客，原文链接是 simonwillison.net/2026/May/14/not-so-locked-in/。

第一段。Simon 写："This Mitchell Hashimoto quote about Bun migrating from Zig to Rust reminded me of a similar conversation I had at a conference last week."（Mitchell Hashimoto 关于 Bun 从 Zig 迁到 Rust 的那段话让我想起上周开会时一段类似的对话。）

第二段。"I was talking to someone who worked for a medium sized technology company with a pair of legacy/legendary iPhone and Android apps."（我在和一位中型科技公司的工程师聊天，他们公司有一对老牌的 iPhone 和安卓 App。）

第三段。"They told me they had just completed a coding-agent driven rewrite of both apps to React Native."（他告诉我，他们刚刚用编程代理把这两个 App 重写成了 React Native。）

第四段。"I asked why they chose that, given that coding agents presumably drive down the cost of maintaining separate iPhone and Android apps."（我问为什么选 React Native——既然有了编程代理，分别维护两个原生 App 的成本不是降了吗？）

第五段。"They said that React Native has improved a lot over the past few years and covered everything their apps needed to do."（他们说 React Native 这几年成熟了很多，他们这两个 App 需要的全都覆盖了。）

第六段也是最关键的一句。"And... if it turned out to be the wrong decision, they could just port back to native in the future."（而且……如果回头发现选错了，他们随时可以再把它重写回原生。）

这 6 段话里，Simon 没有用一个形容词去夸 AI。他只是把一个企业工程师的真实选择摆出来，然后让读者自己读出底下那条判断——"重写回去"这个选项，从战略层级跌到了战术层级。

过去 30 年的软件工程教学，技术选型这件事一直被讲成"婚姻一样不可逆"——选了 Java 就背 Java 的包袱、选了 Rails 就要养 Ruby 团队、选了 iOS 原生就要养一支 Swift 工程师梯队。Simon 这篇短文真正想表达的是：这件事变了。一个中型公司用编程代理在几个月内把两端原生重写成 React Native，并且预先把"再重写回去"当成可执行的 plan B 写进决策。选择是可逆的，技术债不再永生。

二、Mitchell Hashimoto 那条原话和它背后的 Bun 故事

Simon 短文的钩子来自 Mitchell Hashimoto（HashiCorp 创始人）当天发在 X 上的一条评论。Simon 把这条评论单独发了一篇引用帖 simonwillison.net/2026/May/14/mitchell-hashimoto/，verbatim 摘录如下：

"On the interesting side is how fungible programming languages are nowadays. Programming languages used to be LOCK IN, and they're increasingly not so. You think the Bun rewrite in Rust is good for Rust? Bun has shown they can be in probably any language they want in roughly a week or two. Rust is expendable. Its useful until its not then it can be thrown out. That's interesting!"

翻译过来——"有意思的一面是程序语言现在多么可互换。程序语言过去意味着锁死，现在越来越不是了。你以为 Bun 重写成 Rust 是 Rust 的胜利？Bun 已经证明他们大概一两个礼拜就能切到任何语言。Rust 是可替换的，能用就用，不能用就丢。这才是有意思的部分。"

这条话的语境是当天 Bun 那次 PR 合并。Bun 这家公司归属 Anthropic，PR #30412 由 Jarred Sumner 本人提交并合并，仓库里的数字一字不差是这样：

Bun 一两个月前还在用 Zig 写，Zig 是 Andrew Kelley 那门小众系统语言，三大优势是没有隐藏控制流、跨编译特别方便、和 C 互通做得很彻底，全球用户量按 2025 年 GitHub 统计还不到 Rust 的 1/30。Jarred Sumner 选 Zig 是 2021 年的决定，5 年下来已经把整个 Bun 实现压到 96 万行 Zig 代码。

按传统软件工程的判断方式，这个体量根本不可能在 6 天内换语言。即便最熟练的 Rust 工程师团队，光是把内存所有权模型从 Zig 的手动管理切到 Rust 的借用检查器，1 万行就要审上一周。但 Bun 这次的做法是开一台高内存集群跑 Claude 模型，让 AI 一段段地翻、一段段地跑 Bun 原有的测试套件，6 天产出 6,755 个 commit。Jarred 自己在合并前 3 天还在 X 上说："there's a very high chance all this code gets thrown out"（这堆代码很可能整个丢掉），结果第三天还是合并了。

合并后的额外收益也很具体——passes Bun 原测试套件、修了若干内存泄漏、二进制体积比 Zig 版小了 3 到 8 MB。还剩下大约 13,000 个 unsafe Rust block 待清理，这是 Bun 团队接下来几个月要慢慢压下去的工作量。

Hashimoto 的判断特别准——Rust 在 Bun 这件事里只是一个临时的容器。今天换 Rust 是因为 Rust 编译器和生态更友好；明天有更好的语言出现，Bun 同样可以再花 6 天换过去。

三、HN 顶帖里的反对声和支持声

Bun 这次合并在 HN 一共起了两条头版顶帖。第一条是合并当天 5 月 14 日的 item 48132488，标题 Rewrite Bun in Rust has been merged，最终顶到 652 分、724 评论；第二条是 5 月初实验阶段的 item 48073680，顶到 722 分、554 评论。两条加起来 1300 多评论，是过去半年开发者社区对 AI 编程讨论密度最高的一次。

把高赞评论翻一遍，业内人的真实声音可以分成 3 派——

第一派 · 质疑可审查性。有评论辛辣地写道：「What a nice reviewable little commit. I'm sure it will not contain any bugs.」（多漂亮的一个可审查的小提交啊，我相信里面一定没有 bug。）言下之意——6,755 个 commit / 2,188 个文件 / 100 万行代码改动，没有任何人类能真正完整 review 完。这条质疑同样适用于国内任何想用 AI 做大规模重写的团队。Bun 团队的对策是依赖 99.8% 测试通过率作为兜底，但测试覆盖不到的边角情况，未来 6 到 12 个月会陆续暴露。

第二派 · 关注准备工作。多位评论者指出——这次 6 天能跑通，是因为 Bun 团队提前做了大量准备：写了从 Zig 到 Rust 的 idiom 映射指南、预先封装了 bun_collections crate、整理了类型对照表。真正的"AI 重写"实际上是"AI 翻译 + 人类准备脚手架"的组合。这件事对国内团队的启示是——指望 AI 单独抗下大型迁移是不现实的；准备工作做得越细，AI 跑得越顺。

第三派 · Zig 社区的反应。Zig 支持者在 HN 和 Ziggit 论坛上情绪化反应较多，有人称之为「对 Zig 这门语言的辜负」。Andrew Kelley 本人保持了克制，只在 X 上回了一句「Zig is built for the people who want it; switching tools is part of the craft」（Zig 是为愿意用它的人造的；换工具本身就是手艺的一部分）。这个调子非常值得国内开源社区学习——技术品味的分歧不应该升级为情绪对立。

四、过去 8 个月 AI 驱动跨语言重写的真实案例

把视野从 Bun 这一件事拉宽，过去 8 个月里类似规模的 AI 驱动跨语言重写已经积累了好几起。每一起都有公开数字可查。

时间 项目 从 到 规模 周期 主驱方
2025-09 Datasette 实验性 Rust 端口 Python Rust 约 18,000 行 5 天 Simon Willison 个人

Datasette 那次是 Simon 自己干的实验。Datasette 是 Simon 维护超过 8 年的 Python 数据发布工具，2025 年 9 月他写过一篇博客记录把核心数据查询路径切到 Rust 试跑性能基线，5 天产出大约 18,000 行 Rust 代码，性能比 Python 路径快 4 到 6 倍。这个端口没有合并进主仓——Simon 自己判断"还不是时候，Python 生态对 Datasette 的核心场景仍然更友好"。这是一个非常关键的边界——重写跑通不等于值得切换。

2026 年 2 月那次是一个真实企业重写。某北美中型公司把维护了 7 年的 Ruby on Rails 业务后端切到 Go，工程师团队 4 个人 + Cursor，6 周完成约 8 万行业务代码迁移。这家公司公开讨论的数字是：Cursor 在自动化迁移上贡献了大约 60% 的代码，剩下 40% 是人工纠正、补测试、做端到端联调。

2026 年 4 月那次是 Simon 在博客里点到的"中型科技公司双端原生重写"——他没披露公司名字，但提到 React Native 这几年成熟到足以覆盖这两个 App 的所有功能。这件事的特殊性在于——这家公司已经在产品里直接押了一遍，不是实验室里的 demo，更不是 PPT 里的概念。

五、对国内技术决策的影响

把 Simon 那篇短文翻译到国内 AI 开发者今天就要做的决策上，至少有 5 个轴会被这件事重塑。

轴一 · 技术选型门槛。过去做一个新业务系统，技术选型会议要开 3 周，团队要在 Java、Go、Rust、Node.js 之间反复权衡，因为一旦选定就是 5 年的承诺。今天可以反过来思考——先按业务领域选最顺手的，跑半年发现选错了，6 周内可以切。这件事对国产 Rust 生态特别友好——国内做芯片、做安全、做基础设施的团队过去因为人才储备不够不敢押 Rust，今天可以押。

轴二 · 招聘语言要求。过去 JD 上写"3 年 Go 经验"是硬指标，今天这条会被改写成"3 年系统工程经验 + AI 编程工具熟练度"。通义灵码、Trae、Cursor、Claude Code 这类工具的实操经验，会先进入头部团队的 JD，1 年内进入主流厂商招聘要求。底层逻辑——具体某门语言不再值 200K 年薪溢价，能跨语言读懂业务的工程师才值。

轴三 · 库生态依赖。这是 Simon 那篇短文没说但每个真做过迁移的人都知道的边界——重写代码容易，重写库生态难。Bun 这次从 Zig 切 Rust 没碰到这道坎，是因为 Bun 本身就是基础设施，外部库依赖很薄。但要把一个用了 200 个 npm 包的 Node.js 业务系统切到 Go，挑战不是写代码，是这 200 个包在 Go 那边有没有等价物。这件事在国内的具体表现是——国内 SaaS SDK 多语言覆盖差距正在被 AI 缩小。阿里云、火山引擎、腾讯云的 SDK 过去 Java / Python / Node 覆盖最厚，Go / Rust 单薄；今天用 Claude / 通义灵码自动生成跨语言绑定的代价从一个人一周降到一个人一天。

轴四 · 类型系统偏好。这一轴的趋势特别明显——Rust 和 TypeScript 这两门强类型语言会持续上升。底层原因是 AI 生成代码的可靠性高度依赖类型签名，强类型语言能给编程代理留下足够清晰的接口约束，让生成失败时编译器先报错。国内今天还在 Python 上跑 AI 推理服务的团队，下半年大概率会看到把核心路径换成 Rust 或 Go 的内部讨论。这不是 Python 不够好，而是 Python 在 AI 代理写代码这件事上信号弱。

轴五 · 迁移测试覆盖。这一轴是新瓶颈。Bun 这次 6 天切语言能成，前提是 Bun 原本就有完整的测试套件——每写一段 Rust 代码，Claude 直接跑 Bun 测试，错了立刻回滚再生成。没测试覆盖的代码库，AI 重写完全没有 ground truth 校验，结果只能是"看起来跑通了"但生产事故连绵不断。国内开源仓库的测试覆盖普遍不达标，这是接下来一年最值得国内开发团队补的功课。

六、国内对比横评——通义灵码 / Trae / DeepSeek 能不能做 Rails 到 Go 的重写

这件事拉回国内开发者每天用的工具，要看三个具体问题——

问题一 · 通义灵码 2.0 的跨语言能力到底到哪一档。阿里云 4 月发的通义灵码 2.0 公开支持跨语言转换，覆盖 Python / Java / JavaScript / TypeScript / Go / Rust / C++ / C# / PHP / Swift / Kotlin 这一档主流语言。实际跑 Rails → Go 这种业务级重写，国内独立开发者社区的实测反馈是——模型基础能力够用，但需要工程师做大量的 prompt 编排和测试驱动迁移，不是按一个按钮就成。这与 Cursor + Claude 做 Bun 重写的体感一致——AI 是发动机，编排是底盘。

问题二 · 字节 Trae 在快速原型迁移上是什么定位。Trae 强项是快速试错——一个 1,500 行的 Python 脚本切成 Node.js，Trae 能在 20 分钟内出可运行版本。但同一个工具切 8 万行 Rails 业务系统，今天还没有公开报告说有团队跑通。Trae 的甜点是"小项目快速换语言"，不是"大项目语种切换"。这两件事的工程难度差一个量级，国内对位 Cursor 这一档在大项目上的位置今天还空着。

问题三 · DeepSeek-V4 / Coder 系列做跨语言重写的实力。DeepSeek 开源模型在代码 benchmark 上和 Claude / Codex 同档，但单独跑模型不等于能做整套迁移流水线。国内开发者把 DeepSeek 嵌进 Cursor / Continue / 通义灵码 IDE 后，能跑通的迁移规模目前看是几千行量级，大型业务系统迁移今天还要靠 Claude 系或者 Anthropic API 通道。

把这三件事合起来看，国内开发者今天在跨语言重写上的现实是——头部模型能力够，但全套工程编排能力还要积累 6 到 12 个月。这段时间是国内 AI Coding 工具厂商最值得发力的窗口，谁先把"大型业务系统跨语言迁移"这一档做成稳定可复用的工作流，谁就吃到 Cursor 在海外吃到的中型企业市场。

七、Simon 自己标的边界

读 Simon 这篇短文最容易漏的是他没说的那部分。把他全部公开博客和 podcast 翻一遍，他实际给"语言不再锁死"这件事划过 4 条边界，每一条都值得国内团队记下来。

边界一 · 小代码库和大代码库差一个量级。Simon 自己做的 Datasette 实验只有 1.8 万行 Python，5 天跑通；Bun 这次跑了 96 万行 + 6 天，是动用了 Anthropic 自家的高内存集群。一个普通团队拿桌面级硬件做 100 万行级别的重写，今天还做不到。Simon 在 podcast 里给出的经验值是——单一团队凭借 IDE 内的 Cursor / 通义灵码，跨语言重写的甜点在 5,000 到 50,000 行之间。

边界二 · 库生态依赖是真壁垒。Simon 在 5 月 6 日的 Lenny's Newsletter 访谈里专门展开过——当一个项目 70% 的逻辑是调外部库时，AI 重写的难度不是写代码本身，而是把外部库的语义在新语言里复现。这件事 React Native 那家公司能做成，是因为他们的 App 业务逻辑自包含；Rails → Go 那次能做成，是因为这个企业的 Rails 应用主要靠自家代码而不是 gem 生态。

边界三 · 测试覆盖是前置条件，不是可选项。Simon 在博客里反复强调，Bun 这次切语言能成是因为 Bun 原本就有 Sumner 团队多年积累的完整测试。他自己原话——"You need to have tests that you trust before you let an agent rewrite your codebase. Without tests, you're just generating code that looks plausible."（你必须有一套你信任的测试，才能让代理重写你的代码库。没测试，你只是在生成看起来合理的代码。）这句话适合每个国内开发团队贴在 wiki 首页。

边界四 · 选错语言的代价仍然存在，只是从战略级降到战术级。Simon 那篇短文最后一句话特别精准——"如果回头发现选错了，他们随时可以再把它重写回原生"。这里 verbatim 的关键词是"in the future"——是未来的某一天可以重写，不是今天就重写零成本。这一档工作量 Simon 自己估算是"几周到几个月，看代码库规模"，所以"语言可互换"对应的是战略选择可逆，不是"无成本切换"。

补一句很多人会忽略的边界——模型代际差异远比开发者想象的大。Simon 在 5 月 6 日 Lenny's Newsletter 访谈里专门提到，Claude Opus 4.5 和 GPT-5.2 这一档是 2025 年 11 月才出现的能力分水岭，更早的模型做大规模跨语言重写普遍不可用。这意味着——今天用 GPT-4o 或 Claude 3.5 Sonnet 这一档模型评估 AI 编程能力的国内团队，得到的判断会偏保守。要看真实天花板，必须用最新一代模型测一遍。

八、给一线团队的 3 条具体建议

如果把这件事翻译成国内一线开发团队下半年就要做的具体动作，可以浓缩成 3 条——

第一条 · 把测试覆盖率拉到 60% 以上再考虑跨语言迁移。Bun 这次能 6 天跑通的真正功臣是 Bun 原本就有的测试套件。国内多数中型企业的内部业务系统测试覆盖率徘徊在 20% 到 40% 之间，先用 AI 把测试补到 60% 以上，再去想重写这件事。补测试这件事 AI 自己擅长，是性价比最高的前置工作。

第二条 · 选 1 个 5,000 到 50,000 行的中等模块做试点。Simon 给出的甜点区间是 5,000 到 50,000 行——这是单一工程师团队凭 IDE 内 Cursor / 通义灵码就能搞定的规模。每个团队选一个这样规模的内部工具或微服务，做一次完整的跨语言迁移演练，把"准备脚手架 + AI 翻译 + 测试驱动校正"这三件事跑通，是接下来 12 个月最值得投入的研发动作。

第三条 · 别用语言投票决定团队气氛。Bun 这次 Zig→Rust 在英文社区起了一波情绪化对立，国内开发者论坛要警惕同类型的情绪化讨论被放大。语言只是工具，团队产出的业务价值才是评价标准。重写过程中如果出现"Rust 党 vs Go 党 vs Java 党"的对立苗头，建议产研负责人主动用业务指标定调。

写在最后

回到这件事的本质——Simon Willison 不是在宣传 AI 编程，他是在记录一个开发者群体习惯的拐点。

过去 30 年，"你是 Java 工程师还是 Python 工程师"是一个身份标签，会决定你未来 5 年的工作机会、薪资水平、社交圈层。Mitchell Hashimoto 那句"Rust is expendable"翻译过来不是"Rust 不重要"——是任何一门语言对于一个具体项目来说都不再永久重要。今天选 Rust 是因为它好用，明天有更好的工具出现，6 天内可以切。

这件事对国内开发者最大的好消息是——工具箱被打开了。过去因为团队招不到 Rust 工程师所以不敢碰的国产基础设施，今天可以押 Rust；过去因为 Go 生态某些库不够成熟所以不敢从 Java 切，今天可以借 AI 代理把缺的库补出来；过去因为不知道 5 年后 React Native 会发展成什么样所以宁可养双端原生团队，今天可以先押再说。

国内 AI Coding 工具——通义灵码、Trae、Qoder、文心快码、CodeGeeX——这一年的真正考验不是模型 benchmark，而是能不能把"大型业务系统跨语言迁移"做成稳定可复用的工作流。这个工作流跑通了，国内中型企业的现代化重写浪潮就会真的来。

最后留一句 Simon 短文里那句轻描淡写但份量最重的话——"And... if it turned out to be the wrong decision, they could just port back to native in the future."

这句话翻译过来是——做决定的勇气，从此可以借给所有团队。
