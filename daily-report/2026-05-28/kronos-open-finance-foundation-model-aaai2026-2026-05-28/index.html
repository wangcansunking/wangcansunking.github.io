一、26k stars 一周飙起：金融领域为什么需要一个基础模型

shiyu-coder/Kronos 这个仓库 2025-07-01 创建，到 2026-05-27 整十个月，stars 累计 26,832；今日 GitHub Trending 上涨 402 颗星，排在 Python 仓库榜第 8 位。这个增速放在 AI Coding / Agent 项目里不算最快，但放在「金融 + 开源 + 学术论文」这个交叉细分领域属于稀有事件——同类项目过去三年几乎没有一个能突破 1 万 stars。

为什么是现在？要回答这个问题，得先理解金融时间序列建模过去十年的窘境。

问题一：每家机构都从零训自己的小模型

国内私募和券商自营基本都养着一支量化研究员队伍，干的活有 70% 都是「针对某只标的某段时间窗口训一个专门的小模型」。LSTM 派的训 LSTM，注意力派的训 Informer，最近两年又开始往 PatchTST 上换。换一次架构就得重新跑一轮数据清洗、特征工程、超参搜索，半年时间往往就过去了。

更尴尬的是，这些模型彼此互不通用。在 A 股上训出来的 PatchTST 拿到港股几乎不能用；在分钟级数据上训的模型搬到日线就要重训。每个团队、每个市场、每个频率都得来一遍——这是典型的「没有基础模型」时代的状态，跟 GPT-3 之前自然语言处理那种「每个任务一个 BERT 子分支」是同一种结构性问题。

问题二：开源世界长期没有可用的金融底座

自然语言处理领域有 LLaMA、Qwen、DeepSeek 这一堆开源底座，搞 NLP 应用的团队可以拉下来直接微调。视觉领域有 CLIP、SAM、DINO。但金融时间序列长期没有等价物——闭源那边有彭博自家的 BloombergGPT、摩根大通的内部模型，国内蚂蚁 / 平安 / 招商也都有，但全部不公开。

这导致一个奇怪的局面：金融行业是最早大规模用机器学习的行业之一（高频交易圈早在 2010 年就开始上量化模型），但「拿来即用」的开源底座却一直缺位。Kronos 这次填的就是这个空白。

问题三：金融数据高噪音，通用时间序列模型水土不服

近一两年学术界有人尝试用通用时间序列基础模型（比如 TimeGPT、Lag-Llama）去做金融预测，效果都不理想。核心原因是：金融数据的噪声远高于气象、交通、电力这些其他时间序列；同时金融数据带有 fat tail、波动率聚集、跳跃等特殊统计性质，通用 tokenizer 完全 capture 不到这些。

Kronos 给的答案是「专门为金融数据设计 tokenizer」——这一步是它和其他时间序列模型最大的区别点，下一节专门讲。

二、技术核心：把 OHLCV 量化成离散 token 这一步的工程价值

K 线数据的原始形态是 OHLCV 五维浮点向量：开盘价（Open）、最高价（High）、最低价（Low）、收盘价（Close）、成交量（Volume）。每一分钟 / 每一日产出一根 K 线，按时间序列排开就是一支股票或一份合约的完整历史。

LSTM / Informer / PatchTST 这一派模型的做法是：直接把这五维浮点数输入网络，让网络自己学。Kronos 的做法不同——它先训一个 tokenizer，把每根 K 线量化成一组离散 token，然后让 Transformer 在 token 序列上自回归预训。

为什么要先量化成离散 token

这一步的设计灵感直接来自大语言模型那一套。语言模型之所以能 scale 起来，很大程度上是因为「文本天然就是离散 token 序列」，softmax 输出的概率分布意义明确、loss 容易优化、采样过程清晰。时间序列要享受这一套红利，得先把连续值离散化。

视频生成里 VQ-VAE 也是同样思路：把连续像素压成离散 codebook 索引，让 Transformer 在 token 序列上做预训。

Kronos 把这一思路引入金融时间序列。但金融数据的噪音特性决定了不能直接套 VQ-VAE——成交量可能跨越几个数量级（小盘股几千手到大盘股几千万手），价格振幅在不同标的之间差异巨大，简单的均匀 codebook 会被高频低幅噪声吃光。它的做法是「层次离散化」（hierarchical tokenization），论文里把这一步叫做 K-line specific tokenizer，针对金融数据的统计特性做了三处特殊设计：

  1. 分位数归一化：每个 OHLCV 维度单独做基于历史分位数的归一化，而不是用全局 z-score
  2. 层次 codebook：用多层 codebook 表达不同粒度的特征，粗粒度 codebook 记录大趋势，细粒度 codebook 补充局部细节
  3. 噪音感知量化：在 codebook 选择阶段引入噪音感知，避免把无意义的微小波动量化成独立 token

三、Decoder-Only Transformer 在时间序列上的取舍：vs LSTM / vs Informer / vs PatchTST

Kronos 第二阶段用的是 decoder-only Transformer 自回归预训——这套架构跟 GPT、LLaMA 同款。但时间序列领域过去十年的主流不是这套，所以有必要拆开看 Kronos 这个选择的得失。

对照一：vs LSTM

LSTM 是 2015-2020 年金融时间序列建模的标配，门控机制天然适合处理变长序列、记忆短期依赖。但它有两个硬缺点：训练慢（串行计算，无法充分利用 GPU 并行），长程依赖捕捉能力随序列长度衰减。在长会话上下文窗口越来越重要的今天，LSTM 已经不是好的选择。

Kronos 选择 decoder-only Transformer，等于直接放弃 LSTM 这一派——并行训练效率高、长上下文窗口可扩展、scaling law 清晰。

对照二：vs Informer

Informer 2021 年发表，专为长时间序列预测设计，靠 ProbSparse Attention 把注意力复杂度从 O(N²) 降到 O(N log N)。在公开评测集上 Informer 一度是同类最佳水平（SOTA），国内不少券商研究院都跟过这条线。

但 Informer 有个隐患：ProbSparse 只保留 Top-k 关键 query，对金融数据这种「关键信息常常埋在非常微弱的成交量异常里」的场景反而不友好，容易丢掉对预测最有价值的稀疏信号。Kronos 用标准 Transformer 注意力，配合金融专用 tokenizer 把序列长度本身压缩，从另一条路径解决了长序列效率问题。

对照三：vs PatchTST

PatchTST 2023 年发表，把时间序列切成不重叠的 patch 再输入 Transformer，受 ViT 启发。这是当下学术界最热的时间序列基础模型架构之一。

Kronos 跟 PatchTST 的关键区别是：PatchTST 用线性投影把每个 patch 映射成连续 embedding，本质还是「连续值上做注意力」；Kronos 用离散 tokenizer 把每根 K 线量化成 token，本质是「离散 token 上做注意力」。后者更接近大语言模型的预训范式，理论上能享受到语言模型 scaling law 那套红利——更大模型、更多数据、loss 持续下降。

Kronos 选 decoder-only 的代价是什么

代价是 encoder-only 模型在某些下游任务上可能更高效——比如做单点价格预测，encoder 一遍就能出结果，decoder 要自回归生成多步。但 Kronos 的目标是基础模型，要支持的下游任务从单步预测到多步预测、从生成式建模到条件采样都有，decoder-only 是更通用的选择。

四、4 个开源版本怎么选：mini / small / base 各自适配场景

讲完技术架构，回到工程师最关心的问题：实际项目里要选哪一档。这一节按典型团队规模和硬件配置给出选择建议。

个人量化交易者：Kronos-mini 4.1 百万参数

参数量 4.1 百万的 mini 版本可以在 CPU 上推理，下载下来 16GB 内存的笔记本就能跑。适合的场景是：

  • 个人玩家想做一个简单的多空择时模型，比如对沪深 300 ETF 给出未来 5 分钟方向预测
  • 用自己抓的少量 K 线数据做微调，试验不同特征组合
  • 在 Jupyter Notebook 里做学术性质的小实验，不打算上生产

HuggingFace 上 NeoQuasar 账号下有完整的 mini 模型权重，配套的 tokenizer 也在同一个账号下。

中型量化团队：Kronos-small 24.7 百万参数

参数量 24.7 百万的 small 版本，单卡 4090 / 5080 / 3090 就能舒服微调。适合的场景是：

  • 5-10 人量化团队，跑 A 股全市场分钟级数据，做中频策略
  • 团队内部有自己积累的 alpha 因子，想用 Kronos 当底座再叠加因子做监督微调
  • 推理延迟要求在百毫秒级以内，不上 H100 集群的小机构

私募和券商自营：Kronos-base 102.3 百万参数

参数量 102.3 百万的 base 版本是论文推荐的基线版本。多卡 GPU 微调（4 张 4090 或单张 H100）、跨市场对冲、可上生产。适合的场景是：

  • 中大型私募，资产管理规模 10 亿元以上，量化研究员 20 人以上
  • 券商自营业务，需要在 A 股、港股、美股之间做跨市场套保（这里说的是 hedging，不是其他意思）
  • 需要把 Kronos 接入自家因子库、风险管理系统、订单管理系统的完整工程链路

机构客户的 large 闭源版本

参数量 499.2 百万的 large 是闭源的，需要向论文作者团队申请授权。从论文披露的信息看，large 版本在多个公开评测集上比 base 版本有显著精度提升，但需要至少 4 张 H100 才能舒服微调。这一档不是开源版本能直接拿到的，本文不展开。

跨市场训练的工程价值有三层：

价值一：让模型见过更多分布

只见过 A 股的模型遇到熔断、涨跌停板、T+1 等特殊机制天然敏感；只见过美股的模型遇到 A 股的政策驱动就懵。跨市场训练让模型同时见到不同制度、不同流动性、不同波动率特征的数据，泛化能力更强。

价值二：天然支持迁移学习

国内私募拿 Kronos base 做微调时，可以用「先 A 股全市场 fine-tune，再某只标的 task-specific tune」的两步走方案——这是典型大语言模型的 instruction tuning 思路，迁移到金融领域。

价值三：跨市场对冲场景天然可建模

想做 A 股 / 港股 / 美股之间跨市场对冲的团队，过去得训三个独立模型再拼接；用 Kronos 可以直接在同一底座上拿三个市场的同一只标的（比如阿里巴巴 9988.HK + BABA + 影子代码）做联合训练，模型天然理解三个市场之间的价格联动。

六、国内对位：蚂蚁、平安、招商的量化模型为什么都闭源

讲完 Kronos 本身，回到国内视角。同类工作国内有没有？答案是有，但全部闭源。

蚂蚁集团：蚂蚁的财富管理业务部门有一支专门做时间序列建模的算法团队，公开论文显示他们 2023 年开始研发自家的金融基础模型（内部代号未公开），用于支付宝财富页面的基金推荐、收益预测、风险评估。这套模型只在蚂蚁内部使用，不对外提供 API、不开源权重。

平安科技：平安集团旗下平安科技多年深耕保险精算和资产管理领域的时间序列建模，2024 年公开过一个名为「PingTime」的内部模型，用于平安证券的程序化交易和平安信托的资管业务。同样不开源。

招商证券：招商系的量化研究院 2024 年在中文学术期刊发过一篇关于「跨市场时间序列 Transformer」的论文，架构跟 Kronos 思路相似——但模型权重和训练数据都不公开，只发了论文。

为什么国内三家全部闭源？

第一层原因是「这是核心竞争力」。金融机构靠模型赚钱，开源等于把饭碗端出去，机构层面缺乏开源动力。

第二层原因是「合规顾虑」。国内监管对金融机构使用的算法模型有备案要求，开源后第三方拿去做不当用途（比如操纵市场、误导投资者）可能反向连累原机构。

第三层原因是「数据敏感性」。训练金融模型用到的高频 tick 数据、L2 委托数据在国内是受监管资源，机构往往跟交易所有数据使用协议约束，开源会触及协议边界。

Kronos 团队（看 GitHub 主页背景）来自学术机构，能开源是因为：训练数据用的是从公开市场抓的日线 / 分钟级 K 线（不涉及 tick / L2 这种敏感数据）、论文已被 AAAI 2026 收录（合规层面有学术身份背书）、模型权重发布在 HuggingFace 走的是国际通行学术协议。

对国内开发者来说，Kronos 的存在意味着：第一次有了一个可以拉到本地、可以用自家私有 K 线数据微调、可以嵌入自家系统的金融基础模型。这是过去蚂蚁、平安、招商的闭源方案不提供的能力。

七、本地部署落地：从下载 HF 模型到接私有 K 线数据 fine-tune 的实际路径

讲完为什么用，下一步是怎么用。这一节按「准备硬件 → 下载模型 → 拉取数据 → 微调 → 推理」五步走，给出实际可执行的路径。

第一步：准备硬件

按选择的版本不同，硬件要求不同：

版本 推理硬件下限 微调硬件下限 推荐配置
Kronos-mini 4.1M CPU 16GB 内存 单卡 3060 12GB 4090 24GB
Kronos-small 24.7M 单卡 3060 12GB 单卡 4090 24GB 双卡 4090
Kronos-base 102.3M 单卡 4090 24GB 4 卡 4090 / 单卡 H100 80GB 8 卡 H100

显存估算原则：推理需要约 4 × 参数量字节（fp32）；微调需要约 12-16 × 参数量字节（含梯度 + Adam 状态）。

第二步：下载模型

HuggingFace 上 NeoQuasar 账号下有完整的模型权重和 tokenizer。用 huggingface-cli 拉取（国内访问可能需要镜像，hf-mirror.com 或者公司内部代理）。

第三步：准备数据

国内 K 线数据来源主要有三档：

  1. 免费档：AkShare / Tushare Pro 免费版，覆盖 A 股日线、分钟线，适合个人玩家
  2. 付费档：Tushare Pro 付费版、聚宽 JoinQuant、米筐 RiceQuant，覆盖更细粒度数据，适合中型团队
  3. 机构档：万得 Wind、恒生聚源、Choice，覆盖 L2 委托和 tick，仅向机构开放

数据准备阶段需要做的事：

  • 统一时间戳格式（处理交易日 / 非交易日、停牌、复权）
  • OHLCV 五维数据按 Kronos tokenizer 的预处理脚本归一化
  • 切成训练集 / 验证集 / 测试集（金融数据严禁随机切，必须按时间切，否则会有 lookahead bias）

第四步：微调

Kronos 仓库提供了完整的微调脚本，基于 PyTorch + DeepSpeed。常见的微调策略有三种：

  1. 全参数微调：所有参数都更新，适合数据量大（百万 K 线以上）、硬件充足的团队
  2. LoRA 微调：只训低秩矩阵，参数量是全参数微调的 1/100，单卡 4090 也能跑 base 版本
  3. 冻结主干 + 训预测头：只训最后一层，最省资源但效果上限较低

国内中型量化团队推荐 LoRA 微调——在保留 base 版本通用能力的同时，用自家私有 K 线数据做少量样本适配。

第五步：推理与部署

推理阶段把模型加载到 GPU，输入历史 K 线 token 序列，让模型自回归生成未来 N 步预测。生产环境部署通常会做两件事：

  1. 量化加速：用 INT8 或 INT4 量化把推理速度提升 2-4 倍，精度损失通常可控
  2. 批量推理：把多只标的的预测请求合并成一个 batch，充分利用 GPU 算力

对接入自家因子库 / 风控系统的团队，建议把 Kronos 当做一个 alpha 信号源——它给出的预测概率分布加上自家的因子做组合优化，比单纯用 Kronos 预测直接下单要稳健得多。

八、编辑说：金融基础模型对国内量化团队意味着什么

Kronos 是一个迟到了三年但终于出现的金融领域开源底座。

迟到三年是因为：自然语言处理领域 2022 年就有了 LLaMA 这种公开权重的基础模型，视觉领域 2023 年有了 SAM，但金融领域一直没有。中间这三年金融建模一直在零散重复——每家机构都在重训自己的 LSTM 或 PatchTST，研究员的精力大部分耗在重复工程上，而不是在模型创新上。

终于出现是因为：AAAI 2026 这一届有学术机构愿意把训练好的 base 模型权重 MIT 协议开源，这给了产业界一个起点。它不是完美的——Kronos-large 仍然闭源、训练数据覆盖的中国市场细节程度有限、推理延迟还达不到高频交易级别。但它是一个起点，国内量化团队第一次有了一个可以拉到本地、自由微调、自由嵌入产品的金融底座。

对个人量化交易者：Kronos-mini 给了一个低门槛上手的工具，CPU 笔记本就能跑，下载下来一周时间够把自己的小策略升级一次。

对中型团队：Kronos-small 配单卡 4090 LoRA 微调，是过去想都不敢想的配置——用 5 万元硬件投入就能拥有一个跟蚂蚁内部模型同一档的金融时间序列基础模型，再叠加自家的 alpha 因子，竞争力立即提升一档。

对大型机构：Kronos-base 是一个完整的研究起点，跑跨市场对冲、做因子库扩展、训 task-specific 模型都从这里开始。即便自家有闭源模型，对照 Kronos 的开源实现可以重新评估自己技术路线的效率。

国内 AI 圈过去对金融领域的开源关注度不高，机器之心 / 量子位 / 36 氪过去 90 天对 Kronos 零报道——这跟金融机构闭门做研究的传统有关，也跟主流 AI 媒体对垂直行业 vertical foundation model 关注度不足有关。Kronos 之后这块大概率会陆续出现更多开源底座，比如医疗时间序列、工业时间序列、能源时间序列——每一个都可能成为该垂直领域的「LLaMA 时刻」。
