代码变白菜价后 10 条 AI 编程实战守则

一、AI 把写代码这件事打到白菜价之后

Drew Breunig 是数据与 AI 工程方向写了多年的独立博客作者，关注角度偏一线工程。他在 5 月 4 日扔出一篇短文，标题挑明问题：当代码已经便宜到接近免费，开发者还应该做什么。第二天文章被推到 Hacker News 首页，截至本文发出时 203 分 200 条评论，是过去一周英文圈讨论 agentic coding 最密集的一篇。

文章给出 10 条守则，以及一句很扎实的判词：agentic code is "free as in puppies"。这句双关来自开源圈的老梗——免费的小狗看似零成本抱回家，但喂养、生病、训练全是真金白银。Drew 的意思是，AI 帮你把"写代码"这件事打到了白菜价，但代码进入生产之后的支撑、安全、迭代成本，一分都没有省下来。

10 条守则总览先放在这里，后文逐条拆。

国内开发者读这 10 条会有一个特别的好处。Drew 写作时心里默认的工具栈是 Claude Code、Cursor 这一套；而国内同行已经有完整可用的同档替代——通义灵码（Qoder IDE）、字节 Trae、腾讯 CodeBuddy、月之暗面 Kimi Code、字节豆包 MarsCode、百度文心快码，再加上能用海外服务的 Claude Code 拼车与 Cursor 第三方通道。10 条原则在这套国产工具栈上怎么具体执行，正是这篇文章的写作主线。

把数据放上来对照一下：Cursor 月活已经突破 500 万，Claude Code 在 SWE-bench 编程能力评测上突破 80 分，国内字节 Trae 占 41.2% 市场份额，阿里通义灵码占 18.5%，百度文心快码占 12.3%。这套国内外加起来七八个产品的工具栈，是 2025 年这一年从无到有搭起来的——2024 年 5 月这个时候很多产品还没开始做，2026 年 5 月已经是一线开发者每天都用的工具。Drew 写的 10 条守则正好踩在这个节点上：当工具变多变好，工程方法论也要跟着升级。

二、10 条守则逐条翻译成国产工具栈动作

守则 1 实施即学习

Drew 的原话是：写代码会逼出 spec 里没想到的决策。看上去是在用 agent 实现一段功能，本质是在用实现这件事去检查需求文档的盲区。spec 写得越粗，第一次 agent 跑出来的 PR 就越能照出哪里没想清。

这跟传统 waterfall 思路是反的。从前一份 spec 写完会过三轮评审、十轮开会，目标是把所有歧义在写代码之前消灭。在 agent 协作模式下这个流程颠倒了——spec 写到 70 分就让 agent 跑一版，第一份产出本身就是最好的需求评审材料。哪里实现绕了，哪里 agent 选了奇怪的方案，往往就是 spec 没说清楚的地方。

放到国产工具栈上：一份草稿级 spec 喂给通义灵码的智能体模式，让它先一口气生成最小可用版；跑出来的代码不是用来上线的，是用来回看 spec 哪几条歧义、哪几条互相冲突。Trae 的 Builder 模式更激进一点，连项目脚手架都帮你搭好，正适合用在这种"先跑一遍看看需求漏洞"的场景。Kimi Code 长上下文的好处在跨文件 spec 上特别明显，能一次扫完一整个目录的 README 和注释，给出哪几处口径不一致的清单。

国内一线开发者圈子里这种用法已经成了默认习惯。在即刻、V2EX 关于 AI 编程的讨论里反复出现一个说法：spec 不是写出来的，是 agent 第一遍跑完之后改出来的。这个习惯进一步反推 PRD 的写法——产品经理交给开发的需求文档，从前要细到字段级别，现在反而提倡"先把核心目标、核心流程、核心约束写清"，剩下的实现细节交给开发者和 agent 一起跑出来。这种节奏在腾讯、字节内部已经有团队跑通了，反馈是需求迭代周期能压掉接近一半。

守则 2 频繁重构

代码便宜的第一个直接结果，就是重写的心理负担显著下降。Drew 强调这不是"为重构而重构"，是说当一个模块的结构开始让你觉得拧巴时，应该敢于让 agent 重新生成一版完全不同的实现去对比。

回看十年前的工程文化，"重写代码"几乎是技术债的代名词，资深工程师会用各种 SOLID 原则去捍卫旧代码不被推倒。这套审慎是有道理的——重写一次的成本以人天计。但在 agent 时代，重写一个 200 行的模块只要十几分钟，成本曲线变了，决策也应该跟着变。同样一段功能跑两版完全不同实现做对比，再选一版进生产，过去是奢侈品，现在是日常操作。

在 Cursor 或 Claude Code 里这一步成本最低，因为它们对 git 工作流的整合最自然——开新分支、跑一版、不满意就丢掉。国内 CodeBuddy 的 Craft 模式有同样的工作流取向，加上能切换 GLM、Kimi-Thinking、DeepSeek 多个底层模型，同一段需求让两个不同模型各跑一版做对比，是国产工具栈里独有的玩法。比如同一段订单状态机，让 DeepSeek 写一版、让 Kimi-Thinking 写一版，前者偏直接、后者偏完整，对比之后取自己最理解的那一版进 main 分支，剩下的丢掉就行。

需要小心的是 Drew 自己提的护栏：频繁重构不等于无序重写，必须配合守则 3 的端到端测试，不然每次重构都在原地打转。没有 e2e 兜底的频繁重构，只是把混乱换了个写法。

守则 3 押注端到端测试

这一条 Drew 说得最重：测试要测产品做了什么，不要测代码怎么实现。意思是别再花力气写细到方法签名级别的单元测试，那种测试一旦遇到重构就全盘失效。该把测试预算挪到 e2e 层——给一个真实输入、断言一个真实输出。

这个方向其实违反了过去十年单元测试派的主流意见。原本测试金字塔的标准教法是底层单元测试要厚、中间集成测试中等、顶层 e2e 薄。这个金字塔的隐含前提是写代码贵、跑测试便宜——单元测试是为了在写代码阶段尽早捕获错误。在 agent 时代这个前提被掀翻：写代码已经接近免费，e2e 测试反而成了人类对产品形态的最终把关。Drew 提倡的写法事实上把金字塔倒过来——e2e 厚、单元薄。

国产工具栈做 e2e 已经很成熟。前端走 Playwright 或 Cypress，国内云厂提供的 Browser Use、字节家的 Computer Use 这一系列方案也开始有人尝试用 agent 自动写 e2e 用例。后端 e2e 普遍走自家维护的接口快照库。Claude Code 在这块有一个隐藏优势：它的 Plan 模式天然倾向先写 e2e 再实现功能，相当于把 TDD 节奏嵌进了默认工作流。

特别值得提的是，端到端测试套件本身在 agent 时代也变得便宜了。一套覆盖核心用户路径的 Playwright 测试，过去要专门 QA 工程师花一两周写，现在让 agent 看 PRD 加一份 demo 视频，半天能拉出 70 分初稿。剩下的工作就是人类工程师把奇形怪状的边界场景补进去——这正是守则 6 说的"找真正难的事"。

守则 4 记录意图

Drew 区分得很干净：测试讲的是目标，代码讲的是方法，但都没说"为什么"。"为什么"才是项目延续的根。把意图和代码、测试一起留下，agent 在后续维护时才会沿着同一个方向积累决策。

举一个国内常见的反例。某中型电商团队半年前定了一条规则——优惠券核销必须走单独的服务，不能和订单服务耦合。当时是为了防止大促时优惠券逻辑改动牵动订单核心链路。半年之后这条规则的"为什么"没人记得，新进的工程师让 agent 重构时把两个服务合并了，看起来代码更干净，结果下一次大促直接出故障。这就是没有把意图和代码绑在一起的代价。如果当初在服务入口的注释里写一行"分离原因：大促隔离，详见 ADR-2025-031"，agent 重构时就会主动避开这个改动。

国产工具栈里这一步有个文化差异需要点破。海外团队习惯把 intent 写在 ADR（架构决策记录）里，国内团队更多用项目飞书文档或 Notion。两种载体都行，关键是 agent 能不能读到。Kimi Code 长上下文模式在这一条上有天然优势，可以把整本 ADR 喂给它。通义灵码的智能问答模式则配合企业知识库挂载——把 ADR 进知识库，agent 调代码时同步检索意图，是阿里大客户里成熟的玩法。

更轻量的做法是在仓库根放一个 INTENT.md 或 DECISIONS.md，每个重要架构决策一行一句话，再配一个 ADR 详链。这种文件极小、agent 读取成本极低、维护成本约等于 commit message。在用 Claude Code 或 Cursor 的国内团队里这种做法已经很常见，效果远好于把意图藏在飞书深处不更新的设计文档。

守则 5 spec 保持鲜活

Drew 用的词是 keep specs in sync，配的解释是：spec 不是项目立项时写一次的交付物，是要随着代码和测试一起更新的活文档。

这一条对国内中型团队最难。国内团队普遍把 spec 当一次性文档，写完锁进飞书的某个角落，半年不动。问题是 agent 没法读飞书里那份过时 spec，它只能从代码里反向猜测意图，结果就是每次新功能 agent 给出的方案都和 spec 漂移。漂移积累三个月，整个项目就会陷入"spec 一套、代码一套、谁都说不清哪个是真"的混乱。

Drew 给的对策是把 spec 文件也放进仓库，按 markdown 维护，与代码一起 review。具体可以是 docs/specs/ 目录下一个个 .md 文件，每次需求迭代时和实现代码一起 commit。国产工具栈里 Trae 和 CodeBuddy 都开始内置 spec 模板，每次 agent 改完功能会自动建议 spec 增量。这件事最省力的做法是：让 agent 在 PR 描述里强制带 spec diff，review 时一并看。

实践层面还有一个细节。spec 写得越偏目标层面、越避免实现细节，活得越久。如果 spec 把"用 Redis 做缓存"这种实现选择写死，半年后换成 Caffeine 内存缓存时 spec 就会变成累赘。如果 spec 只写"接口响应时间不超过 50ms"这种目标，无论底层实现怎么换，spec 都不需要重写。

守则 6 找真正难的事

Drew 列了一串"真正难"的清单：直觉上对的设计、性能、安全、韧性、系统级架构。这些事 agent 帮得上忙，但替不掉人。守则 6 的潜台词是——别因为 agent 把简单事情做完了，就把简单事情当成全部事情。

国内开发者的真实痛点正是这一层。国产 AI 工具能把 CRUD、表单、列表页几乎全自动化，但真正卡项目命门的事情，比如风控规则、数据权限、千万级表的分库分表、推荐系统的冷启动，依然需要资深工程师盯着。dbreunig 在原文里把这些问题归类成"架构与口味"，国内场景再加一条：合规边界。数据出境、个人信息保护、算法备案、未成年人保护——这些在国内做 to C 项目几乎绕不开的红线，agent 目前完全无法独立判断，必须由懂业务、懂法规的工程师拍板。

更具体地说，资深工程师在 agent 时代的产出方式发生了质变。从前他们的产出主要是代码，现在他们的产出主要是问题清单与判断。一个资深架构师每天可能只写几十行代码，但他给出的"这块逻辑要拆出来""这条 SQL 上线前必须加索引""这个状态机的 race condition 要补"这一类判断，决定了项目能不能稳。这些判断 agent 给不出，因为 agent 没经历过线上事故。

守则 7 把简单都自动化

跟守则 6 是一对：把简单的活全部交给 agent，省下时间投入难的活。Drew 的强调点在"全部"两个字——别留半自动状态，半自动等于每天都要手动一次，反而更累。

这条原则的反面教训很常见。很多团队上了 AI 工具之后只在写新代码时用，code review、commit message、changelog、依赖升级这些"看上去自己做也没多久"的事还在手动。结果一周下来 agent 节省的时间被这些半自动的小事吃光了。要彻底自动化的清单包括：commit message 生成、PR 描述生成、changelog 维护、依赖版本升级、单元测试模板、API 文档同步、错误处理样板、表单校验代码、DTO 转换、SQL CRUD。

国产工具栈能自动化到什么粒度，是 2026 年这一年的真实变化。豆包 MarsCode 在 IDE 插件里把代码片段自动生成、单元测试模板、API mock 数据这三件事打成了套件，开发者只需要按 Tab。通义灵码的智能体模式可以全程托管一类标准任务，比如"给后端接口生成对应的前端类型定义并提交 PR"。这种程度的自动化在 2025 年还要手工拼工具链，2026 年已经是开箱即用。

字节 Trae 在国内市占率冲到 41.2% 的核心原因之一，就是它把这些杂活打包得最狠——一个 Builder 模式覆盖了从需求文字到完整可运行项目的全链路。这种打包式自动化的代价是定制空间小，但对 80% 的中小项目正合适。

守则 8 培养品味

这是 Drew 这篇文章里最值得反复读的一条。他说外部反馈循环正在变快——agent 几秒钟给你一份方案——但产品上线之后的真实反馈循环并没有同步加速。所以开发者的内部判断力，他叫 taste，反而比从前更重要。如果你写代码的速度从一天 100 行变成一天 1000 行，但你判断哪 100 行真正有用的能力没跟上，最后只是多了 900 行待维护的代码。

品味在工程语境里不是审美问题，是一组隐性判断模式：知道什么时候应该退一步用更简单的方案；知道某个看起来优雅的抽象会在三个月后变成累赘；知道哪条 SQL 在数据量上来后会爆；知道这个接口设计会让前端同事骂街。这些都不是 agent 能教给你的，agent 只会给你"看起来合理"的方案——而"看起来合理"和"真正合理"之间，往往隔着两次线上故障的距离。

品味怎么练。Drew 没给方法论，但 HN 评论里有一条高赞回复说得很扎实：去看资深工程师 review 时画的红线、问的反问句，这些是没有被写进文档的判断模式。国内场景下，社区里有大量公开 review 现场——开源项目 PR 区、知乎有经验的工程师答题、即刻上一线 reviewer 的随手吐槽——都是低成本的"品味陪练"。

更主动的练法是：每次 agent 给完方案，自己先在脑子里跑一遍三个月后的状态。如果三个月后这段代码要扩展、要被新人接手、要承受十倍流量，它会出什么问题。这种刻意练习是培养品味最直接的路径。

守则 9 agent 放大经验

资深开发者总是低估自己 prompt 里带了多少直觉：用对了术语、给对了边界、控制了具体度。同样一个需求，资深工程师 prompt 跑一次能得到 80 分实现，新手 prompt 跑五次还在 40 分打转。Drew 说这就是 agent 的杠杆效应——它放大经验，而不是抹平经验。

举个具体例子。同样让 agent 实现一个支付回调，新手开发者会写"实现一个微信支付回调接口"，agent 给出能跑的代码但忽略幂等、忽略验签、忽略并发同单两次的处理。资深工程师的 prompt 会是"实现微信支付异步回调，注意幂等键用 transaction_id，验签按 V3 标准用平台证书，同单两次回调以第一次为准并日志告警"。同一个 agent，两个 prompt，产出质量天差地别。

这一点对国内招聘市场有直接影响。过去一年国内招聘端反复出现"是不是不再需要资深"的疑问，2026 年实测下来恰恰相反——资深工程师配 agent 一个人能顶过去三个人，资历溢价不降反升。重点不是"会不会用 agent"，而是"知不知道项目里哪些坑要提前在 prompt 里堵掉"。

对新人来说这一条听起来残酷，但 Drew 在原文里给了一个温和的视角：经验不是封闭的，它是可以加速积累的。一个新人主动去看资深同事的 prompt、去研究他们 review 的红线、去理解 agent 给出的方案为什么被打回，三到六个月就能把直觉积累的速度从过去的"以年计"压到"以季度计"。

守则 10 代码便宜 维护不便宜

回到那句金句：free as in puppies。这是开源圈的双关——免费的小狗看似零成本抱回家，但喂养、生病、训练、铲屎全是真金白银。Drew 把维护成本拆成两块——support 和 security——任何一块都不会因为代码是 agent 写的就自动消失。

更尖锐的是：agent 写出来的代码量是过去的 5 到 10 倍，维护表面积也跟着同比例放大。如果一个项目过去一年新增 10 万行代码，现在新增 50 万行，那么三年之后需要维护的不是 30 万行而是 150 万行。任何一个对维护成本不警觉的团队，三年之内会被自己的代码量压垮。

代码生命周期里，写作只占很小一块，下面这张占比表能说明问题。

写代码的占比从大约 30% 压到 8%，节省下来的预算并没有蒸发——它流到了维护、安全、合规这几个箱子里。国产工具栈在这一段的供给目前还偏弱，国内做 AI Code Review、AI 漏洞扫描、AI 依赖升级的产品远不如国外丰富。这里面有大量没被填的位置，对国内创业者是清晰的机会窗口。

具体能做的方向至少有四条：自动化的依赖升级机器人（对标 GitHub Dependabot 的国内版）、自动化的 AI Code Review（对标 Greptile / CodeRabbit）、自动化的合规扫描（数据出境、算法备案、敏感词，本土化优势天然存在）、自动化的安全漏洞修复（对标 Snyk 加 agent 自动 PR）。这四个方向 2025 年都还是早期，2026 年到 2027 年会陆续有国产产品冒头。

三、Drew 没说但中文场景必须补的两条

Drew 这篇文章是一线英语技术圈的视角，10 条守则放在硅谷场景下完全够用。但国内开发者每天面对的项目还有两层 Drew 没写到的现实：中文化的工程产物、以及多家国产模型如何组合搭配。把这两条补上，10 条守则才算真正能落到中文项目里。

中文化的 prompt 与产物

英文工具默认输出英文注释、英文 commit message、英文文档。在国内项目里这是个真实摩擦点——中后台代码注释要中文，commit message 要中文，PR 描述要带中文上下文。可执行的做法很简单：在仓库根放一份 .cursorrules 或 CLAUDE.md，明确写清"注释中文、命名英文、commit 中文、不出现港台用语"。通义灵码、CodeBuddy 这些国产工具默认就是中文输出，反而要注意它们偶尔会把变量名也中文化，需要在规则里反向约束。

进阶一点的做法是把团队术语表也写进规则文件。比如某电商项目里"订单"在数据库表叫 trade、在业务文档叫订单、在前端 URL 叫 orders，三套叫法各有历史原因。如果不在规则里把对应关系告诉 agent，每次它生成的代码都会在三套术语之间随机摇摆。把术语表显式写进规则之后，agent 的产出就会和团队约定保持一致。

国产模型链的组合定位

Drew 的文章里 agent 是个抽象概念。国内场景下要落到具体模型——不同模型擅长的活并不一样。一个目前在国内一线团队普遍用得顺手的搭法：长上下文跨文件分析交给 Kimi，SQL / 推理重活交给 DeepSeek，中文文档生成与 review 留给 Qwen，最后端到端的 agent 工作流主框架接 Claude Code 或 Cursor。多模型组合的成本比单挑 GPT-5 / Claude 显著更低，对国内开发者尤其友好。

CodeBuddy 在产品形态上已经把这个事情做出来了——一个 IDE 里直接可选 GLM、Kimi-Thinking、DeepSeek、Hunyuan 四个底座，这是 Cursor 在海外没有的体验。

更进一步说，国产模型组合还能解决一个海外团队不太遇到的问题：合规与数据驻留。海外团队默认所有 prompt 与代码都发到 OpenAI 或 Anthropic 服务器，国内团队在金融、政企、医疗这些场景下没法这么做。国产模型大多支持私有化部署或在国内云环境内调用，在合规这一项上有不可替代的位置。dbreunig 写文章的语境里没这层考虑，但这是国内开发者必须答的题。

四、共勉

Drew 这篇文章值得被国内开发者认真读，是因为它没有把 agent 神化，也没有唱衰开发者岗位。它给的判断很冷静：写代码这件事确实变便宜了，但这份便宜来到生产环境之后，会自动转成对工程素养的更高要求——架构、口味、判断、维护、合规。

放在国内工具栈视角看，这反而是一个友好的时间点。国产工具栈在过去 12 个月把"基础写代码"这一段补齐了，市占率第一的字节 Trae 拿下国内 41.2%，通义灵码在大型企业项目里站住了脚，CodeBuddy 把多模型选择带进 IDE，豆包 MarsCode 把日常自动化打包成套件。10 条守则在这套工具栈上一条不缺，全都能跑。Cursor 与 Claude Code 也通过第三方拼车与企业代理在国内一线工程师手里站稳了位置，海外旗舰能力可触达。

更值得提的是，国产工具栈还在持续往前走。CodeBuddy 的多模型 IDE 是国内独有的形态，Trae 把 Builder 模式的中文优化做到了海外产品做不到的地步，Kimi Code 长上下文给跨文件 spec 维护提供了别处没有的体验，通义灵码深耕大型企业项目，文心快码切政企赛道。这是一个国产工具栈各自把一段做到极致、合在一起就能覆盖国内开发者全部场景的格局。

剩下的真正考验，就是 dbreunig 那句"free as in puppies"——代码白菜价之后，谁能把维护、安全、品味这些不便宜的事做好，谁就握住了 2026 年这一轮的工程红利。这份红利在国内开发者面前是公平的：工具栈齐了、模型齐了、社区也齐了，剩下的就是把 10 条守则一条一条跑通，把每天的工程动作和 Drew 写的"为什么"对齐起来。

整理一下，这 10 条守则真正想说的其实是同一件事——agent 让"快"变得便宜，所以"对"和"久"才更值钱。
