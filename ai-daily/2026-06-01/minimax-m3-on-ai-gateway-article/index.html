奥特曼之外的中国 AI 出海：MiniMax M3 通过 Vercel 直达全球开发者

MiniMax M3 悄悄上线 Vercel：1M 上下文、原生多模态，但最狠的是那行 import 语句

2026 年 5 月 31 日，Vercel 的 changelog 更新了一条不起眼的记录。没有发布会，没有 keynote，没有 CEO 站台。就是一篇 1 分钟能读完的博文，附带两段代码示例。

但如果你点开那两段代码，会发现一件事：第一个示例的 prompt 写的是"复现这个 GitHub issue 里的 bug 并提交修复"，第二个示例传了一张失败测试的截图，让模型"识别根因并写 patch"。

这不是传统 API 文档里那种"写一首关于秋天的诗"或者"解释量子计算"的 demo。Vercel 的工程师在写这行 prompt 的时候，脑子里想的显然不是聊天机器人——他们想的是一个能直接干活儿的 AI 程序员。

而承载这个 demo 的模型，叫 MiniMax M3。

百万 token 的上下文窗口从来不是为了聊天记录——它是为了吃掉整个代码仓库。

谁在 AI Gateway 上架了，谁就拿到了开发者的第一行代码

Vercel AI Gateway 这个产品，2026 年的开发者应该不陌生。它的核心卖点就一句话：一个统一的 API 调用上百个模型，按延迟和成本自动排序，零加价，支持 BYOK。

但这句话背后藏着一个更深的逻辑。AI Gateway 不只是一个"模型路由器"，它是一个开发者心智的入口。当一个模型出现在 AI Gateway 的模型列表里，它面对的不是需要单独注册账号、单独读文档、单独集成 SDK 的繁琐流程——开发者只需要改一行 model 参数，从 'openai/gpt-5' 改成 'minimax/minimax-m3'，一切就绪。

这意味着什么？意味着上架 AI Gateway 的模型，获得的是最低摩擦的试用路径。开发者不需要离开他们已经熟悉的 Vercel 生态，不需要去 MiniMax 官网注册账号，不需要重新配置 API key（如果你在用 BYOK 模式），甚至不需要读 MiniMax 的 API 文档——Vercel 的 AI SDK 已经帮你把接口统一了。

这对 MiniMax 来说，是一次极其精明的渠道选择。2025 年到 2026 年上半年，中国 AI 模型出海的主流路径是两条：要么自己去海外建 developer relations 团队，要么用 AWS Bedrock / Azure AI 这种云厂商的模型市场分发。前者成本高、见效慢；后者要跟几十个模型挤在同一个货架上，还得忍受云厂商的抽成和品牌遮蔽。

Vercel AI Gateway 走了第三条路。它不抽成（"reflects provider pricing with no markup"），不遮蔽品牌（模型名就是 minimax/minimax-m3），而且它的用户群极其精准——全是前端和全栈开发者，正是最愿意尝试新模型、最可能把模型集成到生产环境里的那群人。

在 AI 模型供给过剩的 2026 年，渠道即护城河。

1M token 上下文 + 原生多模态：M3 的硬件参数终于追上了第一梯队

MiniMax 这家公司，国内 AI 圈的人不陌生。2024 年到 2025 年，他们在语音合成（TTS）和视频生成赛道上打出了声量，海螺 AI 的语音效果一度被拿来跟 ElevenLabs 对标。但在大语言模型这条线上，MiniMax 一直处于一个尴尬的位置：产品不错，存在感不强。

M3 可能是他们第一次在模型参数上真正站到台前。

根据 Vercel 博文的描述，M3 有三个关键升级：

第一，1M-token 上下文窗口。 这是 MiniMax 旗下第一个达到百万 token 级别的模型。2025 年 Gemini 1.5 Pro 率先把上下文窗口拉到 1M，随后 Claude 跟进，GPT-5 也在长上下文上做了大幅提升。到 2026 年中，1M 窗口已经成了"旗舰模型"的入场券——没有这个数字，你不好意思说自己是做 agent 的。

为什么 agent 需要 1M token？因为一个真正能在代码库里干活的 AI，需要同时看到项目结构、依赖关系、测试用例、issue 描述、历史 commit 记录。这些东西加起来，轻松超过 128K token。1M 窗口意味着模型可以一次性"吃下"一个中型项目的全部上下文，不需要开发者手动做 RAG 切分和检索。

第二，原生多模态。 博文里给的第二个代码示例很清楚：传一张失败测试的截图，让模型直接识别根因并写 patch。这不是"先 OCR 再分析"的两阶段流程，而是模型内部直接处理图像输入。对于软件工程场景来说，这意味着你可以把 UI bug 的截图、架构图、甚至手绘的流程图直接丢给模型，它自己理解视觉信息并映射到代码逻辑。

第三，MiniMax Sparse Attention（MSA）。 博文提到了这个技术名词，但没有展开技术细节。从命名推测，这是一种稀疏注意力机制，大概率是为了在 1M token 的超长上下文中控制计算成本。全注意力（full attention）在 1M 长度下的计算复杂度是 O(n²)，不做稀疏化根本跑不动。MSA 的具体做到未知，但"sparse attention"这个方向在 2025-2026 年已经是长上下文模型的标配——Gemini 用了，Claude 用了，开源社区也有 MInference 等项目在做类似的事。

有意思的是，博文里没有提 M3 的 benchmark 分数。没有 MMLU，没有 HumanEval，没有 SWE-bench。这在 2026 年的模型发布里相当罕见——通常每个新模型上线都要甩一堆跑分。Vercel 的博文选择只描述能力，不给数字。

这可能是一种刻意的策略：用场景代替跑分，用 demo 代替 benchmark。

软件工程、终端工具使用、多轮协作——M3 的定位不是"聊天"，是"干活"

博文里对 M3 能力描述的原文是这样的：

M3 improves on software engineering, terminal-based tool use, and agentic web browsing, and is tuned for multi-turn collaboration.

拆开来看，这四个词每一个都值得深挖。

"software engineering" —— 不是"代码生成"，是"软件工程"。代码生成是写一个函数、一个类、一个脚本。软件工程是理解整个项目、定位 bug、写测试、重构架构、提交 PR。Vercel 给的两个 demo prompt 都在这个范畴内：复现 GitHub issue 并提交修复，分析失败测试截图并写 patch。这背后需要的不仅是代码能力，还有对软件工程流程的理解。

"terminal-based tool use" —— 终端工具使用。这意味着 M3 被训练成可以直接操作命令行：运行测试、查看日志、执行 git 命令、安装依赖。2025 年 Claude 的 computer use 功能让模型可以操作图形界面，M3 的 terminal use 则是对开发者场景的垂直优化——毕竟大多数后端和 infra 工程师的主要工作界面就是终端。

"agentic web browsing" —— 自主网页浏览。这不是传统的"搜索并返回链接"，而是让模型像人一样浏览网页：打开页面、理解内容、点击链接、填写表单、提取信息。结合 terminal use 和 software engineering，一个可能的场景是：模型自己去 GitHub 上找到一个 issue，理解问题描述，浏览相关 PR 和文档，然后在终端里复现 bug、写修复、跑测试、提交 PR。

"multi-turn collaboration" —— 多轮协作。这个词在 AI 产品语境里经常被滥用，但放在 agent 场景下有特定含义：不是一问一答，而是模型在执行任务过程中持续跟人类开发者交互——遇到不确定的决策时提问，完成一个子任务后汇报，收到反馈后调整方向。这是一种"结对编程"式的协作模式，而不是"下命令-等结果"的黑盒模式。

这四个能力拼在一起，M3 的定位就很清晰了：一个面向开发者的 AI agent，不是聊天机器人。

这跟 MiniMax 过去的打法有明显差异。MiniMax 之前的主力产品海螺 AI 走的是 to C 路线——语音助手、视频生成、虚拟角色。M3 上架 Vercel AI Gateway，是一次明确的 to D（to Developer）转向。而且它没有选择去跟 ChatGPT 抢"通用聊天"的心智，而是直接切入了一个更细分、更刚需的场景：帮开发者写代码、改 bug、跑终端。

零加价、零数据留存——Vercel 的"中立平台"策略正在重塑模型分发

博文里有一段关于 AI Gateway 定价和数据政策的描述，看起来平淡，但细读很有意思：

AI Gateway reflects provider pricing with no markup and does not charge a platform fee on inference, including on Bring Your Own Key (BYOK) requests.

以及：

built-in custom reporting, Zero Data Retention support, dynamic provider sorting by latency & cost, and more.

"No markup"+"Zero Data Retention"+"BYOK"——这三个词组合在一起，Vercel 在告诉模型厂商和开发者：我不是中间商，我是基础设施。

2025 年到 2026 年，模型分发的中间层平台大量涌现。OpenRouter、Together AI、Fireworks AI、Groq——每家都在做"统一 API 调用多个模型"的生意。但大多数平台的商业模式是加价转售：从模型厂商拿批发价，加一层 margin 卖给开发者。

Vercel 选择了另一条路：零加价，靠其他产品（托管、边缘函数、数据库）赚钱。AI Gateway 本身不盈利，它是一个获客和锁客的工具——开发者把模型调用跑在 Vercel 上，自然而然会把前端、后端、数据库也放在 Vercel 上。

这对 MiniMax 这样的模型厂商来说意味着什么？意味着用 Vercel 分发模型，定价权完全在自己手里，不会被中间商吃掉利润。而且"Zero Data Retention"这个承诺，对于有数据合规要求的企业客户来说是一个关键卖点——很多企业不敢用 AI API，就是怕 prompt 和响应被平台拿去训练模型。

但这里有一个隐藏的张力：当 Vercel 的"动态 provider 排序"功能按延迟和成本自动选择模型时，M3 凭什么被选中？

AI Gateway 的 leaderboard 会根据 latency 和 cost 给模型排序。如果 M3 的定价不够低、延迟不够快，它在这个排序机制里就会被 GPT-5 和 Claude 压住，开发者根本看不到它。Vercel 给了 MiniMax 一个零摩擦的渠道，但也把它扔进了一个残酷的对比环境——跟 OpenAI、Anthropic、Google 的模型同台竞技，按性能和成本实时排名。

这对 M3 来说既是机会也是压力。机会在于，如果它的性价比真的够好，开发者会在 leaderboard 上自然发现它；压力在于，如果不够好，上架再多渠道也没用。

2026 年的中国 AI 出海：从"卖模型"到"卖 agent"

把 M3 上线 Vercel AI Gateway 这件事放在更大的背景里看，它代表的可能是一种范式的转变。

2024 年到 2025 年，中国 AI 模型出海的主要叙事是"我们的 benchmark 不输 GPT-4"。DeepSeek、Qwen、Kimi、MiniMax——每一家发布新模型的时候，都要甩出一张跑分表，证明自己在 MMLU、HumanEval、GSM8K 上跟 OpenAI 和 Anthropic 打平甚至超越。

但到了 2026 年，这个叙事开始失效。原因很简单：benchmark 饱和了。 当十几个模型在同一个测试集上都拿到 90%+ 的分数，用户已经分不清谁是谁。开发者选模型的逻辑从"谁分数高"变成了"谁能帮我干活"。

M3 的定位恰好踩中了这个转变。它不强调通用 benchmark 分数，而是在 Vercel 的博文里直接展示两个极其具体的开发场景：复现 bug 和修复测试。它不卖"智能"，卖的是"能帮开发者省多少时间"。

这可能是中国 AI 模型出海的一次叙事升级：从"我们也很聪明"到"我们真的能干活"。

但挑战同样明显。MiniMax 在海外开发者社区的品牌认知度远不如 OpenAI 和 Anthropic。2026 年 5 月，大多数海外开发者在选择模型时，第一反应还是 GPT-5 或 Claude。M3 要想在这个心智战场上撕开一道口子，光靠 Vercel 一个渠道是不够的——它需要更多的开发者关系投入，更多的文档和案例，更多的社区互动。

而且，agent 赛道的竞争正在白热化。2026 年上半年，OpenAI 的 Codex 已经深度集成了 terminal use 和 browser use，Anthropic 的 Claude 在 software engineering 上的口碑有增无减，Devin 等 AI 程序员创业公司也在快速迭代。M3 进入的是一个已经非常拥挤的市场。

但话说回来，那个 1M token 的上下文窗口和原生多模态，如果真能做到"吃下整个代码仓库 + 看懂截图 + 直接写 patch"，那它至少拿到了这场比赛的入场券。

戏谑收尾

我看到那两段代码示例的时候，脑子里闪过一个画面：一个开发者凌晨三点在 Vercel 的 dashboard 上把 model 参数从 'anthropic/claude-sonnet-4-20250514' 改成了 'minimax/minimax-m3'，然后打了个哈欠去睡觉。第二天早上醒来，发现 M3 已经把那个困了他三天的 bug 修好了，PR 都提完了。

这当然是一个浪漫化的想象。现实是，M3 大概率也会犯低级错误、也会在 terminal 里执行错误的命令、也会在浏览网页时迷失方向。但重要的是，那条从"改一行 model 参数"到"模型开始干活"的路径，已经被 Vercel 和 MiniMax 一起铺好了。

接下来就看 M3 自己争不争气了。
