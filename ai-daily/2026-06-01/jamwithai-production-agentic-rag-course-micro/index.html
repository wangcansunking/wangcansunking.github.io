GitHub 7 周 RAG 课程逆潮流：先关键词搜索，再上向量

当大多数 RAG 教程还在让你“pip install chromadb，两分钟搭一个聊天机器人”的时候，GitHub 上一个仓库默默攒出了 7 周的工程量——Docker、PostgreSQL、OpenSearch、Airflow、Redis、Langfuse、LangGraph，最后还接了个 Telegram Bot。

jamwithai/production-agentic-rag-course，项目名叫 The Mother of AI Project，Phase 1 代号 arXiv Paper Curator。仓库的 README 长达数千字，详细到每周的学习目标、架构图、API 端点表、make 命令清单。它不是那种“克隆下来 npm install 就能跑”的 demo，而是一个从基础设施到可观测性全链路打通的工程教学项目。

最狠的是它的核心理念：先做关键词搜索，再上向量。这在 2026 年的 RAG 教程圈，几乎是一种“逆潮流”的教学主张。

从 BM25 开始，而不是从 embedding 开始——这是“专业路线”和“玩具路线”的分水岭

这个课程最让我愣神的设计，是 Week 3 的标题：“Keyword Search First – The Critical Foundation”。在向量检索几乎成为 RAG 标配的当下，作者坚持让学习者先用一周时间啃 BM25 算法、OpenSearch 索引管理、Query DSL 查询构造，以及搜索质量指标（精度、召回率、相关性评分）。

作者在 README 里写了一句毫不客气的话：

We build RAG systems the way successful companies do – solid search foundations enhanced with AI, not AI-first approaches that ignore search fundamentals.

翻译过来就是：真正跑在生产环境里的 RAG 系统，底层是扎实的搜索引擎，AI 是增强层，而不是反过来。那些一上来就教你用 FAISS + OpenAI embedding 搭 demo 的教程，在作者眼里属于“忽略搜索基本功的 AI-first 路线”。

这个判断有行业数据支撑。2025 年以来，多家做企业级 RAG 的公司（包括 Weaviate、Cohere、Elastic）都在公开场合强调过“hybrid search 是生产环境的底线”。纯向量检索在专业术语、缩写、精确编码等场景下召回率会断崖式下跌（一篇 arXiv 论文标题里的“BERT”如果纯靠语义相似度，很可能被“RoBERTa”抢走排名）。BM25 的精确词匹配在这些场景下反而是兜底方案。

课程的 Week 4 才引入语义层：基于 Jina AI 的 embedding 生成，用 RRF（Reciprocal Rank Fusion）做关键词和语义结果的融合。这种“先 BM25 后向量”的顺序，在教学上制造了一个认知阶梯——你先理解了搜索的本质（倒排索引、词频统计、相关性评分），再往上叠加语义理解，才能看清每一层在解决什么问题。

Week 3 和 Week 4 的代码结构也体现了这个理念。src/services/opensearch/ 封装了完整的搜索服务做到，src/routers/search.py 提供 BM25 专用端点，到 Week 4 才新增 src/services/embeddings/ 和 src/routers/hybrid_search.py。API 设计上，单端点支持多种搜索模式切换，而不是粗暴地替换掉旧端点。这种向后兼容的迭代方式，本身就是一堂隐性的软件工程课。

7 周跑通一个“真·生产级”系统：从 Airflow 管道到 Langfuse 追踪，再到 Telegram Bot

如果你只看 Week 1 的架构图，可能会以为这是一个 DevOps 课程。学习者要在一周内搭好 FastAPI（端口 8000）、PostgreSQL 16（端口 5432）、OpenSearch 2.19（端口 9200 + 5601 的 Dashboards）、Airflow 3.0（端口 8080）、Ollama（端口 11434），全部用 Docker Compose 编排。作者甚至贴心地提供了一键健康检查命令：

curl http://localhost:8000/api/v1/health

Week 2 进入数据管道：arXiv API 集成（带速率限制和重试逻辑）、用 Docling 解析科学 PDF、用 Airflow DAG 做每日自动化论文抓取。这里有一个容易被忽略的细节——MetadataFetcher 是主协调器，ArxivClient 和 PDFParserService 各司其职，最后存入 PostgreSQL。这是一个标准的 ETL 微服务拆分，不是那种把所有逻辑塞进一个 Python 脚本的“教程写法”。

Week 5 引入本地 LLM（Ollama），完成完整 RAG 管线。作者在这里做了一个性能优化：用优化 system prompt，把 prompt 长度减少了 80%，速度提升了 6 倍。这个数字被明确写进 README——“80% prompt reduction (6x speed improvement)”。对于一个跑在本地的 LLM 来说，6 倍速度差异意味着从“等得心烦”到“可以对话”的体验跨越。API 设计上做了双端点：/api/v1/ask 是标准请求-响应，/api/v1/stream 用 Server-Sent Events 做流式输出。前端用 Gradio 搭了交互界面（端口 7861），参数可调。

Week 6 是整门课的“工业感”最重的一周。Langfuse 做全链路追踪（end-to-end RAG pipeline tracing），Redis 做精确匹配缓存（exact-match caching with intelligent cache keys and TTL management）。作者给出的性能数据是：缓存命中时速度提升 150-400 倍。这个数字夸张到让我怀疑是不是多写了一个零——但考虑到本地 LLM 推理的延迟通常在秒级，而 Redis 缓存的响应是毫秒级，150 倍确实在物理上成立。

Week 7 是整个课程的终章，也是最“2026 年”的一周。LangGraph 做 agentic RAG：Guardrail 节点做领域边界检测（防止外行问题触发幻觉）、Document Grading 做语义相关性评估、Query Rewriting 在检索结果不足时自动改写查询、Adaptive Retrieval 做多轮检索和智能回退。最后接上 Telegram Bot，把整套系统暴露为移动端可用的对话机器人。作者特别强调了一点——“Full reasoning step tracking for debugging and trust”。Agent 的每一步决策过程都要暴露出来，而不是黑盒输出答案。

技术栈本身也值得列一遍：FastAPI（异步 REST 框架）、PostgreSQL 16（结构化存储）、OpenSearch 2.19（混合搜索引擎）、Airflow 3.0（工作流调度）、Ollama（本地 LLM）、Jina AI（embedding 生成）、Redis（缓存）、Langfuse（可观测性）、LangGraph（agent 编排）、Gradio（交互界面）、Telegram Bot（移动接入）。外加 UV 做包管理、Ruff + MyPy + Pytest 做代码质量。这套技术选型拿出来，已经可以当一家中型 AI startup 的 infra blueprint 用了。

把 7 周课程拆开看，每一周都是独立的工程决策课。合在一起，是一个从“能跑”到“能生产”的完整路径。

课程完全免费，本地运行成本为零（Ollama 跑本地模型），唯一的外部依赖是 Jina AI 的免费 embedding API 和可选的 Langfuse 密钥。两个作者 Shirin Khosravi Jam 和 Shantanu Ladhwe 把仓库放在 MIT 许可证下，Star 曲线还在爬升中。

这门课不会让你两周速成，但它会让你知道，一个真正的 RAG 系统，从 BM25 的倒排索引到 LangGraph 的 decision node，中间隔着多少行代码和多少次 docker compose down -v。
