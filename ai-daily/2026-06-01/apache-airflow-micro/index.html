500家公司的选择：Airflow如何靠生态惯性抵御AI浪潮冲击？

如果你在 2025 年 5 月打开 GitHub Trending，Python 榜上躺着一个 2014 年诞生的项目——Apache Airflow。它没有刷榜的 AI Agent 框架那么性感，也没有大模型推理引擎那么时髦，但它刚刚发布了 3.2.2 版本，同时维护着 2.11.2 这条还在跑的旧主线，Python 版本支持从 3.10 一路拉到 3.14。一个十岁的老项目，依然是数据工程领域的事实标准。

这本身就是一个值得玩味的信号：在 AI 吞噬一切叙事的 2025 年，Airflow 这种“静态工作流调度器”为什么还在 GitHub Trending 上？

真正的护城河不是算法，是生态惯性。

Airflow 3.x 的沉默进化：Python 3.14 都支持了，但没人谈论它

Airflow 3.x 这个版本线在 2025 年 4 月 22 日首次发布，现在是 3.2.2，处于“Maintenance”状态。如果你只看 CHANGELOG，会觉得波澜不惊——但仔细翻 README 里的支持矩阵，会发现一个被所有人忽略的事实。

Airflow 3.x 支持的 Kubernetes 版本从 1.30 一路排到 1.35。PostgreSQL 从 14 支持到 18。Python 从 3.10 支持到 3.14。这不是一个“维护模式”项目该有的兼容性表——这是一个仍然在积极追赶基础设施前沿的项目。

但更有意思的是另一组数字。Airflow 2.x 的生命周期表里写得很清楚：2020 年 12 月 17 日首发，2025 年 10 月 22 日进入“Limited Maintenance”，2026 年 4 月 22 日正式 EOL。也就是说，全球大量企业跑在 Airflow 2.x 上的生产环境，还有不到一年的时间窗口来决定要不要升级到 3.x。

而 3.x 本身的状态是“TBD”——官方还没给出 EOL 日期。这种“新版本还在铺轨，旧版本已经开始倒计时”的时间线，对任何用 Airflow 管着几百条 DAG 的团队来说，都是一个需要开始认真对待的信号。

让我愣神的是另一件事：Airflow 至今仍然不支持 Windows 原生运行。README 里有一句轻描淡写的话——“The work to add Windows support is tracked via #10388, but it is not a high priority.”生产环境只推荐 Linux，CI 测试只跑 Debian Bookworm。一个 2025 年还在 Trending 的项目，居然在操作系统支持上如此“任性”，这反过来证明它的用户群体高度集中在 Linux 服务器端，根本不需要 Windows 的伪需求。

500 个已知用户，和那个“代码即流水线”的核心理念

Airflow 的 README 里写着一句话，放在 2025 年看格外扎眼：

When workflows are defined as code, they become more maintainable, versionable, testable, and collaborative.

这句话在 2014 年 Maxime Beauchemin 在 Airbnb 内部写下第一行 Airflow 代码时是前瞻性的。到了 2025 年，当 Cursor、Copilot、Claude Code 这类 AI 编码工具已经可以帮你写 DAG 文件时，“workflows as code”这个理念反而变得更锋利了——因为 LLM 最擅长理解和生成代码，而不是在 GUI 里拖拽节点。

Airflow 官方说他们知道大约 500 个组织在使用 Airflow，“但实际可能远不止”。考虑到它出现在 GitHub Trending 上，这个“远不止”应该不是谦虚。更值得关注的是官方的语义化版本策略——Airflow Core 本身遵循严格的 SemVer，但 Providers（比如 google 4.1.0 和 amazon 3.1.1）的版本号完全独立于 Airflow 主版本。这意味着你可以把 Airflow 2.1.2 和 google 4.1.0 provider 混装在一起，只要依赖约束允许。

这种“核心稳定、生态灵活”的架构设计，让 Airflow 在 3.x 迁移期有一个关键的缓冲——即使你暂时不敢动 Core 版本，Providers 仍然可以独立升级，继续对接最新的云服务 API。这对那些跑着几百条生产 DAG、不敢轻易动核心调度器的团队来说，是一个被严重低估的保险机制。

另一个容易被忽略的细节：Airflow 的依赖管理策略。README 里花了大量篇幅解释为什么默认不对依赖做上界限制（upper-bound），但对 SQLAlchemy、Alembic、Flask、Celery、Kubernetes 这五个“关键依赖”做了严格的上界约束。这种“抓大放小”的策略，本质上是在稳定性与灵活性之间做了一个精密的权衡——允许用户自由升级大多数依赖来适配自己的环境，但在数据库迁移、Web 框架、分布式执行器这些一旦崩了就会让整个调度系统瘫痪的组件上，保持铁腕控制。

这让我想起 Karpathy 最近反复强调的一个观点：软件工程的核心不是写代码，是管理复杂度。Airflow 用十年时间证明了，一个工作流调度器的终极竞争力不在于它支持多少种算子，而在于它在“不让用户的生产环境炸掉”这件事上花了多少心思。

当然，这并不意味着 Airflow 没有短板。README 里明确写了一句：“Airflow is not a streaming solution.”对于那些想要实时流处理的团队，Airflow 的建议是——去用专门的流处理工具，Airflow 只负责批量拉取数据后触发下游任务。这种“认怂”式的边界声明，反而比那些号称“一站式解决所有数据问题”的平台更值得信任。

至于 Airflow 还能在 Trending 上待多久，这取决于一个更根本的问题：当 AI Agent 开始自己编排任务时，还需要一个人类定义的 DAG 调度器吗？Airflow 的回答似乎是——让 Agent 来写 DAG 文件，然后 Airflow 来执行。毕竟，代码生成的下一站，是代码编排。而 Airflow 已经在那站等了十年了。
