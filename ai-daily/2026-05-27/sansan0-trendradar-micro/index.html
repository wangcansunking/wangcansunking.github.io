sansan0/TrendRadar：开源AI舆情工具358天获2000+星，支持MCP对话

ai-daily · 2026 年 5 月 26 日 21:46 · 来源：GitHub Trending python

2025 年 6 月 2 日凌晨 2:17，一个叫 sansan0 的开发者在 GitHub 上敲下了 TrendRadar 的第一行代码。当时这只是一个「给自己用」的小脚本——聚合几个平台的热搜，过滤掉不想看的噪音，推送到飞书。

他没想太多，甚至 README 都写得潦草。

三天后，100 个 star。又过了一周，200 个。到 2026 年 5 月 23 日发布 v6.8.0 时，这个项目已经迭代了 60 多个版本，拥有 73+ 位「天使轮赞助者」、11 个平台的聚合能力、8 种推送渠道、一套完整的 MCP 架构，甚至还有一个纯前端的可视化配置编辑器。

一个舆情监控工具，在 358 天里，从「个人小脚本」长成了「开源社区的现象级项目」。

开源世界里，最快的增长往往来自最朴素的需求——「我受够了刷屏，只想看真正重要的东西。」

30 秒部署，9 个渠道，1 个执念

TrendRadar 的核心逻辑简单到残忍：你告诉它关键词，它帮你盯 11 个平台（知乎、抖音、B 站、华尔街见闻、贴吧、百度、财联社、澎湃、凤凰网、今日头条、微博），有匹配就推给你。没匹配就安静如鸡。

但就是这个「简单」，让它在 GitHub 上拿到了 2000+ star。

我看到它 README 里那句「最快 30 秒部署的热点助手」时，第一反应是：又一个标题党。但翻完 Docker 部署文档和 GitHub Actions 的一键 fork 流程，我得承认——这可能是中文开源项目里，部署门槛最低的舆情工具之一。不需要服务器、不需要数据库（v4.0.0 之后支持本地 SQLite 和 S3 兼容的远程存储）、不需要写一行 YAML（v5.5.0 新增了可视化配置编辑器，直接浏览器里填表单）。

推送渠道的覆盖几乎是病态的——企业微信、飞书、钉钉、Telegram、邮件、ntfy、Bark、Slack，还有一个通用 Webhook 能对接 Discord 和 IFTTT。v5.0.0 之后甚至支持多账号推送，用分号分隔就能同时发给多个群。一个典型的「我全都要」式产品设计。

但真正的分水岭出现在 v3.0.0——MCP 架构的引入。

MCP：当舆情监控学会「对话」

2025 年 10 月 20 日，TrendRadar v3.0.0 上线了基于 MCP（Model Context Protocol）的 AI 分析服务器。这是我见过最激进的「开源工具 × AI 能力」整合之一。

它不是简单地接个 GPT API 帮你总结新闻——那是 v5.0.0 才做的事。MCP 模块提供的是 14 个工具函数（v3.2.0 增加到 21 个），让 Claude Desktop、Cherry Studio、Cursor、Cline 这类 AI 客户端直接「操作」你的新闻数据库。你可以用自然语言问：「上周 AI 相关话题的热度趋势是什么？」MCP 会调用 get_trending_topics、search_news、compare_periods 等一系列工具，在本地数据库中跑查询，然后把结构化结果返回给 AI 做自然语言输出。

这意味着舆情监控从「被动推送」进化到了「主动对话」——你不需要等推送，你可以直接问数据。

v3.2.0 新增的 read_article 和 read_articles_batch 工具更是把链路补全了——用 Jina AI Reader 读取文章正文，批量最多 5 篇，自动限速。这意味着你可以在 Cursor 里直接问：「搜一下今天关于 OpenAI 的新闻，读前 3 篇正文，帮我对比观点差异。」整个过程在 AI 客户端内闭环完成。

值得一提的是，这个 MCP 模块是独立于推送系统的。你可以只部署 MCP 服务（有独立的 Docker 镜像 wantcat/trendradar-mcp，端口 3333），完全不碰推送功能。这种「功能解耦」的设计思路，在个人开源项目里相当罕见。

到 v6.5.0，AI 能力进一步渗透到了「筛选」环节。你不再需要手动写关键词正则（虽然 v4.7.0 已经支持了 /pattern/ 语法和 => 别名），而是可以在 ai_interests.txt 里用日常语言写：「我想看 AI 和新能源相关新闻」。AI 会自动提取标签、对每条新闻打分（1-10 分），只推送高于阈值的。而且有自动回退机制——AI 筛选挂了，自动切回关键词匹配。

v6.5.0 还做了一个我特别喜欢的设计：不同时段可以用不同的筛选方式。早上用关键词快速过滤科技新闻，晚上换成 AI 兴趣描述做深度筛选。Timeline 配置让「推送策略」变成了一个可编程的东西。

一个开发者，358 天，60+ 个版本

翻完 TrendRadar 的更新日志，我最深的感受不是功能多，而是「迭代密度」。从 2025 年 6 月 2 日到 2026 年 5 月 23 日，平均每 5.9 天一个版本。v5.0.0 那一次，sansan0 在更新日志里写了一句：「致敬那个陪伴我两年多、却在刚续费后反手弹出『This organization has been disabled』的某 C 厂模型。」——说的是某家 AI 厂商突然封禁了他的账号，他被迫切换到 LiteLLM，结果反而统一了 100+ AI 提供商的接口。

这种「被逼出来的架构升级」在开源项目里比比皆是，但 sansan0 的处理方式很值得玩味：他没有抱怨，而是直接重构了 AI 模块（v5.3.0），把 provider 字段砍掉，改用 model: "provider/model_name" 的统一格式，还加了自动重试和备用模型自动切换。一个被平台「背刺」的开发者，最终做出了更健壮的系统。

v6.0.0 引入的 Timeline 调度系统也是同理——原本的 push_window 和 analysis_window 配置越来越复杂，他干脆重构成一套独立的 timeline.yaml，内置 5 种预设模板（全天候、早晚汇总、办公时间、夜猫子、自定义），还做了周视图时间线可视化。Breaking Change 标注得清清楚楚，迁移路径写在 README 里。

这种「对自己狠，对用户负责」的维护风格，在 73 位天使轮赞助者的留言里得到了印证——「这 agent 做得牛逼啊，兄弟」「开源不易，大佬辛苦了」「好项目，相见恨晚，感谢开源！」。最高一笔赞赏是 88 元，最低 0.1 元。sansan0 在 README 里写了一句：「金额随意，1 元也是对开源的鼓励。」

358 天，从一行代码到 60+ 个版本，从一个人到 73 位赞助者。TrendRadar 的故事不是「又一个 AI 工具」，而是一个开发者用最笨的方法——高频迭代、极限兼容、死磕文档——把一个个人需求做成了公共品。

有时候，最好的产品不是「颠覆性创新」，而是「我受够了，自己来」。
