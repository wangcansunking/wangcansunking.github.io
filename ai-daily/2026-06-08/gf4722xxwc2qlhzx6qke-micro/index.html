小红书Seal项目：用OpenClaw一周上线AI助理，全员覆盖

当小红书架构师孟洪进站上 AICon 2026 上海站的讲台，他手里握着的不是一份概念验证报告，而是一个已经跑完全员覆盖的 AI 个人助理系统。Seal——这个名字在 2026 年初还只是基于开源框架 OpenClaw 的一次快速实验，半年后，它成了小红书内部每一个员工日常办公的核心入口。

而同一时间，市面上绝大多数企业还在争论“Agent 能不能用”。

这中间的时间差，大概够一个中型团队做完两轮 PoC。

企业级 AI 的落地速度，从来不是技术问题，是组织胆量问题。

一周内测上线：选 OpenClaw 不选自研的底气

InfoQ 中国在 AICon 2026 上海站的议程披露中，把 Seal 项目的技术细节摊得很开。2026 年初，小红书团队启动 Seal 时做了一件反直觉的事：他们没有自研 Agent 框架，而是直接站到了开源项目 OpenClaw 的肩膀上。

当时 OpenClaw 刚爆发不久，社区生态远谈不上成熟。但孟洪进团队看到的不是“开源项目能不能扛住企业级负载”的疑虑，而是三个硬指标：多通道接入（Hi、飞书、Web 一把梭）、Skill 标准化机制、多 Agent 编排能力。再加上社区正在快速迭代的企业级安全隔离特性，这套组合拳让小红书在一周内完成了从选型到内测上线的全流程。

一周。这个数字放在任何一家千人规模以上的互联网公司，通常只够走完采购审批的前两个节点。

Seal 的技术架构在这个时间压力下做出了几个关键取舍。LLM Gateway 统一路由是第一条命脉——多云多模型接入不绑定单一厂商，推理请求按模型性价比自动分流。容器化多租户隔离是第二条命脉——每个员工一个独立的 Agent 实例，不共享上下文，不串数据。SSO 鉴权直接复用公司现有体系，工具调用时携带员工身份，权限管控不另起炉灶。

这套设计哲学很清晰：不要为了 AI 再建一套基础设施，让 AI 适配现有企业架构，而不是反过来。 这大概是小公司大 APP 型组织被逼出来的务实——小红书员工精简但职能复杂，HR、法务、商业化、研发的需求天差地别，任何需要各部门迁移工作流的方案都会死在推广阶段。

每人一个 Agent 实例：万级调度的工程账本

InfoQ 的报道里有一组数据让我反复看了三遍：Seal 已经做到全公司覆盖，支撑万级 Agent 实例的大规模并发调度。

万级实例意味着什么？不是一万个对话窗口，而是一万个独立的、携带不同身份上下文、调用不同工具链的 Agent 进程在同时运行。每个实例都有自己的记忆、自己的 Skill 配置、自己的权限边界。这在工程上的挑战比单纯的 QPS 要复杂一个数量级。

孟洪进团队在上下文成本控制上的做法尤其值得拆解。长对话叠加多工具调用，单次推理的 token 消耗很容易失控。他们的方案是“无损上下文裁剪”——这个词听起来像营销话术，但技术描述很具体：参考 JVM GC 机制，识别历史工具调用结果中的低价值信息，做压缩摘要而非粗暴截断，保留关键决策节点。同时用 LLM Gateway 的 auto 路由能力，让高性价比模型承接简单任务。

InfoQ 披露的效果数据是成本降低约 50%。假设一个日均万次调用的系统，单次推理成本从 0.02 美元压到 0.01 美元，一年就是 36.5 万美元的差距。这个数字在企业级部署里足够再养一个小型 AI 团队。

更值得琢磨的是 Skill Hub 的定位。孟洪进在演讲提纲里写了一句很关键的话：“Skill 的本质，是把人的经验变成可复用的 Agent 能力。”而且任何员工都能写 Skill，不需要懂代码。这意味着小红书把 Agent 的能力扩展权下放到了全员——法务可以把自己审合同的经验封装成 Skill，HR 可以把入职流程的检查点做成 Skill，商业化团队可以把投放策略的判断逻辑沉淀为 Skill。

这本质上是在用 AI 把隐性知识显性化，而且是众包模式。

从孤岛到入口：一场组织行为的静默革命

Seal 启动前，小红书的各职能团队已经在自建 Bot。HR 有自己的问答机器人，法务有自己的合同审查工具，商业化有自己的数据分析助手。这些 Bot 各自为战，形成了一座座信息孤岛——A 部门的 Bot 调不了 B 系统的接口，C 团队的 Skill 没办法给 D 团队复用，安全管控更是各自为政。

Seal 做的不是“再做一个更好的 Bot”，而是用一个统一的 Agent 入口吃掉所有碎片化需求。这背后的逻辑跟企业 IM 替代个人微信、统一身份认证替代各系统独立账号是同一条线：平台化才有规模效应，碎片化永远是小作坊。

InfoQ 的报道提到，本次 AICon 上海站的核心命题是“构建可信赖、可规模化、可商业化的 Agentic 操作系统”。小红书 Seal 的实践恰好卡在“可规模化”这个关键词上——从 0 到全员覆盖不是技术炫技，而是证明了一套架构能扛住真实组织的复杂度。

清华大学、复旦大学等高校教授，以及阿里、腾讯、蚂蚁、字节、快手、华为、Google Cloud 等数十家头部公司的技术专家也会在 6 月 26 日-27 日的上海站上分享各自的 Agent 工程化实践。2 天、13 大专题、近 60 场议题——这个密度说明一件事：Agent 工程化的讨论已经从“能不能做”转向了“怎么做才对”。

而小红书给出的答案是：别自研，用开源框架快速启动；别新建，复用企业现有基础设施；别封闭，把 Skill 生态开放给全员。这三条原则放在一起，本质上是在重新定义企业内部 AI 产品的研发范式——从“技术团队做工具给业务用”变成“技术团队搭平台，业务自己长能力”。

这大概比任何 benchmark 分数都更能说明 AI 落地的真实进度。
