告别系统提示词：三层记忆架构如何让Claude Code记住用户？

你是一个 AI 记忆系统。你的硬盘里存着 200 条 JSON 文件，每一份都记着用户说过什么、决定过什么、对什么价格敏感。你不需要联网，不依赖任何外部 API，你跑在一台 Android 手机的 Termux 终端里，Python 3.8+，零外部包依赖。用户打开 Claude Code，说了一句“之前聊过什么关于定价的话题”，你在 0.3 秒内扫完索引，吐出五条带标签和置信度的回忆。用户说“帮我把刚才的讨论记住”，你在 .memory/episodic/ 下新建一个 sess-20260607-143022.json，把决策、痛点、用户画像全写进去。

这就是 Memory Agent——一个在 linux.do 社区上由开发者放出完整源码、附 28.9 KB 压缩包和两张 Termux 终端截图的本地 AI 记忆系统。它做的事听起来平平无奇：让 LLM 记住跨会话的信息。但如果你仔细看它的架构设计和集成方式，你会发现，这可能是 2026 年上半年最被低估的 AI 基础设施实验。

一个只有 28.9 KB 的 zip 包，正在重新定义“AI 记忆”这个词在工程上的含义。

三层记忆架构：不是“存下来”，是“提炼出来”

Memory Agent 的数据模型非常清晰，分三层：工作记忆、情景记忆、语义记忆。工作记忆就是 LLM 的上下文窗口，每次会话自动携带，没什么好说的。真正有意思的是后两层。

情景记忆（Episodic Memory）对应每一次对话。每当你跟 AI 说“帮我记住”，系统会在 .memory/episodic/ 下生成一个文件，命名规则是 sess-YYYYMMDD-HHMMSS.json。打开这个 JSON，你会看到 11 个字段：summary 是必填的一句话概括，user_intent 记录用户的核心诉求，solution 记录给出的方案，key_points 是数组形式的关键信息点，decisions 记录达成的决策，open_items 记录待跟进事项，user_context 是一个嵌套对象，可以存行业、规模、痛点、预算范围、决策角色。还有一个 feedback_score 字段，默认 null，等用户反馈后再填。

这意味着什么？意味着 Memory Agent 记住的不只是“用户说了什么”，而是“用户在什么背景下、带着什么诉求、我们给了什么方案、做了什么决策、还有什么没做完”。这跟 ChatGPT 那种“记住我”功能完全不是一回事——后者只是在系统提示词里塞了几条用户偏好，而 Memory Agent 存的是结构化的、可检索的、带时间戳和标签的会话档案。

语义记忆（Semantic Memory）更进一层。当情景记忆积累到 5-10 条后，系统可以从多条会话里提炼出跨会话的规律和知识，存成 km-xxxxxxxx.json。每条语义记忆有 confidence 字段，初始值 0.5，随着用户反馈上下浮动。还有 source_sessions 数组，指向它来自哪些情景记忆，以及 applicable_scenarios 数组，标明这条知识在什么场景下适用。

举个例子：你在三周内分别跟三个不同的 SaaS 客户聊过定价，AI 从这三条情景记忆里提炼出一条语义记忆：“中小企业 SaaS 客户对 per-seat 定价接受度低，偏好 usage-based + tiered 混合，对封顶价格敏感”，confidence 0.85。这条知识会在你第四次跟新客户聊定价时自动注入上下文，AI 直接说“根据历史经验，你的客户群体大概率不喜欢 per-seat”——不用你再说一遍。

这本质上是一个轻量级的知识图谱引擎，只是它用 JSON 文件而不是 Neo4j 做到。

三种集成方式：Skill、CLI、MCP，以及一个隐藏的第四种

Memory Agent 的设计哲学是“数据格式即接口”。它不强迫你用一种方式访问记忆，而是提供了三种集成路径，外加一个隐藏的第四种。

第一种是 Claude Code Skill，已经注册在 .claude/skills/memory-agent/SKILL.md。开发者在 SKILL.md 里定义了六个触发条件：你说“帮我记住”，AI 走工作流 A，生成摘要存情景记忆；你说“之前说过”，AI 走工作流 B，查询记忆注入上下文；你说“总结一下”，AI 走工作流 C，批量提炼存语义记忆；你说“这个建议对不对”，AI 走工作流 D，更新置信度。整个过程对用户完全透明——你只需要用自然语言说话，AI 自己判断什么时候该存、什么时候该查。

第二种是 CLI 命令行。python3 memory.py store session '{"summary":"...","tags":["..."]}' 这种最原始的调用方式，反而暴露了这个项目最硬核的一面：它支持管道模式，支持从文件读取 JSON，支持在 shell 脚本里用 heredoc 传数据。你可以写一个每日自动总结脚本，用 crontab 定时跑，把当天所有会话摘要批量写入。你也可以用 jq 直接查 .memory/episodic/*.json，完全不经过 Python——因为数据就是纯 JSON 文件。

第三种是 MCP 协议。Memory Agent 用 .mcp.json 注册了一个 MCP 服务，由 memory_mcp.py 拉起。任何支持 MCP 的客户端——Claude Desktop、Cursor、以及未来可能出现的各种 AI 工具——都可以调用 7 个标准工具：memory_store_session、memory_store_knowledge、memory_query、memory_list、memory_get、memory_stats、memory_update_confidence。配置简单到只有三行 JSON：

"memory-agent": { "command": "python3", "args": [".claude/skills/memory-agent/scripts/memory_mcp.py"] }

（值得一提的是，这个 MCP 服务不需要手动启动——当 MCP 客户端启动时，会自动拉起进程。这比很多需要手动配置环境变量和端口号的 MCP 方案优雅得多。）

第四种方式是隐藏的，也是最激进的：直接读文件。Memory Agent 的官方文档里专门有一节叫“方式四：直接读文件（数据消费模式）”，给出了 Python 和 jq 的示例代码。这意味着你可以用任何语言、任何工具消费这些记忆数据，不需要用 Memory Agent 的任何接口。你可以用 cat .memory/episodic/*.json | jq '[.summary, .tags]' 一行命令提取所有摘要和标签，也可以写一个 RAG 管道直接把 .memory/ 目录当作文档库做向量检索。Memory Agent 不锁定数据，这是它跟绝大多数 AI 记忆方案最本质的区别。

数据格式即接口——这六个字说出来容易，真正做到的产品，整个 AI 生态里不超过五个。

0.2.0 的思维导图记忆：从“记住对话”到“记住进度”

Memory Agent 的 0.2.0 版本新增了一个叫“思维导图记忆”的模块，这个模块的野心比基础记忆功能更大——它要记住的不是对话，而是“项目开发到哪了”。

这个模块的入口脚本是 mindmap.py，数据落在 ~/.memory/mindmap/<项目名>.{json,md,mmd}。每条快照包含项目名、进度百分比、摘要、节点树（每个节点有 label、status、可选的 note 和 children）、踩过的坑（pitfalls）、下一步计划（next）。节点状态有五种：done ✅、doing 🔄、todo ⏳、bug 🐛、blocked ⛔。

最妙的是它的三位一体渲染机制。同一个快照可以以三种形态呈现：JSON 文件可检索（memory.py query --tag 项目快照 可以直接召回），Unicode 树可以在终端直接看（mindmap.py show 渲染成带 emoji 的树状图），Mermaid 源码可以渲染成真正的思维导图图片（mindmap.py mermaid 输出 Mermaid 代码，贴到任何支持 Mermaid 的工具里就能出图）。

更激进的是它的 SessionStart hook 设计。你可以在 ~/.claude/settings.json 里配置一个 hook，让 Claude Code 每次开新窗口时自动运行 mindmap.py hook，输出一个 JSON，把当前所有项目的进度树注入到新会话的上下文中。这意味着你打开 Claude Code，还没说话，AI 就已经知道你的项目 A 卡在模块 B 的某个子任务上——它会在第一句问候里直接说“上次模块 B 卡在 Y，要不要继续？”

这个功能让 Memory Agent 从一个“会话记忆工具”变成了一个“项目上下文引擎”。

我看到这个设计的时候第一反应是：这不就是 Notion 的数据库视图 + GitHub Projects 的进度看板 + Mermaid 的渲染能力，全部塞进一个本地 Python 脚本里了吗？而且它没有后端，没有数据库，没有同步服务——所有数据都在本地 JSON 文件里，用 Git 就能做版本控制和多设备同步。

商业模型场景：当记忆系统变成 CRM

Memory Agent 的文档里有一整章讲“商业模型场景”，这部分内容暴露了开发者的真实意图——他不仅仅是在做一个开发者工具，而是在做一个轻量级的、本地化的、AI 原生的 CRM 系统。

文档给出了三个完整的使用场景。第一个是客户画像管理：每次跟客户交互后，存储包含 user_context 的会话摘要，user_context 里可以填公司名、行业、规模、痛点、预算范围、决策角色。之后用 memory.py query --tag 客户 A 就能拉出这个客户的所有历史对话，用 --tag 客户画像 --tag SaaS 就能拉出所有 SaaS 客户的画像信息。

第二个是定价策略追踪：记录每次定价讨论的方案、竞品价格、用户倾向、最终决策。当定价讨论积累到 3 条以上，提炼为语义记忆，之后新客户聊定价时自动注入历史经验。

第三个是竞品情报积累：memory.py store knowledge '{"content":"竞品 X：per-seat $29/月，集成能力弱，客户抱怨多","tags":["竞品情报","竞品 X"],"confidence":0.6}'——这条知识的置信度只有 0.6，因为只有一条来源。当第二个客户也提到竞品 X 的集成问题时，置信度会更新到 0.75。当第五个客户提到时，置信度可能到 0.9，AI 会在后续对话中直接说“竞品 X 的集成能力在业内评价较低”——而这句话背后是五条独立来源的交叉验证。

文档还给出了一个推荐标签体系，包含七个核心标签：客户画像、定价策略、竞品情报、决策历史、增长策略、商业模式、技术评估。每个标签下还可以自定义子标签，比如客户名、项目名、竞品名。

（说实话，这个标签体系的设计比很多收费的 CRM 软件都要清晰。而且它不需要任何 onboarding 流程——你不需要先建客户档案、填一堆字段、设置权限，你只需要在跟 AI 对话时说一句“帮我把刚才的讨论记住，标签用：商业模型、定价策略、客户 A”，系统就自动归档了。）

当你的 AI 助手记得每个客户的预算范围和决策角色，它就不再是一个聊天机器人——它是一个销售副总裁的硅基分身。

数据安全与零依赖：一个反潮流的架构选择

在整个 AI 行业都在往云端迁移、往 API 经济靠拢的 2026 年，Memory Agent 做了一个完全相反的架构选择：本地运行，纯文本存储，零外部包依赖（CLI 模式下），零网络请求。

数据目录是 .memory/，结构极其简单：一个 _index.json 轻量索引文件，一个 episodic/ 子目录存情景记忆，一个 semantic/ 子目录存语义记忆。每个记忆是一个独立的 JSON 文件，命名规则清晰（sess-YYYYMMDD-HHMMSS.json 和 km-xxxxxxxx.json）。备份只需要 cp -r .memory .memory.backup-$(date +%Y%m%d)，迁移只需要设置 $MEMORY_DIR 环境变量指向新路径，清空只需要 rm -rf .memory/ 然后 reindex。

文档里明确写了一句：“数据是本地纯文本 JSON 文件。没有网络请求，没有外部依赖。”在 FAQ 里还补了一句：“1000 条以内无性能问题。超过后可考虑定期归档旧数据。”——这意味着开发者对性能边界有清醒的认知，不吹牛，不画饼。

这种架构选择在当下显得格外反潮流。OpenAI 的 Memory 功能跑在云端，数据存在 OpenAI 的服务器上；ChatGPT 的“记住我”功能需要联网，依赖 OpenAI 的 API；绝大多数 AI 记忆方案要么是 SaaS 产品（Mem、Rewind），要么是云服务（Pinecone、Weaviate）。Memory Agent 偏偏选择了一条最原始的路：本地 JSON 文件 + Python 脚本 + 零依赖。它的卖点不是“更聪明”，而是“你完全掌控你的数据”。

（这让我想起 2024 年 Obsidian 和 Logseq 掀起的本地优先运动——用户开始对云端锁定的 AI 工具产生抵触，转而寻求数据主权。Memory Agent 的出现，可能标志着本地优先运动从笔记软件蔓延到了 AI 基础设施层。）

一个社区开发者的野心：从 Skill 到标准

Memory Agent 的发布帖发在 linux.do，一个中文开发者社区。帖子里打了 #开源推广 标签，声明了完整开源、零未开源部分、已链接认可社区。附件是一个 28.9 KB 的 zip 包，两张截图显示 Termux 终端上的命令执行结果。

但如果你仔细看这个项目的文件结构和文档完整度，你会发现它不像是一个随手的 side project。它有完整的 README、REFERENCE、SKILL.md，有 CLI 命令大全（覆盖 store、query、list、get、update confidence、stats、reindex 七个子命令），有 MCP 工具大全（七个标准工具，每个都有参数说明），有故障排除 FAQ（六个常见问题和解决方案），有最佳实践指南（五条建议，从标签规范到定期备份）。甚至附录里还有一张文件路径参考表，把 CLI 脚本、MCP 脚本、Skill 指令、参考文档、MCP 配置、数据目录的路径全部列了出来。

这种文档完整度，在开源项目里属于前 5% 的水平。更关键的是，它的设计思路非常清晰：先定义数据格式（情景记忆和语义记忆的 JSON schema），再定义访问接口（CLI、MCP、Skill），最后定义集成方式（Claude Code、Claude Desktop、Cursor、shell 脚本、直接读文件）。这种“格式优先、接口次之、集成最后”的设计顺序，说明开发者想的不是“做一个好用的工具”，而是“定义一个可以跨工具、跨平台、跨模型使用的记忆标准”。

如果 Memory Agent 的数据格式被更多 AI 工具采纳，它可能成为 AI 记忆层的“Markdown”——一个简单、可读、可迁移、不受任何厂商锁定的开放标准。

当然，现在还远没到那一步。Memory Agent 目前只支持 Claude Code 生态（Skill 注册在 .claude/skills/，MCP 配置在 .mcp.json），对 ChatGPT、Gemini、开源模型的支持还停留在“直接读 JSON 文件”的原始阶段。它的思维导图记忆模块还是 0.2.0 版本，SessionStart hook 还依赖 Claude Code 的环境支持。它的数据量上限是 1000 条，超过后需要手动归档——对于重度用户来说，这个上限可能在三个月内就会触及。

但这些问题都不致命。致命的问题是：当你的 AI 助手开始记得每一个客户的预算范围、每一个项目的卡点、每一个竞品的弱点，你还敢把它托管在别人的服务器上吗？Memory Agent 给出的答案是：不用托管，数据在你自己的硬盘上，用 Git 做版本控制，用 $MEMORY_DIR 做跨设备迁移，用 rm -rf 做彻底删除。这可能是 2026 年最朴素、也最激进的 AI 记忆方案——它不承诺更聪明，它承诺你说了算。
