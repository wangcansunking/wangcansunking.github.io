告别Copilot？Lathe实验：用Claude Code教你，却要你手动完成

当 AI 终于不再帮你写代码，而是逼你亲手敲每一行

"Build a 3D Slicer in Erlang."

这不是某个极客论坛上挖出来的 2012 年老帖，而是一个叫 Lathe 的工具生成的教程。它的作者 Deven Jarvis 在 Hacker News 上发帖时，用了一个让我愣住三秒的比喻："像石器时代那样学习"。

是的，2026 年，一个软件工程师在 Github 上发布了一个开源项目，核心卖点不是"让 AI 帮你干活"，而是"让 AI 逼你亲手干活"。它在你的终端里生成一份完整的手把手教程，然后你坐在屏幕前，一个字一个字敲代码，就像 Jarvis 2007 年捧着 PSP 在 Lua 社区里啃那些满是广告的 cplusplus.com 教程一样。

最激进的反 AI 工具，恰恰是 AI 写的。

Lathe 是什么：一个"反 Copilot"的学习实验

从技术形态看，Lathe 是个组合体：一套 LLM skill 定义（兼容 Claude Code、Cursor 和 Codex）+ 一个 Golang 写的本地 CLI + 一个跑在 localhost:4242 的 Web UI。但它做的事跟所有 AI 编程工具相反。

你给它一个 prompt，比如"build a 3D Slicer in Erlang"，它在你的 LLM session 里生成完整的多章节教程——有目录导航、有侧边注释引导你深度思考、有练习题、甚至有两种"写作语气"可选（plainspoken 和 companion）。然后你关掉 AI，打开教程，自己敲。

Jarvis 在 README 里写了一段让我觉得特别真诚的话：他十几岁时靠 PSP 社区里那些免费教程学会了编程，后来靠 build-your-own-x repo、Crafting Interpreters 和一千个零散的 hands-on 教程从"0 到 1"再"从 1 到 10"。LLM 的出现让写代码变快了，但**"它们替你做了大部分工作，而拿走工作的同时，也拿走了那个让你真正理解新概念的过程"**。

这就是 Lathe 的定位：它不反对 AI，它反对的是 AI 剥夺"顿悟时刻"（aha moment）。它试图用 LLM 的广度（理论上能教你任何领域）去弥补人类教程的稀缺性（比如 g-code 文档难找到让 Jarvis 崩溃），但同时保留手工学习的心流。

一个"vibecoded"项目的自反性悖论

最讽刺的地方来了：Lathe 自己是"vibecoded"的。

Jarvis 在 Hacker News 帖子里直接承认了这一点——"Yep, lathe is vibecoded"。他给出的理由也很坦率：这个项目的风险低，是个个人学习工具，而且他已经每天在使用它，觉得足够稳定才公开。但他同时承诺接下来的几个版本会做"有意的代码和架构清理"。

这就形成了一个奇妙的嵌套结构：一个用 AI 写的工具，教你不要用 AI 写代码。

我不知道 Jarvis 自己有没有意识到这层自反性有多好玩。他甚至在 README 里专门强调，Lathe 的 skill 会"告诉你在它不确定的时候自己也不确定"，而且默认的 plainspoken 语气被设计成"不假装自己是人"——生成的每份教程顶部都会有一行小字：

Generated by Claude Opus 4.8 · voice plainspoken

这种对"AI 身份透明"的执念贯穿了整个项目。教程会记录它查阅过的所有来源 URL（存在 metadata.json 的 sources 字段里），Web UI 上会有一个"Researched against N sources"的面板让你点开检查。验证功能（/lathe-verify）会在一个临时目录里实际跑一遍教程里的每一步代码，把结果写回 verify-result.json——用了显示绿色 verified，失败了就标红 failed，工具链没装就标黄 skipped。

Jarvis 对幻觉问题的态度也很有趣。他承认 Lathe 教程"在用心、个性和架构合理性上远不如人类写的"，但反驳逻辑是："当你亲手敲每一行代码时，你会自然地停下来问'等等，这合理吗？'" 他甚至认为，在陌生领域里主动发现并质疑 LLM 的错误，反而是一种更深度的学习——"我可能因为抓住并反驳 LLM 的失误而更好地内化了概念"。

这个说法没有教育学依据（他自己也承认），但如果你有过"debug 别人代码反而比读文档学得快"的经历，大概能理解他在说什么。

一个比工具本身更大的问题

Lathe 真正让我停下来想的，不是它的技术做到（那个 Golang CLI 只有 4242 端口一个 Web server 和一堆文件管理命令），而是 Jarvis 在 README 末尾抛出的那个问题：

"我怀疑人类还会继续写教程多久，如果只有 LLM 在读它们？"

这个问题比他整个项目都重。2026 年的互联网正在经历一场"内容生产权"的转移——当 AI 能批量生成教程、文档、技术博客，而且速度是人类的上万倍，人类作者还有动力去写吗？如果人类不写了，AI 就只能用 AI 生成的语料训练自己，那"教人学习"这件事的源头活水在哪里？

Lathe 给出的答案是：把 AI 定位成"催化剂"而非"替代品"——它帮你跨越从 0 到 1 的那道坎（尤其是那些人类还没写出教程的冷门领域），但剩下的路你得自己走。Jarvis 举了自己的例子：他最初想做 3D Slicer 软件时，光找 g-code 的文档就痛苦到不行（他向 reprap 社区致敬），Lathe 帮他搭了第一块跳板。写这篇帖子时他正在用 Lathe 学 Zig 嵌入式开发——又是另一个"人类资源近乎为零"的领域。

但这里有个隐含的矛盾：如果 Lathe 真的变流行了，它的教程质量会随着更多人的使用而提高（因为有验证数据和反馈闭环），但人类写教程的动力会进一步下降——因为"反正 Lathe 能生成"。这是一个典型的"工具改善个体体验的同时恶化公共资源"的困境。

Jarvis 似乎意识到了这一点。他在 README 里反复强调："如果你能找到人类写的教程，永远优先选那个。" 他甚至给 Lathe 加了一套防滥用的"护栏"——自定义 voice 功能（/lathe-voice）允许用户创造自己的写作风格，但明确禁止"冒充真实人物、伪造资质或否认 AI 作者身份"。他管这叫"猫鼠游戏"，承认任何安全措施都能被绕过，但"坏演员们已经在全速生产 AI 垃圾教程了，不管我发不发布 Lathe"。

一个"石器时代"工具的 2026 年命运

Lathe 现在是个很早期的项目。它只被 Jarvis 自己在 macOS + Claude Code 的环境下测试过，其他平台"理论上应该能用"。安装方式倒挺齐全：Homebrew（macOS）、curl | sh 脚本、go install（需要 Go 1.25+）、从源码编译。Skills 安装支持 Claude Code、Cursor 和 Codex 三种 agent（Codex 用跟 Claude Code 相同的 SKILL.md 格式，Cursor 转成 slash command）。

但 Jarvis 明确说了一个关键限制：他选择把所有 LLM 交互留在用户的交互式 session 里，而不是让 CLI 自己调 API。原因很现实——Claude Code 的 headless claude -p 模式从 2026 年 6 月 15 日起要按量计费了。他说等价格出来之后可能发现成本很低（"生成教程的 token 消耗远不如 vibecoding"），到时候再考虑把部分功能迁回 UI 里。

这个细节很能说明问题：一个教育工具的架构决策，被 API 定价策略绑架了。 这不是 Jarvis 的问题，这是整个 AI 工具生态的问题——当核心能力依赖第三方模型，你的产品路线图就不是你自己的。

但 Lathe 最打动我的，还是它那股"逆流而上"的劲儿。在所有人都在教 AI 怎么更快地替人写代码的 2026 年，有个人做了个工具，让 AI 教你写代码，然后逼你亲手敲每个字符。他还为此写了全套 CLI、Web UI、voice 系统、source provenance 追踪和 verification pipeline——用 AI 写的。

这就像用印刷术印了一本教人手抄的书，然后告诉全世界：手抄的过程才是重点。

Jarvis 说这是他"活着的论文"（living thesis）。我不知道这篇论文会被多少人读到，但如果哪天你发现自己在 localhost:4242 上对着一个 Lathe 教程敲 Zig 代码，中间停下来骂了一句"等等，这段不对吧？"——那他的实验就成功了。
