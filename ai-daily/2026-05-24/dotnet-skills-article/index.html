微软发布 dotnet/skills：12 个插件让 Claude Code 精通 .NET 开发

ai-daily · 2026 年 5 月 24 日 08:12 · 来源：GitHub Trending any

2026 年 4 月某个周二的凌晨，一个 .NET 开发者在 Slack 里发了一条消息：“我刚让 Claude Code 给我搭了个带 JWT 鉴权的 ASP.NET Core 最小 API，连 Program.cs 都没打开，它就自己把 NuGet 包、EF Core 上下文、MSTest 测试骨架全搞定了。我是不是该慌？”

同一时间，GitHub Trending 上 dotnet/skills 这个仓库的星标数正在以肉眼可见的速度跳动。点进去看，你会觉得自己误入了某种“AI 时代的 .NET 开发者操作手册”——不是传统文档那种死板的 API 罗列，而是 12 个精心编排的插件，每一个都像是一个老法师把自己脑子里的技能包灌进了 AI coding agent 的提示词里。

微软 .NET 团队亲手下场给 AI 写技能包这件事，本身就值得品一品。

AI coding agent 终于从“能写代码”进化到了“知道怎么写对”。

当 AI 学会了“做对的事”，而不只是“做代码”

dotnet/skills 的核心不是一个新框架，也不是一个新 SDK。它是一套遵循 agentskills.io 开放标准的“技能定义文件”。你可以把它理解成：给 AI coding agent 的 .NET 专项培训手册。

这个仓库里塞了 12 个插件，覆盖了从基础 .NET 编码、数据访问（Entity Framework）、性能诊断、MSBuild 构建、NuGet 包管理、.NET 升级迁移、MAUI 跨平台开发、AI/ML 集成（ML.NET + LLM + RAG + MCP）、模板引擎、测试、ASP.NET Core 到 .NET 11 新特性的一整套技能树。每一个插件下面又拆成若干具体技能——dotnet-msbuild 里光构建失败诊断就细分了十几种场景。

让我愣神的是 dotnet-ai 这个插件。它不只是让你“用 .NET 调 OpenAI API”，而是直接把 agentic workflows（智能体工作流）、RAG 管道、MCP 协议这些前沿概念写进了技能定义里。这意味着一个 AI coding agent 在接到“帮我做一个智能客服”任务时，它不会傻乎乎地写个 HTTP 请求就交差，而是会按照技能包里的指引，去判断该用 Semantic Kernel 还是直接调 Azure OpenAI，该不该上向量数据库，什么时候需要 MCP 做工具编排。

（值得一提的是，agentskills.io 是一个独立于微软的开放标准，dotnet/skills 只是它的一个做到。这个标准本身的目标是让不同 AI coding agent 之间可以共享技能定义——Copilot CLI、Claude Code、Cursor、Codex CLI 都能消费同一套技能文件。这就好比 USB-C 之于充电线。）

不是“又一个 SDK”，而是开发者与 AI 之间的协议层

仔细看 dotnet/skills 的安装方式，你会发现一个有意思的设计哲学：它不绑定任何一个特定的 AI 工具。

Copilot CLI 和 Claude Code 用户用 /plugin marketplace add dotnet/skills 一行命令就能接入。VS Code 用户在 settings.json 里加两行配置，勾上 chat.plugins.enabled 就能在 Copilot Chat 里用 /plugins 浏览安装。Cursor 用户直接打开 marketplace 面板搜索 .NET 就完事。Codex CLI 用户更绝，用 skill-installer 命令行工具指定 GitHub URL 就能安装单个技能。

这不是一个工具在抢地盘，这是一个生态在铺铁轨。

dotnet/skills 本质上是一个“协议层”——它定义了一套 .NET 开发的最佳实践，然后用 agentskills.io 这个开放标准把这些实践打包成了 AI coding agent 能消费的格式。Copilot 能读，Claude Code 能读，Cursor 能读，Codex CLI 也能读。谁赢了工具战不重要，重要的是 .NET 的开发者体验在任何一个 agent 里都是一致的。

这种做法的聪明之处在于：微软没有强迫你用 Visual Studio 或者 GitHub Copilot。你用 Claude Code 写 .NET 代码？没问题，技能包照样生效。你用 Cursor？技能包也认。你甚至可以用 OpenAI 的 Codex CLI——它照样能消费 agentskills.io 标准定义的技能文件。这种“工具无关”的姿态，在微软的历史上并不多见。

12 个插件背后，藏着一份 .NET 现代化的路线图

如果你把这 12 个插件摊开来看，会发现它们不是随机拼凑的，而是精确对应了 .NET 生态当前最需要“AI 辅助”的痛点领域。

dotnet-upgrade 这个插件尤其值得细说。.NET 的版本迁移一直是企业开发者的噩梦——从 .NET Framework 4.x 到 .NET 8/9，从传统的 csproj 格式到 SDK-style 项目文件，从旧版 ASP.NET 到 ASP.NET Core，每一步都可能踩坑。dotnet-upgrade 技能包里预置了跨框架版本、语言特性、兼容性目标的迁移规则，AI agent 在接到“把这个项目升级到 .NET 9”的指令时，会按技能包里的 checklist 逐项检查：先看项目文件格式，再看依赖兼容性，然后处理 API 变更，最后跑测试验证。

dotnet-msbuild 则是另一个重头戏。MSBuild 的构建脚本之复杂，连资深 .NET 开发者都经常被劝退。现在这个技能包里塞了构建失败诊断、性能优化、代码质量检查、现代化改造四大类技能。我猜微软内部的数据一定显示，大量 AI coding agent 生成的 .NET 代码在构建阶段就挂了——不是因为代码逻辑错，而是因为 csproj 配置不对、包引用版本冲突、目标框架不匹配这些“工程问题”。dotnet-msbuild 就是来解决这个的。

AI 写代码的正确率再高，过不了构建关就是零。

dotnet-diag 这个插件也值得关注。性能调查、调试、事故分析——这些原本需要资深 SRE 或 .NET 性能专家才能做的事，现在被编码成了技能文件。AI agent 在接到“这个 API 的 P99 延迟突然飙到 2 秒”的告警时，会按技能包里的诊断流程：先检查 GC 暂停时间，再看线程池饥饿，然后排查 EF Core 的 N+1 查询，最后定位到某个 LINQ 表达式生成的 SQL 缺少索引。

微软的阳谋：用技能包锁定开发者心智

站在 2026 年回看，微软在 AI coding agent 这盘棋上的策略其实相当清晰：与其赌哪个 agent 工具会赢，不如确保无论谁赢，.NET 开发者都能得到最好的 AI 辅助体验。

dotnet/skills 的 dashboard 页面上有准确率和效率评分的趋势图（https://dotnet.github.io/skills/），这意味着微软在持续追踪这些技能包的实际效果——哪些技能让 AI agent 的代码质量提升了，哪些场景下 agent 还是会犯错，这些数据会反过来优化技能定义。这是一个闭环：技能包指导 AI agent → AI agent 产生结果 → 结果反馈优化技能包。

这种策略跟微软在 VS Code 上的打法一脉相承——VS Code 不是 Visual Studio 的替代品，而是一个“轻量级入口”，把开发者拉进微软生态后，自然会有一部分人升级到 Visual Studio 或转向 Azure。dotnet/skills 同理：你先用 Cursor + dotnet/skills 写 .NET 代码，写得顺手了，Azure 的集成、GitHub Actions 的 CI/CD、NuGet 的私有源、.NET Aspire 的云原生编排这些“高级功能”就在不远处等着你。

This repository contains the .NET team's curated set of core skills and custom agents for coding agents.

这句话里的 "curated" 一词很关键。它不是自动生成的，不是社区随意提交的，而是 .NET 团队精心筛选和编排的。这意味着技能包的质量和权威性有官方背书——企业 CTO 在做技术选型时，看到“微软官方维护”这个标签，顾虑会少很多。

开放标准 + 官方背书：AI 时代的“开发者关系”新范式

agentskills.io 这个开放标准的存在，让整个故事变得更有意思了。它不是微软的标准，而是一个独立的、跨厂商的技能定义格式。dotnet/skills 只是这个标准在 .NET 生态的落地。

这意味着什么？意味着未来可能出现 python/skills、rust/skills、kubernetes/skills——每个技术生态都可以定义自己的技能包，而任何一个遵循 agentskills.io 标准的 AI coding agent 都能消费它们。开发者不用再在不同工具之间反复切换，也不用担心“这个 prompt 在 Copilot 里好用，换到 Cursor 就失效了”。

工具的壁垒在瓦解，知识的壁垒在重建。

过去，一个资深 .NET 开发者的价值在于他脑子里装了无数踩坑经验——哪个版本的 EF Core 有性能 bug，哪条 MSBuild 配置项在 Linux 容器里会炸，怎么给 MAUI 应用配 Android 的 ProGuard 规则。这些知识分散在无数 Stack Overflow 帖子、GitHub Issues、微软文档和个人博客里，新人要花几年时间才能积累起来。

现在，dotnet/skills 把这些隐性知识显性化了。它不是在教 AI “怎么写 C# 代码”，而是在教 AI “怎么写对的 C# 代码”。这两个之间的差距，就是“能运行”和“能上线”之间的差距。

dotnet-test 这个插件里有一个细节：它包含了“平台检测”技能。AI agent 在生成测试命令时，会自动判断当前是 Windows、Linux 还是 macOS，然后选择对应的测试运行器配置。这东西听起来不起眼，但在真实开发场景里，跨平台测试配置问题能浪费掉一个下午的时间。把这些“工程常识”编码进技能包，才是 dotnet/skills 真正的价值所在。

戏谑收尾：.NET 开发者的“失业焦虑”该换方向了

2026 年的 .NET 开发者可能会面临一种新型焦虑：不是“AI 会不会取代我”，而是“隔壁那个用 dotnet/skills + Claude Code 的家伙，一个上午干了我一周的活，我该怎么跟老板解释”。

但换个角度想，dotnet/skills 本质上是一个“经验杠杆”——它把 .NET 团队和社区里最聪明的那群人的经验，打包成了一个可复制、可分发、可更新的知识模块。以前你必须是微软 MVP 或者某个开源项目的 maintainer 才能接触到这些经验，现在你只要在命令行里敲一行 /plugin install dotnet-upgrade@dotnet-agent-skills，AI agent 就能帮你把 .NET Framework 4.8 的老项目迁移到 .NET 11。

这种感觉很像当年 Git 刚普及时的场景——以前只有核心 maintainer 才能管理 Linux 内核的补丁流，Git 出现后，分布式协作的门槛被彻底打掉了。dotnet/skills 之于 .NET 开发，可能就是类似的东西。只不过这次，被“打掉门槛”的不是版本控制，而是开发经验本身。

微软这步棋走得够早，也够聪明。现在就看其他技术生态能不能跟上了——以及，agentskills.io 这个标准能不能真正成为 AI coding agent 时代的 USB-C。
