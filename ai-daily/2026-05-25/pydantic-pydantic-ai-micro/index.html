Pydantic团队出手：pydantic-ai如何重塑AI Agent开发体验？

ai-daily · 2026 年 5 月 24 日 21:47 · 来源：GitHub Trending python

凌晨三点，一个 Python 后端工程师合上 MacBook，屏幕上还开着那个他写了三天的 AI Agent 项目。LangChain 的依赖冲突、LlamaIndex 的类型警告、某个小众框架的文档黑洞——他把这些坑全都踩了一遍，最后在 Slack 里打了一行字：“有没有一个框架，能让我像写 FastAPI 那样写 Agent？”

他可能不知道，Pydantic 团队也在问自己一模一样的问题。

2025 年 5 月，Python 圈被一个 GitHub 仓库刷了屏——pydantic/pydantic-ai。不是那种“又一个 LLM 包装器”的疲劳感，而是一种久违的“啊，这才对嘛”的释然。这个项目在 Trending 榜上迅速攀升，背后是同一批人：那些年给你 Pydantic Validation、给你 FastAPI 的工程师，现在把矛头对准了 GenAI Agent 开发的混乱局面。

AI Agent 框架已经卷成红海，但 Pydantic 的入场方式不是“我也做一个”——是“你们都用了我的底层，为什么不去源头？”

所有 Agent 框架都在用 Pydantic，但没人做出 Pydantic 的感觉

这件事的讽刺之处，藏在项目 README 的一句轻描淡写里：“Pydantic Validation 是 OpenAI SDK、Google ADK、Anthropic SDK、LangChain、LlamaIndex、AutoGPT、Transformers、CrewAI、Instructor 等无数项目的验证层。”翻译成人话就是：你用的每一个主流 AI 框架，底层都在跑 Pydantic 的数据校验。但当你真的想用这些框架搭一个生产级 Agent 时，那种“丝滑”的感觉就消失了——类型提示不完整、依赖注入靠边站、工具调用像打补丁。

Pydantic 团队自己也在踩这个坑。他们在开发 Pydantic Logfire（一个基于 OpenTelemetry 的可观测性平台）时，需要大量调用 LLM，结果发现市面上的 Agent 框架没有一个能给他们“FastAPI 那样的感觉”。于是他们做了 pydantic-ai，目标极其明确：把 FastAPI 的开发体验带到 GenAI Agent 世界里来。

这个定位很聪明。它不是在功能列表上跟 LangChain 硬拼——LangChain 有几百个集成，pydantic-ai 不会去追那个数量——而是在“开发者手感”这个维度上建立护城河。用 Rust 圈的经典句式来说，就是给你一种“如果编译过了，那它就能跑”的信心。对 Python 这种动态语言来说，这几乎是一种降维打击。

具体怎么做到的？答案藏在类型系统里。pydantic-ai 的 Agent 是泛型的：Agent[SupportDependencies, SupportOutput]。这意味着你的 IDE 和 AI 编程助手（比如 Cursor、Copilot）能在你写代码的时候就告诉你“这个依赖没传对”或者“这个输出字段不存在”，而不是等到运行时爆出一堆 Pydantic ValidationError。整类整类的错误从 runtime 被搬到了 write-time。

（值得一提的是，这种设计哲学跟 FastAPI 一脉相承。FastAPI 当年能火，很大程度上就是因为把 Pydantic 的类型校验和 OpenAPI 文档自动生成做到了极致，让后端开发从“写代码 → 写文档 → 写测试”变成了“写代码 → 自动有文档 → 自动有校验”。pydantic-ai 显然想复制这条路径。）

模型无关 + 可组合能力 + 可观测性：三个引擎同时点火

如果只是类型安全，pydantic-ai 充其量是个“更好的 Instructor”（Instructor 是一个让 LLM 输出结构化数据的库，也基于 Pydantic）。但它真正的野心在另外三个维度：模型无关性、可组合的能力系统，以及原生集成的可观测性。

先看模型支持列表。官方宣称支持“几乎所有模型和提供商”：OpenAI、Anthropic、Gemini、DeepSeek、Grok、Cohere、Mistral、Perplexity 这些模型层；Azure AI Foundry、Amazon Bedrock、Google Cloud、Ollama、LiteLLM、Groq、OpenRouter、Together AI、Fireworks AI、Cerebras、Hugging Face、GitHub、Heroku、Vercel、Nebius、OVHcloud、Alibaba Cloud、SambaNova 这些平台层。如果你用的模型不在列表里，可以自己做到一个 custom model。

这个列表长到让人怀疑是不是把市面上所有叫得上名字的 AI 服务都列了一遍。但重点不是数量——LangChain 也能接很多模型——而是接入方式的一致性。在 pydantic-ai 里，切换模型通常只改一行字符串：'openai:gpt-5.2' 换成 'anthropic:claude-sonnet-4-6'，Agent 的其他代码完全不变。这对那些在多模型之间做 A/B 测试、或者需要根据成本动态切换模型的团队来说，是实打实的效率提升。

再看能力系统。pydantic-ai 引入了一个叫“capabilities”的概念，把工具、钩子、指令、模型设置打包成可复用的单元。内置的能力包括 web search、thinking（推理过程可视化）、MCP（Model Context Protocol，让 Agent 访问外部工具和数据）。开发者还能从 Pydantic AI Harness 能力库里挑选第三方能力包，或者自己写。

这意味着什么？一个银行客服 Agent 的例子很能说明问题。代码里，Thinking() 和 WebSearch() 被作为 capabilities 直接装进 Agent，然后 Agent 就能在回答“今年发现的最大陨石质量是多少”时，自动搜索网络、展示推理过程、给出简洁的一句话答案。整个过程不需要开发者手动写搜索工具的集成代码，也不需要管理推理步骤的状态。

这种设计跟 FastAPI 的依赖注入（Depends）如出一辙——把可复用的逻辑抽成独立单元，用的时候“插”进去，不污染主业务逻辑。对于一个要维护 20 个 Agent 的团队来说，这意味着公共能力（比如日志、权限校验、人机审核）只需要写一次，然后到处复用。

第三个引擎是可观测性。pydantic-ai 跟 Pydantic Logfire 做了深度集成，提供实时调试、基于 evals 的性能监控、行为追踪和成本追踪。如果你的团队已经在用支持 OpenTelemetry 的平台，也可以直接接过去。

（这其实是一个很聪明的商业设计。pydantic-ai 本身是开源的，但 Logfire 是付费的。开源的 Agent 框架培养使用习惯，商业化的可观测性平台负责变现。跟 FastAPI 当年靠文档自动生成吸引用户、靠企业级支持赚钱的逻辑一模一样。）

最有意思的功能可能是 Human-in-the-Loop Tool Approval。Agent 在执行某些工具调用之前，可以根据参数、对话历史或用户偏好来要求人工审批。比如银行客服 Agent 在决定“要不要冻结这张卡”时，风险等级 8 分以上的操作会触发审批，而查余额这种低风险操作自动放行。这个功能在 pydantic-ai 里做到得极其简洁：一个装饰器或者一个配置项就能搞定，不需要自己写审批工作流的状态机。

Durable Execution（持久化执行）也是生产级的标配。Agent 在跨 API 故障、应用错误或重启时能保留进度，支持长时间运行的异步工作流和人机交互场景。对于那些跑一次 Agent 任务要花几分钟甚至几小时、中间还要等人工审批的场景来说，这个功能不是锦上添花，是刚需。

从 Hello World 到银行客服 Agent，pydantic-ai 的示例代码都有一个共同特点：短。不是“省掉了大部分逻辑”的那种短，而是“该有的都有、但一行废话都不写”的那种短。那个银行客服的例子，包括依赖注入、工具定义、动态指令、结构化输出，全部加起来不到 50 行。对于一个被 LangChain 的抽象地狱折磨过的开发者来说，看到这种代码的第一反应可能是：等等，就这么简单？

Pydantic 团队显然很清楚自己在干什么。他们在文档里写：“说实话，任何功能列表都不如你亲自试一试，看看它给你的感觉。”这是在赌开发者的直觉——就像当年 FastAPI 用“自动生成交互式文档”这一个特性就让无数人当场转投阵营一样，pydantic-ai 想用“类型安全 + 依赖注入 + 可组合能力”的组合拳，让你在写完第一个 Hello World 之后产生那种“回不去了”的感觉。

至于这是否能撼动 LangChain 的地位，现在还不好说。LangChain 有先发优势、有社区生态、有几百万行已经上线的代码。但 Pydantic 手里有一张牌是 LangChain 没有的：它本身就是整个生态的地基。当你的底层技术被所有人依赖时，你要做的不是去追别人的功能列表，而是重新定义“什么叫好的开发体验”。这局棋，才刚开始下。
