ai-daily · 2026 年 5 月 25 日 02:37 · 来源：GitHub Trending python

凌晨 3:14，一个安全分析师盯着屏幕上的内存转储文件，光标在终端里一闪一闪。他刚确认了这次入侵——攻击者用 Kerberoasting 拿到了域控制器权限，横向移动的痕迹洒满三朵云的日志。按标准流程，他现在应该打开 Volatility3 跑插件、写 Sigma 规则、对照 MITRE ATT&CK 做 TTP 映射。但今晚只有他一个人值班，而这家公司有 4.8 万个岗位空缺中的一个——他的团队本该有六个人，现在只剩他和一堆告警。

与此同时，他的 AI 编程助手就静静地挂在 IDE 侧边栏里，能写代码、能搜文档、能重构整个微服务架构。但它不知道什么时候该跑 lsass 进程分析，不知道哪条 Sigma 规则能抓 Credential Dumping，更不知道同一个技能动作会在 MITRE ATT&CK、NIST CSF 2.0、MITRE ATLAS、MITRE D3FEND 和 NIST AI RMF 五套框架里分别触发哪条合规项。

这就是 2026 年安全行业的荒诞现实：AI 能帮你写 Python，但教不会它在你睡着的时候顶一个 DFIR 分析师的班。

750 个技能文件，把一个高级分析师的脑子塞进 AI Agent 的上下文窗口里。

一个社区项目，挂了 Anthropic 的名字，但跟 Anthropic 没关系

2026 年 3 月 11 日，一个叫 Mahipal Jangra（GitHub ID：mukul975）的开发者在 GitHub 上发布了 Anthropic Cybersecurity Skills 的 v1.0.0 版本。734 个结构化网络安全技能，跨越 26 个安全域，全部遵循 agentskills.io 开放标准。到了 5 月，这个数字涨到了 754。

但这里有个细节值得玩味——repo 的 README 顶部，赫然挂着一行警告：

⚠️ Community Project — This is an independent, community-created project. Not affiliated with Anthropic PBC.

不是 Anthropic 官方出品。是一个独立开发者，用 Anthropic 的名字命名，做了一个面向 AI Agent 的安全技能库，然后明确告诉你：我跟那家公司没关系。

我看到这个 disclaimer 的时候第一反应是——这操作太聪明了。Anthropic 在 AI 安全领域的品牌势能足够强，Claude Code 又是目前最成熟的 AI 编程助手之一。挂这个名字，等于在搜索引擎和开发者社区里天然获得一个高信任度的入口。但独立项目意味着迭代速度不受大公司 release cycle 的约束，想加 ATLAS 映射就加，想对接 NIST AI RMF 就对接。

（值得一提的是，agentskills.io 这个标准本身也在快速演进——它定义了一套 YAML 前导元数据 + Markdown 本体的技能文件格式，让 AI Agent 能在 30 个 token 内扫描一个技能的适用性，500-2000 token 内加载完整操作流程。这种“渐进式加载”架构是让 754 个技能不撑爆上下文窗口的关键。）

v1.0.0 发布时“只”覆盖了 MITRE ATT&CK 和 NIST CSF 2.0。但 main 分支上的持续增长已经把映射扩展到了五套框架：MITRE ATLAS v5.4（AI/ML 对抗威胁）、MITRE D3FEND v1.3（NSA 资助的防御对策知识图谱）和 NIST AI RMF 1.0（加上 2024 年 7 月发布的 GenAI Profile AI 600-1）。一个技能，五个合规复选框。这在开源安全工具领域是头一回。

五套框架压在一个技能文件上，这不是工程癖，是合规刚需

让我们拆开一个真实的技能文件看看——performing-memory-forensics-with-volatility3，也就是前面那个凌晨值班的分析师最需要的东西。

这个技能的 YAML frontmatter 里写着：

name: performing-memory-forensics-with-volatility3
description: >-
  Analyze memory dumps to extract running processes, network connections,
  injected code, and malware artifacts using the Volatility3 framework.
domain: cybersecurity
subdomain: digital-forensics
tags: [forensics, memory-analysis, volatility3, incident-response, dfir]
atlas_techniques: [AML.T0047]
d3fend_techniques: [D3-MA, D3-PSMD]
nist_ai_rmf: [MEASURE-2.6]
nist_csf: [DE.CM-01, RS.AN-03]
version: "1.2"
author: mukul975
license: Apache-2.0

注意到什么了？同一个内存取证技能，在 MITRE ATT&CK 里对应 T1071（应用层协议），在 NIST CSF 2.0 里触发 DE.CM（持续监控），在 ATLAS 里映射 AML.T0047，在 D3FEND 里对应 D3-NTA（网络流量分析），在 NIST AI RMF 里关联 MEASURE-2.6。

一个分析师跑一次 Volatility3，背后是五套框架的五条合规线索。以前这需要五个人分别填表。

这不是过度工程化。2026 年，科罗拉多州的 AI 法案（Colorado AI Act）已经正式生效，它明确规定：遵守 NIST AI RMF 的组织可以获得法定安全港（safe harbor）保护。换句话说，如果你的 AI 系统出了事，但你能证明自己在 NIST AI RMF 框架下做了尽职调查，法律上你就多一层护身符。而 NIST 自己在 2024 年 2 月发布的 CSF 2.0 里，新增了“治理”（Govern）功能域，把适用范围从关键基础设施扩展到了所有组织。这意味着五套框架的交叉映射不是学术炫技——是实打实的法律责任分摊。

ATT&CK v19 在 2026 年 4 月 28 日落地，把“防御规避”（TA0005）拆成了两个新战术：Stealth 和 Impair Defenses。Jangra 在 README 里承诺后续版本会更新映射。这种对框架版本变化的敏感度，恰恰是企业安全团队在采购工具时最看重的东西——没人想买一个只支持 ATT&CK v14 的僵尸产品。

让我愣神的是 ATT&CK Navigator 图层文件这件事。v1.0.0 的 release assets 里直接附带了一个可导入 ATT&CK Navigator 的 JSON 图层，视觉化展示全部 14 个战术的覆盖强度。Reconnaissance（TA0043）标注为 Strong，Resource Development（TA0042）是 Moderate。这种“我们诚实地告诉你哪里强哪里弱”的态度，在安全产品界属于稀有品质。

26 个安全域，从云取证到工控 SCADA，但有两个域只有个位数的技能

仔细看 26 个安全域的分布，数字本身就在讲一个故事。

Cloud Security 有 60 个技能，Threat Hunting 55 个，Threat Intelligence 50 个——这三个域加起来占了总量的 22%。这是合理的，因为云安全、威胁狩猎和威胁情报恰恰是 AI Agent 最能发挥“速度 × 覆盖面”优势的领域。一个 AI Agent 可以在 30 秒内扫描 754 个 frontmatter，找出 12 个相关的内存取证、凭证窃取和日志分析技能，然后按 Workflow 逐步执行 Volatility3 插件、检查 LSASS 访问模式、关联事件日志证据——最后自动映射到 ATT&CK T1003（Credential Dumping）。这个过程一个初级分析师可能需要 4 小时。

但翻到列表底部，Deception Technology 只有 2 个技能，Compliance & Governance 5 个，Ransomware Defense 7 个。

这不是 bug，是社区贡献的引力场——人人都想做酷炫的攻防，没人想写蜜罐配置和 SOC 2 审计清单。

Jangra 在 CONTRIBUTING.md 里直接把这两个域标记为“最需要帮助”的方向。开源项目的技能分布，本质上是安全从业者集体焦虑的热力图。云安全 60 个技能，是因为 2026 年每一家公司都在往三朵云上迁移，攻击面扩张速度远超安全团队的招聘速度。ISC2 的数据说 2024 年全球有 480 万个网络安全岗位空缺——这个数字到 2026 年只会更大。AI Agent 被寄望于填补这个缺口，但你得先给它一套能用的操作手册。

每个技能文件的结构也值得展开说。一个技能目录下通常有四样东西：SKILL.md（YAML + Markdown 本体）、references/（标准映射和技术深度参考）、scripts/（可执行的 Python 辅助脚本）和 assets/（模板和清单）。拿 performing-memory-forensics-with-volatility3 来说，SKILL.md 的 Markdown 正文分四个小节：When to Use（触发条件——AI Agent 什么时候应该激活这个技能）、Prerequisites（需要的工具、权限、环境配置）、Workflow（分步执行指南，包含具体命令和决策点）、Verification（如何确认执行成功）。

这种结构的设计哲学很清晰：不是给人类看的教程，是给 AI Agent 执行的指令集。Workflow 里的每一步都精确到命令级别，Verification 里定义了 IOC 确认逻辑。人类分析师看这个东西会觉得过于细碎，但对一个 LLM 来说，这种颗粒度恰好是它从“能聊安全”变成“能干安全”的临界点。

20 多个平台兼容，但有意思的是“谁不在列表里”

兼容性列表读起来像一份 2026 年 AI 编程工具的行业黄页：Claude Code、GitHub Copilot、Cursor、Windsurf、Cline、Aider、Continue、Roo Code、Amazon Q Developer、Tabnine、Sourcegraph Cody、JetBrains AI——这是 AI 代码助手阵营。OpenAI Codex CLI 和 Gemini CLI 是命令行 Agent。Devin、Replit Agent、SWE-agent、OpenHands 是自主 Agent。LangChain、CrewAI、AutoGen、Semantic Kernel、Haystack、Vercel AI SDK 是 Agent 框架和 SDK。

总共 20 多个平台，全部用 agentskills.io 标准做到零配置加载。用 npx 一行命令就能把整个技能库注入到 Claude Code 的上下文里：

npx skills add mukul975/Anthropic-Cybersecurity-Skills

但这份兼容性列表最有趣的地方不在于谁在上面，而在于谁不在上面。

OpenAI 的 ChatGPT 不在列表里。Google 的 Gemini（除了 CLI 版本）不在列表里。Anthropic 的 Claude（除了 Claude Code）不在列表里。换句话说，这是一个面向开发者工作流的技能库，不是面向聊天界面的知识库。它假设用户是拿着终端、IDE 和 CI/CD pipeline 的安全工程师，而不是在聊天框里问“怎么防御勒索软件”的普通用户。这种定位选择，让 Anthropic Cybersecurity Skills 更接近“安全版的函数库”而非“安全版的维基百科”。

（值得一提的是，Jangra 还在 repo 里嵌入了一个学术研究项目——GARS-2026，Global Agentic AI Readiness Survey。60 个问题，匿名，由柏林 SRH 大学监督，完成后给 50 个 Casky 代币用于早期访问 casky.ai。一个开源项目同时在做学术数据采集和产品冷启动，这个操作路径值得独立开发者研究。）

一个独立开发者，754 个技能，和一个正在被重新定义的安全行业

回到凌晨 3:14 的那个分析师。

如果他的 IDE 里装了 Anthropic Cybersecurity Skills，流程会变成这样：他把内存转储文件拖进工作区，在 Claude Code 里输入“Analyze this memory dump for signs of credential theft”。Agent 先用 30 token 扫描全部 754 个 frontmatter，匹配 tags、description 和 domain，筛出 12 个相关技能。然后加载前三个最匹配的：performing-memory-forensics-with-volatility3、hunting-for-credential-dumping-lsass、analyzing-windows-event-logs-for-credential-access。接着按 Workflow 执行 Volatility3 插件、检查 LSASS 访问模式、关联 Windows 事件日志。最后用 Verification 里的逻辑确认 IOC，映射到 ATT&CK T1003。

这个过程，从 prompt 输入到结论输出，可能不到 90 秒。

这不是 AI 替代安全分析师的故事。这是“一个人 + 一个 Agent = 一个团队”的故事。

ISC2 那个 480 万岗位空缺的数字，在 2026 年已经不是新闻了。安全行业花了十年时间喊“人才缺口”，但缺口每年都在扩大。Anthropic Cybersecurity Skills 这种项目提供了一条不同的路径：不是培养更多分析师，而是让现有的分析师把重复性的、流程化的、需要对照五套框架填表的工作外包给 AI Agent。人类分析师去做假设驱动的威胁狩猎、做跨域的攻击链重建、做业务影响评估——那些真正需要直觉和判断力的部分。

而这个项目的真正门槛不在于技术做到。754 个 YAML 文件和 Markdown 工作流，任何一个有安全背景的工程师都能写。门槛在于：谁愿意花时间把 26 个域的知识从脑子里倒出来，结构化，标注五套框架映射，然后以 Apache 2.0 协议开源？Mahipal Jangra 做了这件事，而且还在用社区贡献机制让更多人来填 Deception Technology 和 Compliance & Governance 的坑。

开源安全工具的未来，可能不是更强大的扫描器，而是更完整的 AI 操作手册。

至于 Anthropic 官方会不会哪天把这个项目收编——谁知道呢。但眼下，一个挂了 Anthropic 名字的独立项目，正在用 754 个技能文件悄悄改变安全分析师和 AI Agent 之间的关系。凌晨 3:14 的值班，可能很快就会变成凌晨 3:14 的“让 Agent 先跑一遍，我睡 20 分钟再来看结果”。
