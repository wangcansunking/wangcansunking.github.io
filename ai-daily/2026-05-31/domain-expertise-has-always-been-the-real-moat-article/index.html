调度员用Claude三周造系统：不懂代码，但懂领域才是真壁垒

2024 年秋天，一个在物流公司做了十二年调度员的哥们儿给我看他的屏幕。他不会写 Python，分不清 JavaScript 和 Java 的区别，但他用 Claude 生成了一套排班系统——从司机工时校验到路线优化，三周搞定。我问他怎么确认系统没 bug，他愣了一下：“bug？我就是看它排出来的班合不合劳动法啊。不合我就让它重排。”

2025 年底，另一个故事在 Hacker News 上炸了。一个拿了 AWS 认证的全栈工程师接了个医疗账单系统的外包，交付三个月后被客户起诉——系统生成的 CPT 代码组合在表面上用了所有单元测试，但实际提交给保险公司时，拒付率高达 47%。工程师在法庭上说：“代码逻辑没问题，所有测试都绿了。”客户反问：“那你知不知道 99213 和 99214 不能在同一天出现在同一张 claim 里？”

两个故事之间隔了十四个月，隔着一整个 Agentic AI 时代的到来。

代码从来不是壁垒。知道什么是对的，才是。

一、当“能写代码”变成水电煤

Brett Horsting 在 2026 年 5 月 30 日那篇被顶上 Hacker News 首页的文章里，把话说得很直白：写软件的难处从来不在“写”本身，而在于你在脑子里先建好一个能运转的领域模型。在你交付一套薪资系统之前，你得搞懂工资扣押（garnishments）、税前扣除（pre-tax deductions）、以及当一个人的发薪周期跨越费率变更时该怎么算。在你做一个公交 App 之前，你得先知道 GTFS feed 是什么、为什么 trip 和 route 是两个概念、以及为什么一辆“准点”的公交车仍然可能是错的。

代码只是那个理解的转录。获取理解，才是工作本身。

这个判断跟我去年看过的所有“AI 会替代程序员”的论调都不在一个维度上。那些讨论聚焦在“能不能生成代码”“生成质量够不够好”“token 成本降了多少”，但 Horsting 指出的裂痕更底层：Agentic AI 把“理解领域”和“产出软件”之间的那条线切断了。你现在可以不经建模就直接拿到软件。这打破了整个软件行业围绕“先理解、再构建”组织起来的那套前提。

这意味着什么？意味着过去二十年软件工程师最核心的竞争优势——把领域知识翻译成可运行代码的能力——正在以肉眼可见的速度贬值。Horsting 自己去年还写过一篇文章，说这些工具会放大资深工程师的优势，因为他们有判断力。但他现在承认这个说法“正确但不完整”。他在过去一年观察到的东西更具体、也更有趣：约束条件从“你能不能把它做出来”变成了“你能不能判断它做得对不对”。

这个位移是地震级别的。它重新定义了谁在这个行业里有价值。

二、物流调度员 vs 全栈工程师：谁才是 AI 时代的天选之子？

Horsting 在文章里画了两幅肖像。第一幅：一个领域专家，几乎没有软件背景。可能是物流调度员、临床编码员、精算师。他们看不懂 stack trace，分不清 hash map 和 list 的区别。但他们能盯着 Agent 生成的排班表，一眼看出“这个司机不可能合法跑完这趟班”——因为他们花了十年活在这些输入和输出里。你把一个 Agent 塞给他们，他们的效率惊人。因为他们缺的东西——产出代码的能力——恰恰是 Agent 能补上的。而他们拥有的东西——对真实世界正确答案的直觉——Agent 永远不会有。

他们手里握着的，叫 ground truth。

第二幅肖像：一个强悍的通才工程师，但从没在目标领域工作过。他们能架构任何系统，懂可靠性、懂测试、知道怎么让系统在凌晨两点不崩。但一旦掉进临床编码的坑里，他们分不清一个看起来合理的错误答案和一个正确答案。Agent 会高高兴兴地生成一条账单规则——这条规则能编译用，能跑通工程师写的所有测试，但在真实世界里会悄无声息地、以极其昂贵的方式出错。工程师手里没有神谕（oracle）。他们能验证软件“构建得好不好”，但无法验证软件“对不对”。因为在那个领域里，“对”的标准完全定义在一个他们脑子里没有的模型里。

这里有一个残酷的对比：在 Agent 出现之前，工程师有一条路径，调度员没有。工程师可以慢慢学那个领域——跟着专家影子实习、啃规范文档、在生产环境里犯错然后修正——最终在脑子里建起心智模型，然后构建系统。这条路在很多行业里就是整个职业生涯的阶梯。而领域专家没有对等的路径，因为学会构建可靠软件需要数年的功夫，他们永远不会去走。

Agentic 工具碾碎了其中一条路，另一条毫发无损。工程师的优势——把领域模型翻译成工作代码——现在便宜得像自来水。领域专家的优势——知道“对”长什么样——你没法靠写 prompt 获得。没有任何 skill file 能装下一个调和过一千次 payroll 的人脑子里那种默会知识。

于是这个新世界里最有价值的人是同时拥有两种技能的人。他们能在两个层面上做验证：既知道生成的代码结构可靠，又知道代码产出的答案真实。他们能写出“司机不能超过 11 小时”的测试，因为他们知道这条规则；他们也能判断这条测试本身有意义，因为他们知道自己在测什么。Agent 做转录工作。他们做判断工作，而且是两次。

你的机械手艺正在贬值。你的领域嗅觉是最后一道护城河。

三、那些你以为能“学一下”的领域，其实要花十年

Horsting 文章下面有个评论让我愣神了几秒。一个自称做了十五年医疗索赔处理的人写道：“我培训新人最崩溃的部分，不是教他们规则——规则手册有 3000 页，谁都能翻。是教他们什么时候规则手册是错的。保险公司有无数不成文的惯例，比如某家公司在每年三月会系统性地拒掉某几类 claim，因为他们的审计季开始了。你得手动绕过，然后四月再改回来。这种事没有任何文档。”

这就是默会知识。它不是“知道 CPT 代码 99213 的定义”，而是“知道 99213 和 99214 不能在同一天出现在同一张 claim 里”——而且这个限制不在任何公开规范里，它是某家保险公司内部审核系统的历史 bug 演变成的事实标准。你在 Stack Overflow 上搜不到，在官方文档里读不到，Claude 的 training data 里大概率也没有。

这解释了为什么过去一年里，那些“AI 将替代所有初级程序员”的预言落空了——但同时，“AI 将让资深程序员十倍高效”的预言也只对了一半。真正十倍高效的人，不是资深的通用工程师，而是碰巧也懂代码的领域专家。或者反过来，碰巧深耕了某个领域的工程师。

我去年见过一个案例。一个在税务软件公司做了八年的架构师，用 Cursor 在一个周末重写了整个海外资产申报模块——那个模块之前由一个五人团队维护了两年。他的效率不是因为他 prompt 写得好，而是因为他不需要 Agent 给他解释 FATCA 和 CRS 的区别，不需要验证生成的申报逻辑是否符合 IRS 的最新解释，因为他脑子里有一张完整的、实时更新的税务合规地图。Agent 只是在打字。他是在画地图。

而那个接医疗账单外包的工程师呢？他的代码质量可能更高，架构可能更优雅，测试覆盖率可能更漂亮。但他没有那张地图。他只能验证 Agent 输出的代码在语法和逻辑上是否自洽，无法验证它在保险公司的赔付现实里是否成立。47% 的拒付率不是代码 bug 造成的，是领域盲区造成的。

四、这不是新故事，但这次速度不一样

“懂业务比懂技术值钱”这句话在软件行业喊了二十年，但此前它更像一种政治正确。实际薪酬曲线一直是反过来的：一个精通分布式系统的工程师，薪资碾压一个在某个垂直领域深耕十年但不会写代码的业务专家。因为前者的稀缺性有明确的市场定价——能搭高并发系统的人就那么几个——而后者的知识虽然深，但“翻译成软件”这个环节需要前者介入，所以价值被前者截留了一部分。

Agentic AI 改变了这个分配结构。当“翻译成软件”的成本趋近于零时，截留消失了。那个物流调度员不再需要求着 IT 部门排期六个月开发一套排班系统。他打开 Claude，用他脑子里那套“司机不能连续工作超过 11 小时”“危险品运输必须避开人口密集区”“某个客户总是临时改单所以预留 20% buffer”的规则直接驱动 Agent 产出可用的软件。他绕过了整个工程组织。

这一次，拆墙的不是技术革命，是成本革命。

Horsting 的文章没有展开的一点是，这个趋势会怎么重塑软件团队的组织结构。但我看到的方向很明确：未来的团队会从“一个架构师带五个工程师”变成“一个懂代码的领域专家带三个 Agent”。那个专家负责两件事：定义“对”的标准，以及在 Agent 跑偏时纠偏。剩下的“写代码”环节——曾经占工作量 70% 以上的部分——被压缩到几乎为零。

这不意味着工程师会失业。它意味着工程师必须变成那个“懂代码的领域专家”。而“懂领域”这件事的获取成本，比“懂代码”高一个数量级。你可以用六个月 bootcamp 学会写 React。你没法用六个月学会怎么让一笔跨境并购交易用欧盟反垄断审查。

五、如果你现在是一个焦虑的工程师

Horsting 给工程师的建议很直白，甚至有点冷酷：“你为之流汗的机械技能——把一个清晰的想法变成干净代码——已经急剧贬值。仍然稀缺的东西，是对某个真实领域的深度、经过验证的心智模型。去搞一个。挑一个行业、一种工具、一套监管体系、一个物理过程，用你当年学编程语言或框架的方式去学它。这是 Agent 没法替你做的部分，也是现在最值钱的部分。”

这段话如果放在 2023 年，会被喷成“劝人转行做业务是工程师的投降主义”。放在 2026 年 5 月，它被顶上了 Hacker News 首页。

The mechanical skill you sweated for, turning a clear idea into clean code, has gotten dramatically less valuable. The thing that’s still scarce is a deep, verified model of some real domain. Go get one.

我理解为什么这个判断让很多工程师不舒服。它暗示了过去十年你投入的一切——学 Rust、刷 LeetCode、啃 DDIA、拿 AWS 证书——在边际上的回报正在递减。不是没用了，而是不再是差异化因素。当 Claude 能在三十秒内生成一个符合最佳实践的微服务骨架时，你的架构能力从稀缺品变成了标配。

但换个角度看，这也可能是工程师这个职业第一次真正有机会从“打字工”升级为“决策者”。过去二十年，软件工程师的大部分时间花在翻译上：把产品经理的 PRD 翻译成代码，把业务专家的需求翻译成系统设计。翻译这个动作本身不创造核心价值，它只是价值的载体。现在载体可以自动化了，你终于可以去做那个定义“翻译什么”的人——前提是你得先成为那个知道“什么是对的”的人。

这不是一个舒服的转型。它要求你从一个“可以跨行业流动的通才”变成一个“锚定在某个具体领域里的人”。而锚定意味着你放弃了部分流动性——你在医疗索赔领域的深度知识，在跳到一家 SaaS 公司做营销自动化时可能毫无用处。这是真的代价。但反过来，流动性本身也在贬值。当每个人都能用 Agent 写代码时，“什么都能做”的工程师的替代成本急剧下降，而“只有你能做”的专家的溢价急剧上升。

六、那条 Hacker News 评论区里藏着的火药味

Horsting 的文章在 HN 上炸出了几百条评论，我翻了一遍，发现最激烈的争论集中在一个点上：领域知识到底能不能被 AI 学会？

乐观派（主要是 AI 从业者）认为这只是时间问题。“GPT-5 的训练数据里包括了所有公开的税务法规、医疗编码手册、物流调度论文，它为什么不能学会这些领域的 ground truth？”但那些真正在领域里泡了十年以上的人几乎一致反对。一个做了二十年 payroll 的人写道：“你没法从文档里学到 garnishment 的实际处理逻辑，因为每个州的执行方式都不一样，而且每年都在变。我们公司有一个内部 Wiki，专门记录‘文档说 X，但实际做 Y’的 case，那个 Wiki 有 2700 多条记录。”

这个反驳击中了一个关键盲区：AI 的训练数据本质上是对世界的描述，不是世界本身。而领域专家的 ground truth 来自与世界的直接互动——被保险公司拒付过几万次的临床编码员，被劳动稽查部门罚过款的物流调度员，被 IRS 审计过的税务师。他们的知识不是读来的，是疼出来的。

AI 可以读完所有的医疗教科书，但它没被保险公司的拒付函怼过脸。

这也是为什么 Horsting 用“oracle”这个词来描述领域专家的能力——不是数据库，是神谕。数据库存储事实，神谕判断对错。Agent 能访问前者，但无法成为后者。

当然，这场争论没有定论。也许 GPT-6 真的能用强化学习从真实世界的反馈信号中习得这些默会知识。也许未来的 Agent 会在被保险公司拒付四十七次之后自己总结出那条不成文规则。但在那之前，手里有 oracle 的人，就是那个能驱动 Agent 产出正确结果的人。

七、最后一条护城河，也是最古老的那条

我写完这篇文章的时候，突然想起一个细节。2025 年初，Anthropic 发布 Claude Code 的时候，演示视频里用的例子是“帮一个物流公司做排班优化”。演示者输入了一长串约束条件：司机每天不超过 11 小时、每周不超过 60 小时、危险品车辆必须避开学校区域、某个大客户的订单必须优先、跨州运输要考虑各州不同的休息站规定……

弹幕里一片“卧槽这个 prompt 也太长了吧”。但我当时的反应是：写出这个 prompt 的人，才是真正不可替代的人。因为那些约束条件不是从文档里复制粘贴的，是从真实运营里长出来的。写出它们的人，大概率被司机起诉过、被客户骂过、被交警拦过、被 DOT 罚过。

Agentic AI 时代最讽刺的一点是：它让软件生产变得前所未有的便宜，却让“知道该生产什么软件”变得前所未有的昂贵。而后者，恰恰是软件行业诞生第一天起就存在的那个老问题。我们花了五十年发明各种方法论、框架、工具来解决“怎么写代码”的问题，然后 AI 一夜之间让那个问题消失了。但“写什么”和“怎么判断写对了没有”——这两个问题纹丝不动。

它们需要的东西从来都一样：一个人，在一个领域里，待了很久，犯过很多错，然后终于知道什么是对的。

那条护城河不是新挖的。它只是终于被看见了。
