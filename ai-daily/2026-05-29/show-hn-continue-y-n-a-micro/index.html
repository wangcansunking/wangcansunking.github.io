Anthropic 与 OpenAI 的难题：AI 权限弹窗正引发集体决策疲劳

我盯着屏幕右下角的弹窗，已经第 17 次了。

「Continue? Y/N」—— 光标一闪一闪，等着我敲键盘。上一个弹窗是 12 秒前，再上一个大概是 30 秒前。我正在让一个 AI agent 帮我整理三篇论文的参考文献，它每下载一篇 PDF、每打开一个数据库、每尝试读取一个文件路径，系统就会弹一次确认框。我开始怀疑自己到底是在「使用」工具，还是在「当工具的保安」。

就在这种手指肌肉记忆即将接管大脑的临界点，我在 Hacker News 首页刷到了这个链接 —— llmgame.scalex.dev，标题直白得像一记直拳：A 60-second game about AI agent permission fatigue。60 秒，打一局游戏，体验一下被 AI agent 权限请求轰炸是什么感觉。截至 2026 年 5 月 7 日，这个帖子在 HN 上拿了 244 个赞、111 条评论，热度不算爆炸，但评论区里弥漫着一种「被戳中」的集体叹气。

我们正在把人类变成操作系统的 UAC 弹窗处理器。

60 秒的崩溃：为什么一个「无聊」的游戏值得上 HN 首页

我先交代一下这个游戏本身。根据 HN 帖子里的描述，llmgame.scalex.dev 是一个极简的网页游戏：你扮演一个试图完成任务的用户，AI agent 不断向你发出权限请求 —— 读取文件、联网、调用 API、修改配置、发送邮件。60 秒倒计时里，你要快速判断哪些请求该批准、哪些该拒绝。点错太多次，任务失败；点太慢，时间耗尽。

评论区里最高赞的一条留言大概意思是：「我玩了三次，第三次的时候我已经开始无脑按 Y 了。然后我突然意识到，这就是我平时用 AI agent 的真实状态。」

这句话让我愣神了两秒。因为我回想了一下自己下午那 17 次弹窗，至少有一半我根本没看清请求内容就按了回车。不是我不在乎安全，是我的大脑在连续处理第 8 个类似的确认框之后，已经进入了「决策疲劳」模式。心理学上管这叫 ego depletion（自我损耗），通俗点说就是：人的意志力和判断力是消耗品，用一点少一点。

而 AI agent 的设计范式，目前正在系统性地、工业化地消耗这种稀缺资源。

这让我想起 2024 年 Anthropic 发布 Computer Use 功能时的一个细节。当时他们演示 Claude 操控桌面完成订机票任务，全程需要用户手动确认每一个关键步骤。技术博客里用了一个很体面的词叫 human-in-the-loop（人机协同），但实际体验是：你得盯着屏幕看它移动光标、点击按钮、填写表单，每到一个敏感操作就弹窗等你批准。效率呢？可能还不如你自己动手做。

AI agent 的「狼来了」陷阱：当权限弹窗变成白噪音

游戏背后的真问题，在 HN 评论区被拆解得相当透彻。我把 111 条评论翻了一遍，提炼出三条核心焦虑：

第一，频率与敏感度的错配。一个 AI agent 可能在 5 分钟内发出 20 个权限请求，其中 18 个是「读取当前标签页标题」这种无伤大雅的操作，剩下 2 个是「修改系统环境变量」。但因为弹窗长得一模一样，用户在第 15 次确认之后已经放弃区分，直接肌肉记忆按 Y。

第二，上下文缺失。弹窗告诉你「某 agent 请求访问剪贴板」，但它不告诉你为什么要访问、访问之后会干什么、不访问会不会影响任务完成。用户在做安全决策时，手里只有最小化的信息，这几乎是 UI 设计的反面教材。

第三，责任转嫁。AI 公司把「判断安全性」这件事从模型侧转移到了用户侧。模型不知道自己该不该做某件事，于是它问用户。用户不知道该怎么判断，于是 —— 要么无脑批准，要么放弃使用 agent。两种结果都是系统设计的失败。

评论区有一个人说得特别狠：「这本质上是用弹窗把安全风险外包给了最没能力评估风险的人。」（This is essentially outsourcing security risk via dialog boxes to the people least equipped to evaluate it.）

我看到这句的时候，下意识点了点头。因为 2025 年以来，几乎所有主流 AI agent 产品 —— 从 OpenAI 的 Operator 到字节的 Coze 再到各种开源框架如 CrewAI、AutoGPT —— 都在走同一条路：让 agent 能干更多事，然后用权限弹窗来兜底。弹窗越多，说明 agent 的能力边界越模糊；能力越模糊，弹窗就越泛滥。这是个死循环。

更值得警惕的是，这种疲劳感正在训练用户养成一个危险习惯：把「点击确认」当作使用 AI 的默认姿势。就像当年 Windows Vista 把 UAC（用户账户控制）弹窗做到极致之后，用户反而学会了条件反射式地点击「是」，UAC 从安全机制退化成了一种仪式性动作。安全专家 Troy Hunt 在 2015 年就写过一篇文章，标题叫《UAC 唯一教会用户的事就是「看到弹窗就点确定」》。十年过去了，AI agent 正在完美复刻这个失败剧本。

而这个游戏之所以能引发共鸣，恰恰是因为它用 60 秒的极端体验，把这种日常折磨浓缩到了一个让人无法忽视的浓度。游戏设计里有一个概念叫 procedural rhetoric（程序修辞），意思是用规则和交互本身传递观点，而不是用文字说教。llmgame.scalex.dev 的程序修辞极其精准：你不需要读任何说明，玩一局就懂了。

当然，HN 评论区也不是只有抱怨。有几个做 agent 框架的开发者提到了可能的解法：分级权限（像手机 App 那样，普通权限自动放行，敏感权限才弹窗）、时间窗口信任（批准一次，10 分钟内同类请求不再弹）、意图声明（agent 启动时先声明「本次任务需要访问以下资源」，用户一次性审批）。但这些方案各自有坑 —— 分级权限的边界怎么划？时间窗口被恶意利用怎么办？意图声明如果太长，用户会不会直接跳过不读？

这些问题目前没有标准答案，但至少这个 60 秒的小游戏，把问题本身摆到了台面上，而且摆得让人无法假装没看见。

我玩到第四局的时候，发现自己开始对着屏幕骂脏话了。不是骂游戏设计，是骂那个弹窗里写着「Requesting permission to read /Users/me/Documents/report.pdf」的虚拟 agent —— 你他妈已经是第三次请求读这个文件了，上次不是批过吗？

然后我意识到，我骂的那个 agent，和我下午用过的那个，并没有什么本质区别。
