初创公司用3天迁移至Twenty，CRM为何需要“代码化”管理？

ai-daily · 2026 年 5 月 28 日 21:36 · 来源：GitHub Trending any

2025 年 7 月某天深夜，旧金山 SOMA 区一间共享办公室里，三块屏幕同时亮着。左边是 Salesforce 的 Classic 界面，卡在加载状态；中间是终端窗口，npx create-twenty-app 刚跑完；右边的 Discord 频道里，有人丢了一句话：“我们团队用 3 天把整个销售流程搬到了 Twenty，现在法务部问我们有没有买 license。”

发消息的人是某 B2B 初创的 CTO。他们公司 50 人规模，Salesforce 一年的账单能雇两个全职工程师。这件事本身不算新闻——中小企业被 SaaS 巨头按在地上摩擦，早就成了科技行业的背景噪音。但让我愣神的是他后半句：“CRM 终于可以像其他代码一样被 review、branch、rollback 了。”

这大概就是 Twenty 最危险的地方。

当 CRM 变成代码，Salesforce 的护城河就从商业壁垒退化成了技术债务。

一个 GitHub repo 凭什么敢叫板 2000 亿美金巨头

2025 年 7 月，GitHub Trending 榜单上出现了一个异常值。在满屏的 AI agent 框架和 LLM 推理引擎中间，twentyhq/twenty 这个项目显得格格不入——它是个 CRM。不是“用 AI 增强的 CRM”，不是“大模型驱动的销售自动化平台”，就是 CRM。记录客户、跟踪交易、管理销售管道的那类软件。但它的 README 第一行写着：“The open alternative to Salesforce, designed for AI.”

这句话信息密度极高。它同时完成了三件事：定义了竞争对手（Salesforce），声明了分发模式（open source），锚定了技术范式（AI-native）。在开源世界里，“X 的替代品”是个被用滥的句式，但大多数替代品对标的是某个小功能或某个垂类工具。Twenty 对标的是 Salesforce——一家年收入 348 亿美元、市值超过 2000 亿美元的 SaaS 鼻祖。这个定位要么是狂妄，要么是真的看到了某些结构性裂缝。

翻看 Twenty 的架构文档，会发现它的核心设计哲学只有一句话：CRM 是业务逻辑的映射，业务逻辑应该像代码一样被管理。 这句话翻译成工程语言就是——objects、fields、views、workflows、agents，全部定义为代码，存储在 Git 仓库里，用 CLI 部署到 workspace。Twenty 不给你一个配置后台让你点点点，它给你一套 SDK 和 CLI 工具，让你把 CRM 写出来。

这跟 Salesforce 的哲学完全相反。Salesforce 的核心理念是“clicks, not code”——让业务人员用点击和拖拽来配置系统，不需要写代码。这个理念在 1999 年推出时是革命性的，它把 CRM 从 IT 部门的垄断中解放出来，交到了销售 VP 和运营经理手里。但 25 年过去，“clicks not code”变成了另一种枷锁：当业务逻辑复杂到一定程度，点击配置就变成了在迷宫般的 Setup 界面里寻找那个隐藏在第 7 层子菜单下的 checkbox。任何配置过 Salesforce Flow 的人都懂那种痛苦——你以为你在配置自动化流程，实际上你在用鼠标画流程图，而那个流程图的复杂程度已经超过了你用代码写的后端服务。

Twenty 选择了完全相反的路径。它假设使用者是 technical teams，给的是 building blocks，目标是“a CRM you build, ship, and version like the rest of your stack.”这不是在做一个更好的 Salesforce，这是在重新定义 CRM 这个品类跟开发团队的关系。

代码即配置：Twenty 的技术栈藏着什么野心

打开 Twenty 的代码仓库，技术栈清单本身就是一份宣言：TypeScript、Nx monorepo、NestJS 后端、BullMQ 消息队列、PostgreSQL 数据库、Redis 缓存、React 前端、Jotai 状态管理、Linaria CSS-in-JS、Lingui 国际化。这套组合在 2025 年的前端和后端圈子里都算得上主流甚至偏激进的选择。但真正有意思的不是选了什么技术，而是这些技术拼在一起暴露出的产品边界。

BullMQ 的存在说明 Twenty 需要处理异步任务队列——这不是一个简单的 CRUD 应用，它在设计之初就假设了需要执行后台作业。workflows 和 agents 这两个 building block 的引入，进一步证实了这一点。workflows 是自动化流程，agents 是 AI 代理。在 CRM 语境下，这意味着你可以写一个 agent 在交易金额超过阈值时自动触发审批流程，或者写一个 workflow 在新 lead 进入系统时自动抓取该公司的公开财务数据并生成风险评估。

这里有一个容易被忽略的细节。Twenty 的 CLI 命令是 npx twenty app:publish --private。注意 --private 这个 flag。它暗示了 Twenty 的应用生态有一个公私分层的设计：你可以发布只在团队内部使用的 private app，也可以（未来）发布到公共市场。这是一个微妙的信号——Twenty 在模仿的不是 Salesforce 的产品形态，而是 Salesforce 的生态飞轮。AppExchange 是 Salesforce 最深的护城河之一，上面有超过 4000 个第三方应用，覆盖从电子签名到 CPQ 报价的每个环节。如果 Twenty 想在开源世界复刻这个飞轮，--private 就是种子阶段的土壤。

再看 defineObject 那段示例代码：

defineObject({ nameSingular: 'deal', namePlural: 'deals', fields: [{ name: 'name', type: FieldType.TEXT }, { name: 'amount', type: FieldType.CURRENCY }, { name: 'closeDate', type: FieldType.DATE_TIME }] })

这段代码看起来平平无奇，就是一个 schema 定义。但把它放在 CRM 的上下文里，它解决的是一个困扰了企业软件几十年的问题：数据模型的可移植性。在 Salesforce 里，你对一个 object 的定义分散在几十个 Setup 页面的元数据配置里。想把一个 Salesforce org 的数据模型迁移到另一个 org，你需要借助专门的 DevOps 工具（比如 Copado 或 Gearset），还得处理元数据 API 的各种诡异限制。而在 Twenty 里，数据模型就是一个 TypeScript 文件。你可以把它放进 Git，做 code review，跑 CI/CD，出问题就 git revert。任何一个经历过 Salesforce 部署噩梦的工程师看到这里，大概都会有一种“早该如此”的感慨。

Salesforce 用了 25 年把 CRM 从代码变成了配置。Twenty 用一行 import 把它变了回去。

自托管 + AI-native：两个趋势的交叉点

Twenty 的安装方式有三种：Cloud（注册 twenty.com，一分钟内启动 workspace）、Self-hosting（Docker Compose 一键部署）、Local development（本地开发环境）。这三种方式对应三种用户：想快速试用的团队、有合规需求必须自己管数据的公司、以及打算在 Twenty 上做二次开发的 ISV。

自托管这个选项在 2025 年变得格外重要。过去 18 个月里，企业数据隐私的讨论已经从“要不要上云”演变成了“上了云之后数据到底在谁手里”。欧盟的 AI Act、加州的删除权诉讼、以及 Snowflake 和 Databricks 之间关于数据所有权的公开撕扯，都在把“数据驻留”从法务部门的 checklist 推成采购决策的核心变量。Twenty 的自托管选项——用 Docker Compose 在自己的 VPC 里跑起来——对于金融、医疗、法律服务这些行业的团队来说，不是 nice-to-have，是 must-have。

但自托管只是前半句。后半句是“designed for AI”。这个表述在项目文档里多次出现，但 Twenty 并没有在 README 里堆砌 AI buzzword。它的 AI 能力被封装在 agents 这个 building block 里，作为 CRM 原语的一部分，跟 objects、views、workflows 平级。这意味着 AI 不是外挂的 chatbot 侧边栏，而是跟数据模型和业务流程同一层级的基础设施。

这里有一个值得展开的行业背景。2024-2025 年，Salesforce 在 AI 上的投入可以用“疯狂”来形容。Einstein GPT、Einstein Copilot、Agentforce——产品名字换了一茬又一茬，但底层逻辑没变：把 AI 作为一个增值功能层叠在现有的数据平台之上。这种做法的优势是能快速利用已有的客户数据，劣势是 AI 的行为和权限被现有的数据模型和业务流程限制了手脚。一个 Einstein Copilot 可以帮你总结客户邮件，但它不能重新定义“lead”这个 object 的字段结构，不能自动发现销售数据里的新模式并建议你修改 pipeline stage 的定义。

Twenty 的 agents 原语，至少在架构意图上，指向了不同的方向。当 AI 代理跟 objects 和 workflows 处于同一层级时，它理论上可以直接读写数据模型、触发自动化流程、甚至根据业务变化建议修改 schema。当然，这个“理论上”离“实际上”有多远，取决于 Twenty 的 agents 做到到底做到了哪一步——项目文档在这方面着墨不多，但架构设计本身已经暴露了野心。

（值得一提的是，Twenty 的文档里专门有一个“Learn more about AI in doc”的链接，但截至 2025 年 7 月，那个页面的内容还比较骨架化。这要么说明 AI 功能还在早期开发阶段，要么说明团队在有意控制信息披露的节奏——对于一个对标 Salesforce 的开源项目来说，两种可能性都合理。）

从 Salesforce 到 Twenty：迁移不是技术问题，是权力问题

任何一个在真实企业环境里做过系统迁移的人都明白一个残酷的事实：技术兼容性从来不是迁移的最大障碍，组织政治才是。 这句话在 CRM 领域尤其成立。

Salesforce 在企业内部的权力结构里占据了一个非常特殊的位置。它不只是销售团队用来记录客户信息的工具，它是销售 VP 用来做 forecast 的数据源，是市场部门用来做归因分析的触点库，是 CFO 用来算 CAC 和 LTV 的真相来源。Salesforce 管理员（Salesforce Admin）在很多公司里是一个独立的岗位，甚至是一个团队。这些人花了数年时间考取各种 Trailhead 认证，他们的职业身份和 Salesforce 生态深度绑定。你跟他们说“我们可以用一个开源工具替代 Salesforce”，他们听到的不是“更好的技术方案”，而是“你的技能组合可能贬值”。

Twenty 的定位——“for technical teams”——巧妙地绕开了这个问题。它没有试图去说服 Salesforce Admin 放弃他们的配置界面，而是直接去找 CTO 和工程 VP，说：你们的工程师已经在用 Git、写 TypeScript、跑 CI/CD 了，为什么 CRM 不能进入同一个工作流？这个策略的聪明之处在于，它把 CRM 的决策权从业务部门手里重新夺回到工程部门手里。

这不是一个技术主张，这是一个组织权力主张。

过去 15 年里，SaaS 的崛起本质上是业务部门绕开 IT 部门直接采购软件的过程。Salesforce 是这场运动的旗手。Shadow IT 从贬义词变成了默认工作方式。但 AI 时代的到来正在逆转这个趋势。当数据安全、模型训练、agent 权限这些东西变成董事会级别的话题时，IT 和工程部门正在重新夺回技术采购的否决权。Twenty 在这个时间点出现，押注的就是这个权力转移。

再看 Twenty 的社区策略。Discord、Feature requests、Crowdin 翻译——这些是典型的开源项目社区建设工具。但 Star the repo 被放在了 Join the Community 的第一位。GitHub star 在 2025 年已经变成了一种半正式的社交货币，它的数量直接影响项目的可信度、融资估值和招聘吸引力。Twenty 对此毫不掩饰——它就是要把 star 数做上去，用社区热度来对冲 Salesforce 的品牌壁垒。这个策略在开源商业化的历史上被反复使用：MongoDB 当年用开发者社区的狂热倒逼企业采购部门，Grafana 用 GitHub star 和 Docker 下载量撬开了监控市场的裂缝。Twenty 在重走这条路。

开源 CRM 的幽灵：为什么之前没人做成，以及这次有什么不同

如果你对开源 CRM 这个赛道稍有了解，你可能会有一个合理的质疑：这又不是第一次有人想干掉 Salesforce。SugarCRM 2004 年就开源了，后来转向了专有许可；SuiteCRM 作为 SugarCRM 的 fork 活到了现在，但从未进入主流视野；EspoCRM、CiviCRM、Odoo——每一个都在某个细分场景活着，但没有一个撼动过 Salesforce 的地位。Twenty 凭什么不一样？

答案可能在三个维度上。

第一，时间窗口。之前的开源 CRM 诞生在 pre-cloud 或 early-cloud 时代，那个时期的企业软件采购逻辑是“买一个功能完备的套件”。Salesforce 用“clicks not code”在那个时代是降维打击。但 2025 年的采购逻辑变了。企业需要的不是功能最全的 CRM，而是能跟现有数据栈（Snowflake、dbt、Airflow）、现有开发工具（GitHub、VS Code、Vercel）、现有 AI 平台（OpenAI API、Anthropic API、自部署模型）无缝集成的 CRM。Twenty 的技术栈选择——TypeScript、PostgreSQL、Redis——每一项都是现代数据栈的标准组件。它不需要发明新的连接器，它本身就生长在现有的技术生态里。

第二，AI 的介入改变了 CRM 的价值链。传统 CRM 的核心价值是“记录系统”（System of Record）——确保客户数据准确、完整、可审计。这个价值在 AI 时代没有消失，但它变成了基础设施层面的东西，不再是差异化竞争的来源。新的差异化来自“行动系统”（System of Action）——谁能更快地把数据转化为销售动作。Salesforce 的 Agentforce 是在往这个方向走，但它受限于自己的数据模型和权限体系。Twenty 把 agents 作为一级原语，在架构上给了 AI 更大的操作空间。

第三，分发模式的不同。SugarCRM 当年的开源是“open core”——核心功能开源，高级功能收费。这个模式在开发者社区里已经被证明是难以持续的，因为它本质上是一种钓鱼策略，开发者能嗅到那个“免费只是诱饵”的味道。Twenty 目前的模式更接近 GitLab 早期的“open source + cloud hosting”——代码完全开源，自托管免费，云服务收费。这个模式在开发者群体中的信任度远高于 open core。

但话说回来，这三个优势都是结构性的、理论上的。它们能否转化为实际的市场份额，取决于 Twenty 团队的执行力、社区的活力、以及 Salesforce 的反应速度。Salesforce 不是那种会坐以待毙的公司。Marc Benioff 在 2024 年的几次公开演讲里反复提到“开源 CRM 的威胁”，说明他已经意识到了这个赛道的升温。以 Salesforce 的资源，它完全可以在 6-12 个月内推出一个“Salesforce Open Edition”或者大幅降低 Developer Edition 的门槛来应对。Twenty 的窗口期不会很长。

戏谑收尾

Twenty 目前还很小。GitHub star 数、Discord 成员数、Cloud 注册用户数——这些数字在 Salesforce 的季度财报面前不值一提。但我看到它的 README 时，脑子里闪过的不是又一个开源项目挑战巨头的故事，而是一个更具体的画面：某天凌晨，一个 CTO 在 Slack 里发了一条消息：“我们决定把 CRM 迁移到 Twenty。”底下没人反对，因为所有工程师都知道这意味着什么——CRM 终于可以被写进 docker-compose.yml 里了，跟应用服务、数据库、缓存放在一起，用一个 docker compose up -d 就能在本地跑起来。Salesforce 永远给不了这种体验。

Salesforce 让 CRM 变得每个人都能用。Twenty 让 CRM 变得每个工程师都想用。前者是民主化，后者是文艺复兴。

至于这场文艺复兴能不能烧到 Salesforce 的大教堂——那得看 Twenty 的 agents 原语到底能释放出什么，以及 Salesforce 的 AI 税还能收多久。
