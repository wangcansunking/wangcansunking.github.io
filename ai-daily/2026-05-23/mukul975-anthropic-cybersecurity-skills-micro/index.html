754个结构化网络安全技能：GitHub项目如何重塑AI代理决策路径

ai-daily · 2026 年 5 月 23 日 21:41 · 来源：GitHub Trending python

2026 年春天某个凌晨，一位安全分析师被手机震醒。不是告警——是他上周写的 Volatility3 内存取证脚本又跑崩了，CI 流水线发的通知。

他揉着眼睛打开 GitHub，看到一条 trending 仓库：Anthropic Cybersecurity Skills。点进去扫了 30 秒，嘴里蹦出一句脏话，然后点了 Star。

不是因为项目名里挂着 Anthropic 的招牌（实际上这是个社区项目，跟 Anthropic PBC 没半毛钱关系），而是因为他意识到一件事——他花了三年积累的那些“什么时候该跑哪个插件、怎么判断结果是不是假阳性”的肌肉记忆，现在被压缩成了一个 AI agent 可以在 30 个 token 内检索到的结构化技能文件。

754 个技能，5 套框架映射，26 个安全域，一次克隆就能让 AI agent 拥有高级分析师的决策能力。

为什么说这玩意儿戳中了安全行业的肺管子

ISC2 的数据大家都背得出来：2024 年全球网络安全岗位缺口 480 万。这个数字在行业会议上被反复引用，几乎成了一句正确的废话——因为大家都知道缺人，但没人能变出 480 万个有实战经验的分析师。

问题的要害不是“人不够”，而是“经验无法复制”。一个干了五年的 DFIR 分析师，脑子里存着几百个决策分支：看到 LSASS 进程异常访问，是先跑 Mimikatz 检测还是先查事件日志？内存镜像里有 PowerShell 脚本块日志，该不该关联到 ATT&CK T1059.001？这些判断路径靠的是踩坑踩出来的直觉，不是培训手册能教会的东西。

而这个仓库做的事情很直接：把这种直觉拆成结构化的 YAML frontmatter + Markdown 工作流，让 AI agent 在 30 个 token 的扫描成本下完成技能匹配，然后加载 500-2000 个 token 的完整执行流程。

不是教 AI 敲命令，是教 AI 什么时候该敲哪个命令，敲完怎么判断结果对不对。

具体怎么跑的？仓库文档里给了一个例子。用户说“分析这个内存镜像，看有没有凭据窃取的痕迹”，agent 的内部流程是这样的：第一步，扫描全部 754 个技能的 frontmatter（每个约 30 token），用标签和描述匹配出 12 个相关技能；第二步，加载前三个最匹配的——performing-memory-forensics-with-volatility3、hunting-for-credential-dumping-lsass、analyzing-windows-event-logs-for-credential-access；第三步，按照 Workflow 章节一步一步执行 Volatility3 插件、检查 LSASS 访问模式、关联事件日志证据；第四步，用 Verification 章节的标准确认 IOC，把发现映射到 ATT&CK T1003（凭据转储）。

没有这些技能的时候，agent 靠猜命令。有了之后，它走的是一条高级 DFIR 分析师会走的同款路径。

五套框架一张表，合规审计的隐藏杀招

我觉得最有意思的不是技能数量，是那个框架映射表。

看一眼单个技能的映射关系：analyzing-network-traffic-of-malware 这个技能，同时打到 ATT&CK T1071（应用层协议）、NIST CSF DE.CM（持续监控）、ATLAS AML.T0047（AI/ML 模型窃取）、D3FEND D3-NTA（网络流量分析）、AI RMF MEASURE-2.6（AI 系统监测）。

一个技能，五个合规复选框。

这件事的隐藏价值在 2026 年的监管环境里会被放大。科罗拉多州的 AI 法案（Colorado AI Act）已经在 2026 年 2 月生效，其中明确写了一条：组织如果遵守 NIST AI RMF 框架，可以获得法律安全港保护。这意味着那些用 AI agent 做安全运营的团队，如果能证明 agent 执行的每个动作都映射到了 AI RMF 的 72 个子类目，合规审计的难度会断崖式下降。

这个仓库恰好把 NIST AI RMF 1.0 的四个核心功能（Govern、Map、Measure、Manage）和 2024 年 7 月发布的 GenAI Profile（AI 600-1，覆盖 12 个生成式 AI 特定风险类别）都映射进去了。同时还有 MITRE ATLAS v5.4——这个版本是 2025 年底更新的，新增了 agentic AI 攻击向量：AI agent 上下文投毒、工具调用滥用、MCP 服务器入侵、恶意 agent 部署。也就是说，你用 AI agent 跑安全任务，这个库也在帮你防 AI agent 本身被攻击。

（值得一提的是，ATT&CK v19 已经在 2026 年 4 月 28 日落地，把 Defense Evasion 拆成了 Stealth 和 Impair Defenses 两个新战术。仓库文档里明确写了“即将更新映射”，这种对标准演进的敏感度在开源安全项目里不算常见。）

兼容性列表也值得扫一眼。Claude Code、GitHub Copilot、OpenAI Codex CLI、Cursor、Gemini CLI、Devin、Replit Agent、LangChain、CrewAI、AutoGen——二十几个平台全部零配置加载。遵循 agentskills.io 标准这件事意味着它不是一个“给某个特定 agent 写的插件”，而是一套可移植的技能规范。你今天在 Claude Code 里跑威胁狩猎，明天换 Gemini CLI 做同样的任务，技能文件不用改一行。

当然，仓库的短板上也写得很诚实。Deception Technology 域只有 2 个技能，Compliance & Governance 只有 5 个。26 个域里分布极度不均——Cloud Security 有 60 个技能，OT/ICS Security 有 28 个，Ransomware Defense 只有 7 个。这反映的不是项目组偷懒，而是安全行业本身的注意力分配：云安全和威胁狩猎是过去五年砸钱最狠的方向，工控安全和欺骗防御一直相对边缘。现在有人把这些技能缺口用数字标出来了，反而成了一张行业投资热力图。

社区反应也挺有意思。Hasan Toor（@hasantoxr）的原话是“A database of real, organized security skills that any AI agent can plug into and use. Not tutorials. Not blog posts.”这个评价精准地抓住了它和传统安全知识库的区别——不是教你学，是让 agent 直接用。另一个贡献者 fazal-sec 在 Medium 上写的是“structured operational knowledge base designed for AI-driven security workflows”，强调的是“操作知识库”而非“脚本合集”。

项目维护者 Mahipal Jangra 还在同步跑一项叫 GARS-2026 的全球学术调研（Global Agentic AI Readiness Survey），60 道题，匿名，SRH Berlin 监督，结果以 CC-BY 4.0 开放。调研内容直指 MCP 服务器、工具调用、治理和人机协同工作流——说白了就是想知道安全从业者对 agentic AI 到底准备好了没有。这跟仓库本身形成了一个闭环：一边给 agent 喂技能，一边测行业吸收能力。

截止到写这篇文章的时候，仓库的 Star 曲线还在往上走，v1.0.0 是 2026 年 3 月 11 日发的，当时 734 个技能、26 个域、ATT&CK + NIST CSF 2.0 双映射。不到两个月，main 分支已经涨到 754 个技能，ATLAS、D3FEND、AI RMF 三套映射全加上去了。按这个速度，Deception Technology 域从 2 个技能涨到两位数可能也就是下个 PR 的事。

安全行业折腾了这么多年“自动化”，大部分时候其实只是在自动化告警——把日志聚合一下、规则匹配一下、弹窗推一下。真正需要经验判断的环节，还是得人盯着。这个仓库做的事，是把“人盯着”那部分经验拆成了 agent 能执行的步骤。480 万的缺口不会因为一个 GitHub 仓库就填平，但如果每个初级分析师都能带一个装好了 754 个技能包的 AI agent 上岗，那个缺口至少会开始以另一种方式愈合。

安全行业缺的不是工具，是把工具用对的判断力。现在判断力也有开源版了。
