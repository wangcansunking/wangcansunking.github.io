谷歌TimesFM开源2.5版：砍掉六成参数，预测能力反增数倍

ai-daily · 2026 年 5 月 23 日 19:23 · 来源：GitHub Trending python

有时候，一个模型在 GitHub 上静悄悄地更新，比那些发布会上的锣鼓喧天更让我在意。

Google Research 的 TimesFM 就属于这种。它没有占据科技头条，但在过去一年多里，它从一篇 ICML 2024 的论文出发，一路迭代到 2.5 版，顺便把自己塞进了 BigQuery、Google Sheets 和 Vertex AI 这些每天被几百万人在用的产品里。

我看到 2026 年 4 月 9 日那个更新记录的时候愣了一下——这个团队还在给它加 LoRA 微调支持，还在修社区提的 bug。一个时间序列预测模型，活得像个正经的开源项目。

一个 200M 参数的小模型，想干翻你硬盘里那些跑了十年的统计脚本。

从 500M 缩到 200M，反而更能打了

TimesFM 2.5 做了一件反直觉的事：参数砍掉六成，能力却翻了几倍。

2025 年 9 月上线的 2.5 版，把参数量从 2.0 版的 5 亿直接压到 2 亿。但上下文窗口从 2048 拉到 16000——这意味着你可以把更长的时间序列扔给它，让它自己琢磨里面的模式。预测范围也扩到 1000 步，还挂了一个可选的 3000 万参数量化头，能输出连续的分位数预测。

说人话就是：你不仅能拿到一个点预测，还能拿到「10% 分位到 90% 分位」的区间。这对库存管理、电力调度、金融风控这些场景来说，比单个数字有用得多。

他们还干掉了频率指示器（frequency indicator）。老版本需要你告诉模型「这是小时级数据」「这是日级数据」，2.5 版直接不要了。模型自己从数据里推断周期模式。这个改动看着小，实际上把使用门槛砍了一截——你不用再翻文档查频率代码了。

藏在 Google 全家桶里的暗线

TimesFM 最让我觉得有意思的，不是模型架构本身，而是它的分发路径。

Google 把它塞进了三个产品：BigQuery ML 让你在 SQL 查询里直接调时序预测，Google Sheets 让普通人在电子表格里点两下就出预测，Vertex Model Garden 则给开发者一个 Docker 化的端点，方便接到 Agent 工作流里。

这条路径的聪明之处在于，它覆盖了三类完全不同的人。数据工程师在 BigQuery 里写 SQL 的时候顺手把预测跑了，根本不用切到 Python 环境。财务或运营人员在 Sheets 里用，连代码是什么都不用知道。而做 AI Agent 的团队在 Vertex 上直接调 API，把预测塞进自动化流程。

一个模型，三种活法。

而且它开源的部分写得相当干净。用 uv pip install -e .[torch] 一行就能装好，12 行代码跑通预测，支持 PyTorch 和 Flax 双后端。2026 年 3 月还加了对 AI Agent 的 SKILL.md 支持，4 月补了 HuggingFace Transformers + PEFT 的 LoRA 微调示例。这个维护节奏，在 Google 的开源项目里算是相当勤快的。

基础模型这个赛道，时间序列还没打完

时间序列预测一直是个奇怪的领域。传统方法像 ARIMA、Prophet 在企业里扎根极深，深度学习模型喊了很多年要取代它们，但实际落地总差口气——要么训练太麻烦，要么泛化能力不够，换个数据集就崩。

TimesFM 走的是「解码器-only」路线，和大型语言模型的结构一脉相承。它的赌注很清楚：在海量时间序列上预训练，让模型学到跨领域的周期模式、趋势特征和异常形态，然后你拿自己的数据做零样本预测或者轻量微调就行。

这个思路和 NLP 领域 GPT 的崛起逻辑是一样的。只是时间序列的「文本语料」更碎片化——电商的日销量、电厂的分钟级负荷、气象站的小时级温湿度，每家的数据格式和采样频率都不一样。

TimesFM 2.5 用 16000 的上下文窗口和去掉频率指示器这两步，就是在试图解决这个碎片化问题。让模型自己去适应数据的节奏，而不是让人去迁就模型。

至于它能不能真的取代 ARIMA 在企业系统里的位置，现在还不好说。但当一个模型已经嵌入 BigQuery 和 Sheets，每天被几百万次调用的时候，「能不能」这个问题，慢慢就变成了「什么时候」。
