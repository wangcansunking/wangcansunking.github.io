OpenAI庆祝Codex 500万用户，福利重置引发争议，Claude Code占近九成消耗

OpenAI 的 Codex 负责人 Tibo 在 X 上发了一条庆祝帖，语气轻松得像在派发节日礼物：“所有付费版 ChatGPT 订阅用户的 Codex 使用额度已完成重置。你的每周额度和每小时额度都应该已经恢复到 100%。”这场“重置狂欢”的由头是 Codex 用户突破了 500 万——一个值得写进 OKR 的里程碑。

但评论区翻车了。有用户精确计算了自己的损失：“我的每周额度原本应该在 6 号重置，结果现在变成了 7 号。如果额度本来还是 100%，这次统一重置就不应该把我的重置日期往后推。”另一位用户直接下了判词：“实际上，它并没有给大多数人带来任何真正的好处，只是表面上的重置而已。”

一个本该拉拢开发者的“福利”，变成了精准计算个人损益的会计辩论。而与此同时，AI 成本管理平台 CostHawk 放出了一组匿名数据：在追踪的 100 名重度 AI 编程用户中，Claude Code 吃掉了 88.9% 的 Token 消耗量，Codex 只占 11.1%。Cursor 更惨，占比趋近于零。

500 万用户庆祝帖下面吵成一团，隔壁 Claude Code 正悄无声息地吞掉九成 Token。这画面本身就是一个产品寓言。

一场“额度重置”引发的数学战争

OpenAI 这次的操作逻辑本身并不复杂：统一清空所有人的使用计数器，让触顶的用户重新跑起来。问题出在执行精度上。重置时间点对不同用户意味着完全不同的收益曲线——刚重置过的人等于白送 OpenAI 一个免费刷新日，还没重置的人赚了几天额外额度，而像 Jimmni 这样刚好卡在重置前夜的用户，直接亏掉了一整天。

更微妙的是额度的计量机制。有用户指出，在小于 7 天的窗口期内，你不可能真正处于 100% 额度状态，因为只要发送一条 prompt 启动计量周期，额度就会跌破 100%——尽管四舍五入后界面可能仍显示为满格。这意味着“统一重置”本身就是一个近似值操作，而 OpenAI 没有向用户解释这个近似值的误差范围。

这种摩擦本质上暴露了一个更深层的问题：Codex 正在从“ChatGPT 订阅附赠能力”转向“Token 化计费系统”，但用户端的感知仍停留在“我有多少免费额度”的简单心智里。OpenAI 在 4 月初已经将 Codex 定价从“按消息平均扣点”改为按每百万 Token 消耗 credits，输出 Token 明显更贵，不同模型对应不同费率。这意味着长上下文、多轮修复、代码审查这些行为，都会被拆解回更细颗粒度的成本。而“重置额度”这种粗放操作，跟正在搭建的精细计费系统之间，存在明显的产品逻辑断层。

与此同时，100 美元 Pro 档的“双倍额度限时活动”也制造了额外的混乱。帖子下有人追问双倍额度是否仍在生效，得到的回复是“按太平洋时间计算，当天结束前仍然有效”。一场全球用户同时参与的福利活动，截止时间用的是美国西海岸时区——欧洲开发者可能已经在睡梦中错过了窗口期。

Claude Code 的 88.9%，靠的不是“聪明”而是“烧”

CostHawk 那份排行榜的数据粗看是一边倒的碾压：Claude Code 消耗约 3697 亿 Token，Codex 消耗约 462 亿 Token，Cursor 只有 77 万 Token。100 名 operator 中，96 名使用 Claude Code，43 名使用 Codex，仅 2 名使用 Cursor。仅用 Claude Code 的用户占 56%，多工具用户占 40%，仅用 Codex 的只有 3%。

但如果拆开来看，“Token 消耗量”这个指标本身就有误导性。有开发者在一次标准 Figma 集成任务中做了对照实验：相同提示词、相同代码仓库、相同目标输出下，Codex 完成基准任务消耗约 7.2 万个 Token，Claude Code 约消耗 23.5 万个 Token——后者是前者的 3 倍。这不是“用得更多”，而是“更费 Token”。

Theo - t3․gg 对此有一个精准的概括：Claude Code 的底层哲学是“如果更多 token 能解决问题，那就使用更多 token”。sub-agent 并行检查项目、大量 agent 审计代码、在终端里执行更多操作——这些设计确实让用户感觉“非常高产”，终端动画和 token 计数器也在强化一种“事情正在大量发生”的生产力感，非常适合截图发 X。但代价是更高的 Token 账单。

Codex 走的是另一条路。Theo 的描述是“没有试图让自己像老虎机一样令人上瘾”。界面更克制，运行时没有大量动画和多 agent 展示，更多是一个简单的工作状态、计时器和任务输出。但它在工程细节上下了笨功夫：支持 Mac 锁屏状态下继续工作、支持新的 diff marker 设置、支持用快捷键把当前应用画面发送给 Codex 作为上下文。这些功能不适合截图传播，但能切实减少“反复检查”带来的无效 Token 消耗。

OpenAI 的核心思路是：不是用更多 token 反复验证，而是让模型用 computer use 实际查看运行结果，以更少 token 得到更可靠结论。这解释了为什么 Codex 的 Token 效率更高，但也解释了为什么在“开发者心智”上被 Claude Code 压了一头——后者把“我为你做了很多事”这个信号打满了，而前者只给你一个安静的终端。

但 Claude Code 的高消耗模式正在遭遇定价反噬。Claude Mythos 的定价被曝光后——25 美元/每百万输入 Token、125 美元/每百万输出 Token——社区直接炸了。有开发者算了一笔账：“按这个价格，Mythos 上一次深度推理会话的成本，可能就相当于过去 Claude Sonnet 一整个月的使用费用。”评论区出现了“给有钱技术宅准备的新档位”和“不适合穷人”的调侃。Anthropic 正在把高端模型能力瞄准基础设施层客户，而个人开发者被价格墙挡在外面。

这才是 Codex 真正的机会窗口。当 Claude Code 的忠实用户因为几天内耗尽每周 Token 而转向 Codex 时，他们发现 OpenAI 的聊天界面不计入同样的五小时窗口限制——网页端可以做初步规划和小型编码任务，不会烧掉珍贵的订阅额度。而 Claude 网页聊天也会消耗同一订阅窗口。一位此前几乎只用 Claude Code 的开发者坦言：“这就是 Claude 难以与 OpenAI 竞争的原因之一。”

Cursor 以 100% 的月增长率在角落里蛰伏，但基数小到几乎可以忽略。Theo 认为外界低估了 Cursor Cloud 的潜力——从 Slack 中 @Cursor bot 直接启动 agent 修复问题，并在同一 thread 返回修复后的视频证明，这种“从协作工具发起任务再返回可验证结果”的流程，是 Claude Code 和 Codex 当前都难以达到的。但那是“更远的未来”，当下的战场仍然是终端和 IDE。

500 万用户的庆祝帖下，OpenAI 送出的是一把额度重置的撒花，而开发者们掏出计算器开始做减法。Claude Code 烧掉了九成 Token，但烧得越狠，用户对价格的敏感度就越高。Codex 安静、克制、Token 效率高，却还没学会怎么让开发者感觉到“我很努力”。这场 AI 编程工具的战争，目前还没人真正打赢——只是有人在用更多 Token 假装赢了。
