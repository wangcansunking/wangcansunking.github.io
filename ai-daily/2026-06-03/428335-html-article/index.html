OpenSquilla发布Meta Skill：400行代码管理长程Workflow，效率飙升

一个连团队都不公开的项目，正在重写 Agent 的底层逻辑

六一儿童节前一天，我让一个 9 岁小孩（虚构的）去给 GitHub 上突然爆火的一个开源项目写说明书。

这个项目叫 OpenSquilla，当时刚冲到 2000 多 star。我选了一个叫 meta-kid-project-planner 的预制 Skill，塞进 Agent，给了一句 Prompt：让孩子做一本 Meta Skill 魔法书，每页介绍一个咒语。

然后我就去泡咖啡了。

20 分钟后回来，屏幕上躺着一份完整的 7 天项目规划包——从材料清单、安全审查、家长参与度评估，到户外活动自动查天气、甚至给小朋友额外做了一轮安全合规检查。全程没有人在中间点过一下鼠标。

那一刻我意识到，这已经不是「工具在干活」，而是「一个项目经理在替你管外包团队」。

更离谱的是，跑完这一整套长程 Workflow，Token 成本比我预想的低了 60%-80%。有个处理简单查询的子步骤，账单只跳了三分钱。

三分钱。

我盯着那个"老虎机"一样的动画界面愣了五秒——它每跑完一个子任务，屏幕就会弹出一个提示，告诉你这一步省了多少 Token，用的是哪个模型，为什么选它。

那种感觉，就像你雇了个会计兼采购，每一步都帮你死死夹紧钱包。

而这一切的核心，就是 OpenSquilla 刚发布的新功能：Meta Skill。

Meta Skill 不是 Skill，是 Skill 的「项目经理」

OpenSquilla 这个名字，第一次出现在大众视野是 2026 年 5 月初。当时最抓眼球的卖点是「智能模型路由」——同一个任务，能比当时最火的 Agent 框架「龙虾」省 60%-80% 的 Token 成本。

但今天要聊的不是路由。

是 Meta Skill。

名字有点抽象，但你把它理解成"Skill 的 Skill"就对了——一个 Meta Skill 内嵌多个原子 Skill，拼接到一块，就是一本端到端的超级白皮书，能打通整套长程 Workflow。

原文作者 Jay 在量子位的文章里提到，他上上周在筹备 AIGC 产业峰会后台工作自动化时，就踩过这个坑。峰会的 SOP 拆得很细，每个蓝色方块都需要在对话框单独调用一个 Skill——"像癞蛤蟆一样戳一下动一下，全程 Human in the loop，光翻个 Skill 列表就够忙活半天。"

后来他手动把所有上下文炼化成一个超级 Skill，加上心跳机制定时查看状态文档，才勉强端到端跑通。但整个过程很折磨，而且最终交付也粗糙。

这还是建立在他对内容行业有 Know-How 的前提下。如果涉及跨行业专家经验整合，"排列组合 Skill 那真得是灾难了。"

这就暴露了一个被社区集体忽略的问题：Skill 在爆发式增长，但「如何组织 Skill」这件事，还停留在手工作坊阶段。

OpenSquilla 这次的解法，是直接封装了 9 个预制 Meta Skill，内嵌在 Agent 里，开箱即用。

我实测的那个 meta-kid-project-planner，翻开源 SKILL.md 文件，400 多行，工作流清晰到令人发指：

  1. 立项：询问用户偏好——年龄、周期、预算、家长参与度。
  2. 可行性分类：判断安全性、是否需要大人协助、是否需要额外购买材料。
  3. 执行：分步计划 → 材料清单 → 安全提醒 → 家长学习目标 → 最终组装交付。
  4. 如果涉及户外活动：自动调用 web search 查天气。

5 个不同的原子 Skill 拼接而成，最终吐出一份 3000 字左右的项目规划 Markdown。整个过程 Agent 自己跑了 20 多分钟，无需人类介入。

（值得一提的是，它甚至自动做了一轮安全审查——因为接收方是儿童。这个细节让我愣神了好一会儿。）

那个「老虎机」动画背后，是一套正在成型的算力经济学

Meta Skill 解决了"谁来安排活"的问题。但光有项目经理不够。

一个只会排活的 PM，如果每一步都叫最贵的外包来干，公司要不了多久就得倒闭。

这就是智能路由承担的角色。

OpenSquilla 的路由逻辑，不是简单的"复杂任务用大模型，简单任务用小模型"。它在你跑任务的每一步，实时判断这一步的难度层级，然后动态分配模型。

比如那个 kid-project-planner：提取孩子年龄和偏好，这种活 DeepSeek 就够了，成本极低；但生成安全审查方案和 14 天规划这种步骤，才需要分配给参数更大的模型。

结果就是——经常怒省 90% 多的 Token。

原文贴了一张 Token 成本 Benchmark 截图，"还是挺有冲击力的。"（原文没有给出具体 Benchmark 数字，但从"省 60%-80%""怒省 90% 多""三分钱"这些表述来看，成本压缩幅度相当可观。）

而这个路由系统，目前只在 Web 版上才有完整的可视化——一个「老虎机」式的动画，任务跑完弹出来，显示节省了多少 Token，还能溯源各环节消耗量。终端或 IM 端没有这个界面。

"如果用终端或 IM 的话，会少蛮多快乐的。"Jay 在文章里这么写。我觉得这不是快乐的问题——这玩意儿本质上是一个成本可观测性面板，让你在每一次 Agent 调用中都能直观感知到"钱花在哪儿了"。

这对企业级落地来说，比任何技术炫技都更有说服力。

因为大规模跑大模型依然贵。你每次让 Agent 在线上 trial-and-error，反复摸索最优路径，Token 烧掉一大堆。而 Meta Skill 做的事，是把复杂度固化到 Skill 层——优化问题被前置了，线上执行只需要按手册走。

这让我想起一个类比：软件开发从瀑布到敏捷再到 DevOps，本质上都是在解决"怎么把工作流固化、自动化、可观测"。Agent 编排现在正在走同一条路。

9 个 Meta Skill 背后，藏着一个「Skill 推荐引擎」的野心

OpenSquilla 这次发布的 9 个 Meta Skill，每个都有完整的 SKILL.md 源文件，可以直接在 GitHub 上读到。

但最让我在意的，不是这 9 个本身——而是团队已经想好了"当 Skill 膨胀之后怎么办"。

这个问题很现实。现在仓库里只有 9 个 Meta Skill，但如果未来有上百个，你怎么知道哪个最适合你的场景？

OpenSquilla 给出的方案叫「个人 × 社区」的索引协议。

你平时常用哪些 Skill、偏好什么组合顺序、哪个试过不好使——这些数据会作为信号，被 Agent 拿去匹配社区里别人做好的 Skill，然后根据你的工作流缝合出一个新的。

简单说，就是一个自动的 Skill 推荐引擎。

而且这个引擎不是纯靠关键词匹配。它记录的是你的行为信号——你用过什么、弃用过什么、偏好什么顺序组合。这意味着推荐会越来越贴合你的实际工作流，而非泛泛的"热门 Skill 排行榜"。

这步棋下得很深。因为它把"Skill 发现"这个看似边缘的功能，做成了整个生态的护城河。

当社区贡献的 Skill 足够多，推荐引擎足够精准，用户迁移成本会急剧上升——你换一个 Agent 框架，不只是换工具，而是丢掉一整条"你调教了很久的工作流推荐链"。

而在这 9 个 Meta Skill 中，最重要的一个，Jay 认为是 meta-skill-creator——一个创建 Meta Skill 的 Meta Skill。

"这也是我觉得 OpenSquilla 这次发布的 9 个 Skill 中，最重要的一个 Skill。"原文这样说。

逻辑很直白：Meta Skill 是好，但创建门槛太高。那个 kid-project-planner 的 SKILL.md，400 多行，即便跟 AI 迭代也要大概 30 分钟，而且前提是你脑海中已经有了清晰的 SOP。

所以需要一个专门用来创建 Meta Skill 的工具，把"编排 Skill"这件事也自动化掉。到这一步，整个闭环才算真正合上。

5 月初的智能路由，到 6 月的 Meta Skill——这个产品在想什么？

我翻了一下这个仓库的版本历史。原文作者 Jay 也做了同一件事，而且给了一个很有趣的视角：

"可以像抽丝剥茧一样，逐步揭开他们的产品思考。"

5 月初是靴子首次落地，发布了智能路由。那时候外界以为，这只是个做 Token 优化的工具。

但到了 6 月，Meta Skill 出来之后，这个脚印开始清晰了——而且指向了一条反直觉的方向。

它同时踩在三条线的交点上：

  1. 模型能力到位了。 复杂多步骤指令的理解能力这两年飞速拉升，模型已经"听得懂"复杂的编排指令了。这是一切的前提。
  2. Skill 生态在爆发。 从用户手写，到基于数据自动生成，再到社区汇集分享——当可选的 Skill 有成千上万个的时候，需要一个更高的抽象层（Meta Skill）去简化筛选成本。
  3. 成本天花板还没破。 大规模跑大模型依然贵。用 Meta Skill 把复杂度前置固化，线上执行只按手册走，Token 消耗大幅降低。

这三个痛点，同时指向了同一个正在被倒逼出来的范式迭代：Skill 2.0。

单个 Skill 已经不够用了。自动化想要进一步深入，必须要学会对多个子 Skill 排列组合。Agent 下一步要解决的问题，已经从"会不会调用工具"，变成了"会不会组织工具"。

这个判断，跟我最近的观察完全吻合。腾讯有 Marvis、MiniMax 有 Mavis、Kimi 有 Agent 集群——模型层面大家都在推 Agent 团队。但 Skill 层似乎还停留在被 Claude 带火时的那个阶段：社区基本都在为单个模型写 SKILL.md。

"多 Agent 的潜能，其实一直没能被完全释放。"Jay 在文章里写了这么一句。我觉得这话说对了一半——不是潜能没释放，而是释放潜能的"组织层"一直缺位。

Meta Skill 的出现，让我看到了一种可能性：专为 Agent 团队设计的白皮书，赋予模型更宏观的全局上下文。

Agent 和人类在面对的很多核心问题上，其实殊途同归。当员工（Agent）变多，业务（Skill）变多，必然会遇到指数级放大的噪音。此时，如何善用架构和管理去做熵减，就非常有必要了。

团队是谁？——一个连 X 账号都没公开身份的项目

写到这，有个事必须交代。

OpenSquilla 这么火，2000 多 star，第一个版本就被 AK（Andrej Karpathy）转发过。但它的团队一直是个谜。

只有 X 上有官方账号，推文风格克制到近乎冷淡。没有任何团队介绍页面，没有融资公告，没有创始人采访。

Jay 在文章里写，他 AI 挖掘半天无果，最后"选择发动古法，四处找人打听"。

结果挖出了一个名字：王云鹤。

来自他创立的一家叫「基元律动」的公司。团队正在大力招兵买马，所以之前对外保持低调。

"原来只是团队刚刚开始，希望专注打磨产品。"Jay 在结尾处写了这么一句，然后接了一个意味深长的收尾："让子弹再飞一会儿，我们拭目以待。"

这个信息量其实不小。王云鹤这个名字，在 AI 圈并不陌生（原文没有提供更多个人背景，此处不做延伸猜测）。而「基元律动」这个公司名，也暗示了团队对 Agent 底层架构的理解——"基元"是最小单位，"律动"是编排节奏，合在一起，几乎就是 Meta Skill 的产品哲学。

不过更让我好奇的是另一件事：这个团队选择在 GitHub 上裸奔，不开发布会，不发 PR 稿，甚至不公开身份——却第一个版本就被 Karpathy 转发。

这说明他们在开发者社区里的信任积累，不是从零开始的。

（至于这个信任是怎么来的，原文没写，我也不会编。但这是一个值得持续关注的信号。）

写到这，我突然想起 Jay 在文章里用了一个很妙的比喻，来形容 Meta Skill 的定位：

"这是用来指导 Agent 三省六部的白皮书。"

三省六部，中国古代中央行政体系的核心架构——尚书省管执行，中书省管决策，门下省管审核，六部分管兵刑工礼户吏。一个庞大的行政机器，靠的是一套清晰的职责分工和流程编排。

Meta Skill 做的事，本质上就是把这种"三省六部"式的编排逻辑，写进 Agent 的上下文里。

OpenSquilla 让我兴奋的，不是它的路由有多省 Token，也不是那 9 个预制 Meta Skill 有多好用。

而是它踩在了一个被大多数人忽略的断层上：模型和工具都在疯长，但连接它们的"组织力"还是一片荒地。

Skill 1.0 解决的是"Agent 会不会用锤子"。Skill 2.0 要解决的是"Agent 会不会管一个施工队"。

这个断层一旦被填上，Agent 能从"听话的实习生"变成"能独立管项目的初级经理"——而这一步跨越，可能比模型参数从千亿到万亿更有工程落地价值。

当然，这一切的前提是：OpenSquilla 能活到社区贡献出第 100 个 Meta Skill 的那天。

毕竟，一个连团队都还没公开的项目，说它有野心重塑 Agent 编排范式，听起来像是又一个 GitHub 流星。

但那个「老虎机」动画里蹦出来的"怒省 90%"，还有那份 9 岁小孩都能看懂的 7 天项目规划包，让我觉得——这流星，至少值得盯着它飞一会儿。
