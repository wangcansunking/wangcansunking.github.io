IBM：企业AI落地难？Agent Logic让token消耗降30倍

“LLM 很强，但企业级落地还是一塌糊涂。”这句话不是我说的——是 MIT 2025 年那份《The GenAI Divide》报告里的结论。IBM 这篇博客一开篇就把这个伤口重新撕开：无数 AI 试点项目死在了概念验证阶段，不是因为模型不够大，而是因为它们根本踩不进企业工作流的核心。

而 IBM 给出的答案不是换个更大的模型，不是等 GPT-6，也不是多堆几层推理。他们管它叫 agent logic——一套跑在 agent 层、不依赖模型本身能力的软件原语：知识图谱、程序分析库、算法编排、有向无环图。它们的作用只有一个：把 LLM 从“大海捞针”的上下文里拽出来，精准塞进工作流的窄巷子里。

让模型变聪明的方式，是别让它想太多。

四个战场，同一套逻辑——agent logic 不是概念，是已经在跑的生产系统

IBM 在这篇博客里摊开了四个真实落地场景，每一个都指向同一个结论：agent logic 不是在跟 LLM 抢戏，而是在给 LLM 当“导航”——而且效果是硬数字砸出来的。

第一个场景是让 AI 读懂上古代码。IBM 的 watsonx Code Assistant for Z（WCA4Z）里内置了一个 App Insights agent，专门用来理解跑在大型机上的 Cobol 和 PL/1 遗留系统。这些系统动辄 100 万行代码、1000 个程序，纯靠 LLM 去读，token 消耗是天文数字，而且容易产生幻觉。IBM 的做法是：先用深度静态分析把整个应用的代码结构预索引到一个跨越数百张关联表的数据库里，agent 在回答问题时不直接喂源码给 LLM，而是从这个结构化数据库里提取精准信息。

结果呢？在多个关键遗留系统上，应用理解性能跟纯 LLM 方案持平甚至略优，token 消耗却只有后者的 1/30。30 倍——这不是优化，是换赛道。而且他们用的模型是 Mistral Medium 250B，不是最顶配的前沿模型。导航系统足够好，发动机不需要是火箭。

第二个场景是测试生成。IBM 自研了一个叫 Aster 的程序分析库，专门做单元测试、集成测试、API 测试和变更测试的 agent 化生成。这个库已经在 75+ 个 Java 应用（最多 560+ 个类、6.7 万行代码）上跑预生产，搭配的是 Devstral 24B 模型。稳定态数据显示，行覆盖率、分支覆盖率、方法覆盖率比开源工具和零样本 LLM 方案高出 20% 到 45%，token 消耗降低了一个数量级（最高 15 倍）。

让我愣神的是 Aster 的架构思路：它不是让 LLM 去“理解”代码然后写测试，而是先用程序分析产出结构化的代码理解结果，用这个结果去“提示”和“聚焦”LLM，再配上一堆子 agent 专门负责提升覆盖率和修复编译/运行时错误。LLM 在这里更像是一个翻译器——把结构化分析结果翻译成测试代码——而不是一个全知全能的程序员。

从代码到基础设施，agent logic 的边界在扩张

第三个场景把战场从代码层拉到了全栈基础设施。运行时故障定位这件事，LLM 的上下文不再局限于源码，而是要把微服务、数据库、中间件、MELT（指标/事件/日志/追踪）数据全吞进去。IBM 的做法是构建一个知识图谱，把这些实体和领域专家的“部落知识”嵌入进去，然后用本地化推理（local bound reasoning）限制 LLM 的非确定性输出。

这套方案的核心是 Instana 的 I3（Intelligent Incident Investigation）agent，搭配 Instana 数据模型。在 ITBench 基准测试上，I3 agent 比基于 GPT-5.1 的 ReAct agent 性能高出 4 倍。换 Gemini 3 Flash 时，ReAct agent 的表现追到了只差 17%，但 token 消耗多了 1.6 倍。IBM 还把这套方法延伸到了源码层面——代码分析 agent 在定位有问题的微服务上比顶尖 coding agent 强 3 倍，bug 修复强 1.6 倍，token 消耗分别降低 3.7 倍和 5.9 倍。

第四个场景更偏“软”——IT 合规现代化。企业面对的合规要求越来越碎片化，没有集中知识库，修复靠手写，出错风险高得吓人。IBM 的多 agent 系统用算法把复杂合规任务分解成协调步骤，自适应规划、动态分解、工作流排序加持续反馈，把成功率的基线从个位数拉到了 80% 以上（使用 Claude 4 Sonnet），比固定规划策略的旧 agent 强 1.3 到 2 倍。这个系统已经在 IBM Think 大会上作为 Sovereign Core 的一部分发布，包含 16000+ 条数字化控制映射。

这些数字单独拎出来每一个都够写一篇 PR，但放在一起看，它们指向的是一条更深的规律：LLM 是引擎，但引擎需要方向盘。

医疗和工业的边界测试：agent logic 不挑行业

IBM 还塞了两个案例研究来证明这套逻辑的通用性。第一个是医疗健康领域的 CUGA（Configurable Generalist Agent），用“策略即代码”的方式做 agent 治理，在运行时独立于模型提示执行策略，不需要微调。实验结果是：跨 Claude Opus 4.5、GPT OSS 120B 和 GPT-4.1 三个模型家族，任务正确率提升 15% 到 26%。核心设计哲学是“推理自主，决策受限”——智能动作可以提议，但执行权限由策略和监督机制控制。

第二个案例是 IBM 全球房地产部门的物理资产状态维护。Maximo Condition Insights agent 在 120+ 个站点、6000+ 个物理资产上做试点，把资产分析时间从 15-20 分钟压缩到 15-30 秒——97% 的改善。资产审查覆盖率从约 1% 跳到约 30%。在 AssetOpsBench 基准上，无依据声明减少 57%，冗余输出减少 35%，规则合规提升 30%，token 用量平均降低 77%。背后是一个有向无环图（DAG），提供结构化的工程和运维上下文，把 LLM 在天真提示下容易产生的“胡说八道”压了下去。

我觉得最有意思的细节不是数字本身，而是 IBM 在这两个案例里用的模型都不算最新最强——GPT OSS 120B、Claude 4 Sonnet——但配上 agent logic 之后，性能曲线直接被拉上去了。这其实在反过来质问整个行业：我们到底是在拼模型，还是在拼工程？
