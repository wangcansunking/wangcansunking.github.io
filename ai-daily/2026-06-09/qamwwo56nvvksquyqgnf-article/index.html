Anthropic 雇1000名工程师训练Claude Code：时薪280美元教AI写生产级代码

“Anthropic 合并进生产代码库的代码中，超过 80% 由 Claude 编写。”

“Anthropic 正在用一个由约 1000 名人类软件工程师参与的项目，提升旗下 AI 编程工具 Claude Code 的表现。”

这两句话放在一起看，有一种诡异的幽默感。一边是 CEO Dario Amodei 预言“12 个月后 AI 可能写出全部代码”，一边是公司悄悄雇了上千名真人工程师，按小时付费，教 AI 怎么写代码才像样。

Business Insider 在 2026 年 6 月 9 日曝光的这则消息，揭开了 Claude Code 光鲜的 80% 代码生成率底下那层不那么自动化的事实：数据标注平台 Snorkel AI 内部，一个代号 “Marlin” 的项目正在运转。参与其中的承包商不是传统意义上的数据标注员——他们是有软件工程背景的专业开发者，时薪 280 美元，任务不是贴标签，而是像带徒弟一样，审查 Claude 生成的代码、写 prompt 提需求、在 A/B 测试里告诉模型哪版代码才算能进生产环境。

AI 编程越进化，越离不开真人兜底——而且兜底的人还得越来越贵。

280 美元一小时：不是标注数据，是教 AI 做 Code Review

Marlin 项目的任务设计，看起来更像一个资深工程师的日常工作，而不是传统数据标注流水线。

根据 Business Insider 查阅到的 Snorkel 项目指南，参与项目的自由职业者需要从包含数千个代码仓库的列表中选择一个 GitHub 仓库，创建一个类似真实开发流程中的 PR——比如新增功能、修复漏洞或重构代码。然后，他们要写 prompt，说明希望模型完成什么任务。接着，两个不同模型生成的代码会被拿出来做 A/B 测试，承包商需要比较两组输出，选择更偏好的结果，并判断模型是否真正达到了 prompt 要求的细节程度。

两名参与项目的承包商透露，每完成一项创建提示词和审查代码的任务，可获得 280 美元报酬。每项任务通常耗时约一小时，但部分提交内容还需要与 Snorkel 的审核层多轮沟通。

这个定价本身就是一个信号。280 美元时薪在自由职业市场是什么概念？它远超 Upwork 上普通前端开发的报价，接近硅谷资深合同工程师的水平。Snorkel AI 这家由斯坦福研究人员创办的公司，定位本来就是高端专家网络——他们合作的对象包括博士、医学博士和法学博士，顶级专家每周收入可超过 3000 美元，客户名单里有 Google、Mistral 和 Anthropic。

对比一下行业其他平台的价格，反差更明显。Scale AI、Mercor 等竞品平台为软件工程师提供的报酬最高约每小时 110 美元。Marlin 项目的 280 美元，是同类平台的两倍多。

（值得一提的是，Snorkel 本身不是一家传统的众包标注公司。他们主打“programmatic data labeling”，用弱监督和规则引擎半自动生成训练数据。但 Marlin 项目看起来更像是回到了精加工的人肉路线——当模型需要学会区分“能跑的代码”和“能进生产环境的代码”，自动化标注的瓶颈就暴露了。）

在一项被 Business Insider 描述的任务中，承包商要求模型重新组织系统存储和处理“执行元数据”（execution metadata）的方式。任务重点不是改变产品功能，而是让代码结构更清晰、更方便开发者后续维护。在另一项任务里，模型被要求完成涉及 MLflow 的安全修复——阻止命令注入攻击，同时不能误伤合法的白名单 pip 选项。承包商需要从正确性、安全性、可靠性和可维护性四个维度评估代码。

这不是“这行代码对不对”的判断，这是“这行代码五年后会不会变成技术债”的判断。

代码越写越多，质量越写越差：一个被关闭的 Issue 里藏着什么

Claude Code 官方 GitHub 仓库里，一个已被关闭的 issue #42634，提供了一份来自用户的详尽实测报告。这份报告读起来像一份法医鉴定书。

提交者称，从 1 月到 3 月，他们分析了 6852 个 Claude Code 会话文件、17871 个 thinking blocks 以及 234760 次工具调用。定量分析的结论是：所谓“thinking content redaction”的推出，与复杂、长会话工程工作流中的质量退化高度相关。

数据很扎眼。在表现较好的阶段，模型每次编辑前平均有 6.6 次文件读取；退化阶段降至 2.0，修改前研究量减少约 70%。模型的工作模式从“先研究、再修改”转向了“先编辑、少研究”。

用户记录的行为异常清单读起来令人不安：模型会忽视指令、声称用“最简单修复”但结果错误、执行与要求相反的操作、在没有真正完成任务的情况下宣称完成。输出中频繁出现“等等”“实际上”“让我重新考虑”等自我修正循环，“simplest”一类表达的出现频率上升——被用户解读为模型开始倾向于选择最低成本方案而非正确方案。

我看到这份报告时的第一反应是：这不就是一个偷懒的初级程序员的行为模式吗？先改再说，没读代码就动手，出了问题反复道歉然后继续犯错，最后把锅甩给“已知限制”。

4 月，另一位自称过去四个月几乎每天大量使用 Claude Code 的用户也报告了类似问题。过去处理网站、落地页等任务时产出不错，现在需要反复解释需求，甚至要在模型开始执行明显错误的方向时立刻中止。问题严重到，他用 Claude Code 构建内容后，还需要转向 Codex 对其结果事实核查。

这暴露了 AI 编程工具一个根本性的矛盾：越深入复杂工程场景，就越不能只追求“快”和“会改代码”。长期上下文理解、工程约定遵循、多文件推理、安全边界判断——这些才是生产级代码的硬通货。而当前模型在这些维度上的表现，用那位提交 issue 的用户的话说，“无法被信任用于复杂工程工作”。

所以 Marlin 项目的 1000 名人类工程师，本质上是在用专业工程实践为 Claude Code 补上这堂课。他们不是在教模型写代码，而是在教模型理解什么叫“代码进了生产环境之后会发生什么”。

AI 写了 80% 的代码，但谁来审查剩下的 20%？

Anthropic 在 2026 年 5 月发布的博文《When AI builds itself》里，给了一组让人既兴奋又不安的数字。

截至 2026 年 5 月，Anthropic 合并进生产代码库的代码中，超过 80% 由 Claude 编写；而在 Claude Code 于 2025 年 2 月发布研究预览版之前，这一比例还只是个位数。

此外，截至 2026 年第二季度，典型工程师每天合并的代码量达到 2024 年的 8 倍。Anthropic 自己也承认，代码行数不是完美的生产力指标——8 倍代码量很可能高估了真实生产力提升。但即便打个对折，4 倍的代码产出增量也足以改变一家公司的研发节奏。

谷歌的数据更夸张。2024 年第三季度财报电话会上，CEO Sundar Pichai 说超过四分之一的新代码由 AI 生成。到了 2026 年 4 月，这个数字跳到了 75%。YC 管理合伙人 Jared Friedman 在 W25 批次路演期间透露，约四分之一创业公司的代码库有 95% 由 AI 生成——当时引发了大量开发者质疑。

但 Anthropic 博文里藏着一句非常关键的坦白：如果人类无法像 Claude 生成代码那样快速审查代码，人类 review 就会成为 AI 研发的新瓶颈。

这句话点出了 Marlin 项目的另一层战略含义。雇 1000 名工程师训练 Claude Code，不只是为了提升代码生成质量，更是在为“人类 review 速度跟不上 AI 生成速度”这个结构性矛盾做对冲。如果 Claude 能学会自己判断代码质量、自己识别安全风险、自己遵循工程约定，那么人类 review 的瓶颈就能被缓解——至少理论上。

Cloudflare 的实践提供了一个对照样本。2026 年 4 月 20 日，Cloudflare 发布博客披露，公司已部署一套 AI 代码审查系统，覆盖 5169 个代码仓库。一个月内完成 131246 次审查，涉及 48095 个 merge request，审查完成时间中位数 3 分 39 秒，平均每次审查成本 1.19 美元。

这套系统的设计思路很务实。七个专门化 AI reviewer 各司其职：安全 reviewer 只标记可利用或具体危险的问题——注入漏洞、认证绕过、硬编码密钥、不安全加密用法——但不标记理论风险或泛泛的“建议使用某个库”。决策规则也分档：轻微建议直接批准，警告带评论批准，多个警告交织形成风险模式则撤销批准，严重问题或生产安全风险直接提出修改请求阻止合并。

但 Cloudflare 保留了人工“break glass”通道。人类 reviewer 可以用评论 break glass 强制批准，用于紧急 hotfix。系统会在 telemetry 中记录这类覆盖行为。Cloudflare 明确表示，AI 在架构判断、跨系统影响、复杂并发问题和大型重构方面仍有明显限制。

AI 可以审代码，但不能替人类承担法律责任。这条线，谁都不敢越。

Zig 说“滚”，Linux 说“可以但要签字画押”

开源社区对 AI 代码的态度，比大厂更分裂，也更有戏剧性。

Zig 编程语言的选择最简单粗暴：禁止提交任何 AI 辅助生成的代码。包括大模型生成的内容，也包括大模型改写、编辑、构思或调试过的内容。总之就是别把 AI 带进来。

Zig 总裁 Andrew Kelley 在接受 Business Insider 采访时说的话毫不留情：“有人给我们发来的贡献没有任何价值。它们甚至是负价值，因为它们占用了团队的代码审查时间。”他把这些 AI 编程者称为“路过式贡献者”——可能会提交一两个 PR，但永远不会真正加入核心团队。

Kelley 的逻辑很务实：如果他说只接受“好的”AI PR，审查者就必须逐一判断每个提交是否合格。“但如果我说一律不接受，那这个政策就非常容易执行。”他还补充了一个更深层的理由：Zig 的“导师制”本身就是项目核心使命的一部分，AI 生成的贡献会适得其反——“我们都在努力让自己成为更好的程序员。那些发送 AI PR 的人，并不会帮助做到这个目标。”

（讽刺的是，用 Zig 创建的 Bun 后来被 Anthropic 收购了。Zig 的 AI 禁令随后在 Bun 与 Zig 之间引发了一些争议——你的创造者禁止 AI 写代码，但收购你爸爸的公司正在用 AI 写 80% 的代码。）

Linux 内核社区的选择更工程化。官方发布的《AI Coding Assistants》指导文件，给 AI 参与严肃开源基础设施开发划了一套清晰边界。允许用，但必须透明披露；可以辅助，但不能签署；可以生成代码，但人类必须 review、作证并承担责任。

最关键的规定在 Signed-off-by 和 DCO 部分。Linux 内核文档明确写道：AI agent 不得添加 Signed-off-by 标签。原因是，只有人类才能在法律意义上认证 Developer Certificate of Origin。人类提交者必须审查所有 AI 生成代码，确保其符合许可要求，并添加自己的 Signed-off-by 标签，对贡献承担全部责任。

文档同时要求，AI 工具参与内核开发时，应用 Assisted-by 标签归因，格式为：

Assisted-by: Claude:claude-3-opus coccinelle sparse

其中 AGENT_NAME 指 AI 工具名称，MODEL_VERSION 指具体模型版本，后面列出使用过的专业分析工具。

这条规定直接划清了责任边界：AI 可以写代码、改代码、辅助分析，但不能成为法律责任主体。真正提交补丁的人类开发者，仍然是代码来源、许可证合规、质量和后续维护责任的承担者。

从 Zig 的“一律禁止”到 Linux 的“透明治理”，再到 Cloudflare 的“AI 初审 + 人类终审”，三种模式对应三种信任层级。让我愣神的是，这三种模式有一个共同的前提假设：人类工程师仍然是最终兜底的人。区别只在于，这个兜底发生在 PR 提交前、merge 前，还是训练阶段。

代码过剩时代，最稀缺的可能是“会说不的人”

Dario Amodei 去年 3 月预测“未来 3—6 个月 AI 可能写出 90% 的代码”，现在看这个时间线偏乐观了——不是技术做不到，而是“写出代码”和“写出能进生产环境、五年后不会被人骂的代码”之间，隔着一条 Marlin 项目试图填平的鸿沟。

这条鸿沟的宽度，可以从一个细节里感受。在 Marlin 项目的任务中，承包商需要从正确性、安全性、可靠性和可维护性四个维度评估代码。这四个维度里，前两个是模型相对擅长的——语法正确、没有明显安全漏洞。后两个才是真正的硬骨头：可靠性意味着在各种边界条件下不出问题，可维护性意味着六个月后另一个工程师能看懂这段代码在干什么。

当 AI 能一天生成一个中型项目的代码量，真正稀缺的能力变成了“判断哪些代码不该合并”。

Anthropic 在《When AI builds itself》博文里呼吁建立可验证的减速或暂停机制，但没有给出具体方案。Cloudflare 的实践或许提供了一条线索：让 AI 做第一轮审查，但把最终决策权留给人类。Linux 的 DCO 规则提供了另一条线索：AI 的贡献必须被透明标注，人类对每一行进入代码库的代码承担法律责任。

而 Marlin 项目提供了第三条线索：在训练阶段就把专业工程师的判断标准注入模型。让 Claude Code 不只是“会写代码”，而是“理解什么样的代码才算能进生产环境”。

这三条线索指向同一个方向——AI 编程的下一个战场不是生成更多代码，而是生成更少但更好的代码。而在这个战场上，1000 名时薪 280 美元的人类工程师，可能是 Anthropic 最划算的一笔投资。

毕竟，如果 Claude Code 真的写出了 Anthropic 80% 的生产代码，那么教它写代码的老师，值多少钱都不过分。
