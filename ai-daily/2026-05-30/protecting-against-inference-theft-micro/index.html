Vercel警告开发者：一次AI推理2美元，当心“推理盗窃”烧穿账单

当你的 AI 推理成本比你整个网站还贵：Vercel 警告的“推理盗窃”正成为最暴利的黑产

想象一下这个场景：你做了一个 SaaS 产品，辛辛苦苦优化前端性能，把首屏加载压到了 1.2 秒，CDN 边缘节点覆盖全球 300+ 个城市，每月为那几百万次 HTTP 请求付给 Vercel 大概 2 美元——对，你没看错，2 美元。百万次请求，一杯拿铁都买不起的价格。

然后你决定给产品加点 AI 功能。接入一个 Agent，跑在某个前沿模型上。用户输入一句“帮我分析一下这份 50 页的合同并标出所有风险条款”，Agent 开始疯狂调工具、链式推理、RAG 检索、多步验证——单次调用烧掉你 2 美元。

同样的请求量，成本差了六个数量级。这不是技术债，这是商业模式的定时炸弹。

Vercel 在 2026 年 5 月的一篇官方博客里，给这个现象起了个名字：推理盗窃（Inference Theft）。他们不是在写学术论文，是在警告所有在他们平台上挂 AI 功能的开发者：你的 API 端点正被人薅羊毛，而你可能根本不知道。

一个 HTTP 请求 0.000002 美元，一次推理 2 美元：这中间的套利空间够养活一整个黑客产业链

先把这个数字刻进脑子里：Vercel 的定价体系下，一百万次标准 HTTP 请求大约 2 美元，折合每次调用 0.000002 美元。而一个运行在 Claude 或 GPT-4 级别模型上的 Agent，单次复杂推理的成本可以轻松突破 2 美元。这意味着什么？意味着攻击者只需要找到你的 AI 端点，用脚本构造看起来合法的请求，就能以近乎零成本的方式，让你为每一次调用买单。

这跟传统的 DDoS 攻击逻辑完全不同。DDoS 打的是你的带宽和服务器资源，攻击者自己也要付出流量成本（哪怕用的是肉鸡）。但推理盗窃是非对称消耗：攻击者发起请求的成本趋近于零，而你承担的推理成本却是指数级膨胀的。一次调用 2 美元，一千次就是 2000 美元，一万次就是 2 万美元——一个脚本跑一晚上，你的云账单能直接烧穿当月的预算红线。

更可怕的是，这种攻击极难被传统防护机制识别。你的 WAF（Web 应用防火墙）在盯着 SQL 注入和 XSS，你的 Rate Limiting 在按 IP 限制每分钟请求数，但推理盗窃的请求本身就是“合法”的——它就是一个正常的 POST 请求，带着一个看似合理的 prompt，触发的也是你主动开放的 AI 功能。它不会触发 403，不会留下异常日志，唯一的痕迹是你的 OpenAI 或 Anthropic 账单在月底暴增，而你对着那堆“合理使用”的记录完全无从排查。

Vercel 的工程师在博客里指出了一个关键细节：这类攻击往往瞄准的是 Agent 型应用，而不是简单的单轮对话。为什么？因为 Agent 的推理链路长、工具调用多、上下文窗口大，单次消耗的 token 量可能是普通对话的 10-50 倍。攻击者如果直接调你的“聊天”接口，一次可能只烧掉 0.02 美元，但如果他调的是你的“合同分析 Agent”，一次就能烧掉 2 美元。套利效率直接拉满。

攻击者的 ROI 计算公式很简单：找到最贵的那个端点，用最低的成本反复触发它。

从“免费代理”到“模型蒸馏”：推理盗窃的三层变现路径

让我觉得最有意思的是，Vercel 这篇博客其实只讲了冰山一角。推理盗窃本身不是终点，盗窃来的推理能力怎么变现，才是这条黑产链条真正可怕的地方。

第一层最粗暴：免费代理。攻击者把你的 AI 端点包一层壳，做成一个“免费 ChatGPT”网站放出去，用户在前端输入问题，后端偷偷转发到你的 API。用户以为遇到了慈善家，你以为用户在正常使用产品，实际上你正在为全世界的白嫖党买单。这类攻击早在 2023 年就有案例——当时有人把某个初创公司的 GPT-4 接口反代出来，做成 Telegram 机器人，三天烧掉对方 8 万美元的 API 费用，直到 CFO 看到账单才东窗事发。

第二层更隐蔽：模型蒸馏。攻击者不需要把你的端点暴露给公众，他只需要用精心构造的 prompt 集大规模调用你的 Agent，收集高质量的“问题-推理过程-答案”三元组，然后用这些数据去微调他自己的小模型。你的前沿模型充当了免费的标注师，而他最终得到一个性能接近你、但推理成本只有你千分之一的蒸馏模型——然后反过来跟你抢市场。这种攻击的可怕之处在于，你甚至无法从流量模式上察觉异常：攻击者的请求间隔、prompt 多样性、调用频率都可以伪装成正常用户行为，唯一不同的是他每个月调用 50 万次，而你只有 5000 个真实用户。

第三层最恶毒：数据投毒 + 推理盗窃组合拳。攻击者先用推理盗窃摸清你的 Agent 的行为模式和安全边界，然后构造对抗性 prompt 注入，让你的 Agent 在特定场景下输出错误结论或泄露系统提示词。这种事一旦发生，受损的不只是你的钱包，还有你的产品声誉和客户信任。而这一切的起点，不过是你的 API 端点没有做好 AI 层的访问控制。

Vercel 在博客里给出的建议其实相当务实：不要把 AI 端点当成普通 HTTP 端点来防护。传统的 Rate Limiting 要加，但必须叠加 Token 消耗监控——不是限制“每分钟多少次请求”，而是限制“每个用户每小时消耗多少推理成本”。同时，对于 Agent 型端点，必须引入输入复杂度评估：如果一个 prompt 要求你的 Agent 执行超过 5 步工具调用，或者上下文窗口超过 32K tokens，那就应该触发人工审核或硬性配额。（值得一提的是，Vercel 自己并没有原生提供这些 AI 层防护能力，他们更多是在平台层面提醒开发者注意这个风险——但这恰恰说明问题有多严重：连基础设施提供商都开始公开预警了。）

推理盗窃不是安全漏洞，是商业模式的数学漏洞。当你的边际成本远高于边际定价，套利者就会比用户先到。

说到底，这整件事最讽刺的地方在于：整个 AI 行业过去两年都在疯狂卷推理成本的下降——Groq 在卷硬件，Together AI 在卷调度，各种量化、剪枝、投机解码层出不穷——大家的目标是把那 2 美元压到 0.2 美元，再压到 0.02 美元。但在这个过程中，那些把前沿模型能力开放出去的开发者，反而成了成本套利的最佳猎物。你越努力优化用户体验，攻击者的套利空间就越大；你的 Agent 能力越强，你被薅羊毛的速度就越快。

Vercel 这篇博客的标题用的是“Protecting against inference theft”，但我读完的感觉是：这东西短期内根本防不住。它更像是一个行业集体觉醒的前奏——当足够多的开发者被烧掉足够多的钱，当足够多的 CFO 对着 AI 账单暴怒，这个市场才会长出真正的防护层。而在那一天到来之前，每一个在 AI 端点前面只挂了 Rate Limiting 的开发者，都在用自己的云账单，喂养着互联网上最暴利的黑产。
