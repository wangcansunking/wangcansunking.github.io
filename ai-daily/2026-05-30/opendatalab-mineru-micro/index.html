MinerU 3.1.0发布：从AGPLv3切换到Apache 2.0，为企业部署拔掉“毒刺”

一个 200M 参数的小模型，在 OmniDocBench 上跑出 86.2 分，把上一代主力 VLM MinerU2.0-2505-0.9B 按在地上摩擦。这不是什么实验室神话，是 MinerU 3.0.0 版本 pipeline 后端交出的实测成绩——而且它还能在纯 CPU 环境跑。

更狠的是，2026 年 4 月 18 日发布的 3.1.0 版本，直接换了一张开源许可证，从 AGPLv3 跳到了基于 Apache 2.0 的自定义 MinerU Open Source License，同时把 VLM 主模型升级到 MinerU2.5-Pro-2604-1.2B。一个文档解析引擎，两个月内连发两个大版本，每一次都踩在“开源合规焦虑”和“复杂版面理解”这两个行业最痛的穴位上。

许可证一换，企业部署的最后一根刺拔掉了。

从 AGPLv3 到 Apache 2.0：一场精心算计的“去毒”手术

AGPLv3 在开源圈有个外号叫“传染性许可证”——你只要在 SaaS 服务里用了 AGPLv3 的代码，理论上就得把整个服务端代码开源。这对任何想拿 MinerU 做商业化 RAG 流水线的公司来说，都是一颗定时炸弹。

MinerU 团队显然听到了足够多的抱怨。3.1.0 版本直接切到基于 Apache 2.0 的自定义许可证，官方公告里写得很直白：

This change significantly reduces adoption friction for both community users and commercial deployments, making MinerU easier to integrate into real-world workflows.

这不是什么“拥抱开源精神”的道德叙事。这是精确的市场计算——MinerU 已经集成了 LangChain、LlamaIndex、RAGFlow、Dify、FastGPT 等十几个 RAG 框架，MCP Server 直接对接了 Cursor、Claude Desktop、Windsurf 这些 AI 编程工具。如果许可证还卡在 AGPLv3，企业用户只会用脚投票，转向 Unstructured 或者 Azure Document Intelligence。

换许可证的同时，3.0.0 版本还做了一个更彻底的手术：完全移除两个 AGPLv3 模型（doclayoutyolo 和 mfd_yolov8）以及一个 CC-BY-NC-SA 4.0 模型（layoutreader）。这不是简单的依赖清理，是把整个技术栈里所有“不合规”的组件一次性摘干净，让法务审核能一口气用。

值得一提的是，这种“先 AGPLv3 圈社区、再换 Apache 2.0 接商业”的路径，在开源商业化史上并不新鲜。MongoDB 走过，Elastic 走过，但 MinerU 从 AGPLv3 到 Apache 2.0 只用了不到两年（项目诞生于 InternLM 预训练过程中）。这个速度，要么说明团队从一开始就规划好了许可证切换节奏，要么说明外部商业压力已经大到必须立刻动手。

1.2B 小模型 + pipeline 引擎：用“小快灵”打“大而全”

3.1.0 版本的主模型 MinerU2.5-Pro-2604-1.2B 只有 1.2B 参数。对比当前主流 VLM 动辄 7B、13B 甚至更大的体量，这个尺寸小得几乎像个玩具。但它的配置表里写满了“复杂场景”关键词：

  • 图片和图表解析
  • 截断段落合并
  • 跨页表格合并
  • 表格内图片识别

这些都是传统 OCR 引擎的死穴。一个三线表里嵌了一张流程图，普通解析工具要么把图丢掉，要么把表格结构拆碎。MinerU 2.5-Pro 的做法是用 VLM + OCR 双引擎，OCR 负责文字识别和版面检测，VLM 负责理解图文关系和语义连贯性。原文提到：

VLM + OCR dual engine, 109-language OCR recognition

109 种语言覆盖意味着它不只是中英文场景的工具，而是瞄准了全球化企业文档处理的通用基座。

更让我愣神的是 pipeline 后端在 OmniDocBench v1.5 上的 86.2 分。这个分数超过了上一代 VLM MinerU2.0-2505-0.9B，而 pipeline 本身是一个“无幻觉、CPU 可跑”的轻量引擎。3.0.0 版本还给它加了表格内公式/图片识别、印章文字识别、竖排文本支持、行间公式编号识别——这些全是中文企业文档（合同、发票、红头文件）的高频需求。

pipeline 和 VLM 的分工很明确：pipeline 负责“快稳准”，VLM 负责“复杂场景兜底”。3.1.0 版本还加了一个 hybrid-engine，把原生文本提取和 VLM 能力混合，进一步压低幻觉率。这三个引擎的组合，让 MinerU 从“PDF 转 Markdown 的小工具”变成了一个可以按场景选配的文档解析中台。

全格式原生支持：DOCX 不再先转 PDF 再解析

3.0.0 版本最被低估的一个更新，是 DOCX 原生解析。在此之前，几乎所有文档解析工具的 DOCX 处理路径都是“先转 PDF，再解析 PDF”——这会引入两层信息损耗：转换过程中的格式丢失，以及 PDF 解析本身的误差。

MinerU 3.0.0 直接绕过了这个中间步骤，对 DOCX 做端到端原生解析。官方给出的数据是：

Compared with the traditional workflow of first converting DOCX to PDF and then parsing it, end-to-end speed is improved by tens of times.

“数十倍速度提升”这个表述值得拆解：传统路径里，DOCX→PDF 转换本身是计算密集操作（尤其是 LibreOffice headless 模式），再加上 PDF 解析的版面检测和 OCR，整个流水线可能耗时数秒到数十秒。原生解析直接读 XML 结构树，把段落、表格、样式信息直接映射到 Markdown，确实能做到接近零延迟。

3.1.0 版本把原生解析扩展到了 PPTX 和 XLSX，至此 MinerU 覆盖了图片、PDF、DOCX、PPTX、XLSX 五大格式。对于企业 RAG 场景，这意味着从合同（DOCX）到财报（PDF）到投融资 PPT（PPTX）到数据表格（XLSX）的全链路文档处理，不再需要维护多个解析工具。

还有一组数字值得注意：3.0.0 版本优化了长文档解析的峰值内存占用，用滑动窗口机制和流式写入磁盘，让“数万页文档不再需要手动拆分”。同时完成了线程安全优化，支持多线程并发推理，配合 mineru-router 做到一键多 GPU 部署和自动任务负载均衡。这已经不是“开发者工具”的定位了，这是“企业级文档解析基础设施”的肌肉展示。

MinerU 的故事起点其实很朴素——它诞生于上海人工智能实验室 InternLM 预训练过程中，最初只是为了解决科学文献里的符号转换问题。现在它已经长出了 MCP Server、LangChain 原生集成、10+ 国产 AI 芯片支持（昇腾、寒武纪、燧原、沐曦、摩尔线程、昆仑芯、天数智芯、海光、壁仞、平头哥），以及一个可以直接在浏览器里用的 Zero-Install Web 版本。

一个从预训练痛点里长出来的开源项目，两年时间换了许可证、重构了架构、覆盖了五大格式、接入了全球最主流的 RAG 生态——这种进化速度，比大多数拿了融资的商业公司还快。
