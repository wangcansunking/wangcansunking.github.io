前端失落十年重演？AI 正让全栈开发者面临“去技能化”

2025 年某个下午，一个前端老炮在 Slack 群里贴了段代码：一个 Shadcn 的单选按钮组件，展开后 247 行。群里沉默了半分钟，有人回了一句：“十年前我们用 <input type="radio"> 只要一行。”

没人觉得好笑。

因为就在同一天，他们公司的 CTO 刚在全员会上宣布，Q3 要引入 AI coding agent 来“提升工程吞吐量”。理由很直白：既然前端已经证明过，把高度专业化的技能拆解成半熟练工也能操作的流程是可行的，那为什么不全栈推广？

Mauro Bieg 在 2026 年 5 月 23 日发出的这篇博客，像一根针扎进这个行业最敏感的那根神经。他没有讲什么 AGI 取代人类的大叙事，而是翻出了一段前端开发者刻在骨子里的创伤记忆——那个被 Alex Russell 命名为“前端失落十年”的时期——然后平静地说：你看，历史正在重演，只不过这次的范围是全栈。

前端开发者十年前经历的事，现在每一个程序员都在经历。区别在于，这一轮的引擎从 React 换成了 LLM，而速度可能快得多。

“去技能化”：从前端到全栈的同一把刀

Wikipedia 上对 deskilling 的定义，Bieg 直接搬进了文章里。这个词的中文译法很多——“去技能化”“技能降级”“劳动技能消解”——但内核都一样：一种新技术出现，让半熟练或不熟练的工人可以操作它，于是原本需要多年积累的熟练技能被市场抛弃，工人的议价能力随之瓦解。

前端这个工种，在 2010 年代完整经历了一遍这个过程。

Bieg 回忆自己的职业轨迹：从手写 HTML/CSS 加一点 PHP 起步，后来做 Ruby on Rails，再后来成为瑞士一家主流报纸的前端团队负责人（当时用的是 Next.js）。他亲眼看着前端从一个需要精通语义化 HTML、跨浏览器差异、可访问性、渐进增强、网络性能、界面设计和用户测试的高度专业化领域，变成“把浏览器当成编译目标”的流水线。

现在的“全栈开发者”，Bieg 的观察相当刻薄：往往不是一个真正理解前端和后端的人，而是一个通才，刚好知道怎么摆弄一个 JavaScript 框架让两端都能跑。企业喜欢这种安排，因为他们可以轻松把程序员在不同项目间调来调去。同一个通才还能用 React Native 和 Electron 做原生应用。Wikipedia 那句“削弱了工人的议价能力”，在这里得到了完美的现实注脚。

然后 AI 来了。

LLM 辅助编程正在对更广泛的程序员群体做同一件事：手写代码的熟练技能正在被“由半熟练或不熟练工人操作的技术”所替代。Bieg 坦率地说，我们还不清楚在这个转型终局里，操作 agentic AI 的工人到底需要什么技能组合，也不清楚劳动力价格和本地/远程 LLM 的成本最终会落在什么水位。但他确定一件事——

企业绝对会用这项技术来削减成本和削弱工人的议价能力。这是结构性的，不是意外。

Just like artisans and craftsmen that were replaced by assembly line workers more than a century ago, we feel a profound sense of loss.

Bieg 在这里用了一个超出技术讨论的词：grief——哀悼。不是愤怒，不是焦虑，是哀悼。哀悼的是花了半辈子打磨的技能，市场不再需要了。更让人难受的是，新流程产出的质量更低，但“很多人似乎根本不在乎”。

我看到这段的时候，第一反应是想起了一个做性能优化的朋友。他在 2024 年底被裁，公司给出的理由是“AI 工具已经能覆盖 80% 的优化建议”。但几个月后我偶然打开他之前负责的那个网站，Lighthouse 分数从 92 掉到了 47。没人修。因为剩下那 20% 需要真正懂的人，而那个人已经不在了。

Leaky abstractions：塔楼越高，漏水越猛

如果我们换个角度看“去技能化”，可以把它重新包装成“用自动化提高效率”。哪个工程师不喜欢自动化呢？这本来就是我们的核心工作之一。从这个框架出发，新技术不过是工作在一个更高的抽象层级上，让操作者专注于大局，不必纠缠于“不重要”的细节。

但问题恰好出在“不重要”这三个字上。

Bieg 用了一个计算机科学里非常经典的表述：所有抽象最终都会泄漏（all abstractions eventually leak）。关键是，哪些细节被判定为“不重要”，这是一个高度主观且后果严重的决定。而泄漏一旦发生，那些被抽象屏蔽掉的细节会从缝隙里渗出来，淹掉整个系统。

现代前端的抽象塔楼，Bieg 毫不留情地拆解了一遍。用 React 这类重客户端 JS 框架，加上生态里的大量包，你实际上是在把可访问性、低端手机上的性能、慢网络下的加载体验这些事统统抽象掉了。你说你选择了不去想它们，实际上你选择了不去在乎它们。

这在高速网络和旗舰手机上是可行的。但移动端慢网络的现实数据一直在打脸。Bieg 没有给出具体数字（说实话，原文的数据密度确实不高，这是一篇偏论述的博客而非调查报告），但他点出了一个残酷的事实：一个糟糕的网站对商业底线的冲击相对较小。加载慢、弹一堆 cookie 横幅当然会损害转化率，但比起品牌忠诚度和定价，这些影响微乎其微。况且，所有竞争对手的网站也一样慢。

“没人因为选了 React 被开除”这句话，被 Bieg 用做了这个章节的潜台词。它让我想起另一个行业的类似表述：“没人因为买了 IBM 被开除。”当技术选型变成一种保险策略而非质量策略，整个行业的技能结构就不可逆转地改变了。

Agentic coding 面临的抽象泄漏问题，比前端框架更复杂。Bieg 指出的关键差异在于：compiler 是确定性的，LLM 不是。输入或模型的微小变化可能导致完全不同的输出。这导致人们把 AI 比作“初级工程师”——初级工程师也是不确定的。但区别在于，人是可以学习的，而你不需要无休止地调优他们的 AGENTS.md 或 SKILL.md 文件。

（AGENTS.md 和 SKILL.md 是当前 AI coding agent 生态里用来约束模型行为的配置文件，相当于给 AI 写的“员工手册”。Bieg 提到这两个文件名时的嘲讽味，隔着屏幕都能闻到。）

把 LLM 当成“Stack Overflow 复制粘贴”的进化版，这大概是目前最清醒的定位。 Bieg 追溯了一个有趣的谱系：Google 搜索 → Stack Overflow → 复制粘贴 → LLM 生成代码。这条线串起来看，LLM 不过是同一个趋势的延续——工具让知道自己做什么的人稍微快一点，让不知道自己做什么的人也能搞出点勉强能用的东西。他说，这很棒。

但前提是，你得分清什么时候抽象会泄漏。

Google 搜索的命运，就是 LLM 的未来

Bieg 对 Google 搜索演变的观察，放在 LLM 的语境下读，几乎可以当预言看。

他回忆说，Google 搜索曾经是一门需要专门学习的技能：选择精准的关键词，让正确的论坛帖子（后来是 Stack Overflow 帖子）出现在搜索结果第一页。这和 prompt engineering 几乎同构——在超高维空间里做模糊查找，对措辞的微小变化高度敏感。过去对 Google 搜索索引的变化敏感，现在对 LLM 的版本更新敏感。

但 Google 在最近几年大幅改变了搜索行为，对用户输入的词语做了更激进的“规范化”（normalization）。对那些不掌握 Google-fu 黑魔法的人来说，这让搜索变得更容易用了。但对那些掌握了这门技艺的人，Google 搜索变得没那么强大了。专业关键词过去能精准命中答案，现在被规范化成同义词或近义词，最终落到一个更泛化的页面上。

这跟 LLM 目前的发展轨迹几乎一模一样。模型的 RLHF 和对齐训练，本质上就是在做 normalization——把用户的模糊意图“规范化”成模型认为最可能被接受的输出。对新手友好，对专家削弱。Bieg 没有明说但暗示得很清楚：LLM 正在走 Google 搜索走过的同一条路，而且速度更快。

这个类比延伸出一个更让人不舒服的问题。Stack Overflow 出现之后，程序员不再阅读 f***ing manual，而是直接复制粘贴答案，然后惊人地发现往往能勉强跑通。很多程序员从没进阶到“真正理解答案再使用”的阶段。既然能跑，为什么要费劲理解？

公司层面，过去这种行为是不被公开承认的。现在不同了——公司大张旗鼓地宣布自己用了多少 AI，甚至不再假装有人类在审视 AI 的输出。Bieg 说这是一个令人悲伤的生活事实：很多公司在产出糟糕软件的同时，生意做得非常好。商业成功和软件质量极少正相关。其他因素（品牌、定价、渠道、时机）通常压倒性地主导结果。

包豪斯的回答：回到车间，摸材料

文章最后三分之一，Bieg 突然转向了 20 世纪初的包豪斯运动。这个转折初看有点突兀，但读下去会发现这是他整篇论述的支点。

面对工业化让日常用品和建筑可以被大规模生产的现实，前几代手工艺人的反应之一是历史主义——让工厂生产出至少看起来像手工制作的玩意儿。包豪斯走了另一条路。他们不把工厂工人和手工艺人对立起来，而是让两者合作，以工业生产流程为前提重新发展艺术和工艺。

包豪斯敦促设计师回到车间，亲手接触材料。目标仍然是最终会大规模生产的设计，但始终把终端用户放在心里，深切关心他们。Dieter Rams 和 Jonathan Ive 所代表的现代工业设计，可以直接追溯到包豪斯的这条脉络。

Bieg 把软件放在手工艺和工业设计之间的某个位置。一方面，我们写的程序直接交付给用户，没有经过一个制造环节——这像手工艺。另一方面，我们把同一个东西交付给成千上万的用户，我们从来没见过他们如何与产品交互——这像工业设计。

手写代码的必要性，在这个框架下变得清晰。就像工业设计师需要知道产品材料的特性，一个 web 设计师需要深度熟悉 HTML 和 CSS。工具（Google、Stack Overflow、现成库、LLM）让入门变得更容易，这也意味着“让任何东西勉强跑起来”的自然门槛在持续降低。

这里有个反直觉的观察。Bieg 说，当一个领域有很高的进入壁垒时，很难找到绝对糟糕的作品。一个木匠学会了做椅子，他不可避免地也学会了怎么不做烂。工业化让大量便宜塑料制品成为可能，但好的工业设计依然存在。文字处理器的发明让大量排版灾难成为可能，但字体排印和平面设计依然存在。Wix 和 Next.js 让大量加载缓慢且不可访问的网站成为可能，但“前端之巅”的实践者依然存在。

同理，AI 正在让大量 AI slop 成为可能，但这不意味着我们不再需要知道自己在做什么、关心自己在做什么的人。

“做对的事”这个市场份额会越来越小。但因为做烂事变得更容易和便宜了，整个市场的饼会继续变大。 Bieg 说他很难判断，被付费去做对的事的绝对人数会增长还是减少。他的直觉是，设计糟糕的塑料产品已经太多了，新的字体设计也不再是可持续的全职工作——但在所有这些领域，仍然有大量出色的工作在被完成。

快速原型没错，但你得知道自己放弃了什么

Bieg 没有给出非黑即白的结论。他承认，有时候快速搞出一个原型或 MVP 就是正确的选择。如果还没找到 product-market fit，快速迭代和学习比面向未来做过度工程更重要。但你需要知道你试图学习什么，以及如何验证这些学习。

而当时候到了，通常更好的做法是退后一步，从一开始就把事情做对。他举了两个具体例子：好的性能在一个架构糟糕的前端里很难后期做到；从一个简单栈开始然后逐步加功能，比反过来更容易。Mastro——他参与开发的 web 框架——明确鼓励这种理念。

对于系统的任何部分，你需要知道自己在做什么 tradeoff，然后决定是购买服务、使用开源库、让 LLM 吐出来、还是手写。Bieg 预测，等炒作退潮后，行业会意识到 LLM 只是工具箱里多了一把工具。但在那之前，我们会看到很多丑陋的东西：丑陋的代码、断裂的沟通、以及打着 AI 旗号的可怕裁员。

这篇文章没有给终极答案，但它给了历史坐标。前端开发者十年前经历的那一轮 deskilling，没有消灭前端这个工种，但彻底重塑了它的技能分布和薪酬结构。现在全栈程序员正在进入同一台绞肉机，而绞肉机的转速比十年前快了不止一个数量级。

Bieg 在结尾保持了某种务实的乐观：做得好的人永远有空间，但那个空间在变小。饼在变大，但做得好的人分到的比例在缩水。这对个体的决策意味着什么，他留给了读者自己判断。

有意思的是，这篇文章发布在 Hacker News 首页后引发的讨论里，最高赞的评论只有一句话：“我在 2016 年离开了前端，现在我在看这篇文章，觉得 2026 年我可能要离开全栈了。”

历史不会重复，但会押韵。前端押过的韵脚，全栈正在跟着念。而 AI 是那个加速播放键。
