754项Anthropic安全技能开源：让AI像专家一样处理网络攻击

ai-daily · 2026 年 5 月 26 日 02:38 · 来源：GitHub Trending python

2026 年 5 月 7 日，凌晨两点四十七分，东京某安全运营中心的屏幕墙突然红了七块。

一名三级分析师揉了揉眼睛，盯着 SIEM 上跳出的告警序列——Kerberoasting 攻击、LSASS 凭证转储、Cobalt Strike 信标回连，三条攻击链几乎同时展开。他深吸一口气，打开终端，准备手动执行 Volatility3、Sigma 规则匹配、Splunk 横向移动检测三套流程。在他手边，一本翻烂了的 DFIR 手册摊在第 187 页，上面用荧光笔划着：“内存取证步骤：先跑 windows.info，再跑 windows.pslist，然后……”

而在 3000 公里外的班加罗尔，一个叫 Mahipal Jangra 的印度安全研究员刚刚给 GitHub 仓库 push 了第 754 个 skill 文件。这个仓库的名字叫Anthropic Cybersecurity Skills——尽管它跟 Anthropic PBC 这家公司没有任何官方关系（仓库首页明确写着“Community Project, Not affiliated with Anthropic PBC”）。Jangra 的想法很简单：如果 AI agent 能直接读懂那本翻烂的 DFIR 手册，并且能在几秒钟内完成“扫描 754 个技能→匹配前 3 个→按步骤执行→验证结果”的全流程，那个三级分析师是不是就不用熬夜了？

他给这个仓库打上了 Apache-2.0 许可证，然后在 README 里写了一句话——

Give any AI agent the security skills of a senior analyst.

754 个技能，26 个域，5 套框架——这不是教程合集，是 AI 原生作战手册

先看几个数字：754 个结构化网络安全技能，覆盖 26 个安全域，映射到 5 套行业框架，兼容 26+个 AI 平台。如果你觉得这只是一组 GitHub 仓库统计数据，那你就错过了重点。

这个仓库的底层逻辑跟传统安全工具库完全不同。Jangra 没有塞进去一堆 wordlists、payloads 或 exploit 代码——那些东西 AI agent 本来就能自己搜到。他做的是把资深分析师的决策流程编码成机器可读的结构化工作流。每一个 skill 文件都是一个完整的“何时触发→前置条件→分步执行→验证结果”闭环，用 YAML frontmatter 做元数据索引，用 Markdown body 写执行细节，再加上 references/、scripts/、assets/三个子目录提供深度技术上下文。

举个例子。假设一个 AI agent 接到指令：“分析这个内存 dump，找凭证窃取痕迹。”如果没有这个技能库，agent 可能会直接调用 Volatility3 的 help 命令，然后从零开始猜参数。但有了这个库，agent 的流程是这样的——

  1. 扫描 754 个 skill 的 frontmatter（每个约 30 tokens），12 个相关技能被匹配出来
  2. 加载前 3 个最匹配的：performing-memory-forensics-with-volatility3、hunting-for-credential-dumping-lsass、analyzing-windows-event-logs-for-credential-access
  3. 按 Workflow 章节逐步执行：跑 Volatility3 插件、检查 LSASS 访问模式、关联事件日志证据
  4. 用 Verification 章节验证结果：确认 IOC、映射到 ATT&CK T1003（Credential Dumping）

每个 skill 扫描成本仅 30 tokens，完整加载 500-2000 tokens——这种渐进式披露架构让 agent 能在单次遍历中检索全部 754 个技能，而不会撑爆上下文窗口。

我看到这个设计的时候，第一反应是：这跟 RAG（检索增强生成）的思路完全反过来了。RAG 是让 agent 去外部知识库里“搜”答案，而 agentskills.io 标准是让 agent 直接“加载并执行”一个结构化工作流。前者是问“内存取证怎么做”，后者是“按这个 playbook 做内存取证”。区别在于：前者得到一段可能正确的文本，后者得到一套可验证的操作序列。

这也是为什么 Jangra 在 README 里强调：“This is not a collection of scripts or checklists. It is an AI-native knowledge base built from the ground up for the agentskills.io standard.”（这不是脚本或清单的集合，而是从零开始为 agentskills.io 标准构建的 AI 原生知识库。）

五套框架一张图——一个 skill，五个合规复选框

如果说 754 个技能是肌肉，那 5 套框架映射就是骨架。这个仓库把每一个 skill 都映射到了 MITRE ATT&CK（v18）、NIST CSF 2.0、MITRE ATLAS（v5.4）、MITRE D3FEND（v1.3）和 NIST AI RMF（1.0）五套框架上。这不是简单的标签打标——每个映射都精确到具体的战术/技术/子类 ID。

拿 analyzing-network-traffic-of-malware 这个 skill 举例：它同时映射到 ATT&CK T1071（应用层协议）、NIST CSF DE.CM（持续监控）、ATLAS AML.T0047（AI 网络流量分析）、D3FEND D3-NTA（网络流量分析）、AI RMF MEASURE-2.6（AI 系统监控）。一个 skill，五个合规复选框。

没有其他开源技能库能做到五框架全覆盖。这不是技术壁垒，是工程苦力活——754 个 skill 逐个手工映射，每个 skill 要翻阅五套框架文档，找到最精确的对应项。

ATT&CK 覆盖尤其值得单独拿出来说。仓库包含了全部 14 个战术的映射：侦察（TA0043）、初始访问（TA0001）、执行（TA0002）、持久化（TA0003）、提权（TA0004）、防御规避（TA0005）、凭证访问（TA0006）、横向移动（TA0008）、C2（TA0011）、数据渗出（TA0010）、影响（TA0040）等。其中凭证访问、防御规避、横向移动三个战术覆盖最“强”，发现（TA0007）和资源开发（TA0042）为“中等”。

值得一提的是，ATT&CK v19 在 2026 年 4 月 28 日刚刚发布，把防御规避（TA0005）拆成了两个新战术：Stealth 和 Impair Defenses。Jangra 在 README 里明确写了“Skill mappings will be updated in a forthcoming release”——这种对框架版本变化的即时响应，在开源安全工具里并不多见。

NIST CSF 2.0 这边，2024 年 2 月新增的 Govern 功能（GV）有 30+个 skill 覆盖，而 Detect 功能（DE）是重头戏——200+个 skill 集中在威胁狩猎、SIEM 关联、异常检测上。Protect（PR）150+，Respond（RS）160+，Recover（RC）40+。全部 22 个类别、106 个子类别都有对应映射。

MITRE ATLAS v5.4 覆盖了 16 个战术、84 个技术，包括 2025 年底新增的 agentic AI 攻击向量：AI agent 上下文投毒、工具调用滥用、MCP 服务器入侵、恶意 agent 部署。D3FEND v1.3 作为 NSA 资助的防御对策知识图谱，用 267 个防御技术跨 7 个战术类别（Model、Harden、Detect、Isolate、Deceive、Evict、Restore）与 ATT&CK 双向映射。NIST AI RMF 1.0 加上 GenAI Profile（AI 600-1，2024 年 7 月发布）的 12 个生成式 AI 风险类别，覆盖 72 个子类别——而科罗拉多州 AI 法案（2026 年 2 月生效）为遵守 NIST AI RMF 的组织提供了法律安全港，让这些映射直接关联到了合规实务。

从 Claude Code 到 Gemini CLI——26+平台零配置即插即用

这个仓库最让我愣神的地方，不是技能数量，不是框架覆盖，而是它的兼容性列表。

AI 代码助手：Claude Code（Anthropic）、GitHub Copilot（Microsoft）、Cursor、Windsurf、Cline、Aider、Continue、Roo Code、Amazon Q Developer、Tabnine、Sourcegraph Cody、JetBrains AI。

CLI agent：OpenAI Codex CLI、Gemini CLI（Google）。

自主 agent：Devin、Replit Agent、SWE-agent、OpenHands。

Agent 框架和 SDK：LangChain、CrewAI、AutoGen、Semantic Kernel、Haystack、Vercel AI SDK、任何兼容 MCP 的 agent。

一共 26+个平台。安装方式只有一条命令——

npx skills add mukul975/Anthropic-Cybersecurity-Skills

或者直接 git clone。没有配置文件，没有 API 密钥，没有数据库迁移。只要平台支持 agentskills.io 标准，就能在几秒钟内让 agent 获得 754 个安全技能。

这背后的 agentskills.io 标准是个什么东西？从仓库的 skill 文件结构来反推，它至少定义了以下规范：YAML frontmatter 的字段名和格式（name 用 kebab-case 1-64 字符、description 含关键词利于 agent 发现、domain/subdomain/tags 分类体系、五套框架的技术 ID 引用格式）、Markdown body 的四个固定章节（When to Use / Prerequisites / Workflow / Verification）、目录结构（SKILL.md + references/ + scripts/ + assets/）。这是一个典型的“约定优于配置”设计——不发明新语法，用现有的 YAML 和 Markdown 组合出机器可解析的结构化知识。

让我觉得有意思的是，这个仓库已经被多个外部索引收录：VoltAgent 的 awesome-agent-skills（1000+技能索引）、ottosulin 的 awesome-ai-security、RoggeOhta 的 awesome-codex-cli、skillsllm.com 的技能目录、Openflows 的信号分析追踪、NeverSight 的自动化技能 feed。一个发布仅两个月的社区项目（v1.0.0 发布于 2026 年 3 月 11 日，当时有 734 个技能），能在这么短时间内进入多个 awesome list，说明 agent 技能管理这个赛道正在快速成形。

Hasan Toor（@hasantoxr）在社交媒体上评价：“A database of real, organized security skills that any AI agent can plug into and use. Not tutorials. Not blog posts.” Medium 上的 fazal-sec 则说：“This is not a random collection of security scripts. It's a structured operational knowledge base designed for AI-driven security workflows.”

缺口、野心与那个 4.8 百万的数字

ISC2 的数据显示，2024 年全球网络安全职位缺口达到480 万。这不是一个“人才短缺”的温和表述——480 万意味着全球现有安全从业者总数（约 550 万）的基础上，还有接近一倍的岗位没人填。而 AI agent 被认为是可以弥合这个缺口的关键变量。

但问题是：今天的 AI agent 能写代码、能搜网页，却缺乏把通用 LLM 变成合格安全分析师所需的“从业者 playbook”。正如 Jangra 在 README 里写的：“Existing security tool repos give you wordlists, payloads, or exploit code. None of them give an AI agent the structured decision-making workflow a senior analyst follows: when to use each technique, what prerequisites to check, how to execute step-by-step, and how to verify results. That is the gap this project fills.”

480 万岗位缺口不是靠训练更强大的模型来填的，是靠把人类专家的决策流程编码成机器可执行的工作流来填的。

看看 26 个安全域里技能分布最少的几个：Deception Technology 只有 2 个 skill，Compliance & Governance 只有 5 个，Ransomware Defense 只有 7 个。Jangra 在 CONTRIBUTING.md 里明确指出这些域“最需要帮助”。这暴露了一个现实：即使是 754 个技能的体量，对于真实世界的安全运营需求来说，仍然只是冰山一角。

但方向是对的。ATT&CK Navigator 图层文件随 v1.0.0 发布，让团队可以可视化地看到覆盖缺口。GARS-2026 全球 Agentic AI 就绪度调查（60 个问题、匿名、SRH Berlin 监督、CC-BY 4.0 公开发布）正在收集安全从业者对 agentic AI 的实际准备度数据。Casky.ai 的 Playground 让人可以直接在浏览器里跑技能演练——不需要安装，不需要配置，打开就能用。

（顺便说一句，填那个 GARS-2026 问卷能拿 50 个 Casky Tokens，用于 casky.ai 的早期访问——Jangra 显然深谙开发者增长之道。）

这不是 Anthropic 的项目，但 Anthropic 应该关注它

仓库名字叫 Anthropic Cybersecurity Skills，但它跟 Anthropic PBC 没有任何官方关系。Jangra 在 README 顶部用⚠️符号明确标注了“Community Project — Not affiliated with Anthropic PBC.”这种命名方式在开源社区并不罕见——用目标平台或生态的名字做前缀，表明兼容性和设计意图。

但这里有一个微妙的张力。Anthropic 自己的 Claude Code 是这个仓库支持的首批平台之一，而 Anthropic 在 AI 安全领域的品牌定位（Constitutional AI、RLHF、负责任扩展政策）与这个仓库的使命高度重合。如果 Anthropic 官方选择正式支持或收购这个项目，它可能成为 Claude 生态在安全 agent 领域的一个关键基础设施。如果 Anthropic 选择保持距离，这个社区项目将继续独立演化，成为 agentskills.io 生态的一部分——与 GitHub Copilot、OpenAI Codex CLI、Gemini CLI 等竞品平台保持同等兼容性。

敌人的敌人是盟友。但在这个故事里，所有平台都是盟友——因为 agentskills.io 标准不排他。

Jangra 本人在社区里的身份值得注意。GitHub 用户名 mukul975，真名 Mahipal Jangra，从仓库的 commit 历史和 issue 响应速度来看，这是一个高强度的独立维护者。v1.0.0 发布后不到两个月，技能数量从 734 增长到 754，增加了 20 个 skill 和 3 套框架映射（ATLAS、D3FEND、AI RMF 都是 post-release 加入的）。这种迭代速度对于一个单人项目来说相当惊人。

仓库的 Star 历史曲线和多个 awesome list 的收录表明，社区正在注意到这件事。但真正的考验不在于 Star 数，而在于：有多少安全团队真的把这些 skill 部署到了生产环境的 AI agent 里？有多少 SOC 分析师真的在凌晨三点让 agent 跑了第一条自动化内存取证流程？有多少 480 万缺口里的岗位，真的被这 754 个 skill 填上了一点点？

这些问题现在还回答不了。但 2026 年 5 月 7 日这个凌晨，班加罗尔的 Mahipal Jangra push 完第 754 个 skill 之后，东京那位三级分析师可能还不知道，他手边那本翻烂的 DFIR 手册，已经被另一个人用完全不同的方式重写了一遍——不是给人看的，是给 AI agent 执行的。

这才是最值得盯着看的事。
