Claude Code、Codex 并行编程：Superset 如何成为 AI Agent 调度层？

ai-daily · 2026 年 5 月 25 日 21:42 · 来源：GitHub Trending typescript

2026 年 1 月 15 日凌晨，旧金山某间公寓的屏幕还亮着。一个全栈开发者同时开着 6 个终端窗口，左边 Claude Code 在重构用户认证模块，右边 Codex CLI 在写单元测试，第三个窗口里 Gemini CLI 正在改前端组件——他像个空中交通管制员，不停地在 Alt+Tab 之间切换，脑子快要烧了。

这种画面在过去半年里越来越常见。自 2025 年下半年 Anthropic 发布 Claude Code、OpenAI 推出 Codex CLI 以来，“终端里的 AI 程序员”从一个实验玩具变成了生产工具。但问题也随之而来：当你想同时跑 3 个、5 个、10 个 AI agent 时，终端切换的认知负担直接拉满——context switching 不再是人类开发者的专属痛苦，管理 AI agent 本身也成了一门手艺。

管理一群 AI，比写代码本身更让人头秃。

一个终端管 10 个 agent，这工具想把“AI 监工”做成标准姿势

Superset 就是在这个时间点出现在 GitHub Trending 上的。2026 年 1 月 15 日，这个名为 superset-sh/superset 的 TypeScript 项目冲上榜单，定位很直接：“Code Editor for the AI Agents Era”——不是给人类用的编辑器，是给 AI agent 用的控制面板。

核心技术思路是 git worktree 隔离。每个 agent 任务被分配到一个独立的工作目录和分支，互不干扰。这意味着你可以让 Claude Code 在 feature/auth 分支上重构登录逻辑，同时让 Codex CLI 在 fix/payment-bug 分支上修支付漏洞，再加一个 Gemini CLI 在 experiment/new-ui 上改组件——三个 agent 并行跑，不会互相踩文件、不会污染彼此的工作区。

Superset 内置了一个 diff viewer，当某个 agent 完成修改后，你可以直接在应用里检查代码变更，决定接受还是丢弃。它还支持一键把当前工作区甩到外部编辑器（VS Code、Cursor 等）或终端里继续手动调整。整个逻辑链条是：分配任务→agent 干活→通知你审查→一键打开编辑器→人工微调→合并。

“Wait less, ship more”这句话，第一次听起来不像营销口号。

兼容一切 CLI agent，但真正的野心是“调度层”

Superset 的兼容列表值得仔细看：Claude Code、OpenAI Codex CLI、Cursor Agent、Gemini CLI、GitHub Copilot、Amp Code、OpenCode、Pi——官方用了一个“Fully supported”标签全覆盖。更关键的是最后一行：“Any CLI agent — If it runs in a terminal, it runs on Superset”。

这意味着 Superset 不想绑定任何一个 AI 模型提供商。它的定位是中间调度层：你选你的 agent，我管我的工作区和任务队列。这种“agent-agnostic”的姿态在 2026 年初的 AI 工具生态里并不多见——大多数产品都在试图锁定用户到自家 agent 上（比如 Cursor 的 agent 模式只能在 Cursor 里用）。

技术栈方面，Superset 基于 Bun v1.0+ 运行时、Git 2.20+、GitHub CLI（gh）、Caddy 做本地反向代理（用于 Electric SQL streams），目前只正式支持 macOS。Windows 和 Linux 被标注为“untested”。桌面应用用 bun run build 构建，输出在 apps/desktop/release 目录。

键盘快捷键的设计透露出这个工具的真实使用场景——高频操作。⌘1-9 直接切到第 1-9 号工作区，⌘N 新建工作区，⌘T 在终端里开新标签，⌘D 和 ⌘⇧D 分别向右和向下分屏。这种快捷键密度让人想起 iTerm2 或 tmux 的重度用户配置——Superset 显然假设你会同时盯着多个 agent，并且需要在它们之间快速跳转。

工作区预设（Workspace Presets）功能允许你在 .superset/config.json 里定义 setup 和 teardown 脚本。比如 setup 脚本执行 cp ../.env .env && bun install，teardown 脚本清理临时文件。每个工作区创建时自动跑 setup，删除时自动跑 teardown——这让批量创建 agent 任务变得可行：你可以预先配置好环境变量和依赖安装，然后一口气启动 5 个 agent 分别处理不同的 issue。

（值得一提的是，Superset 的许可证是 Elastic License 2.0（ELv2），不是 MIT 也不是 GPL。ELv2 的限制主要在“不能把 Superset 本身作为托管服务卖给别人”——对大多数个人开发者和企业内部使用来说影响不大，但对想做“AI agent 调度 SaaS”的创业者来说，得先跟团队谈谈。）

我看到这个项目时的第一反应是：它解决了一个“还没被命名”的痛点。2025 年下半年以来，开发者社区里关于“如何管理多个 AI agent”的讨论在 Discord 和 Twitter 上零星出现，但没有人把它定义成一个产品品类。Superset 的 README 里没有写“我们发明了 X”，而是直接摆功能——这种“先做出来再说”的风格，反而比那些动辄“重新定义开发范式”的 PR 稿更让人信服。

不过，Superset 目前的状态也暴露了一些现实限制。macOS only + Bun 运行时 + Caddy 反向代理 + gh CLI 的依赖组合，意味着它暂时不是“下载即用”的类型——你得先配好一套工具链。对于已经在这套生态里的开发者（用 Bun、用 gh、用 macOS）来说门槛很低，但对于 Windows 上的 Node.js 开发者或者习惯了 GitHub Desktop 的用户，可能还得等社区贡献兼容性补丁。

社区方面，Superset 团队在 Discord 和 Twitter 上建立了沟通渠道，GitHub Discussions 用于问答和想法分享。从提交历史和 Star 增速来看（2026 年 1 月 15 日登上 Trending），这个项目的关注度正在快速爬升——但它能不能从“开发者玩具”变成“团队标配”，取决于后续的稳定性、跨平台支持，以及最重要的：当 agent 本身出错时，Superset 能不能提供足够的调试信息。

2026 年的开发者正在从“写代码的人”变成“管 AI 写代码的人”。这个转变比大多数人预想的更快——而 Superset 想成为这个新角色的控制台。至于它能不能做到，等 Windows 版本出来再说吧。
