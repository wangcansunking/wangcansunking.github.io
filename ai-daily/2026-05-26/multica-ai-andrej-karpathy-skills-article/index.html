卡帕西吐槽被开源团队做成文件：Claude Code四原则少犯80%

ai-daily · 2026 年 5 月 26 日 08:08 · 来源：GitHub Trending any

2026 年 5 月 7 日凌晨 2:14，Andrej Karpathy 发了一条推文。不是宣布新公司，不是展示新模型，而是吐槽 —— 精准地、外科手术式地吐槽了 LLM 在写代码这件事上的一堆毛病。

这条推文很快在开发者圈子里炸开了。不是因为 Karpathy 说了什么惊天秘密，而是因为他说得太对了。每一个跟 Claude Code、Cursor、Copilot 打过交道的程序员，都在他的描述里看到了自己昨天的遭遇。

“模型会替你做出错误的假设，然后直接跑下去，根本不检查。” 他在推文里写道。“它们不管理自己的困惑，不寻求澄清，不暴露不一致，不呈现权衡，在该反驳的时候不反驳。”

这段话像一把手术刀，精准地剖开了 AI 编程助手的核心病灶。而更妙的是，一个叫 Multica 的开源团队，在 GitHub Trending 上扔出了一个解决方案 —— 不是新模型，不是新框架，而是一个文件。一个 CLAUDE.md 文件。

四句话就能让 AI 编程助手少犯 80% 的错误，你敢信？

一、Karpathy 的吐槽清单：模型到底在哪些地方翻车

Let me 把 Karpathy 原文里提到的三个问题掰开揉碎了说。

第一个问题：模型会悄无声息地做错误假设，然后一路狂奔。举个场景 —— 你让 Claude Code 写一个用户认证模块。你没说要支持 OAuth 还是 JWT，你只说了“做到登录功能”。模型不会停下来问你“你要哪种认证方式？”，它会自己选一个，吭哧吭哧写完 300 行代码，然后你发现它选的根本不是你项目里在用的方案。Karpathy 的观察是，LLM 在这类模糊指令面前，“不管理自己的困惑，不寻求澄清”。它不告诉你“这里有 3 种可能的理解”，而是直接 pick 一个，然后祈祷你接受。

第二个问题：过度复杂化。Karpathy 的原文是这么说的 —— “它们真的很喜欢把代码和 API 搞复杂，膨胀抽象，不清理死代码……用 1000 行做到一个 100 行就能搞定的膨胀结构。” 这简直是每个用过 AI 编程工具的人都见过的噩梦。你让它写一个简单的数据解析函数，它给你整出一套完整的工厂模式 + 策略模式 + 抽象基类，还附赠一个“将来可能会用到”的配置文件解析器。1000 行 vs 100 行 —— Karpathy 给的这个比例不是夸张修辞，是真实发生过的事。

第三个问题：乱动不该动的代码。Karpathy 写道：“它们有时还是会修改/删除它们理解不够充分的注释和代码，产生副作用，即使这些跟任务完全正交。” 你让模型改一个按钮的颜色，它顺手把你精心写的注释删了，因为它觉得“这段注释描述的逻辑已经过时了”。或者它把旁边一个它觉得“写得不好”的函数重构了，然后你的测试挂了。正交改动 —— 这是软件工程里最让人血压升高的 bug 来源，AI 编程助手偏偏特别擅长制造这类问题。

这些毛病单个看不致命，叠在一起能让一个下午的生产力归零。

二、一个文件四个原则：Multica 的 CLAUDE.md 到底写了什么

Multica 团队做的事异常简单。他们把 Karpathy 的观察翻译成了四个原则，塞进一个 CLAUDE.md 文件里。这个文件可以安装成 Claude Code 插件，也可以直接丢进项目根目录。

四个原则，每一个都直接对应 Karpathy 列出的一个或多个问题。

原则一：Think Before Coding（编码前先思考）。这条直接打第一个问题 —— 错误假设和隐藏困惑。文件里写了四条硬规则：明确陈述假设（如果不确定，问，不要猜）；呈现多种理解（有歧义时不悄无声息地 pick 一个）；该反驳时就反驳（有更简单的方案就直接说）；困惑时就停下来（说出哪里不清楚，要求澄清）。这不是什么高深的 AI alignment 理论，这是把程序员之间 code review 的基本礼仪写进了给 LLM 的 system prompt。

原则二：Simplicity First（简单优先）。这条打第二个问题 —— 过度复杂化和膨胀抽象。规则也很直白：不加用户没要求的功能；不为单次使用的代码建抽象；不搞没被要求的“灵活性”或“可配置性”；不处理不可能发生的错误场景；如果 200 行能精简到 50 行，重写。文件里甚至给了一个检验标准：“一个有经验的工程师会说这过度复杂吗？如果是，就简化它。”

这可能是 AI 时代最值钱的一句话：Don't tell it what to do, give it success criteria and watch it go.

原则三：Surgical Changes（外科手术式改动）。这条直接打第三个问题 —— 正交编辑和乱动不该动的代码。规则是：不“改进”相邻的代码、注释、格式；不重构没坏的东西；匹配现有风格，即使你觉得你的写法更好；如果注意到不相关的死代码，提一嘴，但别删。当你的改动产生了孤儿代码（你改完后没人调用的 import / 变量 / 函数），只清理你自己制造的，别碰之前就存在的。检验标准：“每一行改动都应该能直接追溯到用户的需求。”

原则四：Goal-Driven Execution（目标驱动执行）。这是整个文件里最有意思的一条，也是 Karpathy 本人特别强调的一点。原文引用他的原话：“LLM 非常擅长循环直到达成特定目标……不要告诉它做什么，给它成功标准，然后看它跑。” 这条原则要求把命令式指令转换成可验证的目标。比如“添加验证”要变成“为无效输入写测试，然后让测试用”；“修这个 bug”要变成“写一个能复现它的测试，然后让它用”；“重构 X”要变成“确保重构前后测试都用”。对于多步骤任务，要求模型先陈述一个简短计划，每步带验证检查点。

这个原则的妙处在于，它利用了 LLM 的一个真正强项 —— 循环迭代能力 —— 来弥补它的弱项 —— 一次性搞对复杂需求。强成功标准让模型能独立循环，弱标准（“把它搞对”）则需要不断澄清。Multica 团队在文件里特别标注了这一点，说这是从 Karpathy 的洞察中提炼出的核心。

三、安装方式暴露了一个有趣的生态信号

Multica 给这个 CLAUDE.md 文件提供了两种安装方式，而这两种方式本身就在讲一个关于 AI 工具生态的故事。

选项 A 是 Claude Code 插件市场安装。在 Claude Code 里敲两行命令：/plugin marketplace add forrestchang/andrej-karpathy-skills 然后 /plugin install andrej-karpathy-skills@karpathy-skills。装完之后，这个技能会在你所有的项目里生效。

选项 B 是传统的 CLAUDE.md 文件方式。新建项目就 curl 下载，已有项目就追加到现有的 CLAUDE.md 后面。

（值得一提的是，Multica 自己是一个开源平台，专门做编程 agent 的可复用技能管理 —— 这个 CLAUDE.md 文件本质上是他们平台理念的一次“病毒式传播”。用一个文件来展示整个平台的价值主张，这招挺聪明的。）

更有意思的是，这个仓库还包含了一个 Cursor 项目规则文件 —— .cursor/rules/karpathy-guidelines.mdc。也就是说，同样的四个原则可以同时作用在 Claude Code 和 Cursor 两个不同的 AI 编程环境里。这说明这套指导原则是模型无关的 —— 它不是对某个特定模型的 prompt engineering 技巧，而是对 LLM 编码行为模式的通用修正。

好 prompt 的价值正在从“模型专属”走向“跨平台通用”，这可能是 AI 工程化最被低估的趋势。

Multica 的创始人（仓库里显示是一个 X 账号 @jiayuan_jy 的开发者）显然深谙此道。他没有去训练一个新模型，没有去做一个复杂的 agent 框架，他只是把 Karpathy 的观察翻译成了 LLM 能理解和遵循的结构化指令。这个操作的成本几乎为零，但效果 —— 如果你去看看 GitHub 上那些“Claude Code 让我疯了”的 issue，你就会明白 —— 可能比换一个更贵的模型还管用。

四、如何判断这四个原则生效了？一个简单到残忍的检验清单

Multica 在仓库里给了一个“How to Know It's Working”清单，列出了四个信号。

第一个信号：diff 里的不必要改动变少了 —— 只有被要求的改动出现。第二个信号：因为过度复杂化而导致的返工变少了 —— 代码第一次就足够简单。第三个信号：澄清问题在做到之前出现 —— 而不是在搞砸之后。第四个信号：PR 干净、最小化 —— 没有顺手重构，没有“顺便改进”。

这四条标准，坦白说，放在人类程序员身上也是顶级的职业素养。一个能主动澄清需求、写简洁代码、不做正交改动、提交最小化 PR 的工程师，在任何团队里都是宝贝。Multica 做的事情，本质上是用一份 prompt 文件，给 AI 编程助手装上了一套“高级工程师行为规范”。

但这里有一个 Multica 自己都承认的 tradeoff。他们在文件里写了一段“Tradeoff Note”：这些指导原则偏向“谨慎优先于速度”。对于琐碎任务（简单的拼写错误修复、明显的一行代码改动），你得自己判断 —— 不是每次改动都需要全套严谨流程。目标是减少非琐碎工作上的昂贵错误，而不是拖慢简单任务。

这段话很重要。因为它说明 Multica 团队知道自己在做什么 —— 他们不是在提供一个“万能 prompt”，而是在提供一个“默认行为修正”。这个修正的代价是，在某些场景下模型会变得“过于谨慎”，可能会多问你几个问题，可能会拒绝做一些它觉得“过度复杂”的事情。但对于大多数非琐碎的编码任务来说，这个代价完全值得。

我看到这个 tradeoff 的时候，第一反应是，这其实就是把人类 code review 的 overhead 嫁接到了 AI 上。你让一个高级工程师 review 每一行 AI 写的代码，时间成本太高。但你让 AI 自己内置一套“自我 review”的行为准则，overhead 就变成了 prompt 里多出来的几百个 token。几百个 token 换一个下午不被正交改动搞崩测试 —— 这笔账怎么算都是赚的。

五、为什么这件事值得你花 10 分钟去装一个插件

让我觉得最有意思的，不是这四个原则本身 —— 说实话，任何一个有经验的工程师都能写出类似的东西。让我愣神的是，这件事在 GitHub Trending 上登顶的速度。

一个 CLAUDE.md 文件。没有新模型。没有新技术。没有融资公告。就是一个文件，用几百字把 Karpathy 的吐槽翻译成了结构化的行为指令。然后它登上了 GitHub Trending。

这说明什么？说明 AI 编程助手的“行为问题”已经严重到了开发者愿意主动寻找解决方案的地步。不是“模型不够聪明”的问题 —— Claude 3.5 Sonnet 和 GPT-4 已经足够聪明了。问题是“聪明但没有判断力”。就像一个智商 160 但社会经验为零的实习生，他能写出你要求的东西，但他不知道什么时候该停下来问一句“老板，你说的这个需求有 3 种理解，你要哪一种？”

Multica 的 CLAUDE.md 就是给这个实习生配了一个随身携带的导师手册。手册上没写什么高深的技术知识，只写了四句话：做事前先想、保持简单、别乱动别人东西、搞清楚什么叫“做完了”。

Karpathy 自己在推特里说的那句话，可能是对这件事最好的总结：“LLM 非常擅长循环直到达成特定目标。” 但前提是，你得给它一个正确的目标。不是“写个登录功能”，而是“写一个能用这 5 个测试用例的登录功能，不要引入新的依赖，不要修改 auth 模块以外的任何文件，如果有不清楚的地方先问”。第二种指令，就是 Multica 试图用 system prompt 让 LLM 自动补全出来的那种。

AI 编程的下一个战场，可能不是更大的模型，而是更好的指令。一个 CLAUDE.md 文件，四个原则，MIT 开源许可 —— 这可能是 2026 年 5 月你花 10 分钟就能装上的、ROI 最高的开发工具。当然，如果哪天 Anthropic 直接把这四个原则炼进 Claude Code 的默认行为里，Multica 的这个仓库就可以功成身退了。但在那之前，手动给 AI 装上“判断力”的活，还得开发者自己来。
