第7课：上下文管理

学习目标

  1. 理解 Context Window 限制与 Token 增长问题
  2. 实现滑动窗口：只保留最近 N 轮对话
  3. 实现摘要压缩：用模型自己压缩旧历史
  4. 了解三家 Provider 的 Prompt Caching 机制差异

问题：Token 随轮数线性增长

每轮对话，你发送的 Token 数都在增加：

轮次 1：system(100) + 对话(50)   =  150 tokens
轮次 5：system(100) + 对话(500)  =  600 tokens
轮次 20：system(100) + 对话(2000) = 2100 tokens

各 provider 的上下文上限：

Provider 模型 上限
DeepSeek deepseek-chat 64K
Kimi moonshot-v1-128k 128K
Anthropic claude-opus-4-7[1m] 1M

不管多大都有上限，且费用随 token 数线性增长。

策略一：滑动窗口

最简单的策略——只保留最近 N 轮，丢弃更早的历史：

import OpenAI from "openai";
import { client, model } from "../shared/llmClient";

type Msg = OpenAI.ChatCompletionMessageParam;

class SlidingWindowChat {
  private history: Msg[] = [];

  constructor(
    private systemPrompt: string,
    private maxTurns = 10  // 保留最近 10 轮
  ) {
    this.history.push({ role: "system", content: systemPrompt });
  }

  async send(userMessage: string): Promise<string> {
    this.history.push({ role: "user", content: userMessage });

    // 如果超出窗口，从中间删除旧消息（保留 system + 最近 N 轮）
    while (this.history.length > 1 + this.maxTurns * 2) {
      this.history.splice(1, 2); // 删除最早的一轮（user + assistant）
    }

    const response = await client.chat.completions.create({
      model,
      max_tokens: 2048,
      messages: this.history,
    });

    const text = response.choices[0].message.content ?? "";
    this.history.push({ role: "assistant", content: text });
    return text;
  }
}

适用场景：聊天机器人、近期上下文足够的任务 缺点：丢失旧信息，模型可能"忘记"早期重要内容

策略二：摘要压缩

当对话过长时，让模型把旧历史压缩成一段摘要：

class SummaryChat {
  private history: Msg[] = [];
  private summary: string = "";

  constructor(
    private systemPrompt: string,
    private compressionThreshold = 20  // 超过 20 轮就压缩
  ) {}

  private async compressHistory(): Promise<void> {
    if (this.history.length < this.compressionThreshold * 2) return;

    // 取前半段历史进行压缩
    const toCompress = this.history.slice(0, this.history.length / 2);
    const recentHistory = this.history.slice(this.history.length / 2);

    const compressionResponse = await client.chat.completions.create({
      model,
      max_tokens: 1024,
      messages: [
        { role: "system", content: "你是对话摘要助手。请将以下对话历史压缩为简洁的摘要，保留所有重要信息。" },
        {
          role: "user",
          content: `请压缩这段对话历史：\n\n${JSON.stringify(toCompress)}\n\n已有摘要：${this.summary}`,
        },
      ],
    });

    this.summary = compressionResponse.choices[0].message.content ?? "";
    this.history = recentHistory; // 只保留近期历史

    console.log(`🗜️ 历史已压缩，保留 ${recentHistory.length / 2} 轮近期对话`);
  }

  async send(userMessage: string): Promise<string> {
    await this.compressHistory();

    this.history.push({ role: "user", content: userMessage });

    // System prompt = 原始提示 + 历史摘要
    const systemWithSummary = this.summary
      ? `${this.systemPrompt}\n\n【对话历史摘要】\n${this.summary}`
      : this.systemPrompt;

    const response = await client.chat.completions.create({
      model,
      max_tokens: 2048,
      messages: [
        { role: "system", content: systemWithSummary },
        ...this.history,
      ],
    });

    const text = response.choices[0].message.content ?? "";
    this.history.push({ role: "assistant", content: text });
    return text;
  }
}

适用场景：长期对话、需要保留整体语境 缺点：压缩有信息损失，额外消耗 token

策略三：Prompt Caching（三家机制不同）

对重复的长 system prompt（如大型代码库、文档）启用缓存可以省 ~90% Token 费用，但三家机制完全不同：

DeepSeek（自动 Hard Disk Cache）

DeepSeek 自动 缓存重复的 prefix（system 提示 + 早期 messages），无需任何代码改动：

// 你什么都不用改，DeepSeek 自动检测重复 prefix 并缓存
const response = await client.chat.completions.create({
  model,
  max_tokens: 2048,
  messages: [
    { role: "system", content: veryLongSystemPrompt },  // 几千字
    ...history,
  ],
});

// 检查缓存命中（DeepSeek 在响应里给出）
console.log(`缓存命中 token: ${(response.usage as any)?.prompt_cache_hit_tokens}`);
console.log(`未命中 token:   ${(response.usage as any)?.prompt_cache_miss_tokens}`);

DeepSeek 缓存命中价格 0.1 元/百万 token（vs 未命中 1 元），降 90%。

Kimi（Context Caching API）

Kimi 提供显式 Context Caching API，需要先创建 cache 再引用：

// 1. 创建 cache（一次性）
const cache = await fetch("https://api.moonshot.cn/v1/caching", {
  method: "POST",
  headers: { Authorization: `Bearer ${apiKey}`, "Content-Type": "application/json" },
  body: JSON.stringify({
    model: "moonshot-v1-32k",
    messages: [{ role: "system", content: veryLongSystemPrompt }],
    ttl: 3600,  // 1 小时
  }),
}).then(r => r.json());

// 2. 后续请求引用 cache_id
const response = await client.chat.completions.create({
  model: "moonshot-v1-32k",
  max_tokens: 2048,
  messages: [...history],
} as any, { headers: { "X-Msh-Context-Cache": cache.id } });

Anthropic（cache_control 显式标记）

Anthropic 用 cache_control 字段显式标记可缓存的 prefix：

// 注意：这里需要用 Anthropic 原生 SDK，OpenAI 兼容端点不支持 cache_control
import Anthropic from "@anthropic-ai/sdk";
const ant = new Anthropic();

const response = await ant.messages.create({
  model: "claude-opus-4-7",
  max_tokens: 2048,
  system: [
    { type: "text", text: veryLongSystemPrompt, cache_control: { type: "ephemeral" } },
  ],
  messages: history,
});

console.log(`缓存读取: ${response.usage.cache_read_input_tokens}`);
console.log(`缓存写入: ${response.usage.cache_creation_input_tokens}`);

策略对比

策略 实现复杂度 信息保留 Token 节省 适用场景
滑动窗口 ⭐ 简单 只保留近期 中 简单聊天
摘要压缩 ⭐⭐ 中等 有损压缩 中高 长期对话
DeepSeek 自动缓存 ⭐ 零代码 完整保留 高（~90%） 固定长 prefix
Kimi Context Caching ⭐⭐ 需创建 cache 完整保留 高 重复长文档
Anthropic cache_control ⭐⭐ 显式标记 完整保留 高（~90%） 固定长 system

建议：如果你的应用 system prompt 较长且重复，直接用 DeepSeek 最省事——零改动就有 90% 的成本节省。
