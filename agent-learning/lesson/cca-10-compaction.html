第10课：Context 自动压缩与 State Re-injection

介绍

模型的 context window 是稀缺资源。长会话怎么不"撞墙"？

很多项目用最简单的"超过截断点就丢前面"。Claude Code 的方案是两阶段压缩 + 状态重启点补偿：自动判断阈值 → 后台 forked agent 总结 → 把"被丢掉的工具能力 / 正在读的文件 / 进行中的 Plan"重新声明回去。本课拆这个机制。

源码定位

文件 角色
src/services/compact/autoCompact.ts 触发判定 + 熔断器
src/services/compact/compact.ts 压缩主逻辑 + State Re-injection
src/services/compact/sessionMemoryCompact.ts 用 Session Memory 替代 LLM 总结

Effective Context Window：留出 Summary 预算

真实源码（src/services/compact/autoCompact.ts:33）：

export const MODEL_CONTEXT_WINDOW_DEFAULT = 200_000;
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000;

export function getEffectiveContextWindowSize(model: string): number {
  const reservedTokensForSummary = Math.min(
    getMaxOutputTokensForModel(model),
    MAX_OUTPUT_TOKENS_FOR_SUMMARY,
  );
  let contextWindow = getContextWindowForModel(model);
  return contextWindow - reservedTokensForSummary;
}

200K 窗口里永远预留 20K 给 summary 输出。否则触发压缩时 summary 自己就会撞墙（PTL: Prompt Too Long）。

这是个简单但容易忘的细节——你以为窗口有 200K 可用，其实最多用到 180K。

CAPPED_DEFAULT_MAX_TOKENS：基于真实数据的优化

// src/utils/context.ts:18
// Capped default for slot-reservation optimization. BQ p99 output = 4,911
// tokens, so 32k/64k defaults over-reserve 8-16× slot capacity. With the cap
// enabled, <1% of requests hit the limit; those get one clean retry at 64k
export const CAPPED_DEFAULT_MAX_TOKENS = 8_000;
export const ESCALATED_MAX_TOKENS = 64_000;

这是真实数据驱动的优化：

  • 业务真实 p99 输出长度是 4911 tokens
  • 默认 32K / 64K 过度预留 8-16 倍 slot
  • 卡到 8K 后，<1% 请求会撞限制 → 那些自动 retry 到 64K

效果：API 排队效率显著提升、token 浪费显著减少。

这就是工程价值：默认值不是拍脑袋的"安全大数"，而是测量出来的 p99。

Auto-Compact 触发与熔断器

真实源码（src/services/compact/autoCompact.ts:241）：

export async function autoCompactIfNeeded(
  messages: Message[],
  toolUseContext: ToolUseContext,
  cacheSafeParams: CacheSafeParams,
  querySource?: QuerySource,
  tracking?: AutoCompactTrackingState,
  snipTokensFreed?: number,
): Promise<{ wasCompacted: boolean; compactionResult?: CompactionResult }> {

  // ⚠️ 熔断器：连续失败 ≥3 次 → 完全停止该会话的 autocompact
  // 防止"用户单张超大图片就超限"导致无限重试
  if (tracking?.consecutiveFailures !== undefined &&
      tracking.consecutiveFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES) {
    return { wasCompacted: false };
  }

  const shouldCompact = await shouldAutoCompact(messages, model, querySource, snipTokensFreed);
  if (!shouldCompact) return { wasCompacted: false };

  try {
    const compactionResult = await compactConversation(messages, ...);
    return { wasCompacted: true, compactionResult, consecutiveFailures: 0 };
  } catch (error) {
    const nextFailures = (tracking?.consecutiveFailures ?? 0) + 1;
    return { wasCompacted: false, consecutiveFailures: nextFailures };
  }
}

两个保护：

  1. 缓冲界限 AUTOCOMPACT_BUFFER_TOKENS = 13_000：剩余 13K 时就开始压缩，不等真撞墙
  2. 熔断器 MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3：连续 3 次失败就停掉该会话的 autocompact

注释里写了：这一行 if 每天给大盘省了约 250K 次死锁 API 调用。看似简单的 3 行代码，节省巨大。

Session Memory 直挂：不再调 API 总结

这是 Claude Code 压缩中最巧妙的设计之一。

如果系统已开启 Session Memory 并已积累足够内容，不再调用模型 API 总结：

// 调 trySessionMemoryCompaction 直接读后台 sub-agent 写好的 session memory 文件
// 充当 SummaryMessage

为什么这么省？因为 Session Memory 提取本来就在后台跑（参考第 9 课的 shouldExtractMemory）。当主线程需要 compact 时，"上一次摘要"已经在磁盘上，直接读就行。

省下来的：

  • 一次完整的 summary API 调用
  • 几万 token 的 summary prompt
  • 几秒钟的延迟

这是"后台积累 + 前台复用"思路在 Agent 工程上的典型应用。

精确裁切：保护 tool_use / tool_result 链

calculateMessagesToKeepIndex() 决定"从哪里切"：

1. 必须从后向前留足配置的下限 token（默认约 10K-40K）
2. 截断不能落在 tool_use / tool_result 链中间
3. 截断不能截断与 assistant 共享 message.id 的 thinking 流

adjustIndexToPreserveAPIInvariants() 强制把 index 向头部平移，把这些孤立片段一并保留——绝不让 API 收到孤立的 tool_result（API 会直接报错）。

这是个非常细的工程细节。它体现了"前向兼容 API 严格契约"的工程纪律。

State Re-injection：本课重头戏

总结完成后，原本的长 messages 被压缩成一条短 SummaryMessage。但 agent 不能"失忆"——它正在读的文件 / 正在做的 Plan / 已激活的 Tools 必须保留。

// src/services/compact/compact.ts:517

// 1. 重新加入刚才通过 FileReadTool 看过的文件（带截断上限）
const fileAttachments = createPostCompactFileAttachments(preCompactReadFileState, ...);

// 2. 重新加入仍在进行中的 Plan / Skill
const planAttachment = createPlanAttachmentIfNeeded(context.agentId);
const skillAttachment = createSkillAttachmentIfNeeded(context.agentId);

// 3. 把丢掉的 Deferred Delta 工具协议重新发给模型
for (const att of getDeferredToolsDeltaAttachment(...)) {
  postCompactFileAttachments.push(createAttachmentMessage(att));
}

最终模型醒来后看到的是：

[System 边界宣告]
  +
[精简文本摘要]
  +
[正在查看的文件内文截取]
  +
[正在做的 Plan]
  +
[仍然激活的 MCP / Tools 完整声明]

模型既释放了冗余历史，依然如同长时程 Agent 那样"身处工作台旁边、手持刚刚用到的工具"。

这种"压缩后状态重建"的思路非常通用，可以套用到任何"长时记忆受限"的系统上。

PTL Fallback：救命稻草

万一 summary 请求自己也撞 PTL（Prompt Too Long）怎么办？

// src/services/compact/compact.ts:462
// CC-1180: 一次剥掉 20% 旧分组重试。最后的"救命稻草"，有损但能解锁
const truncated = ptlAttempts <= MAX_PTL_RETRIES
  ? truncateHeadForPTLRetry(messagesToSummarize, summaryResponse)
  : null;

剥洋葱：每次切掉 20% 头部，最多 N 次。即使丢失早期细节，也保住会话不死锁。

Pattern 提炼

Pattern 1：可用窗口 = 总窗口 - summary 预算

const effective = total - reservedForSummary;

不要"等到撞墙才压缩"。

Pattern 2：默认值基于真实 p99

// p99 = 4,911 tokens，所以 cap = 8,000

数据驱动的默认值，比拍脑袋大数省下大量资源。

Pattern 3：熔断器 = 3 行代码省 25 万 API 调用/天

if (consecutiveFailures >= MAX_FAILURES) return;

任何重试逻辑都该有上限。

Pattern 4：后台积累 + 前台复用

Session Memory 后台写好后，前台需要 compact 时直接读，省掉一次 API。

Pattern 5：State Re-injection 三件套

file 状态 + plan 状态 + 工具协议

任何"压缩 / 截断 / 重启"的场景都该考虑这三类状态怎么不丢。

Pattern 6：API 不变量是硬约束

切割消息时绝对不能留下孤立的 tool_result。

自己实现：80 行的最小压缩管线

// my-compaction.ts
type Message = { role: 'user' | 'assistant'; content: any; uuid: string };

const SUMMARY_RESERVE = 20_000;
const COMPACT_BUFFER  = 13_000;
const MAX_FAILURES    = 3;

export async function autoCompactIfNeeded(
  messages: Message[],
  modelWindow: number,
  state: { failures: number },
  llmSummarize: (msgs: Message[]) => Promise<string>,
): Promise<Message[] | null> {
  // 熔断
  if (state.failures >= MAX_FAILURES) return null;

  const effective = modelWindow - SUMMARY_RESERVE;
  const used = estimateTokens(messages);
  if (used < effective - COMPACT_BUFFER) return null;  // 还没到压缩点

  try {
    return await compact(messages, llmSummarize);
  } catch (e) {
    state.failures += 1;
    return null;
  }
}

async function compact(
  messages: Message[],
  llmSummarize: (msgs: Message[]) => Promise<string>,
): Promise<Message[]> {
  // 1. 剔除图片等非关键素材（节省 summary 输入）
  const cleaned = messages.map(m => stripImages(m));

  // 2. 决定保留多少最近消息
  const keepFrom = calculateKeepIndex(cleaned, 10_000);  // 至少留 10K token 的最近消息

  // 3. 调 LLM 总结被丢掉的部分
  const toSummarize = cleaned.slice(0, keepFrom);
  const summary = await llmSummarize(toSummarize);

  // 4. 拼回：[summary 边界] + [最近消息] + [激活的 tools/files/plan 重新声明]
  const summaryMessage: Message = {
    role: 'assistant',
    content: `[Compacted] ${summary}`,
    uuid: 'summary-' + Date.now(),
  };
  const stateRebuild: Message = {
    role: 'user',
    content: 'Active files: a.ts, b.ts\nActive plan: ...\nActive tools: Read, Bash',
    uuid: 'rebuild-' + Date.now(),
  };

  return [summaryMessage, stateRebuild, ...cleaned.slice(keepFrom)];
}

function stripImages(m: Message): Message {
  if (Array.isArray(m.content)) {
    return {
      ...m,
      content: m.content.map((b: any) =>
        b.type === 'image' ? { type: 'text', text: '[image]' } : b
      ),
    };
  }
  return m;
}

function calculateKeepIndex(messages: Message[], minRecentTokens: number): number {
  let total = 0;
  for (let i = messages.length - 1; i >= 0; i--) {
    total += estimateTokens([messages[i]!]);
    if (total >= minRecentTokens) {
      // 不能切在 tool_use / tool_result 中间
      while (i > 0 && isInToolChain(messages[i]!)) i--;
      return i;
    }
  }
  return 0;
}

function estimateTokens(m: any[]): number { return JSON.stringify(m).length / 4; }
function isInToolChain(m: Message): boolean {
  return Array.isArray(m.content) && m.content.some((b: any) => b.type === 'tool_use' || b.type === 'tool_result');
}

80 行就抓住了 Claude Code 压缩的核心：预算 + 熔断 + 自然断点保护 + State Re-injection。

本课小结

  • 总窗口 200K，永远预留 20K 给 summary 输出
  • CAPPED_DEFAULT_MAX_TOKENS = 8000 是基于业务 p99=4,911 的数据决策
  • 熔断器 3 行代码每天省 25 万次 API 调用
  • Session Memory 直挂：后台已写好就别再花 token 总结
  • 切割时绝不留下孤立 tool_result（API 会拒）
  • State Re-injection：file / plan / 工具协议三件套，让模型醒来后还能继续工作
