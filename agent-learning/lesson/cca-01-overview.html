第1课：泄露事件与全景导览

介绍

2026 年 3 月 31 日凌晨，安全研究者 Chaofan Shou 在 X 发布推文：Anthropic 发布到 npm 的 Claude Code 包没有删除 source map 文件。

这意味着完整的 TypeScript 源码全部公开：

1902 个源文件
513,237 行代码

这是观察一个生产级 Agent 平台到底如何被构建的稀有样本。本课先建立"鸟瞰图"，知道 Claude Code 是个什么形状的系统，后面 13 课每课一个零件，逐个拆开看。

核心结论

Claude Code 不是命令行聊天工具，而是一套本地 Agent 平台。它有三个识别度极高的特征：

  1. 多入口：CLI、REPL、SDK、MCP Server、bridge、remote
  2. 多层次：命令 / 执行内核 / 工具 / 权限 / memory / 扩展
  3. 多形态协作：单 agent / subagent / background / teammate / swarm

如果只把它当成"调 LLM 的脚本"，就会忽略它真正的工程价值。

默认交互的完整链路

打开终端敲 claude，发生了什么？

1. 进程启动
   entrypoints/cli.tsx
     -> 快路径分流（--version / --dump-system-prompt / remote-control / daemon）
     -> 否则 import('./main.tsx')

2. 主启动器
   main.tsx
     -> init(argv)                    // trust 前的安全初始化
     -> 解析 permissionMode / model
     -> 选择执行路径（headless / bridge / remote / 默认 REPL）
     -> fetchBootstrapData()          // 拉取远端配置
     -> getMcpToolsCommandsAndResources()
     -> getTools(permissionContext)
     -> initBundledSkills()
     -> getAgentDefinitionsWithOverrides()
     -> initializeTelemetryAfterTrust()  // trust 后才启动 telemetry
     -> launchRepl()

3. REPL 启动
   replLauncher.tsx
     -> 动态加载 App + REPL 组件
     -> Ink render

4. 用户输入
   PromptInput
     -> processTextPrompt
     -> 进入 query.ts 主循环

5. 执行内核 query.ts (AsyncGenerator)
   for (;;) {
     -> normalizeMessagesForAPI(messages)
     -> claudeApi.stream(...)
     -> 提取 tool_use blocks
     -> runTools() / StreamingToolExecutor
        -> partitionToolCalls() (并发安全分批)
        -> toolExecution.ts (Schema → Validate → Hook → Permission → call())
     -> 工具结果回流到 messages
     -> executePostSamplingHooks() / shouldCompact()
   }

每一行都对应后续课程的一个细节，但先记住整体形状。

为什么这样分层很重要

很多 Agent 项目把"主循环 + 工具调度 + UI 渲染"全揉在一起。Claude Code 把这三件事严格分开，结果是：

  1. REPL 模式和 headless SDK 模式走同一个 query()，只是消费 AsyncGenerator 的 UI 不同
  2. SubAgent 和主 Agent 走同一个 query()，只是 ToolUseContext 被替换
  3. Bridge / Remote 模式也走同一个 query()，只是消息源从 stdin 换成 WebSocket

"本地 REPL 里 Claude 怎么调工具" = "后台自动化任务里 Claude 怎么调工具" = "远程 bridge 里 Claude 怎么调工具"

这三件事走的是完全相同的代码路径，不存在两套实现的行为差异风险。

这就是"统一执行内核"的真正意义。

Pattern 提炼

Pattern 1：双阶段启动（Two-Phase Boot）

// entrypoints/init.ts
export async function init(argv) {
  applySafeEnvironmentVariables()    // trust 前：只应用安全 env
  initializeCertificates()
  initTelemetrySkeleton()            // 注册 sink，但不发事件
}

export async function initializeTelemetryAfterTrust() {
  applyFullEnvironmentVariables()    // trust 通过后：应用全部 env
  attachAnalyticsSink()              // 开始处理 telemetry 队列
}

为什么值得学：trust 建立前不应用任意 env、不发 telemetry，防止"配置文件 / includes 本身就是攻击面"。这是一种最小可信初始化思想，凡是涉及"用户配置"的系统都应该这么做。

Pattern 2：同一个 AsyncGenerator 喂多个消费者

query() 返回 AsyncGenerator<StreamEvent>：

  • REPL 消费它 → 渲染 UI
  • SDK 消费它 → yield 给调用方
  • bridge 消费它 → 通过 WebSocket 推到远端

只要你的 Agent 主循环也设计成生成器，就能用同一份逻辑驱动 CLI / Web / API 三种形态。

Pattern 3：能力按需装配，不是写死

const tools = getTools(permissionContext)        // 工具池随权限变
const mcpTools = await getMcpToolsCommandsAndResources()  // 拉取外部 MCP
const skills = initBundledSkills()               // Skill 也是动态装配
const agents = getAgentDefinitionsWithOverrides() // Agent 定义可覆盖

每一个能力都来自一次"装配调用"，方便：单测、远程模式裁剪、Feature Flag 控制可见性。

自己实现：30 行的"分层雏形"

// my-agent.ts
import { Anthropic } from '@anthropic-ai/sdk';

// ===== 第 1 层：CLI 引导 =====
async function cli(argv: string[]) {
  if (argv.includes('--version')) {
    console.log('0.0.1');
    return;          // 快路径退出
  }
  await main(argv);  // 否则进入完整启动
}

// ===== 第 2 层：初始化 =====
async function main(argv: string[]) {
  const tools = getTools();             // 装配工具
  const systemPrompt = getSystemPrompt();
  await launchRepl({ tools, systemPrompt });
}

// ===== 第 3 层：REPL =====
async function launchRepl(opts: { tools: Tool[]; systemPrompt: string }) {
  const userInput = await prompt('> ');
  for await (const ev of query({ messages: [...], ...opts })) {
    console.log(ev);  // UI = 消费 AsyncGenerator
  }
}

// ===== 第 4 层：执行内核 =====
async function* query(opts): AsyncGenerator<StreamEvent> {
  let messages = opts.messages;
  while (true) {
    const stream = anthropic.messages.stream({ messages, system: opts.systemPrompt, tools: opts.tools });
    for await (const ev of stream) yield ev;

    const toolUses = extractToolUses(messages);
    if (!toolUses.length) break;

    for await (const update of runTools(toolUses, opts.tools)) {
      yield update;
      messages = appendResult(messages, update);
    }
  }
}

这就是 Claude Code 最核心的骨架。后面 13 课要做的，就是把这 30 行里每一个 runTools / getTools / query 一个个打开，看里面塞了多少工程细节。

本课小结

  • Claude Code 是 Agent 平台，不是聊天工具加壳
  • 六层分层 + 多入口 + 多形态，但所有形态共用 query()
  • 双阶段启动和按需装配是值得迁移到自己项目的两个 pattern
  • 后续 13 课会按"启动 → 内核 → 工具 → 权限 → 扩展 → 记忆 → 持久化 → 高阶"的顺序展开
