第11课：Prompt 管理 — 不是字符串，是 runtime

介绍

写 Agent 的人最初都会把 system prompt 当成一个字符串常量。Claude Code 不是这样。

它把 prompt 做成了6 层装配的 runtime：默认主提示 / 组装器 / 上下文注入 / 启动期附加 / 缓存边界 / 专项 prompt 家族。本课拆这套设计，重点看它的 cache 边界和优先级覆盖规则。

源码定位

文件 角色
src/constants/prompts.ts 默认 system prompt（多段）
src/utils/systemPrompt.ts buildEffectiveSystemPrompt 优先级覆盖
src/context.ts userContext / systemContext 运行时注入
src/constants/systemPromptSections.ts section cache + dynamic boundary

6 层结构

1. 默认主系统提示
   src/constants/prompts.ts

2. 有效 system prompt 组装器
   src/utils/systemPrompt.ts
   - override   ← 完全替换
   - coordinator ← coordinator mode 专用
   - agent       ← agent 定义
   - custom      ← 用户自定义
   - append      ← 永远挂尾

3. 运行时上下文注入
   src/context.ts
   - CLAUDE.md
   - currentDate
   - git status
   - cache breaker

4. 启动期附加指令入口
   src/main.tsx
   - --system-prompt
   - --append-system-prompt
   - systemPromptFile / appendSystemPromptFile
   - proactive / chrome / teammate addendum

5. Prompt 缓存与失效管理
   src/constants/systemPromptSections.ts
   - section cache
   - dynamic boundary
   - cache break

6. 专项 prompt 家族
   compact / session memory / extract memories / hooks / insights

也就是说，这里管理的不是 prompt 文本，而是：

  • 哪些 prompt 属于主循环
  • 哪些属于子任务
  • 哪些要长期缓存
  • 哪些必须逐轮重算
  • 哪些允许外部覆盖
  • 哪些可以被导出审计

getSystemPrompt 返回数组而不是字符串

export async function getSystemPrompt(
  tools: Tools,
  model: string,
  additionalWorkingDirectories?: string[],
  mcpClients?: MCPServerConnection[],
): Promise<string[]>

返回 string[]，每个元素是一个 section。这件事本身就说明设计意图：

  • system prompt 被拆成多个 section
  • 每个 section 可以单独缓存、单独插拔、单独统计 token
  • 还能在 section 级别做 cache boundary 和动态失效

主体结构：静态主干 + 动态 section

return [
  getSimpleIntroSection(outputStyleConfig),
  getSimpleSystemSection(),
  outputStyleConfig === null || outputStyleConfig.keepCodingInstructions === true
    ? getSimpleDoingTasksSection()
    : null,
  getActionsSection(),
  getUsingYourToolsSection(enabledTools),
  getSimpleToneAndStyleSection(),
  getOutputEfficiencySection(),

  ...(shouldUseGlobalCacheScope() ? [SYSTEM_PROMPT_DYNAMIC_BOUNDARY] : []),  // ⚠️ 边界

  ...resolvedDynamicSections,
].filter(s => s !== null);

注意中间那个 SYSTEM_PROMPT_DYNAMIC_BOUNDARY——它不是文字，是一个缓存边界标记：

[静态主干]                ← 这一段 cache 命中率高，长期不变
[DYNAMIC_BOUNDARY]        ← 边界标记
[动态 section]            ← 这一段每轮都可能变化

这意味着 prompt 不是按"内容"组织，而是按"变更频率"组织——把不变的放前面、把变的放后面，配合 prompt cache 命中率最大化。

静态主干：完整的"执行政策包"

举几个静态段：

  • getSimpleIntroSection()：身份声明 + 安全指令
  • getSimpleSystemSection()：基础规则（输出格式、工具拒绝处理、prompt injection 警告、hooks 反馈、上下文压缩说明）
  • getSimpleDoingTasksSection()：coding agent 的工作规则（不过度设计、读再改、避免漏洞）

这些几乎不会变，是 prompt cache 的稳定锚点。

动态 section：依赖运行态

const dynamicSections = [
  systemPromptSection('session_guidance', ...),
  systemPromptSection('memory', ...),
  systemPromptSection('ant_model_override', ...),
  systemPromptSection('env_info_simple', ...),
  systemPromptSection('language', ...),
  systemPromptSection('output_style', ...),
  DANGEROUS_uncachedSystemPromptSection('mcp_instructions', ...),  // ⚠️ 不可缓存
  systemPromptSection('scratchpad', ...),
  systemPromptSection('frc', ...),
  systemPromptSection('summarize_tool_results', ...),
];

注意 DANGEROUS_uncachedSystemPromptSection——这个名字本身就是设计文档：MCP instructions 是不可缓存的，因为它依赖当前连接的 MCP server 配置，缓存会出错。

buildEffectiveSystemPrompt：优先级覆盖

注释直接写明：

/**
 * 0. Override system prompt
 * 1. Coordinator system prompt
 * 2. Agent system prompt
 * 3. Custom system prompt
 * 4. Default system prompt
 * Plus appendSystemPrompt is always added at the end
 */

伪代码：

if overrideSystemPrompt:
    final = [overrideSystemPrompt]
else if coordinator mode:
    final = [coordinatorPrompt] + [appendPrompt?]
else:
    base =
      agentPrompt
      or customSystemPrompt
      or defaultSystemPrompt

    if proactive mode and agentPrompt exists:
        final = defaultSystemPrompt + ["# Custom Agent Instructions" + agentPrompt]
    else:
        final = [base]

    if appendSystemPrompt:
        final += [appendSystemPrompt]

最值得注意的两条：

  1. customSystemPrompt 替换默认，不是追加——用户写 --system-prompt "..." 是想完全控制，不该被默认 prompt 干扰
  2. appendSystemPrompt 永远挂尾——不管主体 prompt 来自哪里，挂尾都是允许的"安全注入点"

这个区分很重要：override（替换语义）vs append（追加语义）必须严格分开。

userContext / systemContext：运行时注入

不是每段 prompt 都写在 prompts.ts 里。src/context.ts 负责运行时注入：

// 伪代码
function buildUserContext(): string {
  const claudeMd = readClaudeMd();          // 项目根 CLAUDE.md
  const userClaudeMd = readUserClaudeMd();  // ~/.claude/CLAUDE.md
  return [userClaudeMd, claudeMd, currentDate].join('\n\n');
}

function buildSystemContext(model): string {
  return [
    `Model: ${model}`,
    `Current dir: ${cwd}`,
    `Git status: ${runGitStatus()}`,
    `Cache breaker: ${getCacheBreakerToken()}`,
  ].join('\n');
}

注意 cacheBreaker——这是个特殊的"打破缓存"令牌。当系统知道某些条件改变（settings 变了、agents 变了）但 prompt 内容没变时，需要主动让缓存失效，就在 systemContext 里塞一个变化的 token。

专项 prompt 家族

主 prompt 之外，还有一组专项 prompt 服务于后台任务：

用途 文件
Compact 总结 src/services/compact/prompt.ts
Session Memory 更新 src/services/SessionMemory/prompts.ts
Auto Memory 提取 src/services/extractMemories/prompts.ts
Hooks src/hooks/prompts.ts
Insights src/commands/insights/prompts.ts

每个专项 prompt 都独立维护，不和主 prompt 共享上下文。这避免了"主 prompt 改一行，会话压缩 prompt 突然行为变了"这种耦合事故。

可观测性：dumpPrompts

src/services/api/dumpPrompts.ts 提供了 --dump-system-prompt CLI 标志，可以把任意时刻的 system prompt 完整输出。

这是 prompt 系统的审计能力——你不需要猜模型看到了什么，可以直接 dump 出来读。

很多 Agent 项目缺这个能力，调试时只能"靠想象"。这一行 CLI 标志背后的设计哲学是：prompt 是数据，应该可观测。

Pattern 提炼

Pattern 1：prompt 是 string[]，不是 string

每段独立缓存、独立插拔、独立审计。

Pattern 2：变更频率分组 + cache boundary

[静态主干（高 cache 命中）]
[DYNAMIC_BOUNDARY]
[动态 section（每轮可能变）]

让 cache 命中率最大化。

Pattern 3：Override / Append 严格分开

不要让用户写 --system-prompt 时被默认 prompt 干扰。

Pattern 4：cacheBreaker 主动打破缓存

当语义变化但 token 不变时，主动加变化的 token 让缓存失效。

Pattern 5：dump-system-prompt 是必备能力

prompt 是数据，不是黑盒。任何 Agent 项目都应该提供 dump 入口。

Pattern 6：专项 prompt 不共享上下文

compact / memory extraction / hooks 都是独立 prompt，避免耦合事故。

自己实现：50 行的分段 prompt 管理器

// my-prompt-manager.ts
import * as fs from 'fs/promises';

type Section =
  | { type: 'static'; key: string; render: () => string }
  | { type: 'dynamic'; key: string; render: () => Promise<string> }
  | { type: 'cache_boundary' };

const DEFAULT_SECTIONS: Section[] = [
  { type: 'static', key: 'identity', render: () => 'You are a helpful assistant.' },
  { type: 'static', key: 'rules', render: () => 'Follow safety guidelines.' },
  { type: 'static', key: 'tools', render: () => 'You have Read/Write/Bash tools.' },
  { type: 'cache_boundary' },
  { type: 'dynamic', key: 'env', render: async () => `cwd: ${process.cwd()}\ndate: ${new Date().toISOString()}` },
  { type: 'dynamic', key: 'memory', render: async () => {
    try { return await fs.readFile('CLAUDE.md', 'utf-8'); } catch { return ''; }
  }},
];

export async function buildEffectivePrompt(opts: {
  override?: string;       // 完全替换
  custom?: string;         // 替换默认
  agent?: string;          // agent 专属
  append?: string;         // 永远挂尾
  proactive?: boolean;
}): Promise<string[]> {
  // 0. override：完全替换
  if (opts.override) return [opts.override];

  // 1. 选 base
  let base: string[];
  if (opts.proactive && opts.agent) {
    const defaults = await renderSections(DEFAULT_SECTIONS);
    base = [...defaults, `# Custom Agent Instructions\n\n${opts.agent}`];
  } else if (opts.agent) {
    base = [opts.agent];
  } else if (opts.custom) {
    base = [opts.custom];
  } else {
    base = await renderSections(DEFAULT_SECTIONS);
  }

  // 4. append 总是挂尾
  if (opts.append) base.push(opts.append);
  return base;
}

async function renderSections(sections: Section[]): Promise<string[]> {
  const out: string[] = [];
  for (const s of sections) {
    if (s.type === 'cache_boundary') {
      out.push('---CACHE_BOUNDARY---');   // 给上层知道缓存边界
    } else if (s.type === 'static') {
      out.push(s.render());
    } else {
      out.push(await s.render());
    }
  }
  return out;
}

// dumpSystemPrompt: 调试入口
export async function dumpSystemPrompt(opts: Parameters<typeof buildEffectivePrompt>[0]): Promise<void> {
  const sections = await buildEffectivePrompt(opts);
  console.log(sections.join('\n\n'));
}

50 行就有了 Claude Code prompt 系统的核心 pattern：分段、cache 边界、Override/Append 分离、dump 入口。

本课小结

  • prompt 是 string[] 而不是 string，让 cache / 审计 / 测试更可控
  • 静态主干在前、动态 section 在后，中间用 DYNAMIC_BOUNDARY 划线
  • buildEffectiveSystemPrompt 严格区分 override / coordinator / agent / custom / append 5 类语义
  • 专项 prompt（compact / memory）独立维护，不和主 prompt 耦合
  • --dump-system-prompt 让 prompt 可观测，调试时不必靠想象
