第2课：多轮对话

学习目标

  1. 理解 LLM API 是无状态的 —— 每次必须发送完整历史
  2. 用 OpenAI.ChatCompletionMessageParam[] 正确管理对话历史
  3. 掌握 system prompt 的设计技巧
  4. 封装可复用的 ChatSession 类

核心概念：无状态 API

这是理解所有 AI 框架的关键。LLM API 不记得上一次请求：

第1次请求：[user: "我叫 Alice"]
第2次请求：[user: "我叫什么名字？"]  ← 模型不知道 Alice 是谁！

正确做法：每次请求都把完整对话历史一起发送：

第2次请求：[
  user: "我叫 Alice",
  assistant: "你好，Alice！",
  user: "我叫什么名字？"    ← 现在模型能回答了
]

LangChain、Vercel AI SDK 等框架帮你管理历史，但底层都是这个逻辑。

消息历史的规则

import OpenAI from "openai";

// ✅ 正确：system / user / assistant 交替
const messages: OpenAI.ChatCompletionMessageParam[] = [
  { role: "system",    content: "你是一个 TypeScript 助手" },
  { role: "user",      content: "你好" },
  { role: "assistant", content: "你好！有什么能帮你？" },
  { role: "user",      content: "帮我写段代码" },
];

// ❌ 错误：连续两条 user（多数 provider 会报错或合并）
const bad = [
  { role: "user", content: "你好" },
  { role: "user", content: "在吗？" },
];

System Prompt

system 角色是对话之外的"元指令"，用来设定模型的角色和行为。在 OpenAI 兼容协议下它是 messages 数组的第一条：

import { client, model } from "../shared/llmClient";

const response = await client.chat.completions.create({
  model,
  max_tokens: 1024,
  messages: [
    { role: "system", content: "你是一个专注于 TypeScript 的技术助手。回答要简洁，必要时提供代码示例。" },
    ...history,
  ],
});

好的 system prompt 应该包含：

  • 角色定位（你是谁）
  • 行为风格（如何回答）
  • 约束条件（不做什么）

封装 ChatSession 类

import OpenAI from "openai";
import { client, model } from "../shared/llmClient";

type Msg = OpenAI.ChatCompletionMessageParam;

class ChatSession {
  private history: Msg[] = [];

  constructor(private systemPrompt: string) {
    // system 消息只放一次，永远是第一条
    this.history.push({ role: "system", content: systemPrompt });
  }

  async send(userMessage: string): Promise<string> {
    // 1. 把用户消息追加到历史
    this.history.push({ role: "user", content: userMessage });

    // 2. 发送完整历史（system + 所有轮次）
    const response = await client.chat.completions.create({
      model,
      max_tokens: 1024,
      messages: this.history,  // ← 包含 system + 所有历史
    });

    const assistantText = response.choices[0].message.content ?? "";

    // 3. 把助手回复也追加到历史（下次请求必需！）
    this.history.push({ role: "assistant", content: assistantText });

    return assistantText;
  }

  get turnCount() {
    // history[0] 是 system，所以减 1 后再除 2
    return (this.history.length - 1) / 2;
  }
}

使用示例

const session = new ChatSession(
  "你是一个 TypeScript 专家助手，回答简洁直接。"
);

const r1 = await session.send("什么是类型守卫？");
console.log("第1轮:", r1);

const r2 = await session.send("能给个具体例子吗？");
console.log("第2轮:", r2); // 模型记得上文讨论的是类型守卫

console.log(`已对话 ${session.turnCount} 轮`);

Token 消耗增长问题

每轮对话，发送的 token 数都会增加：

第1轮：发送 50 tokens
第2轮：发送 150 tokens（含第1轮历史）
第3轮：发送 300 tokens（含前两轮历史）
...
第N轮：发送 N * 平均长度 tokens

解决方案（第7课会详细讲）：

  • 滑动窗口：只保留最近 N 条
  • 摘要压缩：把旧历史压缩成摘要
  • Prompt Caching：DeepSeek / Kimi 自动缓存重复 prefix；Anthropic 用 cache_control 显式控制
  • Compaction：超长对话自动摘要

小结

概念 要点
无状态 API 每次请求必须携带完整历史
消息顺序 user / assistant 严格交替，system 放在最前
system 角色 设定角色和行为，整个对话只放一次
token 增长 随对话轮数线性增加，需要管理策略
