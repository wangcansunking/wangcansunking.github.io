第2课：启动路径剖析

介绍

claude --version 应该多快返回？理论上越快越好——但很多 CLI 工具会先把整个应用初始化完，才发现"哦，用户只想看版本号"。

Claude Code 这件事做得很优雅。本课拆解它的启动路径，看 cli.tsx → main.tsx → init.ts + setup.ts 这条链上每个角色到底干了什么。

源码定位

角色 文件
入口分流器 src/entrypoints/cli.tsx
主启动器 src/main.tsx
早期初始化 src/entrypoints/init.ts
运行环境初始化 src/setup.ts
REPL 启动 src/replLauncher.tsx

启动链全景

node bin/claude.mjs <args>
       │
       ▼
src/entrypoints/cli.tsx          ← 入口分流器（轻量）
       │
       ├─ argv 命中 --version         → console.log + exit
       ├─ argv 命中 --dump-system-prompt → 输出 prompt + exit
       ├─ argv 命中 remote-control     → runRemoteControl()
       ├─ argv 命中 daemon / bg / runner → runDaemonOrBackground()
       │
       └─ 兜底：动态 import('./main.tsx')
                    │
                    ▼
            src/main.tsx          ← 编排中心（重）
                    │
                    ├─ init(argv)                   ← entrypoints/init.ts
                    │   └─ trust 前的安全初始化
                    │
                    ├─ 解析 CLI 参数
                    ├─ 选择执行路径（headless / bridge / remote / REPL）
                    │
                    ├─ fetchBootstrapData()
                    ├─ getMcpToolsCommandsAndResources()
                    ├─ getTools(permissionContext)
                    ├─ initBundledSkills()
                    ├─ getAgentDefinitionsWithOverrides()
                    │
                    ├─ initializeTelemetryAfterTrust()  ← trust 之后再启用
                    ├─ settingsChangeDetector.start()
                    │
                    └─ launchRepl(...)

cli.tsx：快路径分流的工程价值

cli.tsx 是入口分流器，不是完整应用。它的职责是：识别快路径，提前退出，避免加载完整应用。

伪代码：

// src/entrypoints/cli.tsx
async function main() {
  const argv = parseArgs(process.argv);

  // 快路径 —— 命中即执行并退出
  if (argv['--version']) {
    console.log(version);
    process.exit(0);
  }
  if (argv['--dump-system-prompt']) {
    await dumpSystemPrompt();
    process.exit(0);
  }
  if (argv['remote-control']) {
    return runRemoteControl(argv);
  }
  if (argv['daemon'] || argv['bg'] || argv['runner']) {
    return runDaemonOrBackground(argv);
  }

  // 兜底：进入完整主启动器
  await import('./main.tsx').then(m => m.main(argv));
}

关键技巧：使用动态 import() 而非 import 语句。这意味着：

  • --version 命中时，main.tsx 这条链上的所有依赖（React、Ink、MCP Client、80+ 模块）根本不会加载
  • TTY 启动时间从1.5s 降到0.05s

这是一个非常普遍但又经常被忽视的工程模式：给 CLI 加快路径，靠动态 import 拆模块加载边界。

main.tsx：编排中心

main.tsx 才是真正的总控入口。从它的 import 列表（约 80 个模块）就能看出职责范围：

// src/main.tsx —— 关键 import 反映职责
import { init, initializeTelemetryAfterTrust } from './entrypoints/init.js';
import { launchRepl } from './replLauncher.js';
import { fetchBootstrapData } from './services/api/bootstrap.js';
import { getMcpToolsCommandsAndResources } from './services/mcp/client.js';
import { getTools } from './tools.js';
import { getAgentDefinitionsWithOverrides } from './tools/AgentTool/loadAgentsDir.js';
import { initBundledSkills } from './skills/bundled/index.js';
import { showSetupScreens, exitWithError } from './interactiveHelpers.js';
import { settingsChangeDetector } from './utils/settings/changeDetector.js';
// ... 还有约 80 个 import

主流程伪代码：

export async function main(argv: ParsedArgs) {
  // 1. 早期初始化（不依赖 trust）
  await init(argv);

  // 2. 解析 CLI 参数 → 决定 permission mode、模型、工具等
  const permissionMode = initialPermissionModeFromCLI(argv);
  const model = resolveModel(argv);

  // 3. 条件分支：选择执行路径
  if (argv['--print'] || argv['--sdk']) {
    return runHeadless(argv, model, permissionMode);
  }
  if (argv['bridge']) return runBridge(argv);
  if (argv['remote']) return runRemote(argv);

  // 4. 默认路径：完整运行时初始化
  const bootstrap = await fetchBootstrapData();
  const mcpTools  = await getMcpToolsCommandsAndResources();
  const tools     = getTools(permissionContext);
  const skills    = initBundledSkills();
  const agents    = getAgentDefinitionsWithOverrides();

  // 5. trust 后再启用 telemetry
  await initializeTelemetryAfterTrust();

  // 6. 配置文件热更新监听
  settingsChangeDetector.start();

  // 7. 进入 REPL
  await launchRepl(root, appProps, replProps, renderAndRun);
}

注意几件事：

  1. headless / bridge / remote 都在 main.tsx 里分流，但都共享相同的"装配阶段"（bootstrap / mcp / tools / skills / agents）
  2. 能力装配是顺序的——这里没有 Promise.all，因为某些后续装配依赖前面的结果（比如 agent 定义可能引用 MCP server）
  3. telemetry 开启时机受 trust 控制——这是双阶段启动的核心

init.ts + setup.ts：双阶段初始化

项目把初始化分成逻辑独立的两个文件，职责严格区分：

// src/entrypoints/init.ts —— 逻辑初始化（trust 前）
export async function init(argv) {
  applySafeEnvironmentVariables();    // 只应用安全的 env var
  initializeCertificates();           // 证书与 HTTPS 代理
  initializeHttpAgent();              // HTTP agent 配置
  initTelemetrySkeleton();            // 注册 telemetry sink，但不发事件
  // ⚠️ 不调 applyFullEnvironmentVariables() / 不 attachAnalyticsSink()
}

export async function initializeTelemetryAfterTrust() {
  applyFullEnvironmentVariables();    // trust 通过后才应用全部 env var
  attachAnalyticsSink();              // 开始处理 telemetry 事件队列
}

// src/setup.ts —— 运行环境初始化（trust 之后）
export async function setup(argv, permissionContext) {
  setCwd(resolvedWorkingDir);         // 设置工作目录
  startHooksWatcher();                // 监听 hooks 配置变化
  initWorktreeSnapshot();             // tmux/worktree 快照
  initSessionMemory();                // session memory 系统
  startTeamMemoryWatcher();           // team memory 文件监听
}

为什么要这样切？

Trust 建立前只应用安全的环境变量，防止"配置文件 / includes 本身是攻击面"的风险。

举例：用户克隆了一个恶意 repo，repo 的 .claude/settings.json 里塞了 env: { CLAUDE_API_KEY: '<exfiltrate>' }。如果不分阶段，这个 env 会在用户被询问"是否信任此目录？"之前就被应用——攻击成立。

这套思路适用于一切"接受用户提供配置"的工具：先做安全的部分，等用户确认信任后再加载危险的部分。

Pattern 提炼

Pattern 1：动态 import 拆解启动成本

// 不要这样：
import { main } from './main.tsx';
if (!isFastPath) main(argv);

// 这样：
if (!isFastPath) {
  await import('./main.tsx').then(m => m.main(argv));
}

这一行差别，决定了你的 CLI 从 1500ms 启动还是 50ms 启动。

Pattern 2：init / setup / trust 三段式

init     ←—— 安全的、不依赖任何用户输入的初始化
trust    ←—— 用户决策点
setup    ←—— 完整环境初始化

凡是接收用户配置的工具，都建议把"加载用户配置之前能做的事"和"加载用户配置之后才做的事"拆成两组函数。

Pattern 3：装配与执行分离

main.tsx 用一组 getX() / initX() 做能力装配，再把装配结果传给 launchRepl(...)。装配阶段没有副作用，REPL 阶段才开始消耗资源。这种"build-then-run"结构有两个好处：

  1. 单测可以替换装配函数
  2. headless / bridge / remote 等不同形态可以共用装配逻辑、只换执行函数

自己实现：双阶段启动的最小骨架

// my-cli.ts

// ===== 入口分流器 =====
async function cli() {
  const argv = process.argv.slice(2);
  if (argv.includes('--version')) {
    console.log('0.1.0');
    return;
  }
  // 动态 import：版本号路径不会加载下面这一堆
  const { main } = await import('./main.js');
  await main(argv);
}

// ===== 双阶段初始化 =====
let trusted = false;

async function initBeforeTrust() {
  // 只做安全的事：证书、HTTP agent、telemetry skeleton
  applySafeEnvVars();
  initTelemetrySkeleton();
}

async function initAfterTrust() {
  applyFullEnvVars();
  attachAnalyticsSink();
}

// ===== 编排中心 =====
async function main(argv: string[]) {
  await initBeforeTrust();

  const userTrust = await confirmTrust(process.cwd());
  trusted = userTrust;
  if (!trusted) {
    console.warn('Untrusted directory; running in safe mode');
  }
  if (trusted) await initAfterTrust();

  // 装配
  const tools = await getTools();
  const mcp = trusted ? await loadMcp() : [];

  // 执行
  await runRepl({ tools: [...tools, ...mcp] });
}

cli();

本课小结

  • cli.tsx 用动态 import 把启动成本切成两段，给快路径让路
  • main.tsx 是真正的编排中心，负责装配能力 + 选择执行路径
  • init / setup 双阶段初始化是处理"用户配置即攻击面"的工程答案
  • 装配阶段无副作用、执行阶段才开消耗，让 headless / REPL / bridge 复用同一套装配
