第14课（番外）：隐藏命令、Feature Flag 与彩蛋

介绍

读过前 13 课，你已经摸清了 Claude Code 的"主干"。本课转向"隐藏角落"——那些不显眼但能看出工程文化的细节：

  1. INTERNAL_ONLY_COMMANDS：内部版与外部版的编译分流
  2. 89 个 feature(...) 编译期开关
  3. 四类隐藏命令（内部专属 / 隐藏真命令 / stub / 品牌彩蛋）
  4. 负面情绪关键词检测的真实用途

源码定位

文件 角色
src/commands.ts 命令注册 + INTERNAL_ONLY_COMMANDS
src/types/command.ts isHidden 语义定义
src/utils/undercover.ts 内部身份判断
src/utils/userPromptKeywords.ts 负面词正则
src/utils/processUserInput/processTextPrompt.ts 把负面词标签写入埋点

"隐藏能力"的三层机制

源码中的能力
    |
    +-- 编译期分流
    |      |
    |      +-- USER_TYPE === ant
    |      |      → 注入 internal only commands
    |      |
    |      +-- feature('...')
    |             → 决定是否把命令/模块编进产物
    |
    +-- 运行期分流
           |
           +-- isEnabled()
           |      → 依赖账号类型、平台、实验 gate、环境变量
           |
           +-- isHidden
                  → 即使存在，也不出现在 help / typeahead

INTERNAL_ONLY_COMMANDS：内部版独享

commands.ts 的注释直接写："Commands that get eliminated from the external build"：

const INTERNAL_ONLY_COMMANDS = [
  'backfillSessions', 'breakCache', 'bughunter',
  'commit', 'commitPushPr', 'ctx_viz',
  'goodClaude', 'issue', 'initVerifiers',
  'mockLimits', 'bridgeKick', 'version',
  'resetLimits', 'onboarding', 'share',
  'summary', 'teleport', 'antTrace',
  'perfIssue', 'env', 'oauthRefresh',
  'debugToolCall', 'agentsPlatform', 'autofixPr',
];

这些命令外部构建会被裁掉——但代码就在仓库里，作为一等公民被维护。

意义：

  • 内部团队有自己的 commit / issue / debug 流程
  • 但外部用户拿不到（构建剪除）
  • 不是"残留代码"，是显式分流

89 个 feature flag

仓库里有 89 个 feature(...) 调用——典型用法：

const tools = [
  BashTool,
  ...(feature('LSP_TOOL') ? [LSPTool] : []),
  ...(feature('VOICE_MODE') ? [VoiceTool] : []),
  ...(feature('BUDDY') ? [BuddyTool] : []),
];

feature() 是编译期判定——通过 bun:bundle 把不启用的代码 tree-shake 掉。意味着：

  • 内部测试版可以同时存在多个未发布特性
  • 外部产物只包含开关启用的代码
  • 单一代码库，多种 build profile

这是处理"实验性能力 / AB 测试 / 多版本发布"的标准答案。

隐藏命令的四种类型

类型 1：内部专属命令

如上 INTERNAL_ONLY_COMMANDS——外部构建直接剪除。

类型 2：隐藏但仍是真命令

// src/commands/heapdump/index.ts
export default {
  name: 'heapdump',
  description: 'Dump the JS heap to ~/Desktop',
  isHidden: true,    // 不在 help/typeahead 出现，但能用
};

典型例子：

命令 用途
heapdump 调试用：dump JS heap
rate-limit-options 内部限流配置
output-style 已废弃但保留兼容入口
thinkback-play 由 thinkback skill 自动调用

不出现在 help 列表，但功能完整、能调用。这是"调试入口"或"过渡兼容"。

类型 3：纯 stub 占位

// src/commands/good-claude/index.js
export default { isEnabled: () => false, isHidden: true, name: 'stub' };

很多目录只有这一行：

  • good-claude/
  • share/
  • summary/
  • onboarding/
  • bughunter/
  • oauth-refresh/
  • ...

含义：外部源码快照里不提供真实实现。要么在内部仓库、要么被构建流程替换、要么已下线但接口位还在。

类型 4：品牌人格化彩蛋

像 buddy、Clawd、stickers、passes 等——产品化得很完整的"性格层"功能，部分是隐藏入口。

isHidden vs isEnabled vs feature() 的语义区分

type Command = {
  name: string;
  isHidden?: boolean;          // 是否在 help/typeahead 中隐藏
  isEnabled?: () => boolean;   // 当前是否可用（运行期）
};

// feature() = 编译期开关
const cmd = feature('NEW_THING') ? newCommand : null;

三层语义清晰：

层 控制时机 失败结果
feature() 编译期 代码根本不进产物
isEnabled() 运行期 命令存在但不能调用
isHidden 渲染期 命令可调用但 UI 不展示

这种分层设计的好处：可以做"对部分用户隐藏但保留能力"、"在某些环境下完全禁用"、"在新版本里完全去掉"——三种粒度独立。

负面词埋点：不是审查器，是产品信号

src/utils/userPromptKeywords.ts：

export function matchesNegativeKeyword(input: string): boolean {
  const lowerInput = input.toLowerCase();
  const negativePattern = /\b(wtf|wth|ffs|omfg|shit(ty|tiest)?|dumbass|horrible|awful|piss(ed|ing)? off|piece of (shit|crap|junk)|what the (fuck|hell)|fucking? (broken|useless|terrible|awful|horrible)|fuck you|screw (this|you)|so frustrating|this sucks|damn it)\b/;
  return negativePattern.test(lowerInput);
}

第一眼看像内容审查器。实际不是。

调用点：

// src/utils/processUserInput/processTextPrompt.ts
const isNegative = matchesNegativeKeyword(userPromptText);
const isKeepGoing = matchesKeepGoingKeyword(userPromptText);
logEvent('tengu_input_prompt', {
  is_negative: isNegative,
  is_keep_going: isKeepGoing,
});

它的唯一用途是：在用户输入进入系统时，给输入打"负面情绪标签"，写进 analytics 事件 tengu_input_prompt。

它不会：

  • 改写 prompt
  • 阻止 prompt 发送
  • 降权模型回答
  • 触发权限拒绝
  • 弹用户界面

它只是个布尔判定器。配合 matchesKeepGoingKeyword（"continue"、"keep going"），形成一对"用户情绪信号"。

下游用途：

  • 产品分析"用户什么情况下烦躁"
  • 在某种情况下触发"反馈问卷"
  • 决定是否提示用户"分享 transcript 给我们排查"

这是产品 telemetry + frustration sensing 的一部分，不是文本处理技巧。

Pattern 提炼

Pattern 1：编译期 / 运行期 / 渲染期三层开关

feature()    ← 编译时不进产物
isEnabled()  ← 运行时不能调
isHidden     ← UI 上不展示

不要把三件事用一个 if 解决。

Pattern 2：同一个仓库支持多个产品形态

通过 INTERNAL_ONLY_COMMANDS + feature() + USER_TYPE，同一份代码可以构建出"内部版"、"外部版"、"实验版"等多种产物。这比"维护多个分支"省力得多。

Pattern 3：stub 是治理手段，不是 dead code

保留命令名 + 接线位，让"外部缺少能力"对齐"内部存在能力"——升级 / 替换时不用改命令命名空间。

Pattern 4：埋点不只采"成功事件"

logEvent('tengu_input_prompt', { is_negative, is_keep_going });

挫败感、情绪信号也是产品数据。

Pattern 5：正则比 LLM 调用便宜得多

判断"用户是不是在抱怨"用一行正则——不用每条 prompt 都调一次模型分类。把决策从模型搬到代码上，省钱、低延迟、可审计。

自己实现：30 行的三层开关

// my-feature-flags.ts

// 1. 编译期：通过 process.env 在 bundle 时替换
export const FEATURES = {
  LSP_TOOL:    process.env.NEXT_FEATURE_LSP_TOOL === '1',
  VOICE_MODE:  process.env.NEXT_FEATURE_VOICE_MODE === '1',
};

// 2. 运行期：基于环境的 isEnabled
type Command = {
  name: string;
  isEnabled?: () => boolean;
  isHidden?: boolean;
};

const HeapDumpCommand: Command = {
  name: 'heapdump',
  isEnabled: () => process.env.DEBUG === '1',  // 只在 DEBUG 环境可用
  isHidden: true,                              // 永不在 help 出现
};

// 3. 装配
const COMMANDS_BUILDTIME: Command[] = [
  HeapDumpCommand,
  ...(FEATURES.LSP_TOOL  ? [LSPCommand] : []),
  ...(FEATURES.VOICE_MODE ? [VoiceCommand] : []),
];

export function getAvailableCommands(): Command[] {
  return COMMANDS_BUILDTIME.filter(c => c.isEnabled?.() ?? true);
}

export function getVisibleCommands(): Command[] {
  return getAvailableCommands().filter(c => !c.isHidden);  // 给 help 用
}

30 行就抓住了三层开关的核心 pattern。

番外课小结

  • INTERNAL_ONLY_COMMANDS + 89 个 feature flag 让单一代码库支持多产物
  • 隐藏命令分四类：内部独享 / 隐藏真命令 / stub / 彩蛋
  • 三层开关：feature() 编译时 / isEnabled() 运行时 / isHidden 渲染时
  • 负面词正则不是审查器，是 产品 telemetry——情绪信号也是数据
  • 决策能用代码就不用 LLM，省钱低延迟可审计

整门课程小结

读完 14 课，你应该对 Claude Code 的工程实践有了完整认识：

主题 一句话 pattern
启动 双阶段启动 + 快路径分流（动态 import）
内核 query() AsyncGenerator，多形态共用
工具 Fail-Closed 默认值 + 6 步流水线
调度 相邻合批 + contextModifier 延后应用
权限 四层结构 + 子任务专属闭包
Skill Markdown + YAML + 内嵌 Shell + 信任分层
MCP Auth Cache 防雪崩 + 白名单 + 踩坑注释
Memory 文件化 + 索引 + 硬截断 + 轻量召回
Compaction State Re-injection + 熔断器 + 自然断点
Prompt 是 runtime 不是字符串，分静态/动态/cache 边界
Multi-Agent 三层模型 + 拓扑约束 + fork 复用字节
Session append-only JSONL + 写简单读复杂
Hidden 三层开关 + 埋点信号 + stub 治理

更重要的是，你现在能在自己的 Agent 项目里主动套用这些 pattern 了。比如：

  • 给你的 Tool 接口加 Fail-Closed 默认
  • 给远端调用加 Auth Cache
  • 给长会话加 State Re-injection
  • 给 prompt 加 cache boundary

把 Claude Code 不当"产品"看，当工程范本看，它的价值能放大 10 倍。
