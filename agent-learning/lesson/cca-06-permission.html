第6课：权限模型与 Sandbox 隔离

介绍

模型说"跑 rm -rf /"的那一刻，谁来拦它？

很多人会想到 sandbox。但 Claude Code 的源码里写得非常直白：

excludedCommands is a user-facing convenience feature, not a security boundary.

The sandbox permission system (which prompts users) is the actual security control.

本课拆解 Claude Code 的权限模型与 Sandbox 隔离——它不是单点功能，而是四层结构协作。

源码定位

文件 角色
src/tools/BashTool/shouldUseSandbox.ts 决定某条命令是否进沙箱
src/utils/sandbox/sandbox-adapter.ts settings → sandbox runtime config 翻译

第 1 层：shouldUseSandbox

真实源码（src/tools/BashTool/shouldUseSandbox.ts）：

export function shouldUseSandbox(input: Partial<SandboxInput>): boolean {
  if (!SandboxManager.isSandboxingEnabled()) {
    return false;
  }

  if (
    input.dangerouslyDisableSandbox &&
    SandboxManager.areUnsandboxedCommandsAllowed()
  ) {
    return false;
  }

  if (!input.command) {
    return false;
  }

  if (containsExcludedCommand(input.command)) {
    return false;
  }

  return true;
}

四个 if，决定一条命令是否进沙箱：

  1. 全局开关关了？不进
  2. 调用方显式 dangerouslyDisableSandbox 且策略允许？不进
  3. 命令为空？不进
  4. 命中 excludedCommands（用户配置的豁免列表，如 bazel、docker）？不进

否则进沙箱。

"excludedCommands 不是安全边界"

这条注释非常重要：

// NOTE: excludedCommands is a user-facing convenience feature, not a security boundary.
// It is not a security bug to be able to bypass excludedCommands — the sandbox permission
// system (which prompts users) is the actual security control.

它在说什么？

概念 含义
便利特性（Convenience） 让用户对兼容性差的命令做豁免（比如 docker、IDE 工具）
安全边界（Security Boundary） 系统抗攻击的真正护栏

excludedCommands 属于第一类。它不是用来"防止恶意命令绕过沙箱"的——它只是"告诉系统这类命令不要自动进沙箱"。

真正的安全边界是：

  • Sandbox runtime 本身（bwrap / macOS sandbox）
  • Permission System（用户弹窗确认）

这是非常成熟的安全工程思维：清晰区分"防错误"和"防恶意"。

第 2 层：settings → sandbox config 翻译

convertToSandboxRuntimeConfig() 不是简单字段映射，而是语义重解释：

// settings.json 用户写的：
{
  "permissions": {
    "allow": ["Read(./src/**)"],
    "deny": ["Bash(rm -rf *)"]
  },
  "sandbox": {
    "network": "deny",
    "writeable_dirs": ["/tmp"]
  }
}

// 翻译后给 sandbox runtime（bwrap / macOS sandbox）：
{
  "fs": { "ro": ["/usr", "/etc"], "rw": ["/tmp", "<cwd>"] },
  "net": { "allow": [], "deny": ["*"] },
  "envOverlays": { ... }
}

为什么需要这层？因为：

  • Claude Code 的 settings 是用户友好的语义（"允许读 src/"）
  • Sandbox runtime 接受的是系统级配置（具体的目录路径、syscalls）
  • 翻译层让用户不用学 bwrap / seatbelt 的细节

第 3 层：bashPermissions

这一层处理"沙箱里命令能不能直接放行"的判断。它把以下三类规则揉在一起：

// 伪代码
function checkSandboxAutoAllow(command: string, settings) {
  // a) 显式 deny：永远拦
  if (matchesDeny(command, settings.permissions.deny)) return 'deny';

  // b) 显式 allow：直接放
  if (matchesAllow(command, settings.permissions.allow)) return 'allow';

  // c) 沙箱保护下的"安全默认"：不弹窗也能跑
  if (isInSandbox && isReadOnlyCommand(command)) return 'auto_allow';

  // d) 否则要弹窗问
  return 'ask';
}

注意 (c)：沙箱里跑的"已知只读"命令可以不弹窗——因为沙箱本身已经保护宿主机，问用户没意义。这是沙箱与权限系统的协作：沙箱让某些权限决策可以省掉。

第 4 层：Shell.ts 真正包 runtime

// src/utils/Shell.ts
if (shouldUseSandbox) {
  commandString = await SandboxManager.wrapWithSandbox(
    commandString,
    sandboxBinShell,
    undefined,
    abortSignal,
  );
}

这一行才是把命令真正变成"沙箱内命令"。

wrapWithSandbox 会根据平台调用不同 runtime：

  • Linux：bwrap (Bubblewrap)
  • macOS：sandbox-exec + seatbelt 配置
  • 不支持的平台：直接拒绝（fail-closed）

平台级权限：CanUseToolFn

除了 sandbox，每次工具调用还要走显式权限判定（CanUseToolFn）。

来看一个非常优雅的例子——Session Memory 提取子 agent 的权限：

真实源码（src/services/SessionMemory/sessionMemory.ts:460）：

export function createMemoryFileCanUseTool(memoryPath: string): CanUseToolFn {
  return async (tool: Tool, input: unknown) => {
    if (
      tool.name === FILE_EDIT_TOOL_NAME &&
      typeof input === 'object' && input !== null &&
      'file_path' in input &&
      typeof input.file_path === 'string' &&
      input.file_path === memoryPath    // ⚠️ 精确路径匹配
    ) {
      return { behavior: 'allow', updatedInput: input };
    }
    return {
      behavior: 'deny',
      message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`,
      decisionReason: { type: 'other', reason: ... },
    };
  };
}

这个权限函数的语义是：

  • 只允许 FileEditTool
  • 且 file_path 必须精确等于 memoryPath
  • 其他一切（FileRead / FileWrite / Bash / Glob / Edit 别的文件…）全部 deny

这是 Claude Code 极致 最小权限原则 的实现：Session Memory 提取 subagent 是个严密沙箱化的摘要代理，连 FileReadTool 都被拒绝。

Pattern 提炼

Pattern 1：分清"防错误"和"防恶意"

// excludedCommands 是 convenience，不是 security boundary

文档化你的安全护栏到底防什么。让"漏洞报告"能被正确分类。

Pattern 2：四层防御纵深

配置 → 翻译 → 决策 → 包装

每层职责单一。某一层错了不至于直接破防。

Pattern 3：沙箱与权限系统协作

if (isInSandbox && isReadOnlyCommand(command)) return 'auto_allow';

沙箱保护宿主机后，权限决策可以适度放宽。两套机制不是冗余，而是协作。

Pattern 4：CanUseToolFn 是闭包

function createMemoryFileCanUseTool(memoryPath: string): CanUseToolFn {
  return async (tool, input) => { ... };
}

针对特定子任务生成专属权限函数，路径硬编码到闭包里。比"权限规则字符串匹配"更可靠。

自己实现：60 行的最小权限模型

// my-permissions.ts

type Tool = { name: string; isReadOnly?: boolean };
type CanUseToolFn = (tool: Tool, input: any) =>
  Promise<{ behavior: 'allow' | 'deny' | 'ask'; message?: string }>;

// ===== 默认权限：白名单优先 + Fail-Closed =====
function createDefaultPermission(allowList: string[], denyList: string[]): CanUseToolFn {
  return async (tool, input) => {
    const sig = `${tool.name}(${JSON.stringify(input)})`;

    // a) 显式 deny 优先
    if (denyList.some(p => matchPattern(sig, p))) {
      return { behavior: 'deny', message: `denied by rule` };
    }
    // b) 显式 allow
    if (allowList.some(p => matchPattern(sig, p))) {
      return { behavior: 'allow' };
    }
    // c) 只读默认 allow（在沙箱前提下可以这么宽松）
    if (tool.isReadOnly) {
      return { behavior: 'allow' };
    }
    // d) 其他：问用户
    return { behavior: 'ask' };
  };
}

// ===== 子任务专属权限（闭包） =====
function createNarrowPermission(allowedTool: string, allowedPath: string): CanUseToolFn {
  return async (tool, input) => {
    if (tool.name === allowedTool && input?.file_path === allowedPath) {
      return { behavior: 'allow' };
    }
    return { behavior: 'deny', message: `only ${allowedTool} on ${allowedPath}` };
  };
}

function matchPattern(sig: string, pattern: string): boolean {
  // 简化：把 * 当 .*
  const re = new RegExp('^' + pattern.replace(/\*/g, '.*').replace(/\(/g, '\\(').replace(/\)/g, '\\)') + '$');
  return re.test(sig);
}

// ===== 用法 =====
const defaultPerm = createDefaultPermission(
  ['Read(*)', 'Glob(*)'],          // 全允
  ['Bash(rm -rf*)']                // 危险拒
);

const sessionMemoryPerm = createNarrowPermission('FileEdit', '/tmp/session.md');
// 这个 perm 给 session-memory 子 agent 用 → 只能编辑那一个文件

60 行就有了 Claude Code 权限模型的核心 pattern：默认规则 + 子任务专属闭包。

本课小结

  • 安全是四层结构：shouldUseSandbox → 翻译 config → bashPermissions → Shell wrap
  • excludedCommands 是 convenience，不是 security boundary——文档化清楚
  • 沙箱让权限系统可以适度放宽，两者协作而非冗余
  • 子任务权限用闭包硬编码比规则字符串更可靠（createMemoryFileCanUseTool 是教科书级例子）
