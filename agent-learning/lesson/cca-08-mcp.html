第8课：MCP 集成 — 4 种 transport 与 Auth Cache 防雪崩

介绍

MCP（Model Context Protocol）是 Anthropic 主导的开放标准，让模型统一接入外部工具和数据源。

很多项目把 MCP 当成"加个 stdio 进程就行"，Claude Code 的实现把它做到了生产级：4 种 transport、OAuth 认证、Auth Cache 防雪崩、Session 重连、IDE 工具白名单……本课逐项拆开。

源码定位

文件 角色
src/services/mcp/client.ts MCP 客户端核心
src/services/mcp/auth.ts OAuth + Step-up 检测

工具命名：buildMcpToolName

// src/services/mcp/mcpStringUtils.ts
export function buildMcpToolName(serverName: string, toolName: string): string {
  return `mcp__${serverName}__${toolName}`;
  // 例：mcp__filesystem__read_file
  //     mcp__puppeteer__screenshot
}

简单一行，但意图清晰：

  • 前缀 mcp__ 让来源一目了然
  • 中间 serverName 让命名空间不会冲突
  • 模型看到的工具列表是统一 schema，不需要区分内建还是 MCP

4 种 transport

真实源码节选（src/services/mcp/client.ts:595）：

export const connectToServer = memoize(
  async (name, serverRef, serverStats?) => {
    let transport;

    if (serverRef.type === 'sse') {
      const authProvider = new ClaudeAuthProvider(name, serverRef);
      transport = new SSEClientTransport(serverRef.url, {
        authProvider,
        fetch: wrapFetchWithTimeout(
          wrapFetchWithStepUpDetection(createFetchWithInit(), authProvider)
        ),
      });
    } else if (serverRef.type === 'ws' || serverRef.type === 'ws-ide') {
      transport = new WebSocketTransport(serverRef.url);
    } else if (serverRef.type === 'http' || serverRef.type === 'streamable-http') {
      transport = new StreamableHTTPClientTransport(serverRef.url, {
        fetch: wrapFetchWithTimeout(createClaudeAiProxyFetch(baseFetch)),
      });
    } else {
      // 默认：stdio
      transport = new StdioClientTransport({
        command: serverRef.command,
        args: serverRef.args,
        env: serverRef.env,
      });
    }

    const client = new Client({ name: 'claude-code', version: ... });
    await client.connect(transport);
    return { client, transport, ... };
  },
  getServerCacheKey
);
Transport 适用场景 特点
stdio 本地进程（最常用） 最简单，子进程通信
sse / sse-ide 远程 HTTP 长连接 OAuth + Step-up 检测
ws / ws-ide 长久 WebSocket（IDE 集成） 可推可拉，最适合 IDE
http / streamable-http HTTP + claude.ai 代理 适合无状态调用

注意 connectToServer 用 memoize 包裹——同一 server 配置只建一次连接。

wrapFetchWithTimeout：踩过坑的代码

真实源码（src/services/mcp/client.ts:492）：

export function wrapFetchWithTimeout(baseFetch: FetchLike): FetchLike {
  return async (url, init?) => {
    const method = (init?.method ?? 'GET').toUpperCase();

    // GET 不加超时 —— SSE 是长连接 GET，不能被超时切断
    if (method === 'GET') return baseFetch(url, init);

    // ⚠️ 用 setTimeout 而非 AbortSignal.timeout()
    // 原因：AbortSignal.timeout() 在 Bun 中内存泄漏（每请求 ~2.4KB 在 GC 前残留）
    const controller = new AbortController();
    const timer = setTimeout(
      c => c.abort(new DOMException('The operation timed out.', 'TimeoutError')),
      MCP_REQUEST_TIMEOUT_MS,
      controller,
    );
    timer.unref?.();   // 不阻止进程退出

    try {
      const response = await baseFetch(url, { ...init, signal: controller.signal });
      clearTimeout(timer);
      return response;
    } catch (error) {
      clearTimeout(timer);
      throw error;
    }
  };
}

两个非常工程化的细节：

  1. GET 不加超时：因为 SSE 长连接是 GET 请求，超时会把它切断
  2. 不用 AbortSignal.timeout()：原本的简洁写法，但在 Bun 运行时有内存泄漏

注释直接写明原因——这种"踩坑后留痕"的注释比"添加超时"这类无意义注释有价值得多。

MAX_MCP_DESCRIPTION_LENGTH：防 prompt 爆炸

// src/services/mcp/client.ts:218
// 注释原文：OpenAPI-generated MCP servers have been observed dumping 15-60KB
// of endpoint docs into tool.description; this caps the p95 tail without losing the intent.
const MAX_MCP_DESCRIPTION_LENGTH = 2048;

OpenAPI 自动转 MCP 的工具，描述常常长达 60KB（Swagger 的完整 docs）。如果不截断，5 个这样的工具就把 200K context window 吃掉一半。

2048 这个数字是测过 p95 的——既不破坏意图，又不让 prompt 爆。

并发连接控制

// src/services/mcp/client.ts:552
export function getMcpServerConnectionBatchSize(): number {
  return parseInt(process.env.MCP_SERVER_CONNECTION_BATCH_SIZE || '', 10) || 3;
}

function getRemoteMcpServerConnectionBatchSize(): number {
  return parseInt(process.env.MCP_REMOTE_SERVER_CONNECTION_BATCH_SIZE || '', 10) || 20;
}

通过 pMap(servers, connectToServer, { concurrency: batchSize }) 控制连接并发：

  • 本地默认 3：避免一次起太多子进程
  • 远端默认 20：网络 IO 并发价值高

Auth Cache 防雪崩 — 全文最值得学的设计

问题：100 个并发工具子调用同时遇到过期 Token，100 个都尝试刷新，目标 OAuth 服务被打爆。这就是"认证雪崩"。

解法：本地文件缓存"已知需要认证"的状态。

真实源码（src/services/mcp/client.ts:259）：

type McpAuthCacheData = Record<string, { timestamp: number }>;

// 缓存文件：~/.claude/mcp-needs-auth-cache.json
function getMcpAuthCachePath(): string { ... }

// 读取：Promise 结果 memoize，避免并发读重复 fs.readFile
function getMcpAuthCache(): Promise<McpAuthCacheData> {
  authCachePromise ??= readFile(getMcpAuthCachePath(), 'utf-8')
    .then(data => jsonParse(data) as McpAuthCacheData)
    .catch(() => ({}));
  return authCachePromise;
}

// 写入：标记某 serverId 认证失败
function setMcpAuthCacheEntry(serverId: string): void {
  // 异步写入，不阻塞调用方
}

// 校验：若服务 15 分钟内已知需要认证，直接返回 needs-auth 不重试
async function isMcpAuthCached(serverId: string): Promise<boolean> {
  const cache = await getMcpAuthCache();
  const entry = cache[serverId];
  if (!entry) return false;
  const MCP_AUTH_CACHE_TTL_MS = 15 * 60 * 1000;
  return Date.now() - entry.timestamp < MCP_AUTH_CACHE_TTL_MS;
}

效果：一个 Server 认证失败一次 → 后续 15 分钟所有对它的调用直接短路返回 needs-auth，不消耗额外的 Token / 网络请求。

这是断路器模式（Circuit Breaker）的最小化实现：

状态 触发条件 行为
Closed（正常） 默认 走完整 OAuth 流
Open（失败） 一次 401/403 15 分钟内直接返回 needs-auth

15 分钟 TTL 既给"用户主动刷新认证"留窗口，也防止"一次性失败永久烙印"。

Session 过期检测

// src/services/mcp/client.ts:193
export function isMcpSessionExpiredError(error: Error): boolean {
  const httpStatus = (error as Error & { code?: number }).code;
  if (httpStatus !== 404) return false;

  // MCP 规范：Session 过期 → HTTP 404 + JSON-RPC 错误码 -32001
  return (
    error.message.includes('"code":-32001') ||
    error.message.includes('"code": -32001')
  );
}

检测到过期 → 清缓存 (connectToServer.cache.clear()) → 重新调用 connectToServer() 建新连接。

注意它同时检查 HTTP 404 和 JSON-RPC -32001——这是协议规范，符合就是过期。

IDE 工具白名单

IDE 集成（ws-ide）能力很强——能推 LSP 诊断、文件树、活动编辑器。但不是所有 IDE 工具都该自动信任：

// src/services/mcp/client.ts:569
const ALLOWED_IDE_TOOLS = [
  'mcp__ide__getDiagnostics',
  'mcp__ide__getOpenEditorFiles',
  // ... 仅少数高权限工具通过白名单
];

function isIncludedMcpTool(tool: Tool): boolean {
  // non-IDE 工具全部允许；IDE 工具只允许白名单内
  return !tool.name.startsWith('mcp__ide__') || ALLOWED_IDE_TOOLS.includes(tool.name);
}

这是默认拒绝 + 显式允许模式（与之前 Tool Fail-Closed 一致）。新增 IDE 工具必须显式加进白名单，不能"反向通过 deny 列表防御未知"。

Pattern 提炼

Pattern 1：踩坑注释比解释代码更有价值

// AbortSignal.timeout() leaks ~2.4KB per request in Bun before GC
const controller = new AbortController();

不要写"添加超时控制"。要写"为什么不用看起来更优雅的写法"。

Pattern 2：Auth Cache = Circuit Breaker 简化版

第一次失败 → 15 分钟内同 server 短路。最小成本，最大效果。

Pattern 3：白名单 > 黑名单

return !isIdeTool || ALLOWED.includes(tool.name);

新增 IDE 能力默认拒绝，不能靠"我相信开发者会更新 deny list"。

Pattern 4：长度上限基于真实数据

const MAX_MCP_DESCRIPTION_LENGTH = 2048;  // 测过 p95

不要拍脑袋，要看真实数据 p50 / p95 / p99。

自己实现：60 行的 Auth Cache

// my-mcp-auth-cache.ts
import * as fs from 'fs/promises';
import * as path from 'path';

const CACHE_FILE = path.join(process.env.HOME!, '.my-mcp-auth-cache.json');
const TTL_MS = 15 * 60 * 1000;

let cachePromise: Promise<Record<string, { ts: number }>> | null = null;

async function readCache(): Promise<Record<string, { ts: number }>> {
  cachePromise ??= fs.readFile(CACHE_FILE, 'utf-8')
    .then(d => JSON.parse(d))
    .catch(() => ({}));
  return cachePromise;
}

async function writeCacheEntry(serverId: string): Promise<void> {
  const cache = await readCache();
  cache[serverId] = { ts: Date.now() };
  cachePromise = Promise.resolve(cache);  // 立刻更新内存版
  fs.writeFile(CACHE_FILE, JSON.stringify(cache)).catch(() => {});  // 异步落盘
}

export async function isAuthFailedRecently(serverId: string): Promise<boolean> {
  const cache = await readCache();
  const entry = cache[serverId];
  if (!entry) return false;
  return Date.now() - entry.ts < TTL_MS;
}

export async function markAuthFailed(serverId: string): Promise<void> {
  await writeCacheEntry(serverId);
}

// ===== 用法 =====
async function callMcp(serverId: string, request: any) {
  if (await isAuthFailedRecently(serverId)) {
    return { error: 'needs-auth', cached: true };
  }
  try {
    return await actuallyCallMcp(serverId, request);
  } catch (e: any) {
    if (e.status === 401 || e.status === 403) {
      await markAuthFailed(serverId);
      return { error: 'needs-auth' };
    }
    throw e;
  }
}

60 行就阻止了认证雪崩。把这个 pattern 用到任何"远端调用 + 易过期凭证"的场景，都能省下大量 token / API 调用。

本课小结

  • 4 种 transport：stdio / sse / ws / http，按场景选
  • wrapFetchWithTimeout 用 setTimeout 而非 AbortSignal.timeout()——这是踩过坑的代码
  • MAX_MCP_DESCRIPTION_LENGTH = 2048 基于真实 p95 数据
  • Auth Cache 防雪崩是全文最值得抄的 pattern，15 分钟 TTL 短路同 server 调用
  • IDE 工具用白名单而非黑名单
