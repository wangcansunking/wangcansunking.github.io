第1课：设计哲学 — 为什么 Bash is All You Need

介绍

本课解答一个根本问题：为什么 Claude Code 这么强大，却只依赖少量工具？

理解这个问题，你就理解了 AI Agent 设计的核心哲学，而不只是学会"怎么用 API"。

遇到什么问题

你想让 AI 帮你写代码、修 Bug、跑测试、提交 Git。最直觉的想法是：

给 AI 配一堆专用工具：
  WriteCodeTool → 写代码
  RunTestTool   → 跑测试
  GitTool       → 提交代码
  DocTool       → 查文档
  ...

但这条路走不通，原因有三：

  1. 维护地狱：每新增一个操作就要新写一个工具，永无止境
  2. 描述困难：工具越多，AI 越难判断该用哪个
  3. 组合爆炸：现实任务需要工具组合，专用工具无法灵活组合

有哪些方案

方案 A：每个操作一个工具（专用工具路线）

WriteFileTool、ReadFileTool、RunPythonTool、RunNodeTool、
GitAddTool、GitCommitTool、GitPushTool、NpmInstallTool...

✅ 工具语义清晰 ❌ 工具数量爆炸 ❌ 难以覆盖所有场景

方案 B：只给 Bash + 少量高频工具（极简路线）

Bash   → 执行任意 Shell 命令
Read   → 读文件（流式，不依赖 Bash）
Write  → 写文件
Edit   → 精确修改文件片段
Glob   → 文件模式匹配
Grep   → 内容搜索

✅ 覆盖 95% 以上的编程操作 ✅ AI 更容易决策 ✅ 可组合性极强 ❌ 需要精心设计安全边界

方案 C：无工具，纯文本生成

AI 只输出代码文本，用户手动复制执行

✅ 最简单 ❌ 无法闭环，用户仍然是瓶颈

怎么选择

选方案 B，理由：

Bash 是操作系统的通用接口。所有你"想给 AI 的工具"，本质上都是在 Bash 里早就有的命令：

你想要的工具 Bash 怎么做
运行测试 npm test / pytest
搜索代码 grep -r "function" src/
安装依赖 pip install requests
查看差异 git diff HEAD
重命名文件 mv old.ts new.ts
查找大文件 find . -size +1M

专门给 Read/Write/Edit 是因为文件操作频率极高，且需要流式处理，Bash 做起来啰嗦，专用工具更高效。

怎么执行

Claude Code 的工具哲学落地成三个设计原则：

原则 1：工具最小化（Minimal Tools）

核心工具 = Bash + Read + Write + Edit + Glob + Grep
（必要时加 WebSearch、WebFetch）

"够用"比"大而全"更重要。

原则 2：上下文是最重要的资产（Context is King）

工具决定 AI 能"做什么"，上下文决定 AI "做得对不对"。

普通 Chatbot：
  用户消息 → 回答（无记忆，无项目背景）

Claude Code：
  CLAUDE.md（项目规范）
  + 对话历史
  + 相关文件内容
  + 工具执行结果
  → 深度理解项目，做出正确决策

CLAUDE.md 是关键创新：放在项目根目录，记录约定和禁忌，Claude 每次都会读取：

# CLAUDE.md 示例

## 项目规范
- 使用 TypeScript strict 模式
- 测试覆盖率必须 > 80%

## 禁止操作
- 不能删除 config/ 目录
- 不能修改 .env 文件

原则 3：Human-in-the-loop（精细权限）

不是"全自动"，也不是"每步都问"，而是只在危险处停下：

读文件、写代码、跑测试  →  自动执行（低风险）
git push、rm -rf        →  必须用户确认（高风险）
修改配置文件            →  可配置（中风险）
