第12课：多 Agent 三层模型

介绍

很多人看到 AgentTool 会下结论："Claude Code 支持开 subagent"。

这个理解只摸到了 1/3。源码里实际上有三层并存的多 Agent 运行模型：

  1. 普通 subagent——主线侧链
  2. Coordinator Mode——主线变调度器
  3. Swarm Teammates——显式 team / mailbox / 共享 task list

本课逐层拆开。

源码定位

文件 角色
src/tools/AgentTool/AgentTool.tsx 三种多 Agent 的统一入口
src/tools/AgentTool/runAgent.ts 真正的 agent 执行器
src/tools/AgentTool/forkSubagent.ts fork 模式（继承 prompt 字节）
src/coordinator/coordinatorMode.ts Coordinator Mode
src/utils/swarm/spawnInProcess.ts Swarm 后端
src/utils/teammateMailbox.ts 团队消息系统
src/utils/swarm/leaderPermissionBridge.ts leader / worker 权限桥

三层模型一览

一、普通 AgentTool 子代理
   - 单个主 agent 派一个 subagent
   - 同步 / 后台 / fork 三种模式

二、Coordinator Mode
   - 主线程变成 coordinator
   - 通过 AgentTool 持续派多 worker
   - worker 结果用 task-notification 回流

三、Swarm / Teammates
   - 创建 team
   - 产生 lead + teammates
   - 支持 in-process / tmux / iTerm2 后端
   - 支持 mailbox、共享 task list、权限回流

更准确地说，Claude Code 不是"加了个 AgentTool"，而是做了一个小型 agent runtime：

  • 有 agent 身份模型
  • 有 agent 生成与调度
  • 有 agent 间通信
  • 有 leader / worker 权限桥
  • 有共享任务平面
  • 有 UI 层 task / teammate 可视化

AgentTool 是统一入口

const baseInputSchema = z.object({
  description:      z.string(),
  prompt:           z.string(),
  subagent_type:    z.string().optional(),
  model:            z.enum(['sonnet', 'opus', 'haiku']).optional(),
  run_in_background: z.boolean().optional(),
});

const multiAgentInputSchema = z.object({
  name:      z.string().optional(),
  team_name: z.string().optional(),
  mode:      permissionModeSchema().optional(),
});

字段含义：

字段 决定
subagent_type agent 类型（用哪个角色 prompt）
run_in_background 同步 vs 异步
name 让 agent 可被后续寻址
team_name 进入 swarm / teammate 路径
mode spawned teammate 继承的 plan 等模式

call() 里的核心分支：

if (teamName && name) {
  return await spawnTeammate({ ... }, toolUseContext);
} else {
  return await runAgent({ subagent_type, prompt, ... });
}

同时带 teamName + name → swarm 路径；只有 subagent_type → 普通子代理。

拓扑约束：防止 agent graph 失控

if (isTeammate() && teamName && name) {
  throw new Error('Teammates cannot spawn other teammates ...');
}

if (isInProcessTeammate() && teamName && run_in_background === true) {
  throw new Error('In-process teammates cannot spawn background agents ...');
}

作者不是简单堆功能，而是在约束多 agent 拓扑：

  • teammate 不能无限嵌套 teammate（防爆炸）
  • in-process teammate 不能再起自己的 background agent（防资源失控）

如果不加这种约束，三层就会迅速膨胀成百层，资源、debug、UI 都会失控。

第一层：普通 subagent —— 主会话的侧链

runAgent() 是真正的执行器：

runAgent
  ├─ 初始化 agent-specific MCP servers
  ├─ 构造子 agent 的 ToolUseContext
  ├─ executeSubagentStartHooks()
  ├─ recordSidechainTranscript(initialMessages, agentId)  ← 单独 sidechain 文件
  ├─ writeAgentMetadata(agentId, { ... })
  └─ for await (msg of query({ ... }))                    ← 复用主 query()

注意最后一行：subagent 不是"主线程里跑个函数"，它复用了同一个 query() 主循环。

fork subagent：复用父 prompt 字节

forkSubagent.ts 的核心注释：

When enabled:
- Omitting `subagent_type` triggers an implicit fork
- the child inherits the parent's full conversation context and system prompt
- All agent spawns run in the background

更关键的细节在于"为什么 fork 要复用父 prompt 的原始字节"：

The getSystemPrompt here is unused: the fork path passes
`override.systemPrompt` with the parent's already-rendered system prompt bytes ...
Reconstructing by re-calling getSystemPrompt() can diverge ... and bust the prompt cache

也就是说：不是重新生成，而是直接拿父会话已经渲染好的 prompt 字节。

为什么？为了 prompt cache 命中稳定性。重新调 getSystemPrompt() 哪怕只多一个空格、字段顺序不同，都会让 cache miss——多花钱、多延迟。

这表明 multi-agent 实现和 prompt/caching 基础设施是深度耦合的。设计 multi-agent 时不能只想"逻辑对就行"，还要想"会不会破坏 cache"。

第二层：Coordinator Mode —— 主线变调度器

getCoordinatorSystemPrompt() 把主线程身份直接改写：

You are Claude Code, an AI assistant that orchestrates software engineering tasks across multiple workers.

主线程不再主要写代码，而是：

  • 调度多个 worker
  • worker 负责 research / implementation / verification
  • 主线程综合结果、继续派工、对用户汇报

这是真正的 orchestrator 模式。它不是工具，是模式切换——通过环境变量 / feature flag 在启动时决定。

export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) return true;
  if (process.env.CLAUDE_COORDINATOR === '1') return true;
  return false;
}

进入这个模式后，主 prompt 就变了，AgentTool 行为也变了——更像"派单中心"而不是"通用 Agent"。

第三层：Swarm Teammates —— 轻量 agent 组织

swarm 进一步：不是"临时起几个 worker"，而是显式创建一个 team。

元素 用途
team_name 团队标识
lead agent 团队的负责 agent
teammate roster 团队成员清单
inbox / mailbox 成员间发消息
共享 task list TaskCreate / TaskStop 同步
leaderPermissionBridge teammate 工具权限回流到 leader

这一层已经接近一个轻量 agent 组织系统。

Swarm 后端可扩展

const BACKEND_REGISTRY: Record<string, TeammateBackend> = {
  'in-process': InProcessBackend,
  'tmux':       TmuxBackend,
  'iterm2':     ITerm2PaneBackend,
};

export function spawnTeammate(config: TeammateConfig): TeammateHandle {
  const Backend = BACKEND_REGISTRY[config.backendType];
  return new Backend(config).spawn();
}

支持三种后端：

  • in-process：同进程，最轻量
  • tmux：每个 teammate 一个 tmux 窗格
  • iterm2：iTerm2 标签页

这种"backend registry"模式让"在哪里运行 agent"和"怎么协调 agent"完全解耦。

Mailbox 通信

// SendMessageTool 给某个 teammate 发消息
// useInboxPoller 让 teammate 轮询自己的 inbox
// teammateMailbox 是底层存储

每个 teammate 有自己的 inbox 文件，发消息 = 往对方 inbox append；轮询 = 读自己 inbox。简单、可靠、可观察。

Leader Permission Bridge

teammate 想跑高权限工具时，权限决策回流到 leader：

teammate 调 BashTool('rm -rf x')
  -> teammate 没权限
  -> bridge 把请求发给 leader
  -> leader UI 弹窗问 lead 操作员
  -> leader 决定 allow/deny/ask
  -> bridge 把决策推回 teammate
  -> teammate 继续/拒绝

这让 swarm 既能并行干活，又能集中管控关键决策——团队治理结构的工程映射。

Pattern 提炼

Pattern 1：拓扑约束写在入口

if (isTeammate() && teamName) throw new Error('Teammates cannot spawn other teammates');

不要等运行时崩溃才发现层级失控。

Pattern 2：fork 复用 prompt 字节

不是"逻辑等价"就行，要保字节等价——cache 命中靠的是字节比对。

Pattern 3：Mode 是配置，不是工具

Coordinator Mode 不是某个 toolUse，而是启动时切换的整个 prompt + AgentTool 行为。

Pattern 4：Backend Registry

const BACKENDS = { 'in-process': X, 'tmux': Y };
spawn(name => new BACKENDS[name]());

让"运行环境"成为可替换的策略对象。

Pattern 5：权限回流（Permission Bridge）

teammate → leader 决策回流。集中管控 + 分布式执行。

自己实现：60 行的最小 swarm

// my-swarm.ts
import * as fs from 'fs/promises';
import * as path from 'path';

type Teammate = {
  name: string;
  inboxPath: string;   // mailbox 文件
};

const TEAMS = new Map<string, { lead: string; members: Teammate[] }>();

export async function createTeam(teamName: string, leadName: string) {
  const dir = `/tmp/swarm/${teamName}`;
  await fs.mkdir(dir, { recursive: true, mode: 0o700 });
  TEAMS.set(teamName, { lead: leadName, members: [] });
}

export async function spawnTeammate(teamName: string, name: string): Promise<Teammate> {
  const team = TEAMS.get(teamName);
  if (!team) throw new Error(`Unknown team: ${teamName}`);
  // 拓扑约束：teammate 不能再 spawn teammate
  if (process.env.IS_TEAMMATE === '1') throw new Error('Teammates cannot spawn other teammates');

  const inboxPath = path.join(`/tmp/swarm/${teamName}`, `${name}.inbox.jsonl`);
  await fs.writeFile(inboxPath, '', { mode: 0o600, flag: 'wx' }).catch(() => {});

  const t: Teammate = { name, inboxPath };
  team.members.push(t);
  return t;
}

export async function sendMessage(to: Teammate, from: string, text: string) {
  const entry = JSON.stringify({ from, text, ts: Date.now() }) + '\n';
  await fs.appendFile(to.inboxPath, entry);
}

export async function* pollInbox(t: Teammate) {
  let lastSize = 0;
  while (true) {
    try {
      const stat = await fs.stat(t.inboxPath);
      if (stat.size > lastSize) {
        const fd = await fs.open(t.inboxPath, 'r');
        const buf = Buffer.alloc(stat.size - lastSize);
        await fd.read(buf, 0, buf.length, lastSize);
        await fd.close();
        lastSize = stat.size;
        for (const line of buf.toString('utf-8').split('\n').filter(Boolean)) {
          yield JSON.parse(line);
        }
      }
    } catch {}
    await new Promise(r => setTimeout(r, 500));
  }
}

// ===== 用法 =====
await createTeam('alpha', 'lead-1');
const tm1 = await spawnTeammate('alpha', 'tm-A');
const tm2 = await spawnTeammate('alpha', 'tm-B');

await sendMessage(tm1, 'lead-1', '请研究 X');
await sendMessage(tm2, 'lead-1', '请实现 Y');

// teammate-A 的 loop
(async () => {
  for await (const msg of pollInbox(tm1)) {
    console.log('[tm-A] 收到:', msg);
  }
})();

60 行就有了 swarm 的核心：team 注册 + mailbox + 拓扑约束 + 轮询通信。

本课小结

  • 三层多 Agent：subagent / coordinator / swarm，AgentTool 是统一入口
  • 拓扑约束写在入口：teammate 不能嵌套 teammate
  • fork 复用父 prompt 字节——cache 命中靠字节，不是语义
  • Coordinator Mode 是模式切换，改变主线身份
  • Swarm 用 mailbox + 共享 task list + 权限桥 做轻量 agent 组织
  • Backend Registry（in-process / tmux / iterm2）让运行环境可替换
