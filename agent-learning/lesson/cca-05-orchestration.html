第5课：工具调度 — partitionToolCalls 与流式执行器

介绍

模型一次输出 5 个 tool_use 请求时，应该并发跑还是串行跑？

[ Read(a.ts), Read(b.ts), Write(c.ts), Read(d.ts), Bash(rm -rf x) ]

朴素答案是"全部并发"——但这会出错：Write 和 Bash 之间可能存在状态依赖。Claude Code 的解法是 partitionToolCalls() + StreamingToolExecutor，把工具按"并发安全性"分批，相邻安全工具合一组，遇到不安全就单独成组。

源码定位

文件 角色
src/services/tools/toolOrchestration.ts partitionToolCalls + runToolsConcurrently / runToolsSerially
src/services/tools/StreamingToolExecutor.ts 流式执行器，状态机
src/services/tools/toolExecution.ts 单个工具的 6 步流水线

partitionToolCalls：相邻合批，不安全单飞

真实源码节选（src/services/tools/toolOrchestration.ts）：

function partitionToolCalls(
  toolUseMessages: ToolUseBlock[],
  ctx: ToolUseContext,
): Batch[] {
  return toolUseMessages.reduce((acc: Batch[], toolUse) => {
    const tool = findToolByName(ctx.options.tools, toolUse.name);
    const isConcurrencySafe = Boolean(tool?.isConcurrencySafe(parsedInput.data));

    // 若上一批次也是并发安全，合入同一批
    if (isConcurrencySafe && acc[acc.length - 1]?.isConcurrencySafe) {
      acc[acc.length - 1]!.blocks.push(toolUse);
    } else {
      // 否则开新批
      acc.push({ isConcurrencySafe, blocks: [toolUse] });
    }
    return acc;
  }, []);
}

举例：模型返回 [A(Read), B(Read), C(Write), D(Read), E(Read)]：

工具 isConcurrencySafe 决策 批次
A(Read) true 起新批 Batch1: [A]
B(Read) true 上一批也是 safe，合入 Batch1: [A, B]
C(Write) false 不安全，起新批 Batch2: [C]
D(Read) true 上一批是 unsafe，起新批 Batch3: [D]
E(Read) true 上一批是 safe，合入 Batch3: [D, E]

最终：[Batch1: A,B] → [Batch2: C] → [Batch3: D,E]，三个批次依次执行，每批内部并发。

runTools：批次执行入口

export async function* runTools(
  toolUseMessages: ToolUseBlock[],
  assistantMessages: AssistantMessage[],
  canUseTool: CanUseToolFn,
  toolUseContext: ToolUseContext,
): AsyncGenerator<MessageUpdate, void> {
  let currentContext = toolUseContext;

  for (const { isConcurrencySafe, blocks } of partitionToolCalls(toolUseMessages, currentContext)) {
    if (isConcurrencySafe) {
      // 并发批次：先收集 contextModifier，批次完再按序应用
      for await (const update of runToolsConcurrently(blocks, ...)) {
        yield update;
      }
    } else {
      // 串行批次：每个工具必须等上一个完成
      for await (const update of runToolsSerially(blocks, ...)) {
        yield update;
      }
    }
  }
}

注意一个关键细节：并发批次的 contextModifier 是延后应用的。

contextModifier 延后应用是什么意思？

工具执行后，可能会修改全局上下文（比如 BashTool 改了 cwd）。如果并发批次里的两个工具都修改 context，并发应用就会冲突。

伪代码示意：

async function runToolsConcurrently(blocks, ctx) {
  // 1. 并发执行，每个工具返回 result + 可能的 contextModifier
  const outcomes = await Promise.all(
    blocks.map(b => executeTool(b, ctx))   // ctx 在并发期间不变
  );

  // 2. 全部完成后，按声明顺序串行应用 modifier
  for (const outcome of outcomes) {
    if (outcome.contextModifier) {
      ctx = outcome.contextModifier(ctx);
    }
  }
  return ctx;
}

这样做保证：

  • 并发期间，所有工具看到的 ctx 是同一个快照
  • 修改在批次结束时按顺序应用，不会出现 race
  • 即便两个工具都改 cwd，结果是确定的（按声明顺序）

这是分布式系统里 **MVCC（多版本并发控制）**思路在 Agent 工具调度上的应用。

StreamingToolExecutor：边收边跑

runTools() 默认是"等模型完整 yield 完所有 tool_use 再开跑"。但有时候模型会流式输出多个 tool_use——能不能边收边跑？

StreamingToolExecutor 就是为此而生：

// 状态机：每个 tool 的状态
type ExecState = 'queued' | 'executing' | 'completed' | 'yielded';

class StreamingToolExecutor {
  private states = new Map<string, ExecState>();

  async *consume(stream: AsyncGenerator<ToolUseBlock>) {
    for await (const toolUse of stream) {
      // schema 解析通过的并发安全工具，立刻 launch
      if (toolUse.tool.isConcurrencySafe(toolUse.input)) {
        this.launch(toolUse);   // 不等下一个就启动
      } else {
        this.queue(toolUse);    // 不安全，等独占权
      }
    }
    // ...
  }
}

具体规则：

  1. 并发安全的工具：schema 解析通过 → 立刻启动
  2. 不安全的工具：进入排队，等独占
  3. 任一工具抛错：可以 cancel 同批未完成的兄弟节点，提早回报模型

这个机制对"模型输出 10 个并行 Read"这类场景特别有用——不必等第 10 个 toolUse 解析完才启动第 1 个。

toolExecution.ts：单个工具的 6 步流水线

每个工具的 call() 真正落地前要走 6 步：

toolUse
   │
   ▼
1. Zod schema 校验          ← inputSchema.safeParse
   │ 失败：抛 schema 错给 LLM 让它重试
   ▼
2. 独立语义校验             ← validateInput()
   │ 比如检查"old/new 字符串是否相等"、目录黑名单
   ▼
3. backfill 隐式依赖        ← backfillObservableInput()
   │ 比如 expandPath() 注入到 input
   ▼
4. PreToolUse Hooks         ← 用户配置的 hook，可拦截
   │
   ▼
5. 权限检查                  ← canUseTool()
   │ 三态：allow / deny / ask
   │
   ▼
6. tool.call()              ← 真正执行
   │
   ▼
toolResult / progress

每一步都是独立的可拦截点。比如第 4 步 PreToolUse Hooks 让用户可以注入 lint / format 等自定义逻辑，不必改 Tool 源码。

Pattern 提炼

Pattern 1：相邻合批，不安全开新批

partitionToolCalls 的 reduce 模式简洁有力：

arr.reduce((acc, item) => {
  if (canMerge(item, acc[acc.length - 1])) {
    acc[acc.length - 1].push(item);
  } else {
    acc.push([item]);
  }
  return acc;
}, []);

很多"按某种条件分组"的场景都能套这个模板。

Pattern 2：并发期快照 + 批次结束串行应用

const outcomes = await Promise.all(items.map(i => exec(i, snapshot)));
for (const o of outcomes) snapshot = apply(o, snapshot);

避免 race 的优雅写法。

Pattern 3：管线代理（Pipeline）

工具执行不是直接 call，而是 6 步流水线（schema → validate → backfill → hooks → permission → call）。每一步都是切面，可以单独拦截、扩展、审计。

Pattern 4：状态机驱动的流式执行

queued → executing → completed → yielded。明确的状态转移让 cancel / abort / retry 都有据可循。

自己实现：partitionToolCalls 60 行

// my-orchestration.ts

type ToolUse = { name: string; input: any };
type Tool = { name: string; isConcurrencySafe: (input: any) => boolean; run: (input: any) => Promise<any> };
type Batch = { isConcurrencySafe: boolean; uses: ToolUse[] };

function partitionToolCalls(uses: ToolUse[], tools: Tool[]): Batch[] {
  return uses.reduce((acc: Batch[], use) => {
    const tool = tools.find(t => t.name === use.name);
    const safe = tool?.isConcurrencySafe(use.input) ?? false;

    const last = acc[acc.length - 1];
    if (safe && last?.isConcurrencySafe) {
      last.uses.push(use);
    } else {
      acc.push({ isConcurrencySafe: safe, uses: [use] });
    }
    return acc;
  }, []);
}

export async function* runTools(uses: ToolUse[], tools: Tool[]) {
  const byName = new Map(tools.map(t => [t.name, t]));

  for (const batch of partitionToolCalls(uses, tools)) {
    if (batch.isConcurrencySafe) {
      // 并发：用 Promise.all
      const results = await Promise.all(
        batch.uses.map(async u => {
          const tool = byName.get(u.name);
          if (!tool) return { use: u, error: 'unknown tool' };
          try { return { use: u, output: await tool.run(u.input) }; }
          catch (e) { return { use: u, error: String(e) }; }
        })
      );
      for (const r of results) yield r;
    } else {
      // 串行：for await
      for (const u of batch.uses) {
        const tool = byName.get(u.name);
        if (!tool) { yield { use: u, error: 'unknown tool' }; continue; }
        try { yield { use: u, output: await tool.run(u.input) }; }
        catch (e) { yield { use: u, error: String(e) }; }
      }
    }
  }
}

// ===== 用法 =====
const tools: Tool[] = [
  { name: 'Read', isConcurrencySafe: () => true, run: async (i) => `<${i.path}>` },
  { name: 'Write', isConcurrencySafe: () => false, run: async (i) => 'OK' },
];

const uses: ToolUse[] = [
  { name: 'Read', input: { path: 'a.ts' } },
  { name: 'Read', input: { path: 'b.ts' } },
  { name: 'Write', input: { path: 'c.ts', content: '...' } },
  { name: 'Read', input: { path: 'd.ts' } },
];

for await (const r of runTools(uses, tools)) console.log(r);
// 输出：[Read(a), Read(b)] 并发 → [Write(c)] 串行 → [Read(d)] 单批

60 行就有了 Claude Code 工具调度的核心骨架。

本课小结

  • partitionToolCalls：相邻并发安全工具合一批，遇到不安全就开新批
  • 并发批次内 contextModifier 延后串行应用，避免 race
  • StreamingToolExecutor 让"边收边跑"成为可能，不安全工具仍排队
  • 每个工具落地要走 6 步流水线（schema / validate / backfill / hooks / permission / call）
