第3课：Query 执行内核

介绍

如果只允许读 Claude Code 一个文件，应该读 src/query.ts。

它是整个 Agent 平台的心脏——所有运行形态（REPL / headless SDK / subagent / bridge）最后都会汇聚到这一个 query() 函数。本课拆解它为什么写成 AsyncGenerator，以及它怎么把 tool_use 闭环回流到下一轮模型调用。

源码定位

文件 角色
src/query.ts 单次 query 主循环
src/QueryEngine.ts 跨多轮会话的状态包装
src/services/api/claude.ts 实际的 API 调用
src/services/tools/toolOrchestration.ts 工具调度入口
src/utils/messages.ts normalize / parentUuid 修复

主循环骨架

// src/query.ts —— 主循环骨架（伪代码）
export async function* query(
  userMessages: Message[],
  systemPrompt: SystemPrompt,
  toolUseContext: ToolUseContext,
  deps: QueryDeps,
): AsyncGenerator<StreamEvent> {
  let messages = userMessages;

  while (true) {
    // 1. 装配上下文（memory 注入也发生在这里）
    const apiMessages = normalizeMessagesForAPI(messages);

    // 2. 调用 Claude API，流式接收
    for await (const event of deps.claudeApi.stream(apiMessages, systemPrompt)) {
      yield event;
    }

    // 3. 提取模型输出的 tool_use 列表
    const toolUseBlocks = extractToolUseBlocks(messages);
    if (toolUseBlocks.length === 0) break;     // 无工具调用 → 结束

    // 4. 执行工具（并发 / 串行由 partitionToolCalls 决定）
    const toolResults = [];
    for await (const update of runTools(toolUseBlocks, ...)) {
      yield update;                            // 实时 yield 给 UI
      toolResults.push(update.message);
    }

    // 5. 把工具结果追加到 messages → 进入下一轮
    messages = [...messages, ...toolResults];

    // 6. compact 检查、hook 执行
    await executePostSamplingHooks(messages, toolUseContext);
    if (shouldCompact(messages)) await compact(messages, toolUseContext);
  }
}

每一行都有讲究。下面逐个剖析。

设计 1：返回 AsyncGenerator<StreamEvent> 的真正价值

如果你之前写过 Agent，你的循环函数大概率是 async function run(): Promise<Result>。Claude Code 不是这样：

export async function* query(...): AsyncGenerator<StreamEvent>

返回异步生成器意味着：

  1. 流式产出：每个 token、每个工具进度、每个 hook 都即时 yield，UI 不用等整轮跑完
  2. 同一份代码喂多种消费者：
    • REPL：消费 yield 渲染 Ink 组件
    • SDK：消费 yield 转交给调用方代码
    • bridge：消费 yield 通过 WebSocket 推到远端
  3. 天然支持中断：消费者只要 break 出 for await，生成器就被 GC，副作用清晰

这是 Claude Code "统一执行内核" 这个工程亮点的技术基础——不是因为它很巧妙地抽象了什么接口，而是因为它选了 AsyncGenerator 这种数据结构。

设计 2：while(true) + break 而不是递归

while (true) {
  ...
  if (toolUseBlocks.length === 0) break;
  ...
}

你可能想过递归实现：

async function step(messages) {
  const result = await callApi(messages);
  if (hasToolUse(result)) {
    return step([...messages, ...await runTools(result)]);
  }
  return result;
}

Claude Code 没有这么做，原因：

  1. 栈深度：长任务可能跑 50+ 轮，递归会占栈
  2. 可观察性：循环写法可以在每轮开始 / 结束插日志、metric、hook
  3. 可中断：从外面控制 abortController 时，循环更容易在轮次边界优雅退出

工程上循环优于递归是 Agent 主控代码的常见选择。

设计 3：extractToolUseBlocks + appendToolResults 的解耦

注意主循环里没有"工具执行"和"消息处理"的耦合：

const toolUseBlocks = extractToolUseBlocks(messages);   // 数据提取
for await (const update of runTools(toolUseBlocks, ...)) { ... }  // 工具执行
messages = [...messages, ...toolResults];               // 数据追加

extract / run / append 三件事严格分开。这意味着你可以：

  • 测试 extractToolUseBlocks：给一个 messages 数组，断言提取出对的 tool_use
  • 替换 runTools：测试时用 mock 工具调度
  • 审计 appendToolResults：分析消息累积模式

很多 Agent 项目把这三件事写在一个大函数里，就丢失了这种可测性。

设计 4：normalizeMessagesForAPI 的清理职责

API 接受的消息格式有严格约束：

  • tool_result 必须紧跟 tool_use
  • 图片不能超过 5MB
  • 不能有空字符串内容
  • parentUuid 链不能断

但 Claude Code 内部的 messages 数组里夹了大量 UI 元素：progress 标记、本地附件元数据、富文本片段……这些 API 都不认。

normalizeMessagesForAPI() 就是这层清理器。它的真正价值不只是"剔除多余字段"，更重要的是：

transcript 可以保留 UI 需要的所有元数据，但发送给 API 时只保留 API 认的字段。

这是一个常见 pattern：两种数据视图——一种给 UI / 持久化、一种给 API。中间用 normalize 函数做投影。

设计 5：每一轮都跑 executePostSamplingHooks + shouldCompact

await executePostSamplingHooks(messages, toolUseContext);
if (shouldCompact(messages)) await compact(messages, toolUseContext);

这两件事在每一轮工具执行之后都跑：

  • executePostSamplingHooks：用户配置的 hook（比如"每次工具调用后跑一下 lint"）
  • shouldCompact：检查 token 是否快满了，要不要压缩

注意它们都在循环里，而不是只在循环开始时检查一次。原因是：长任务可能在循环中间消耗大量 token（比如一次 Read 超大文件），必须在每轮都检查，否则下一轮 API 调用就会撞墙。

QueryEngine：跨多轮的薄壳

QueryEngine.ts 是 query() 的对象包装，给 SDK 用：

// src/QueryEngine.ts 结构伪代码
export class QueryEngine {
  private state: { messages: Message[]; ... };

  async *query(userInput: string): AsyncGenerator<StreamEvent> {
    this.state.messages.push({ role: 'user', content: userInput });
    for await (const ev of query(this.state.messages, this.systemPrompt, this.ctx, this.deps)) {
      yield ev;
      this.state.messages = mergeFromEvent(this.state.messages, ev);
    }
  }
}

它做的事很简单：维护跨轮次的 messages 状态。query() 本身是无状态的（每次都要传完整 messages），QueryEngine 是给那些不想自己管状态的调用方用的。

Pattern 提炼

Pattern 1：主循环 = AsyncGenerator

async function* run(): AsyncGenerator<Event> { ... }

只要你的 Agent 主循环返回 AsyncGenerator，CLI / Web / 测试就都能复用同一份代码。

Pattern 2：while(true) + 每轮检查 hook / compact

不要在循环开始时一次性做所有检查；每个回合 boundary 都是一个完整的检查点。

Pattern 3：API 视图与 UI 视图分离

const apiMessages = normalizeMessagesForAPI(internalMessages);

让你的 transcript 可以包含一切 UI / 调试需要的元数据，发 API 时再投影成纯净版。

Pattern 4：装配-执行分离

extract / runTools / append 三件事独立。每件事都可以单独测、单独换。

自己实现：80 行的 query 主循环

// my-query.ts
import Anthropic from '@anthropic-ai/sdk';

export type StreamEvent =
  | { type: 'text'; text: string }
  | { type: 'tool_use'; name: string; input: any }
  | { type: 'tool_result'; toolUseId: string; output: string }
  | { type: 'done' };

export async function* query(opts: {
  messages: Anthropic.MessageParam[];
  system: string;
  tools: { name: string; run: (input: any) => Promise<string> }[];
}): AsyncGenerator<StreamEvent> {
  const client = new Anthropic();
  let messages = opts.messages;

  while (true) {
    // 1. API 调用
    const stream = client.messages.stream({
      model: 'claude-opus-4-7',
      max_tokens: 4096,
      system: opts.system,
      messages: normalizeMessagesForAPI(messages),
      tools: opts.tools.map(t => ({ name: t.name, description: '', input_schema: { type: 'object', properties: {} } })),
    });

    let finalMessage: Anthropic.Message | null = null;
    for await (const ev of stream) {
      if (ev.type === 'content_block_delta' && ev.delta.type === 'text_delta') {
        yield { type: 'text', text: ev.delta.text };
      }
    }
    finalMessage = await stream.finalMessage();
    messages = [...messages, { role: 'assistant', content: finalMessage.content }];

    // 2. 提取 tool_use
    const toolUses = finalMessage.content.filter((b): b is Anthropic.ToolUseBlock => b.type === 'tool_use');
    if (toolUses.length === 0) {
      yield { type: 'done' };
      break;
    }

    // 3. 执行工具
    const results: Anthropic.ToolResultBlockParam[] = [];
    for (const tu of toolUses) {
      yield { type: 'tool_use', name: tu.name, input: tu.input };
      const tool = opts.tools.find(t => t.name === tu.name);
      const output = tool ? await tool.run(tu.input) : `Unknown tool: ${tu.name}`;
      yield { type: 'tool_result', toolUseId: tu.id, output };
      results.push({ type: 'tool_result', tool_use_id: tu.id, content: output });
    }

    // 4. 追加结果，进入下一轮
    messages = [...messages, { role: 'user', content: results }];

    // 5. 简化的 hook / compact 占位
    if (estimateTokens(messages) > 100_000) {
      messages = await compactMessages(messages);
    }
  }
}

function normalizeMessagesForAPI(m: Anthropic.MessageParam[]): Anthropic.MessageParam[] {
  return m.filter(msg => msg.content != null && msg.content !== '');
}
function estimateTokens(m: any[]): number { return JSON.stringify(m).length / 4; }
async function compactMessages(m: any[]): Promise<any[]> { return m.slice(-10); }

这就是 Claude Code query.ts 的最小可运行版本。后续课程要做的，是把这里被缩成一行的 runTools / compactMessages 等等一一打开。

本课小结

  • query() 是所有运行形态共用的主循环
  • 选 AsyncGenerator<StreamEvent> 而不是 Promise<Result>，是为了让 UI / SDK / bridge 共用代码
  • while(true) + 每轮 hook / compact 检查，比递归更可观察、更可中断
  • API 视图与内部视图分离，靠 normalizeMessagesForAPI 投影
