第13课：Session 持久化与 Resume — append-only JSONL

介绍

聊到一半电脑重启了，怎么继续？

很多 Agent 项目把会话写成"内存里聊完就没"，重启即丢失。Claude Code 把它做成了append-only JSONL 事件流 + 恢复时按需修复——写入端尽量简单，恢复端负责复杂性。本课拆这个机制。

源码定位

文件 角色
src/utils/sessionStorage.ts append-only 写入 + 路径解析 + 分流
src/utils/conversationRecovery.ts 恢复流水线 + 链路修复
src/screens/ResumeConversation.tsx UI 入口
src/services/api/sessionIngress.ts 远端 ingress 副本

五层结构

一、主 transcript                  → 每个 session 一个 .jsonl
   user / assistant / attachment / system 都 append-only

二、附加元数据条目                  → summary / custom-title / tag / agent-setting / mode / worktree-state / pr-link
   写进同一文件，但单独按类型恢复

三、subagent sidechain transcript  → 每个 agent 独立 .jsonl
   用于 fork / teammate / subagent 恢复

四、远端 ingress 副本               → 主 transcript 的远端 append 链
   用于 hydrate 和跨进程恢复

五、resume 恢复流水线              → JSONL 读 → compact/snip 修复 → metadata/file/contextCollapse 恢复 → REPL 接管

把"会话状态"拆成两层：

  • 写入层尽量简单：只追加，不原地改写
  • 恢复层负责复杂性：读取时补齐、修复、重建

什么算 transcript message

真实源码（src/utils/sessionStorage.ts:139）：

export function isTranscriptMessage(entry: Entry): entry is TranscriptMessage {
  return (
    entry.type === 'user' ||
    entry.type === 'assistant' ||
    entry.type === 'attachment' ||
    entry.type === 'system'
  );
}

注释还明确说：

  • progress 不是 transcript message
  • progress 不能进入 parentUuid 主链
  • 旧版本把 progress 混进 transcript 后，恢复时会把真实对话链截断

设计目标很清楚：

  • 保留真正影响上下文重建的消息
  • 把高频 UI 状态从持久化层排除

transcript 路径联动 session 切换

// src/utils/sessionStorage.ts:202
export function getTranscriptPath(): string {
  const projectDir = getSessionProjectDir() ?? getProjectDir(getOriginalCwd());
  return join(projectDir, `${getSessionId()}.jsonl`);
}

export function getAgentTranscriptPath(agentId: AgentId): string {
  const projectDir = getSessionProjectDir() ?? getProjectDir(getOriginalCwd());
  const sessionId = getSessionId();
  const subdir = agentTranscriptSubdirs.get(agentId);
  const base = subdir
    ? join(projectDir, sessionId, 'subagents', subdir)
    : join(projectDir, sessionId, 'subagents');
  return join(base, `agent-${agentId}.jsonl`);
}

三个关键：

  1. 主 transcript 是 <projectDir>/<sessionId>.jsonl
  2. subagent transcript 单独 sidechain 文件，不和主链混写
  3. 路径解析联动 sessionId，resume / branch / switch 都能落到正确目录

不是"当前目录写聊天记录"，而是带 session 路由能力的日志存储层。

写入：批量 flush + 0o600 权限

底层写盘器：

// src/utils/sessionStorage.ts:634
private async appendToFile(filePath: string, data: string): Promise<void> {
  try {
    await fsAppendFile(filePath, data, { mode: 0o600 });
  } catch {
    await mkdir(dirname(filePath), { recursive: true, mode: 0o700 });
    await fsAppendFile(filePath, data, { mode: 0o600 });
  }
}

private async drainWriteQueue(): Promise<void> {
  for (const [filePath, queue] of this.writeQueues) {
    const batch = queue.splice(0);
    let content = '';
    for (const { entry } of batch) {
      content += jsonStringify(entry) + '\n';
    }
    await this.appendToFile(filePath, content);
  }
}

特点：

  • 进入内存队列，由 drainWriteQueue() 批量 flush
  • 文件 / 目录权限固定 0o600 / 0o700
  • 文件不存在时 catch → mkdir 重试一次

典型的日志系统设计：追加简单、崩溃恢复容易、不必每次重写整份 transcript。

appendEntry：分流器

appendEntry() 决定一条 entry 写到哪：

async function appendEntry(entry, sessionId) {
  if (当前 session 文件还没 materialize) {
    pendingEntries.push(entry); return;
  }

  if (entry 是 metadata: summary/title/tag/mode/worktree/pr-link) {
    enqueueWrite(mainSessionFile, entry); return;
  }

  if (entry.type === 'content-replacement') {
    target = entry.agentId ? agentSidechainFile : mainSessionFile;
    enqueueWrite(target, entry); return;
  }

  // 剩下的：transcript message
  target = entry.isSidechain ? agentSidechainFile : mainSessionFile;

  if (target 是 sidechain 本地文件) {
    // ⚠️ 允许写入与主链重复 UUID，保证 fork 后上下文完整
    enqueueWrite(target, entry); return;
  }

  if (uuid 之前没写过) {
    enqueueWrite(mainSessionFile, entry);
    messageSet.add(entry.uuid);
    persistToRemote(sessionId, entry);
  }
}

最巧妙的是 sidechain 允许重复 UUID：

  • 主 transcript 不能重复（否则 resume 链路有 409 冲突）
  • sidechain 必须允许重复（否则 fork 出来的子 agent 会丢继承的父消息）

源码里写得很直白：

// src/utils/sessionStorage.ts:1212
const isAgentSidechain = entry.isSidechain && entry.agentId !== undefined;
const targetFile = isAgentSidechain ? getAgentTranscriptPath(...) : sessionFile;

const isNewUuid = !messageSet.has(entry.uuid);
if (isAgentSidechain || isNewUuid) {
  void this.enqueueWrite(targetFile, entry);
  if (!isAgentSidechain) {
    messageSet.add(entry.uuid);
    if (isTranscriptMessage(entry)) {
      // 同步到远端 ingress
    }
  }
}

恢复流水线

读 JSONL 不是简单 for of：

read JSONL
  -> 跳过 progress 等非 transcript message
  -> 修复链路：
     - compact 边界还原
     - snip / progress 留下的孤立 tool_result 处理
     - parallel tool result 重新关联
  -> 恢复 metadata（summary / title / mode / worktree）
  -> 恢复 fileHistory（哪些文件还在 active）
  -> 恢复 contextCollapse（compact 后哪些被折叠）
  -> 恢复 worktree / agent 状态
  -> 把还原好的 message 数组交给 REPL

最关键的一句话：写入端尽量简单，恢复端负责复杂性。这是 append-only 日志系统的通用思路——日志是单向的、不可变的，所有"修复 / 重组"在读取时做。

为什么这样？因为：

  • 写入端复杂 → 崩溃时容易留半截烂帐
  • 读取端复杂 → 崩溃只是少了几行 entry，下次 resume 再补救

举一个具体场景：fork subagent 恢复

主线程派出 subagent A，A 又调了 5 个工具，然后用户 Ctrl+C 退出。

主 transcript:                   subagent sidechain:
[user: "do x"]                  [继承的父消息: user "do x"]   ← 重复 UUID 允许
[assistant: "I'll fork"]        [继承的父消息: assistant ...]
[tool_use: AgentTool(...)]      [tool_use: Read(a.ts)]
                                [tool_result: ...]
                                [tool_use: Edit(a.ts)]
                                [tool_result: ...]

resume 时：

  1. 读主 transcript → 知道有个 subagent A
  2. 读 sidechain → 拿到 A 的完整对话 + 工具调用历史
  3. 链路修复：把孤立 tool_use（用户中断时还没收到 result）补完整
  4. 恢复 A 的 ToolUseContext，让它能继续执行

如果不允许 sidechain 重复 UUID，第 3 步就会因为找不到父消息而出错。

Pattern 提炼

Pattern 1：append-only JSONL 是黄金存储格式

  • 简单
  • 崩溃容忍
  • 每行独立可解析
  • 可流式读

任何"事件流 / 操作日志"都该首选 JSONL。

Pattern 2：写入端简单，恢复端复杂

write:  append entry → 完事
read:   修复 + 重组 + 还原状态

这是分布式系统里 LSM-Tree、日志结构文件系统的核心思想。

Pattern 3：metadata 和 message 同表存

不为 metadata 单建一张表。同 JSONL 文件 + 类型字段过滤。读取时按类型分流。

Pattern 4：sidechain 允许重复 UUID

主链不允许，sidechain 必须允许。两个文件，两套约束。

Pattern 5：什么进 transcript 决定了恢复能力

function isTranscriptMessage(e) {
  return e.type === 'user' || ...;
}

什么算"该持久化的 message"必须明确定义，否则恢复链路会被乱七八糟的 UI 事件污染。

Pattern 6：0o600 文件权限是默认值

会话内容是敏感数据。

自己实现：80 行的 append-only session

// my-session.ts
import * as fs from 'fs/promises';
import * as path from 'path';

type Entry =
  | { type: 'user'; content: string; uuid: string; ts: number }
  | { type: 'assistant'; content: any; uuid: string; ts: number }
  | { type: 'tool_use'; name: string; input: any; uuid: string; ts: number }
  | { type: 'tool_result'; toolUseId: string; output: string; uuid: string; ts: number }
  | { type: 'metadata'; key: 'title' | 'tag' | 'summary'; value: string; ts: number };

export class SessionLogger {
  private writeQueue: Entry[] = [];
  private flushing = false;
  private uuids = new Set<string>();

  constructor(private sessionPath: string) {}

  async ensureDir() {
    await fs.mkdir(path.dirname(this.sessionPath), { recursive: true, mode: 0o700 });
  }

  async append(entry: Entry) {
    if ('uuid' in entry) {
      if (this.uuids.has(entry.uuid)) return;   // 主链不重复
      this.uuids.add(entry.uuid);
    }
    this.writeQueue.push(entry);
    if (!this.flushing) {
      this.flushing = true;
      setImmediate(() => this.flush());
    }
  }

  private async flush() {
    await this.ensureDir();
    const batch = this.writeQueue.splice(0);
    const content = batch.map(e => JSON.stringify(e)).join('\n') + '\n';
    await fs.appendFile(this.sessionPath, content, { mode: 0o600 });
    this.flushing = false;
    if (this.writeQueue.length) setImmediate(() => this.flush());
  }

  async readAll(): Promise<Entry[]> {
    let raw = '';
    try { raw = await fs.readFile(this.sessionPath, 'utf-8'); } catch { return []; }
    return raw.split('\n').filter(Boolean).map(l => JSON.parse(l));
  }

  /** 恢复：把日志重组成可继续的 messages 数组 */
  async resume(): Promise<{ messages: any[]; metadata: Record<string, string> }> {
    const entries = await this.readAll();
    const messages: any[] = [];
    const metadata: Record<string, string> = {};

    for (const e of entries) {
      if (e.type === 'metadata') {
        metadata[e.key] = e.value;
      } else if (e.type === 'user' || e.type === 'assistant') {
        messages.push({ role: e.type, content: e.content, uuid: e.uuid });
      }
      // tool_use / tool_result 这里简化处理，真实场景要修复链路
    }

    return { messages, metadata };
  }
}

// ===== 用法 =====
const session = new SessionLogger(`/tmp/my-agent/sessions/${Date.now()}.jsonl`);
await session.append({ type: 'user', content: 'hello', uuid: 'u1', ts: Date.now() });
await session.append({ type: 'metadata', key: 'title', value: 'Greeting', ts: Date.now() });

// 重启后
const next = new SessionLogger('/tmp/my-agent/sessions/.../...jsonl');
const { messages, metadata } = await next.resume();
console.log('Resumed:', metadata.title, messages);

80 行就有了 Claude Code session 的核心 pattern：append-only + 队列批量 flush + UUID 去重 + resume 恢复。

本课小结

  • transcript 是 append-only JSONL 事件流，不是数据库快照
  • 主链 UUID 不能重复，sidechain 必须允许重复——两个文件两套约束
  • 什么进 transcript 必须明确定义（progress 不进，否则污染恢复）
  • 写入端简单、恢复端复杂：日志结构存储的黄金法则
  • metadata 与 message 同表存，按类型分流
  • 0o600 文件权限是基本动作
